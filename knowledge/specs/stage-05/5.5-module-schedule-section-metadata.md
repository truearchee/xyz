## Why this stage exists

`week_number`, `session_date`, and `due_at` exist in the schema but are never populated. Module creation currently emits a **fixed 4-section template** instead of the schedule-driven structure Slice 1 specifies. Stage 6's recap and exam-prep quizzes resolve their scope (`coveredWeeks`) entirely through `week_number` on lecture/lab sections — without this stage, Stage 6 stops at a findings note in its first week.

This stage implements the schedule-driven generation Slice 1 always described, makes the three metadata fields real and editable, and exposes the week→sections read model Stage 6 consumes.

## Product model (confirmed with product owner)

Reference course: **11 May → 26 Jun 2026**, weekly pattern repeating identically:

```
Mon, Tue, Wed  → Lecture
Thu            → Lab
Fri            → Quiz/test  (covers that week's Mon–Wed lectures + Thu lab)
```

- The weekly pattern is identical every week; the admin sets it up once and the system duplicates it across the date range.
- **Lectures and Labs are the only generated section types.**
- **Friday produces nothing in this stage.** The Friday quiz is Stage 6 and resolves from `week_number`. The quiz day is *recorded* in the schedule config (Stage 6 intent) but generates no section here.
- **Assignments are not a generated section type.** A practice assignment is a PDF an instructor attaches to a lecture/lab section through the existing content flow (Slice 1). Practice-only, no grade.
- **Lab sections carry a deadline (`due_at`)** set by the instructor at lab-material upload. Lab materials may include **non-PDF files** (e.g. `.ipynb`) — Sub-scope C.

### Expected output for the reference course (test oracle — verified)

```
7   teaching weeks
21  lecture sections (3 × 7)
7   lab sections     (1 × 7)
28  total generated sections
0   Friday sections
```

Boundary facts this count protects: **11 May 2026 is a Monday** (clean week-1 start); the final lab is **Thu 25 Jun**; the end date **Fri 26 Jun is in range but generates nothing**; max `week_number` is **7** ((25 Jun − 11 May)=45 days → floor(45/7)+1=7). This exact count goes into both the generation unit tests and the browser gate so the assertion cannot be silently weakened.

## Locked decisions (with rationale)

```
D1  Generation is WEEKDAY + DATE-RANGE driven.
    Input: course start/end dates + a weekday→sectionType pattern. Only
    (start date × weekday pattern) yields real session_dates. Slice 1's "12
    lectures" is an emergent OUTPUT count, never an input.

D2  Structure is generated ONCE at module creation, then IMMUTABLE.
    No add / delete / reorder / regenerate. Re-running generation over a module
    that already has transcripts, summaries, quizzes, and PDFs attached is the
    real disaster this guards against. Only per-section METADATA is editable.

D3  week_number and session_date are STORED, EDITABLE fields.
    Generation seeds defaults; admin/lecturer curate. The resolver READS the
    stored value and NEVER recomputes it — an exam scope stays stable even if a
    date is later nudged. Structural truth over derived truth. (See D13 for the
    edit-consistency rule that prevents silent date/week drift.)

D4  Metadata is editable by ADMIN and ASSIGNED LECTURER. Structure is not.
    Editing week_number / session_date / due_at: allowed for both roles.
    Creating / deleting / reordering sections: still excluded (Slice 1 permanent
    rule). Instructor-managed structure, if ever needed, is its own future slice.

D5  due_at is set by the lecturer at lab-material upload time — not by the admin
    at generation. This stage makes the field settable + editable and exposes it.
    due_at is LAB-ONLY and has ONE source of truth: the lab upload sets it; the
    metadata-edit endpoint corrects it later. Same validation in both paths.

D6  session_date is a CALENDAR DATE (no time-of-day, no timezone in 5.5).
    Times and timezones are Stage 11's calendar problem. 5.5 does NOT compute
    overdue/late state from due_at — it stores and displays it only.

D7  Holiday / no-class weeks are NOT modeled (accepted MVP limitation).
    A stray generated session stays empty and unpublished. Recorded, not solved.
    The resolver visibility rule (§Resolver) ensures Stage 6 does not pull empty
    unpublished sections as quiz sources.

D8  The week→sections resolver lives in platform/query (read model). Generation
    is a PURE, unit-testable function in the modules/admin domain. Domains own
    writes; the quiz domain (Stage 6) consumes the read model, never the writer.

D9  Code is authoritative. The agent VERIFIES the exact existing ModuleSection
    columns (names/types/nullability for week_number, session_date, due_at) and
    the course/module table shape BEFORE writing migrations. Schema disagreement
    is resolved in favour of the codebase.

D10 The module schedule config is CREATION-TIME PROVENANCE.
    courseStartDate / courseEndDate / weekStartDay / sessionPattern / quizDay are
    displayed and readable after creation but NOT edited or regenerated in 5.5.
    All later corrections happen through per-section metadata edits only. Extends
    D2 to the stored config so config and generated structure cannot drift.

D11 due_at KEEPS THE EXISTING CODEBASE TYPE.
    If due_at is timestamp/timestamptz in the schema, 5.5 preserves it and does
    NOT downgrade to DATE. session_date is DATE only. due_at is nullable until set.

D12 enum + asset semantics are additive only.
    5.5 does not delete assignment/supplementary section-type enum values; it only
    stops GENERATION from creating them. Legacy sections, if present, remain
    readable and are IGNORED by the week resolver. asset_kind=processable means
    "eligible for a FUTURE section-material processing pipeline" — it does NOT mean
    Transcript and does NOT enter the transcript ingestion pipeline. PDF section
    assets keep the existing immediate-completed behavior in 5.5. (So a practice-
    assignment PDF is processable-but-inert; a .ipynb is an attachment.)

D13 Metadata edit CONSISTENCY (added v1.2 — prevents date/week drift).
    PATCHing session_date RECOMPUTES week_number from the stored course anchor,
    UNLESS week_number is supplied in the same request (explicit override). This
    stops the footgun where a moved date and a stale week label disagree and
    silently corrupt Stage 6 quiz scope. The resolver still trusts the stored
    week_number (D3); D13 only governs how that stored value is kept honest on edit.

D14 Generation is SYNCHRONOUS and ATOMIC (added v1.2).
    Section generation runs inside the module-creation transaction — no worker,
    no partial "module with zero sections," and no double-generation on a retried
    creation request. Generating ~28 rows is trivial; the heavy async pipeline
    pattern does not apply here.
```

## Week-number computation (deterministic)

```
weekStartDay      = Monday (default; stored, configurable — single field, no branching)
anchor            = most recent weekStartDay on-or-before courseStartDate
week_number(D)    = floor((D - anchor).days / 7) + 1
session_date(D)   = D
order             = global ascending by (session_date, type-priority lecture<lab)
```

A week's Mon/Tue/Wed lectures, its Thu lab, and its Fri quiz all share one `week_number`. Partial first/last weeks (course not starting on weekStartDay) are allowed and simply yield fewer sections — expected, not an error. Default title (stored, not separately editable): `"{TypeLabel} — Week {week_number} ({WeekdayAbbr} {DD Mon})"`. Editing week/date does not rewrite the stored title; live week/date fields are shown alongside it.

## Backend scope

**Sub-scope A — Schedule-driven generation (core)**
- Module creation accepts schedule params: `courseStartDate`, `courseEndDate`, `weekStartDay`, `sessionPattern` (`[{weekday, sectionType}]`), optional `quizDay`. Persisted on the module (JSON `sessionPattern` acceptable for MVP — read rarely, provenance only).
- **Replace** the fixed 4-section template with a pure generator `generateSections(schedule) -> [section drafts]` stamping `type`, `session_date`, `week_number`, `order`, default `title`. Admin-only; runs inside module creation; **synchronous and atomic in the creation transaction (D14)**.
- **No silent fallback:** module creation WITHOUT schedule params returns **422**. The fixed 4-section path is removed, not left behind an optional branch.
- **Call-site audit (suite-breaking — do this first):** every place a module is created in tests, fixtures, seeds, and E2E setup must be updated to supply a schedule. Until done, the active suite fails rule 14. This is an explicit, tracked task, not an afterthought.

```
Schedule validation:
- courseStartDate, courseEndDate required, parsed as CALENDAR dates (YYYY-MM-DD),
  never JS Date timestamps (prevents timezone off-by-one).
- courseStartDate <= courseEndDate.
- weekStartDay required or defaults to Monday.
- sessionPattern non-empty; weekdays unique; sectionType ∈ {lecture, lab}.
- quizDay optional; if present, must NOT overlap any sessionPattern weekday;
  recorded; generates no section.
- Generated sessions include only dates within [start, end] INCLUSIVE.
```

**Sub-scope B — Metadata edit + resolver**
- Per-section metadata edit endpoint — admin + assigned lecturer; lecture/lab only; backend authorization mandatory.

```
Metadata edit validation:
- Accepts ONLY week_number, session_date, due_at. Extra fields REJECTED, not ignored.
- type, order, title, courseModuleId, publishStatus are NOT patchable here.
- week_number: positive integer. session_date: valid calendar date.
- PATCH session_date without week_number → week_number recomputed from anchor (D13).
- due_at: editable only for lab sections in 5.5; same validation as the lab-upload path (D5).
- Authorization (match the existing API-wrapper convention; agent verifies exact split):
    · student role on an edit endpoint        → 403 (role denial).
    · lecturer not assigned to the module / cross-module section → 404 (no existence leak).
```

- **week→sections resolver in `platform/query`:**

```
resolveSectionsByWeeks(moduleId, coveredWeeks, includeUnstamped=false):
- Reads STORED week_number; never recomputes from session_date.
- Includes section types lecture | lab only; ignores legacy/other types.
- Orders by (week_number, session_date, order).
- Returns SECTION ids + metadata ONLY. It does NOT join summaries — mapping a
  section to its active detailed summary is the consumer's job (Stage 6 uses the
  4.7 ActiveTranscriptSummaryResolver). Keeps the domain boundary clean.
- includeUnstamped=false (default, Stage-6 mode): omits null week_number/session_date.
- includeUnstamped=true (admin curation mode): returns ALL sections incl. null/anomalous
  metadata so the by-week admin view can surface sections that need fixing.
- Pure read; mutates nothing; generates no quizzes.
Visibility:
- The structural resolver MAY return unpublished sections to admin/assigned-lecturer views.
- Stage 6 student-facing quiz generation MUST apply student access + publishStatus +
  completed-detailed-summary filters BEFORE creating a QuizDefinition. This closes the
  D7 holiday-empty-section loophole.
```

**Sub-scope C — Lab non-PDF materials (folded in per product decision)**
- `SectionAsset.asset_kind: processable | attachment` (default `processable`; existing rows backfilled to `processable` in the migration; see D12). Attachments are a **structural exclusion** from any AI pipeline — never parsed/chunked/embedded/summarized. `processingStatus` set to `completed` immediately on upload (Slice 1 no-pipeline behavior).
- Server-side allowlist: PDF (processable); at minimum `.ipynb` (attachment). Validated by **both extension and content-type**, size-capped. Stored in object storage. Notebooks never executed or rendered as code.

```
Attachment serving:
- Downloads use Content-Disposition: attachment and X-Content-Type-Options: nosniff.
- PROOF SPLIT (two surfaces):
    · Browser gate proves the student can download the attachment.
    · A DETERMINISTIC BACKEND/DB test (mirroring the 4.6d fencing pytest) asserts NO
      transcript/embedding/summary/AI-job row is created for an attachment asset.
      Absence-of-processing is a backend invariant, not a UI label, and cannot be
      proven by Playwright alone.
```

- The lab section's `due_at` is set/edited by the assigned lecturer at upload (D5, D11).
- **Implementation note (from 5.5a Plan agent):** a Supabase signed URL can carry
  `Content-Disposition: attachment` (via the `download` param the codebase already uses) but CANNOT
  carry `X-Content-Type-Options: nosniff`. Therefore attachments require a **backend streaming proxy**
  endpoint (reusing `storage_provider.get_object` + the existing download authz); PDFs keep the
  signed-URL path. The no-pipeline test asserts zero rows in `transcripts`/`ingestion_jobs`/
  `transcript_segments`/`transcript_chunks`/`generated_lecture_summaries` for the attachment (those
  FK to transcripts, never to assets).

## Thin UI scope

- **Admin module-creation screen:** course start/end + weekly-pattern picker (which weekdays are lectures, which labs, which the quiz day).
- **Creation preview before submit:** because structure is immutable (one chance), the admin sees the computed result before committing, e.g. `11 May–26 Jun 2026 · 7 weeks · 21 lectures · 7 labs · 0 Friday sections`. **The preview calls the same `generateSections` pure function as a dry-run (no persistence)** — never a separate count estimator — so preview and result cannot diverge.
- **Admin/lecturer "by week" view:** sections grouped by `week_number` with `session_date`. **Must be backed by the platform/query resolver (`includeUnstamped=true`), not client-side regrouping of a generic all-sections response** — otherwise the Stage 6 contract is unproven and anomalous sections stay hidden.
- **Per-section metadata edit:** inline edit of `week_number` / `session_date`, and `due_at` on lab sections (admin + assigned lecturer).
- **Lab upload:** lecturer uploads PDF + non-PDF lab files and sets the deadline.
- **Student lab view:** sees lab files (PDF + attachments), can **download** each, sees the deadline (or "No deadline set" under Option B). Read-only; published sections only.

## UI proof obligation

An admin sets schedule values in a real browser and the resolver-backed by-week view **visibly resolves the correct sections**; a lecturer edits a section's week and uploads a non-PDF lab file with a deadline; a student opens that published lab and downloads the file.

   lecturer/student views unchanged.

### Backfill → reseed (deviation, recorded; corrected in v1.2)

The roadmap lists a "backfill path for existing modules (29 in dev)." A self-review found that **in-place metadata stamping cannot serve those modules**: a legacy module has ~4 template sections, while any real schedule produces a different shape (28 for the reference course). No schedule reproduces a 4-section template, so an alignment-based stamp would refuse on every legacy module — and after 5.5a, generation always stamps metadata, so nothing else ever produces "schedule-shaped-but-unstamped" sections. A stamping script would therefore have **no real consumer** (YAGNI).

**Resolution (fulfils the roadmap's intent via a better mechanism):** dev modules are **reseeded** — recreated through the new schedule-driven creation flow. The dev DB is throwaway (staging deploy deferred; no real users yet), and the E2E suite creates its own per-run modules (`e2e/{runId}`), so reseeding does not break the suite. Any dev module with attached test content that someone wants to preserve must be manually recreated; for MVP dev, a clean reseed is the recommendation. Recorded as an ADR.
