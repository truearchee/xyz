## Goal
Module creation generates lecture+lab sections from a schedule (weekday × date-range) with correct
`week_number`/`session_date`, synchronously and atomically; the fixed 4-section template is gone and a
missing schedule returns 422 — proven by the 28-section reference oracle in unit tests.

## Why now
First sub-session of Stage 5.5. Everything downstream (metadata edit, resolver, attachments, UI)
builds on schedule-driven generation + the schedule config columns. Sub-scope A from the master spec.

## Build
- Migration 0020: nullable `week_start_day` (Text), `session_pattern` (JSONB), `quiz_day` (Text) on
  `course_modules` (reuse existing `starts_on`/`ends_on` for course dates). down_revision='0013'.
- Pure `generate_sections(...)` (D8) replacing the fixed template + ORM adapter
  `generate_initial_sections` reading the module's persisted schedule (D14: in the creation txn).
- Required `schedule` on `CreateModuleRequest` → 422 when absent; schedule validation; `ModuleResponse`
  surfaces the config (D10 provenance).
- Call-site audit: update API-path module creators to supply a schedule.

## Do not build
- Metadata edit endpoint + resolver (5.5b). asset_kind + attachments + proxy download (5.5c).
- Dev reseed (5.5d). The weekly-pattern picker UI, creation preview, by-week view, browser gate,
  full E2E run (5.5e). No section add/delete/reorder; no schedule re-edit after creation (D2/D10).

## Data model changes
- `course_modules`: +`week_start_day` text NULL, +`session_pattern` jsonb NULL, +`quiz_day` text NULL.
  `module_sections.week_number/session_date/due_at` already exist (no change).

## API changes
- `POST /admin/modules`: `CreateModuleRequest` now requires `schedule {courseStartDate, courseEndDate,
  weekStartDay?, sessionPattern[], quizDay?}`; drops loose `startsOn`/`endsOn`. 422 without schedule.
- `ModuleResponse` adds `weekStartDay`, `sessionPattern`, `quizDay`.

## Worker / job changes
None (generation is synchronous in the creation transaction — D14).

## Authz rules
Unchanged — module creation stays admin-only (`AdminUser`).
