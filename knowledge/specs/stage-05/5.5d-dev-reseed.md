## Goal
The dev database can replace legacy fixed-template modules with schedule-driven modules, using a documented reference schedule and seeding one published lab with both PDF and notebook assets for the 5.5e browser gate.

## Why now
5.5a removed the fixed 4-section creation path and 5.5c added attachment assets. Existing dev modules were created before that shift, so their structure and metadata no longer represent any schedule Stage 6 or 5.5e can safely consume.

## Build
- Add a dev-only reseed path that snapshots existing `course_modules`, deletes their dependent rows, and recreates the modules through the schedule-driven section generation flow.
- Preserve logical module metadata and memberships where possible: title, description, owner, timezone, active flag, and non-duplicate membership rows.
- Require an explicit confirmation flag before destructive reseed execution.
- Refuse to run unless the environment is non-production, the database URL is local/container-scoped unless explicitly overridden, and Alembic is at `0021`.
- Use the documented Stage 5.5 reference schedule for every recreated module:
  - `courseStartDate=2026-05-11`
  - `courseEndDate=2026-06-26`
  - `weekStartDay=monday`
  - `sessionPattern`: Monday/Tuesday/Wednesday lecture, Thursday lab
  - `quizDay=friday`
- Seed one published lab on a recreated module with:
  - one processable PDF asset served by the existing signed-URL path
  - one attachment `.ipynb` asset served by the 5.5c backend download path
- Produce a deterministic summary that reports modules replaced, generated section counts, fixture module/section IDs, and asset IDs.
- Add backend tests for schedule selection, destructive safety guards, module replacement, membership preservation, and lab asset fixture seeding.

## Do not build
- No new migrations.
- No in-place metadata stamping/backfill for legacy 4-section modules.
- No production/staging data operation.
- No UI or browser gate; 5.5e owns the product/browser proof after the reseed.
- No quiz generation, submissions, grading, overdue calculation, or schedule editing/regeneration feature.
- No notebook execution, parsing, rendering, transcript creation, chunking, embedding, summarization, or AI job enqueueing.

## Schedule source decision
No authoritative per-module schedule list exists in the repository, attached session transcript, attached 5.5 plan, seed fixture, or current knowledge files. The "29 modules" reference is a historical dev row count, not a curriculum schedule map. Therefore 5.5d intentionally uses the Stage 5.5 reference course schedule for every recreated dev module. This is a stated dev-data decision, not an inferred production rule.
