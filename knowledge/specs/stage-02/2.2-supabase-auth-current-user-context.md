## How auth works in this system — read before building

Supabase Auth handles password checking and issues JWTs. The app database owns
authorization.

The flow on every authenticated request:

```
Client sends:  Authorization: Bearer <supabase_jwt>
                        ↓
backend/app/platform/auth/jwt.py
  - fetch/cache the Supabase JWKS public keys
  - decode and verify the JWT using the retrieved public key
  - verify signature, expiry, audience, and issuer
  - extract sub (Supabase user ID) from the token claims
                        ↓
backend/app/platform/auth/dependencies.py
  - query app_users WHERE auth_provider_id = sub
  - reject if user not found
  - reject if is_active = false
  - load active module memberships
                        ↓
CurrentUserContext returned to the route handler
```

The backend never calls the Supabase API to validate tokens dynamically per user.
Verification is done locally using the cached JWKS public keys. This keeps auth
fast, offline-capable (mostly), and free of Supabase Auth API rate limits.

### JWT strategy — Asymmetric Keys (JWKS)

This session explicitly assumes the Supabase project uses **asymmetric signing keys (JWKS)**.
The backend must fetch the public keys from the Supabase JWKS endpoint to verify token
signatures. Do not implement legacy shared-secret HS256 verification.

### Source of trust — DB vs JWT

The JWT is used **only** to authenticate the provider identity via the `sub`
claim. Everything else about the user is read from `app_users`:

- `email`, `full_name`, `role`, `timezone`, `is_active` → trusted only from
  `app_users`.
- JWT claims (including the JWT's own `role`) are never used for app
  authorization. JWT metadata can be stale; the app database is authoritative.

### 1. JWT verification — `platform/auth/jwt.py`

Responsibilities:
- Accept a raw JWT string.
- Dynamically fetch the signing key from `SUPABASE_JWKS_URL` using the token's `kid` header.
- Verify the signature using the fetched public key.
- Verify expiry (`exp`), audience (`aud`), and issuer (`iss`).
- Require the presence of `sub`, `exp`, and `role` claims.
- Return the decoded claims on success.
- Raise `401 Unauthorized` on any failure.

Requirements:
- Use `PyJWT[crypto]`. Add it to `pyproject.toml`. **Do not add `supabase-py`** — this
  session never calls the Supabase database API. The Supabase client is introduced in
  Session 2.3 where admin provisioning needs it.
- Use `jwt.PyJWKClient` to handle fetching and caching the keys.
- Decode with explicit validation:

```python
jwks_client = jwt.PyJWKClient(settings.SUPABASE_JWKS_URL)
signing_key = jwks_client.get_signing_key_from_jwt(token)

jwt.decode(
    token,
    signing_key.key,
    algorithms=["RS256", "ES256"], # Allow asymmetric algorithms used by Supabase
    audience=settings.SUPABASE_JWT_AUDIENCE,
    issuer=settings.SUPABASE_JWT_ISSUER,
    options={"require": ["sub", "exp", "role"]},
)
```

- The JWT `role` claim is extracted for logging/debugging only. It is never
  used for app authorization.

### 2. CurrentUserContext — `platform/auth/context.py`

Define **frozen** dataclasses representing the resolved, trusted identity for a
request. Frozen because this object is the request's source of truth and must
never be mutated after construction.

```python
@dataclass(frozen=True)
class ModuleMembership:
    module_id: UUID
    role: Literal["student", "lecturer"]
    is_owner: bool       # course_modules.owner_id == this user's id
    can_publish: bool    # role == "lecturer" and membership is active


@dataclass(frozen=True)
class CurrentUserContext:
    user_id: UUID                              # app_users.id (UUIDv7)
    auth_provider_id: str                      # Supabase Auth sub claim
    email: str
    full_name: str
    role: str                                  # 'student' | 'lecturer' | 'admin'
    is_active: bool
    timezone: str
    module_memberships: tuple[ModuleMembership, ...]
```

Rules:

- `module_memberships` contains only **active** memberships
  (`course_memberships.status = 'active'`). Archived memberships are excluded.
- `is_owner` is computed by comparing `course_modules.owner_id` to the current
  user's id.
- `can_publish` is `True` only when the membership role is `lecturer` and the
  membership is active.
- These flags exist now (not later) because Stage 3 content publishing depends
  on them. Without them, Stage 3 invents its own ownership checks — the exact
  fragmentation the roadmap forbids.
- This object is constructed once per request in the FastAPI dependency and
  passed down. It is never re-queried or mutated.

The membership query joins `course_memberships` to `course_modules` so
ownership can be resolved in a single query.

### 3. FastAPI dependency — `platform/auth/dependencies.py`

`get_current_user` is a FastAPI dependency that:

1. Extracts and validates the `Authorization` header format. Must be exactly
   `Bearer <token>`. Returns `401` on missing or malformed header.
2. Calls `jwt.py` to verify and decode the token. Returns `401` on any failure.
3. Queries `app_users` by `auth_provider_id = sub`. Returns `401` if not found.
4. Checks `is_active = true`. Returns `403` if inactive.
5. Queries active memberships (joined to `course_modules` for ownership) for
   this user.
6. Constructs and returns the frozen `CurrentUserContext`.

**Performance:** steps 3 and 5 run within a single async DB session. Do not
open two sessions per request. Use the existing `get_db_session` dependency
from `platform/db/session.py`.

**Authorization header parsing** — reject all of these with `401`:

- No `Authorization` header
- Blank token (`Bearer ` with nothing after)
- Wrong scheme (`Basic <token>`)
- `Bearer` with no token
- Malformed multi-part values (more than two whitespace-separated parts)

**Error responses** — the public detail is always generic for credential
failures. Keep specific reasons in internal logs only.

| Condition | HTTP status | Public detail |
| --- | --- | --- |
| Missing Authorization header | 401 | "Authorization header required" |
| Malformed Authorization header | 401 | "Invalid authentication credentials" |
| Invalid or expired JWT | 401 | "Invalid authentication credentials" |
| User not found in app_users | 401 | "Invalid authentication credentials" |
| User is inactive | 403 | "Account is inactive" |

All `401` responses include the header `WWW-Authenticate: Bearer`.

Do not distinguish "expired" from "bad signature" from "user not found" in the
public response. A single generic message prevents enumeration and timing
leaks.

## Tests required

`backend/tests/test_auth.py`. Tests must monkeypatch `SUPABASE_JWKS_URL`,
`SUPABASE_JWT_AUDIENCE`, and `SUPABASE_JWT_ISSUER` to test-only values.

Because we are using JWKS, the tests must **not** require a live network call.
In `conftest.py`, generate a local RSA or EC keypair. Mock `jwt.PyJWKClient` (or
the HTTP requests via `httpx`/`responses`) to return the generated public key in
JWKS format. Use the generated private key to sign the test JWTs.

```python
# Test 1 — valid JWT decodes correctly
# Generate a JWT signed with the test private key, correct aud + iss.
# Assert decoded claims contain correct sub and exp.

# Test 2 — expired JWT raises 401
# exp in the past. Assert HTTPException 401.

# Test 3 — invalid signature raises 401
# Signed with a different, unknown private key. Assert 401.

# Test 4 — wrong audience raises 401
# Correct signature, aud = "wrong". Assert 401.

# Test 5 — wrong issuer raises 401
# Correct signature, iss = "https://evil.example.com". Assert 401.

# Test 6 — missing Authorization header returns 401
# GET /health/authed with no header. Assert 401 and WWW-Authenticate: Bearer.

# Test 7 — malformed Authorization header returns 401
# Try: "Basic xxx", "Bearer", "Bearer ", "Bearer a b c".
# Assert each returns 401.

# Test 8 — valid token for active user returns 200
# Insert active user. Valid JWT with sub = auth_provider_id.
# Assert 200, body has correct user_id, role, email.

# Test 9 — valid token for inactive user returns 403
# Insert user with is_active = false. Assert 403.

# Test 10 — valid token for unknown user returns 401
# sub = random UUID not in app_users. Assert 401.

# Test 11 — JWT role is ignored; app role wins   ← proves Stage 2's core claim
# JWT role claim = "admin", app_users.role = "student".
# Assert CurrentUserContext.role == "student".

# Test 12 — CurrentUserContext memberships: active included, archived excluded
# Insert user with two active memberships and one archived membership.
# Assert module_memberships contains exactly the two active ones.

# Test 13 — is_owner and can_publish resolve correctly
# Insert a lecturer who owns module A and is a member of (not owner of) module B.
# Assert membership for A has is_owner=True, can_publish=True.
# Assert membership for B has is_owner=False, can_publish=True.
# Insert a student membership; assert can_publish=False.
```

Use `TEST_DATABASE_URL` for all DB-touching tests. Do not make real Supabase
API calls.
