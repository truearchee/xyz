## Goal
Resolve module access from the database on a per-module basis and expose minimal authorization-gated participant module views that prove users see only active module memberships.

## Why now
Session 2.4 closes the Stage 2 gate by making `require_module_access` the reusable module authorization primitive later content, quiz, transcript, progress, and assistant-context domains must consume.

## Build
- Make `CurrentUserContext` identity-only by removing eager-loaded module memberships. Authentication and active-user state stay in `get_current_user`; module authorization moves to the per-module guard.
- Add frozen `ModuleAccessContext` with `module_id`, `is_active`, `global_role`, `can_publish`, and `membership_id`.
- Add `require_module_access` in `backend/app/platform/auth/guards.py`. It consumes the already-resolved `CurrentUserContext`, the `module_id` path parameter, and the DB session. It does not decode JWTs again.
- Add `backend/app/platform/query/modules.py` as a pure read/context-query layer. It must use projection dataclasses, not raw ORM objects.
- Add participant routes:
  - `GET /modules` returns the caller's active modules.
  - `GET /modules/{module_id}` returns module detail only after `require_module_access` authorizes access.
- Add typed, data-driven frontend presentational components under `frontend/src/features/modules/` using generated DTO types.
- Regenerate the TypeScript API client with `scripts/generate-api-client.sh`.

## Corrected schema facts
The pasted pre-review spec referred to `course_modules.status` and full `(user_id, module_id)` uniqueness. Live code and migrations are the source of truth:
- `course_modules.is_active` is the module active flag.
- `course_memberships.status` and `course_memberships.archived_at` track membership archival.
- `course_memberships` has a partial unique index on active `(user_id, module_id)` rows.
- Reassignment after removal preserves historical archived rows by inserting a new active row.
- There is no membership capability column, so `canPublish` is role-derived for 2.4: active lecturer membership returns `true`; active student membership returns `false`.
- No migration is allowed in this session.

## Do not build
- No database migration.
- No section listing, content, or section count.
- No publish or unpublish actions.
- No eager-loading the full membership list into `get_current_user`.
- No per-module role columns.
- No Redis membership cache.
- No pagination on `/modules`.
- No admin branch in `/modules`; admin module management remains under `/admin`.
- No frontend token storage, local auth/session fetching, fake auth provider, route-level API fetching, or frontend auth plumbing.

## API changes
```text
GET /modules
  -> 200 ModuleSummary[]
  -> 401 unauthenticated

GET /modules/{module_id}
  -> 200 ModuleDetail
  -> 401 unauthenticated
  -> 404 authenticated active user without active access
  -> 422 malformed UUID
```

DTOs use camelCase:
```text
ModuleSummary { id, title, isActive, globalRole }
ModuleDetail  { id, title, isActive, globalRole, canPublish, createdAt }
```

`globalRole` is the caller's global app role, not a per-module role. `isActive` reflects `course_modules.is_active`.

## Authz rules
- Authentication and active-user state are enforced upstream by `get_current_user`.
- Module access is enforced only by `require_module_access`.
- Active module access requires both:
  - `course_memberships.status == "active"`
  - `course_modules.is_active is true`
- No active row means `404`; archived membership, inactive module, non-member, and cross-tenant access all look identical.
- `403` is reserved for future role-forbidden actions by a valid member. Session 2.4 reads should not raise `403` except inherited inactive-user behavior from auth.
- Admins have no participant-module access; `/modules` returns `[]` for list and `404` for detail.

## Tests required
- Active member plus active module returns module access context and `200`.
- Non-member, archived membership, inactive module, admin detail, and valid non-existent module return `404`.
- Malformed module UUID returns `422`.
- Unauthenticated requests return `401`.
- Lecturer and student lists include only active assigned modules.
- Admin list returns `[]`.
- Role-derived `canPublish` is `true` for lecturers and `false` for students.
- Revocation proof: after admin archives membership, the same token gets `404` and list no longer includes the module.
- Reassignment with historical rows: archived historical membership plus new active membership for the same user/module returns `200`, and `require_module_access` resolves the active `membership_id`, not the archived one.
- Co-lecturer parity: a second lecturer assigned to the same module gets `200` on detail and `canPublish: true`.
