## Context carried from 2.2 — read before building

Session 2.2 (canonical) verifies Supabase JWTs using **asymmetric keys via
JWKS** — `jwt.PyJWKClient` against `SUPABASE_JWKS_URL`, decoding with
`algorithms=["RS256","ES256"]`, validating `aud`/`iss`. Two consequences here:

1. **`supabase` (the Python SDK) is introduced HERE for the first time.** It is
   NOT in `pyproject.toml` yet — 2.2 was built with PyJWT only.
2. **The privileged key is the new secret key, not the legacy `service_role`
   JWT.** This project uses the new Supabase API key system
   (`sb_publishable_…` / `sb_secret_…`). The Admin API methods (`auth.admin.*`)
   require a secret key, and `sb_secret_…` is the supported substitute for the
   old `service_role` key. We use `SUPABASE_SECRET_KEY`.

## Authorization model

Every admin endpoint enforces, in order:

1. **Authentication** — valid JWT, active user (`get_current_user` from 2.2).
2. **Role check** — `current_user.role == 'admin'`, else `403`.

`require_role()` **wraps** `get_current_user` and **returns** the resolved
`CurrentUserContext`. Admin routes therefore use a single dependency:

```python
current_user: CurrentUserContext = Depends(require_role("admin"))
```

Do **not** also add `Depends(get_current_user)` on the same route — that is
redundant double resolution. `require_role()` is the one entry point.

Authz rules (enforced, not aspirational):

- Every `/admin` route resolves identity via `require_role("admin")`.
- Role comes from `app_users.role` — never the JWT `role` claim.
- Route handlers contain **no** inline role checks.
- The service layer owns business-rule enforcement: no admin creation, no
  self-deactivation, no admin course membership, lecturer-only module owner,
  membership role must equal the user's global role.

## Build

Work through these tasks in order.

### 1. Role guard — `platform/auth/guards.py`

- `require_role(*roles)` returns a FastAPI dependency that **depends on
  `get_current_user` and returns the `CurrentUserContext`**.
- If `current_user.role` is not in `roles`, raise `403 "Insufficient
  permissions"`.
- Accepts multiple roles: `require_role('admin', 'lecturer')` allows either.
- Handlers receive the context from this dependency; they never separately
  depend on `get_current_user`.

### 2. Supabase Admin client — `platform/supabase_client.py`

- Initialize an **async** Supabase client (`acreate_client` / `AsyncClient`)
  using `SUPABASE_URL` and `SUPABASE_SECRET_KEY`.
- Single shared instance created at application startup or module-level
  singleton; never one per request.
- The async client's `auth.admin.*` methods are awaited. Do not block the event
  loop. If the async admin surface proves unstable, the ONLY acceptable fallback
  is wrapping sync calls in `anyio.to_thread.run_sync` — never a bare sync call
  inside an async handler.
- This module is the ONLY place the Supabase client is constructed.

### 3. Admin service — `domains/admin/service.py`

Service owns all logic. Handlers are thin.

#### 3a. Create user — `create_user(db, payload) → AppUser`

- **Reject admin creation:** if `payload.role == 'admin'` → `400`. The request
  model permits `admin`, so this is a business rejection, not a `422`.
- **Duplicate-email pre-check:** if a row with `payload.email` exists in
  `app_users` → `409`, and **do not call Supabase**.
- `await supabase.auth.admin.create_user({ "email", "password",
  "email_confirm": True })`; capture the Supabase user id. If Supabase returns
  an error (password policy, duplicate in Supabase, etc.) → surface `400`/`409`
  and **do not** insert into `app_users`.
- Insert `app_users` (UUIDv7 id, `auth_provider_id` = Supabase id, email,
  full_name, role, timezone default `'UTC'`, `is_active = true`).
- **Backstop + rollback:** `app_users.email` is UNIQUE — if the insert hits a
  unique violation despite the pre-check (race), treat as `409`. On **any**
  failed insert after a successful Supabase create, call
  `await supabase.auth.admin.delete_user(supabase_user_id)` to roll back, and
  log the outcome. The two stores must not drift.
- Allowed roles: `student`, `lecturer`.

#### 3b. Deactivate user — `deactivate_user(db, user_id, current_user) → AppUser`

- Set `is_active = false`. Do not delete.
- `400` if `user_id == current_user.user_id` (no self-deactivation).
- `404` if not found.
- No Supabase-side change: 2.2 rejects inactive users with `403` live on every
  request, so the local flag fully blocks access. Supabase-side banning is
  deferred — do not build it.

#### 3c. Reset password — `reset_password(db, user_id, new_password) → None`

- Look up the user in `app_users`; `404` if absent.
- `await supabase.auth.admin.update_user_by_id(auth_provider_id,
  { "password": new_password })`.
- Never store passwords or hashes in the app DB.

#### 3d. Create course module — `create_module(db, payload, current_user) → CourseModule`

- `owner_id` must reference an `app_users` row with `role = 'lecturer'` → else
  `400`.
- Insert `course_modules` (UUIDv7 id, `is_active = true`). `starts_on`/`ends_on`
  optional.
- **Auto-owner-membership, same transaction:** create an **active lecturer**
  `course_memberships` row for `owner_id` on the new module unless an active one
  already exists. Both writes commit in one DB transaction; if the membership
  insert fails, the module insert rolls back.

#### 3e. Assign user to module — `assign_to_module(db, user_id, module_id, role, current_user) → CourseMembership`

- `404` if user or module not found.
- **Admin can't be a member:** if target `app_users.role == 'admin'` → `400`.
- **Role must match global role:** if `payload.role != target_user.role` →
  `400`. Student only as student; lecturer only as lecturer.
- `409` if an **active** membership already exists for the pair.
- If only an **archived** membership exists, insert a NEW active row. UUIDv7 id.

#### 3f. Remove user from module — `remove_from_module(db, user_id, module_id, current_user) → None`

- Archive the active membership: `status='archived'`, `archived_at=now()`,
  `updated_at=now()`. Never hard-delete.
- `404` if no active membership exists.

#### 3g. List helpers (pagination)

```
list_users(db, limit, offset)   → list[AppUser]
list_modules(db, limit, offset) → list[CourseModule]
```

- `limit` default 50, min 1, max 100. `offset` default 0, min 0.
- Order by `created_at, id` for fully deterministic pagination.
- `list_users` returns all users (active and inactive).

### 4. Admin router — `api/routers/admin.py`

All routes prefixed `/admin`, all guarded by
`Depends(require_role("admin"))` (which supplies `current_user`). Bodies are
**camelCase**.

```
POST   /admin/users
       Body: { email, fullName, role, password, timezone? }
       201: UserResponse   400: role=='admin'   409: email exists

GET    /admin/users?limit=&offset=
       200: list[UserResponse]

GET    /admin/users/{user_id}
       200: UserResponse   404: not found

POST   /admin/users/{user_id}/deactivate
       200: UserResponse   400: self   404: not found

POST   /admin/users/{user_id}/reset-password
       Body: { newPassword }
       200: { "status": "ok" }   404: not found

POST   /admin/modules
       Body: { title, description?, ownerId, timezone?, startsOn?, endsOn? }
       201: ModuleResponse   400: owner not a lecturer

GET    /admin/modules?limit=&offset=
       200: list[ModuleResponse]

POST   /admin/modules/{module_id}/members
       Body: { userId, role }
       201: MembershipResponse   400: bad role / admin user / role mismatch
       404: user or module not found   409: already an active member

DELETE /admin/modules/{module_id}/members/{user_id}
       200: { "status": "ok" }   404: no active membership
```

`limit`/`offset` are FastAPI `Query` params with the bounds from 3g. Handlers
stay thin.

### 5. Pydantic schemas — `domains/admin/schemas.py`

DTOs are **camelCase on the wire, snake_case in Python**:

```python
from pydantic import BaseModel, ConfigDict, EmailStr, Field, model_validator
from pydantic.alias_generators import to_camel

class CamelModel(BaseModel):
    model_config = ConfigDict(alias_generator=to_camel, populate_by_name=True)
```

All requests/responses inherit `CamelModel`.

**Requests:**

```python
class CreateUserRequest(CamelModel):
    email: EmailStr
    full_name: str = Field(min_length=1, max_length=200)
    role: Literal['student', 'lecturer', 'admin']
    password: str = Field(min_length=8, max_length=128)
    timezone: str = 'UTC'

class ResetPasswordRequest(CamelModel):
    new_password: str = Field(min_length=8, max_length=128)

class CreateModuleRequest(CamelModel):
    title: str = Field(min_length=1, max_length=200)
    description: str | None = None
    owner_id: UUID
    timezone: str = 'UTC'
    starts_on: date | None = None
    ends_on: date | None = None

    @model_validator(mode='after')
    def _dates_ordered(self):
        if self.starts_on and self.ends_on and self.starts_on > self.ends_on:
            raise ValueError('startsOn must be on or before endsOn')
        return self

class AssignMemberRequest(CamelModel):
    user_id: UUID
    role: Literal['student', 'lecturer']
```

**Responses:** `UserResponse`, `ModuleResponse`, `MembershipResponse` — all
`CamelModel`. `UserResponse` exposes `id, email, fullName, role, isActive,
timezone, createdAt` (never `auth_provider_id`, never any password field).
`ModuleResponse`: `id, title, description, ownerId, timezone, startsOn, endsOn,
isActive, createdAt`. `MembershipResponse`: `id, userId, moduleId, role, status,
createdAt`.

`role: Literal[..., 'admin']` on `CreateUserRequest` is deliberate — it lets the
service return a business `400` instead of FastAPI's `422`.

2. Every admin endpoint returns `401` with no token and `403` with a non-admin
