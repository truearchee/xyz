## ID strategy — read before building
All primary keys use **UUIDv7**, generated in application code (not by the database).

- Add the `uuid7` library (e.g. the `uuid6` package, which provides `uuid7()`) to `pyproject.toml`.
- Models set their `id` default to a Python callable that returns a UUIDv7.
- Do **not** use `gen_random_uuid()` as a database-side default. The database column type is `UUID`, but the value is supplied by the application.
- Reason: UUIDv7 is time-ordered, which keeps inserts roughly sequential and avoids index fragmentation as tables grow. UUIDv4 (`gen_random_uuid()`) is random and fragments b-tree indexes over time.

If a prior architecture decision explicitly locked UUIDv4, stop and flag it before proceeding — do not silently mix the two.

### 2. The five tables
Define each model in its own file. All models inherit from the base above. All `id` columns are `UUID` type with a UUIDv7 application-side default.

#### Table 1 — `app_users`
The local authorization anchor. **Not** the Supabase Auth user record. Links to Supabase via `auth_provider_id` and carries the app's own role, active state, and timezone.

```
app_users
- id                  UUID        primary key (UUIDv7, app-generated)
- auth_provider_id    TEXT        unique, not null  ← Supabase Auth user ID
- email               TEXT        unique, not null
- full_name           TEXT        not null
- role                TEXT        not null          ← 'student' | 'lecturer' | 'admin'
- is_active           BOOLEAN     not null, default true
- timezone            TEXT        not null, default 'UTC'
- created_at          TIMESTAMPTZ not null, default now()
- updated_at          TIMESTAMPTZ not null, default now()
```

Constraints:
- `role` check constraint: `student`, `lecturer`, `admin`.
- `auth_provider_id` unique — one Supabase user maps to exactly one app user.
- `email` unique.

`timezone` is required now (not premature): `CurrentUserContext` in Stage 2.2 includes it, and streaks, attendance days, deadlines, and workload planning all depend on per-user timezone later.

#### Table 2 — `course_modules`
The module container. One module per course offering.

```
course_modules
- id            UUID        primary key (UUIDv7, app-generated)
- title         TEXT        not null
- description   TEXT        null
- owner_id      UUID        not null, FK → app_users(id)
- timezone      TEXT        not null, default 'UTC'
- starts_on     DATE        null
- ends_on       DATE        null
- is_active     BOOLEAN     not null, default true
- created_at    TIMESTAMPTZ not null, default now()
- updated_at    TIMESTAMPTZ not null, default now()
```

Constraints:
- `owner_id` references a user with `role = 'lecturer'`. Enforced at the service layer in Stage 2.3 — not as a DB constraint for MVP.
- `is_active` determines whether the module is live to members.
- Future / in-progress / past states are **derived** from `starts_on` / `ends_on` at query time — never stored as status columns.

`starts_on` / `ends_on` are nullable now because the admin course-creation flow that populates them lands in Stage 2.3. The columns exist now so derived-state logic has real data to derive from.

#### Table 3 — `course_memberships`
Connects students and lecturers to modules. Admins are never members. Archived memberships preserve enrollment history — they are never hard-deleted.

```
course_memberships
- id              UUID        primary key (UUIDv7, app-generated)
- user_id         UUID        not null, FK → app_users(id)
- module_id       UUID        not null, FK → course_modules(id)
- role            TEXT        not null  ← 'student' | 'lecturer'
- status          TEXT        not null, default 'active'  ← 'active' | 'archived'
- archived_at     TIMESTAMPTZ null
- created_at      TIMESTAMPTZ not null, default now()
- updated_at      TIMESTAMPTZ not null, default now()
```

Constraints:
- Partial unique index on `(user_id, module_id) WHERE status = 'active'` — at most one active membership per user per module. Archived memberships do not block re-enrolment.
- `role` check constraint: `student`, `lecturer`.
- `status` check constraint: `active`, `archived`.
- Admin users cannot be assigned to modules. Enforced at the service layer in Stage 2.3 — not as a DB constraint for MVP.

**Archive over delete:** memberships are archived, not hard-deleted. When a user is unenrolled, set `status = 'archived'` and `archived_at = now()`. This preserves enrollment history for auditing, reporting, and re-enrolment workflows. Future queries filter by `status = 'active'` to exclude archived memberships.

#### Table 4 — `module_sections`
Stores the resolved content instances within a module. Auto-generated from the course schedule when a module is created (that logic is Stage 2.3 — this session defines the table only).

```
module_sections
- id               UUID        primary key (UUIDv7, app-generated)
- course_module_id UUID        not null, FK → course_modules(id)
- title            TEXT        not null
- type             TEXT        not null  ← 'lecture' | 'lab' | 'assignment' | 'supplementary'
- order_index      INTEGER     not null
- week_number      INTEGER     null
- session_date     DATE        null
- due_at           TIMESTAMPTZ null
- publish_status   TEXT        not null, default 'draft'  ← 'draft' | 'published' | 'unpublished'
- lecturer_notes   TEXT        null
- status           TEXT        not null, default 'active'  ← 'active' | 'archived'
- archived_at      TIMESTAMPTZ null
- created_at       TIMESTAMPTZ not null, default now()
- updated_at       TIMESTAMPTZ not null, default now()
```

Constraints:
- `type` check constraint: `lecture`, `lab`, `assignment`, `supplementary`.
- `publish_status` check constraint: `draft`, `published`, `unpublished`.
- `status` check constraint: `active`, `archived`.
- `CHECK (order_index >= 0)`
- `CHECK (week_number IS NULL OR week_number > 0)`
- Unique constraint on `(course_module_id, order_index)` — section order within a module must be unambiguous.
- The three scheduling fields (`week_number`, `session_date`, `due_at`) are intentionally separate. Do not merge or derive one from another.

Indexes:
```sql
(course_module_id, week_number)
(course_module_id, session_date)
(due_at) WHERE due_at IS NOT NULL
(course_module_id, publish_status)
```

#### Table 5 — `section_assets`
Stores metadata for files uploaded to a section. The actual file lives in object storage — only the reference and metadata live here.

```
section_assets
- id                    UUID        primary key (UUIDv7, app-generated)
- module_section_id     UUID        not null, FK → module_sections(id)
- file_url              TEXT        not null    ← see note below
- file_name             TEXT        not null
- mime_type             TEXT        not null
- file_size             BIGINT      not null
- processing_status     TEXT        not null, default 'uploaded'
                                    ← 'uploaded' | 'processing' | 'completed' | 'failed'
- uploaded_by_id        UUID        not null, FK → app_users(id)
- created_at            TIMESTAMPTZ not null, default now()
- updated_at            TIMESTAMPTZ not null, default now()
```

Constraints:
- `processing_status` check constraint: `uploaded`, `processing`, `completed`, `failed`.
- `CHECK (file_size > 0)`
- `processing_status` is the technical file-processing state. It is completely separate from `module_sections.publish_status` (student visibility). Never merge these two concepts.

**Note on `file_url`:** the slice names this field `fileUrl`, so the column is kept as `file_url` here for alignment. It must store a stable storage reference (a storage key/path), **never** a short-lived signed URL. The storage-provider abstraction and the final decision on whether to rename this to `storage_key` are resolved in Spec 3.1, where file upload is actually implemented. Do not wire any storage logic in this session.

### 3. Migration
`alembic/versions/0002_db_spine.py`:

- Creates all five tables in dependency order: `app_users` → `course_modules` → `course_memberships` → `module_sections` → `section_assets`.
- Creates all indexes, unique constraints, and check constraints listed above.
- `upgrade()` creates everything. `downgrade()` drops everything in reverse order.
- Both directions must work cleanly.

**Extensions:** add `CREATE EXTENSION IF NOT EXISTS pgcrypto;` to `docker/postgres/init/001-create-vector.sql`, alongside the existing vector line. Do not create extensions inside the migration.

> Deployment caveat: the Docker init script only runs on the local container. Managed/hosted Postgres will not run it. Before the first hosted deployment, extension bootstrap (vector, pgcrypto) must be handled explicitly for the target database. Flag this — do not silently assume the init script covers production.

Note: because IDs are generated in application code (UUIDv7), the migration does not need pgcrypto for UUID defaults. pgcrypto is included for forward compatibility; vector is required by later stages.

## Tests required
`backend/tests/test_db_spine.py` — three tests, no more.

```python
# Test 1 — migration round-trip
# alembic upgrade head → downgrade base → upgrade head again. All three exit 0.

# Test 2 — all five tables exist after upgrade head
# Inspect information_schema.tables; assert app_users, course_modules,
# course_memberships, module_sections, section_assets are all present.

# Test 3 — a check constraint is enforced
# Insert into app_users with role='superuser'; assert the DB raises IntegrityError.
```

**Test database isolation — required:**

- These tests run destructive migrations (downgrade to base). They must run against an **isolated test database**, never the developer's working DB.
- Use `TEST_DATABASE_URL` from the environment. If it is not set, the tests must skip or fail with a clear message — they must **not** fall back to `DATABASE_URL`.
- Add `TEST_DATABASE_URL` to `.env.example` with a placeholder pointing at a separate database name (e.g. `xyz_lms_test`).
- Use a real PostgreSQL database — do not mock. The point is to verify the real schema and real constraints.
