# Welcome to XYZ LMS Development

You're working on a university AI-powered learning management system. This document orients you to how we work together.

---

## How Development Works Here

**Three things matter:**

1. **You follow a strict spec → plan → code → report workflow.** No skipping steps.
2. **You build only what's in the spec.** Scope creep is the enemy; if something's not in "Build" or it's in "Do not build," flag it and stop.
3. **You leave a trail.** Every session produces three linked documents: what we intended (spec), how you'd do it (plan), and what actually happened (report). These live in `knowledge/` and they're the continuity layer for the next session.

---

## Before You Start Any Task

**Read these files in order:**

1. **`AGENTS.md`** (repo root) — your operating rules for every session. Read it every time.
2. **`knowledge/dev-workflow.md`** — the full logic of how the memory system works. Skim it once, reference it when confused.
3. **`knowledge/STATUS.md`** — where we are right now. Read this first every session so you know the context.
4. The **spec file** you're given — this is WHAT + WHY + what done means.

---

## The Workflow (Every Session)

### Step 1: Understand the spec
You'll be handed a spec file (or the developer will ask you to write one and they'll approve it). Read it. The spec answers:
- What are we building?
- What are we explicitly NOT building?
- What files must I read first?
- What counts as "done"?

**If anything is ambiguous, ask before proceeding.** No spec, no implementation.

### Step 2: Write a plan (`/plan`)
Before you write a single line of code, enter planning mode. Output a plan that says:
- How you'll approach this (the technical strategy)
- Which files you'll change and why
- The order you'll do things in
- How you'll prove it works (mapped to the spec's verification commands)
- Any risks you see

The developer will review and approve the plan. **No plan, no code.**

### Step 3: Implement
Build exactly what the spec says. Stay inside the scope. If scope genuinely must change mid-flight, amend the spec (add a dated entry to the `## Amendments` section) and re-confirm with the developer before continuing.

### Step 4: Verify
Run the spec's verification commands. Capture the actual output. No hand-waving: "tests pass" is not acceptable without the command and its result showing on screen.

### Step 5: Write the report
After you're done, write a report from `git diff` and command output — not from memory. The report records:
- What you actually built
- Files changed
- Real verification output
- Any places you deviated from spec and why
- Risks introduced
- Follow-ups for next time

### Step 6: Close the loop
Update the linking between spec ↔ plan ↔ report. Overwrite `STATUS.md`. Append a line to `knowledge/log.md`. Update `knowledge/architecture/` only if you touched source paths. Add an ADR to `knowledge/decisions/` only if you made a durable architectural decision.

## The Sacred Rule

**Docs are a cache of engineering intent. Code is the truth.**

When documentation contradicts the code, the code is right. Flag it, fix the doc. Never bend code to match a stale doc.

## The Gate Rules (Non-Negotiable)

- **No spec → no implementation.** Intent must be committed before work.
- **No plan → no code.** Approach must be approved before editing source.
- **No evidence → no report.** The record comes from `git diff` and command output, never memory.
- **No drift → only amendments.** Scope changes are logged in the spec, never silent.
- **Code wins.** Docs that contradict code are stale; fix the doc.

---

## What You Don't Do

- Don't commit or push to Git. The developer handles that. You just edit files and verify.
- Don't write LLM-maintained descriptions of code (per-file, per-function summaries). That layer goes stale fastest and lies to the next agent.
- Don't create new folders or systems outside `knowledge/`. Keep everything tight.
- Don't skip verification or backfill reports from memory. Evidence only.

## Starting Your First Session

1. Read `AGENTS.md` (takes 5 min).
2. Read `knowledge/STATUS.md` (takes 1 min — see where we are now).
3. Read the spec you're handed (takes as long as it takes).
4. Write your plan and get it approved.
5. Build.
6. Write the report.
7. Close the loop.

Questions? Ask before you move to the next step. Ambiguity is expensive.
