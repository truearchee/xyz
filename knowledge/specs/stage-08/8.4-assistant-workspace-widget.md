## Goal
One place to see and continue every assistant conversation (a ChatGPT-style workspace) and a
floating assistant button reachable from any **student** page — *without changing what the assistant
knows or how it grounds answers.*

## Scope boundary — READ FIRST
8.4 is a **navigation / UX + conversation-management** stage. In its build form (Option A) it adds
**no new AI surface**: no new prompt, no new model route, no new grounding mode. Every AI answer is a
**8.2 grounded answer reached through a new entry point**. Grounding stays scoped to a **single
lecture**. Cross-lecture / cross-week studying is **Stage 8.6**; an ungrounded general chat is
**Appendix B (out of MVP)**.

## Role scope
The Workspace and the floating button are **student-role surfaces only**. Lecturer/admin views are
unaffected (lecturer chat monitoring is an explicit Stage 8 exclusion). Do not mount these on
lecturer or admin pages.

## Core invariants (the bug-prone spots — each gets a test)
```
A. One active conversation per (student, section, kind)
   - transactional GET-OR-CREATE endpoint (exists; reuse) — never two creates racing
   - partial UNIQUE index WHERE deleted_at IS NULL (REBUILD the existing index to add this)
   - concurrent open from the inline entry + the floating widget returns the SAME row
   - soft-deleted rows are excluded, so delete-then-reopen creates a FRESH row

B. Send-message idempotency
   - user-message creation carries a clientMessageId (idempotency key), unique per
     conversation (exists at DB level: uq_assistant_messages_user_idempotency)
   - double-click, retry-after-timeout, and "send from floating while inline is open
     in another tab" must NOT create duplicate user messages

C. Access change after a conversation exists — CURRENT ACCESS WINS
   If the student NO LONGER has access to the linked section/module:
   - the conversation is FILTERED OUT of the Workspace list, AND
   - direct open returns 404 (not 403 — consistent with 8.2's "don't reveal existence")

D. Supersession (Stage 4.6) does NOT revoke access
   If the section stays published but its transcript/summary is superseded:
   - the conversation REMAINS accessible (access keyed on section/module visibility)
   - historical answers stay as history (their snapshot reflects material at the time)
   - a NEW message grounds against the CURRENT (replacement) material

E. Delete while a response is pending (create-then-poll)
   - polling returns 404; the UI closes/removes the conversation
   - a worker that finishes afterward MUST NOT resurrect the conversation, and
     last_activity_at must not make a deleted conversation reappear
```
All new edge-case tests use **deterministic, E2E-only** setup (seeded unpublished section, seeded
pending job) — never random failure or manual DB edits (carried testability rule).

## Backend scope (Option A)
Every endpoint is **student-owned**: list / open / rename / delete only the caller's own
conversations.
```
Get-or-create lecture conversation   POST — transactional (EXISTS; reuse). Returns the single
                                     active conversation for (student, section, lecture_default)
                                     or creates it (invariant A).
List conversations                   GET — paginated via the standard Stage 5 offset envelope.
                                     Per row: id, title, course/module, lecture/lab title, last
                                     message preview, last_activity_at, message count, a "Lecture
                                     grounded" status chip. Excludes soft-deleted AND
                                     access-revoked (invariant C). Newest activity first.
Conversation messages                GET — KEYSET / cursor pagination: before=<created_at,id>,
                                     limit. Initial load = newest page in display order; older on
                                     scroll. RULE-10 ESCALATION (offset envelope can't express a
                                     cursor) → resolved with a KeysetPage sibling envelope.
Send message                         Creates a user message with a clientMessageId (invariant B),
                                     then existing 8.2 grounded generation runs at interactive
                                     priority. NO new AI infra.
Rename conversation                  PATCH title. Sets titleSource = manual. Plain text,
                                     length-bounded, Unicode-safe, whitespace-collapsed, escaped.
Delete conversation                  SOFT delete (deleted_at). Removed from list; reopen -> 404.
                                     No hard delete (retention note below).
```
```
Title rules
  - auto-derived from the lecture name or a plain TRUNCATION of the first user message
  - auto-title may update ONLY while titleSource = auto; a manual rename is never overwritten
  - NO AI call to generate a title — a wasted interactive request (rule 15). Forbidden.
last_activity_at
  - updates on user-message creation and assistant-response completion ONLY
  - rename and delete do NOT reorder the list
Access control
  - ownership enforced everywhere; 401 -> clear session + /login; 403 -> unauthorized state
    (rules 4, 5); not-yours / soft-deleted / access-revoked -> 404
```
**Migrations:** the one-active partial-unique index (`WHERE deleted_at IS NULL`); soft-delete column;
`titleSource`; (title + `clientMessageId` unique already exist). Single head after 0039. No
8.1-duplicate collapse (prereq #4 resolved).

**Generated client:** OpenAPI change -> regenerate the TS client + commit (rule 3).

## Thin UI scope (Option A)
Match the existing assistant-panel styling (the "inline idiom"; the monochrome system is still not in
code — `findings-design-doc-reality-gap`). Do NOT block on the unbuilt 4.9 Tailwind foundation.
Most-visible assistant UI so far -> a design-review pass is mandatory.

**Workspace (new student page; give it a nav home):** conversation list newest-first grouped by
course (row hierarchy + "Lecture grounded" chip; no user folders); empty state (CTA to start from a
published lecture); open -> full view reusing the 8.1/8.2 message UI + the 8.2 "Where did this answer
come from?" panel, keyset lazy-loading; a persistent **context pill** ("Chatting about: <lecture> ·
Grounded in published lecture material" + "Open lecture" action); new chat -> **lecture picker**
(states Available / Disabled / Unavailable / Hidden + empty/error states); inline rename; delete with
a confirm step.

**Floating widget:** persistent button on student pages (deterministic placement: bottom-right
default; quiz attempt offset/never over Submit; sticky-bar aware; mobile safe-area; keyboard
viewport overlap is not verified in 8.4 unless a future visualViewport/mobile gate is added). Lecture/lab page -> THAT lecture's grounded conversation (invariant A; same row as inline;
message visible in both; no double-submit — invariant B). Non-lecture page -> compact drawer (recents
+ "Start with a lecture" + "Open full workspace"). Drafts persist per conversation for the page
session. Delete-then-reopen starts a FRESH conversation. Poll only while a response is pending.

**Accessibility (in scope):** keyboard-reachable, focus trap in the drawer, ESC closes, ARIA on
list/panel, and an **aria-live region announcing assistant messages as they arrive** (async via
polling). Consistent with Stage 9's accessibility hardening.

**Current-lecture chat starters** (the renamed "breakdown" chips), strictly about the current lecture:
```
Explain the key ideas from this lecture simply.
What should I revise first from this lecture?
What are the most assessment-relevant points in this lecture?
```
They must NOT imply cross-lecture exam prep (that is 8.6).

## UI proof obligation
> From any student page the student opens the floating assistant, holds a grounded conversation on a
> lecture, finds that same conversation in the Workspace, renames one and deletes another — and a
> second student never sees any of it.

A panel rendering against a mock does not satisfy this. Real browser -> real backend -> real result.

## Browser gate
```
Lecture page -> floating button -> THAT lecture's grounded chat (8.2 grounding + 4.7 visibility
  intact; context pill + "where from" shown) -> inline entry and floating panel show the SAME
  conversation (no duplicate) -> a message sent in one view is visible in the other
Race: open from inline AND widget near-simultaneously -> ONE conversation (invariant A)
Idempotency: double-click send / retry after timeout -> ONE user message (invariant B)
Workspace -> lists the student's chats -> empty state on a fresh account -> open -> long thread
  lazy-loads via cursor -> send -> it lands -> "Open lecture" navigates to the linked lecture
New chat -> lecture picker shows Available/Disabled/Unavailable states correctly
Rename -> title persists across reload (plain text); a later auto-title does NOT overwrite manual
Delete -> drops from list -> reopen 404 -> reopen the LECTURE -> a FRESH chat
Delete while a response is pending -> polling 404s, UI closes, no resurrection (invariant E)
Access revoked: lecturer unpublishes the section (seeded) -> chat drops from Workspace AND direct
  open returns 404 (invariant C)
Isolation (separate contexts, rule 9): two students on the SAME lecture each have their OWN chat;
  student-2's Workspace shows only theirs; direct id access to student-1's chat -> 404
```
Plus: **Rule 14** full active Playwright suite re-runs green at close (was 17/17 after 8.2).
**Rule 11** one real-provider smoke against the **reissued** K2 key — at least one grounded message
through a new entry point — with the model-ID echo asserted; recorded in
`knowledge/steps/stage-08/8.4-real-provider-smoke.md`.

**Retention note + delete copy.** Soft-deleted conversations retain their messages and
`context_snapshot` rows — fine for staging, but it feeds the post-MVP watchlist item *"transcript
retention/deletion policy BEFORE any real-student deployment."* The confirm copy must NOT promise
permanent deletion:
```
Remove this conversation from your chat list?
(It is hidden from you; this is not a permanent data purge.)
```
"permanently delete" copy is forbidden.
