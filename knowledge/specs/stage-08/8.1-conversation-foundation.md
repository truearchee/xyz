## Goal
From a published lecture/lab page a student can press **"Start chat"** to open (or re-open) a
conversation **attached to that section**, ask questions, and get **real** AI answers (through the
4.5 gateway, not a hardcoded echo) that **persist and reload**. Conversations never bleed between
lectures. Answers may arrive as a single block (no streaming, no grounding yet).

## Why now
Stage 8 (Personal AI Assistant) per the roadmap. 8.1 lays the **conversation-list-ready** data
foundation so the 8.4 standalone window + sidebar grow on top with no rework, and is the first
consumer of the **interactive** limiter headroom (rule 15).

## Build
- Conversation + message store, shaped as a **list of conversations** each with a
  `conversationKind` (`lecture_default`/`manual`; reserve `floating_widget`/`workspace`) and an
  **optional** attached section. One `lecture_default` per (student, section) via a partial-unique index.
- "Start chat" = open-or-create the lecture_default conversation (race-safe).
- Availability follows decision 9: published + assigned + transcript **chunks + embeddings** present.
- Message send: save the user message first + a `pending` assistant message; enqueue an `ai`-queue
  job at **interactive** priority; the job runs `LLMGateway.complete(feature="assistant")` and
  persists the answer + provenance + `ai_request_log_id`; frontend polls (4.5d backoff, no hard
  timeout) until terminal. Bounded conversation history is sent to the model (decision 1).
- Double-send-safe via a client idempotency key (decision 8); failed turns never duplicate the user
  message and offer inline Retry (decisions 7, 11).
- Thin chat UI in the **existing inline-style idiom**: message list (user/assistant by alignment),
  textarea (Enter sends, Shift+Enter newline, labelled), pill-ish send button, empty/processing/
  unavailable states reusing the summary surface treatment, answers via `SummaryMarkdown`.

## Do not build
- No grounding/retrieval (8.2), no streaming (8.3), **no conversation-list sidebar / standalone
  window (8.4), no floating widget (8.4)**, no modes/workspaces (8.4–8.6), no save-to-glossary (8.5).
- No Tailwind / component library / monochrome repaint (that is Stage 4.9).
- No lecturer/admin assistant (student-only this stage).

## Data model changes
Migration **0032**: `assistant_conversations`, `assistant_messages`, partial-unique indexes; widen
`ck_ai_request_logs_feature` to add `'assistant'`. `grounding_status` column added now (nullable;
populated in 8.2). No further migration for 8.2.

## API changes
New student-only routes (all `Cache-Control: private, no-store`):
`GET /student/sections/{section_id}/assistant/availability`,
`POST /student/sections/{section_id}/assistant/conversation`,
`GET /student/assistant/conversations/{conversation_id}/messages` (paginated, Stage 5 envelope),
`POST /student/assistant/conversations/{conversation_id}/messages`,
`POST /student/assistant/messages/{message_id}/retry`. OpenAPI client regenerated.

## Worker / job changes
New `ai`-queue job `generate_assistant_answer(message_id)` (interactive priority via the gateway;
bounded RQ retry for transient/invalid_output only).

## Authz rules
Student-only (403 before lookup). Visibility via `get_visible_student_section` (published + assigned;
zero rows → 404, never 403). Conversation private to owner (another student's id → 404). Access
re-checked on every load (decision 5).
