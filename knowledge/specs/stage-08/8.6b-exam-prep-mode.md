# Stage 8.6b — Exam-prep mode

The second assistant mode, built on the 8.6a coordinator seam. Parent spec (the three modes, shared design,
guardrail philosophy): [[specs/stage-08/8.6a-mode-coordinator-homework]]. **Conversational only** — no saved
study plan/calendar/artifact (Stage 11 owns those). Migration block **0043–0047** (0042 = 8.6a). Pause for
review after 8.6b (D1=B); do not start 8.6c until the owner says go. Nothing merges (all of Stage 8 in one
PR at the end).

## Behavior
A student enters "Exam prep" → selects a **NAMED `AssessmentScope`** (Midterm/Final/…; UX #2) whose covered
weeks/sections are shown **read-only**. The assistant then, conversationally:
- summarizes what the exam covers (the scope's covered weeks),
- explains/reviews concepts grounded in the **covered weeks' permitted summaries** (published+assigned+
  generated only; says so if partial),
- highlights focus/weak areas from the **student's own progress** (Stage 9, read-only),
- **points to the Stage 6 exam-prep quiz** with the precise states below — it NEVER generates a quiz.

### Quiz pointer (precise — never generate)
Sourced from the existing quiz domain (`resolve_exam_prep_scope` → `ScopeResolution.available/reason_code`,
surfaced by `GET /student/modules/{id}/exam-prep-scopes`). The CTA lives on the conversation surface and is
computed FRONTEND-side via the existing `api.quiz.listExamPrepScopes` — the assistant domain never imports
quiz and never starts/generates a quiz:
```
available = true                 → CTA "Practice with the exam-prep quiz" → /student/modules/{moduleId}
reason_code = "processing"       → "Your practice quiz is being prepared"  (do NOT generate)
reason_code = "no_eligible_…"    → "Practice quiz not available yet"        (do NOT generate)
```

- **D2 = A (resume-or-create):** one active exam-prep conversation per `(student, assessment_scope)`,
  double-click-safe via a partial-unique index (the lecture_default/homework pattern).
- **Route — confirm against the routing table (the 8.6a lesson):** start at the spec's Think/Nvidia, run
  the rule-11 exam-prep smoke; if K2-Think-v2 returns `not_json` (reasons inline + rambles, F-6e — exactly
  what 8.6a's homework hit), **reroute to V2/Cerebras** and record it in ADR-057 + the smoke note. An
  interactive turn that emits a compact JSON answer is V2's job; Think is unreliable for it.
- **Grounding output:** reuse `AssistantGroundedAnswer` (`{answer, isStudyRelated}`) + `decide_grounding`.
  Exam-prep always discusses the scope when it is visible (`ready=True`); a question with no relevant chunk
  answers generally, off-topic redirects, lost access → access_denied. Partial readiness (some covered
  sections still generating) is noted in the context, not a grounding status.

## Thin UI scope

## UI proof obligation
A student picks a named exam, sees the assistant discuss exactly that scope grounded only in permitted
summaries, sees the correct quiz CTA/unavailable state, and another module's scope cannot be selected/accessed.

## Browser gate (8.6b)
```
open "Exam prep" → select a NAMED scope → covered weeks shown read-only
→ assistant discusses only that scope, grounded only in permitted summaries
→ another module's AssessmentScope cannot be selected/accessed (404)
→ unpublished/unassigned/other-student material never appears
→ quiz CTA when available; a real "being prepared"/"not available yet" state otherwise (no fake CTA,
  no generation) → NO saved study-plan artifact created
Regression → the existing general/lecture chat + homework still pass
```
Close with: the 8.6b gate + the general/lecture regression + the **full active Playwright suite (rule 14)** +
the **rule-11 exam-prep smoke** (model-ID echo asserted; route confirmed) — all on the non-disruptive
alt-port stack (:8005/:3005; never touch the :8000/:3000 stack Stages 10/11 use).

## Exclusions
No quiz generation (Stage 6 owns it; reference only); no saved plan/calendar/.ics; no mid-conversation mode
switching; no analytics/gamification/Stage-11 touch; no new retrieval index.
