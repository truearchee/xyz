# Stage 8.5 — Save-to-Glossary from the Assistant

> Source: owner spec filed 2026-06-19 (`.context/attachments/.../pasted_text_2026-06-19_22-33-08.txt`).
> Approved this session with decisions D1–D4 locked (see below). Implemented as ONE combined session
> (the owner spec permits it); the a/b split is retained as section headings, not separate trios.

## Why / What
Bring Stage 7's "highlight → save to glossary" feature into the assistant's chat replies. A student
highlights a term in a **completed assistant reply**, saves it, and gets a standard glossary entry —
reusing Stage 7's save flow, duplicate detection, definition cache, background job, and practice tools
end to end. The only genuinely new pieces: (1) `<SaveToGlossary>` mounted in assistant message
rendering, and (2) a glossary entry can record that it originated from a conversation. Everything else
is reuse — treat 8.5 as wiring two finished features across a thin seam.

## Hard prerequisites (confirmed by read-first)
Stage 7 core (7a–7c), Stage 8.1 + 8.2, Stage 8.4 all verified. Read-first confirmed: the source-reference
model already carries a `source_type` discriminator (`summary|manual|quiz`) + cross-domain FK precedent
(`source_quiz_attempt_id`); **the Stage 7 definition job DOES inject `selected_text` as prompt context on
a cache miss** (triggered the rule-10 escalation → resolved as D1); unbound conversations are NOT reachable
after 8.4 (so the unbound case is defensive + backend-rejected, not a UI surface).

## Locked decisions (owner-confirmed)
- **D1 — Definition content = subject-level only.** Chat saves feed NO conversation text to the prompt
  (`definition_context=""` → identical to the manual-add AI path). Cache key unchanged. Proof: cache key
  excludes context, input hash includes it ⇒ empty-context chat save shares the manual-add cache row +
  input hash ⇒ no new AI behavior ⇒ **no rule-11 smoke**. (See [[decisions/adr-055-conversation-sourced-glossary-save]].)
- **D2 — Destination read-only.** Subject/folder resolved server-side from the conversation's bound
  section; shown read-only. Entry type editable.
- **D3 — Duplicate = attach + confirm.** On a duplicate term, attach the conversation+message as an
  additional source **idempotently** (same conversation+message never twice); confirm "Already in your
  glossary."
- **D4 — Unbound chats = affordance hidden.** No save button when a conversation has no bound section;
  the backend rejects an unbound source regardless.

## 8.5a — Backend scope
- Extend `POST /student/glossary/highlight` (the ONE write path) with an optional discriminated
  `conversation: {conversationId, messageId}` source; `moduleSectionId` becomes optional; a validator
  requires exactly one. The assistant domain writes no glossary tables and gets no parallel endpoint
  (rule 8). Cross-domain reads via a new `platform/query/assistant_save_source_read`.
- **Server-side verification (anti-spoofing):** completed assistant message; message belongs to the
  conversation; conversation owned + not soft-deleted + section-bound; bound section published+assigned;
  `selectedText` occurs in the message after the same markdown-strip + whitespace-collapse normalization
  used for rendering; stored snippet server-derived, ≤500 chars.
- **Reuse unchanged:** subject/folder resolution, duplicate detection, term normalization, the definition
  cache, the gateway-routed background definition job (background priority), the `glossary_term_saved`
  event (atomic, idempotent). On duplicate, attach the conversation source idempotently.
- **Migration 0041** (only because the source model needed it): `source_conversation_id` +
  `source_message_id` FKs, widened `source_type` CHECK (+`'conversation'`), partial-unique idempotency
  index. Single head 0040→0041; fresh-DB round-trip green.
- **No new model call, prompt, queue, or cache-key change** (rule 6). Contract changed ⇒ client
  regenerated + committed (rule 3).

## 8.5b — Thin UI scope
- Generalize `<SaveToGlossary>` backward-compatibly to also accept the `conversation` source (a11y
  preserved). Mount it ONCE in the shared assistant render surface (`ConversationView` →
  `AssistantAnswerBody`), so the affordance appears wherever assistant replies display (inline panel +
  workspace + widget) — from a single mount point. Assistant replies only; never on the student's own
  message; shown only in section-bound conversations (D4). Read-only destination (D2); entry type
  editable. In-place confirmation; no navigation away.

## UI proof obligation
A student, inside a real assistant chat about an enrolled, published lecture, highlights a term in a
completed reply, saves it, and — without leaving the chat — that term appears in their glossary, its
definition fills in on its own, and they can practise it. Saving the same term again does not create a
duplicate. The affordance never appears on the student's own message or in an unbound chat.

## Browser gate (real browser → real backend → real DB, rule 9)
Open enrolled published lecture → open assistant → ask → grounded reply completes → highlight term in
the reply → "Save to glossary" appears → save → entry appears (subject = this lecture) → definition
fills async (background; AIRequestLog on cache miss, deterministic adapter at the provider boundary in
CI) → practise it → save same term again = "already saved", no duplicate, chat attached idempotently →
`glossary_term_saved` event in the same txn → another student cannot save against this conversation.
**Visible negative assertions:** no affordance on the student's OWN message; no affordance in an UNBOUND
conversation (UI-unreachable → covered by backend rejection + client gating).

## Required backend negative tests
user message rejected; pending/failed message rejected; messageId-not-in-conversation rejected;
selectedText-not-in-message rejected (+ positive markdown-straddle match); unbound rejected; not-owned
rejected (404); unpublished/unassigned rejected; double-submit idempotent (no dup entry/source/event);
cache key has no conversation/message component.

## Done means
`<SaveToGlossary>` works in assistant replies (assistant-only, completed-message, bound-only), single
mount point, a11y preserved; server verifies the save; standard subject-level definition (cache key
unchanged); duplicate handling idempotent ("already saved"); source records the OWN conversation; event
atomic+idempotent; ONE write path (glossary owns it); all negative tests pass; browser gate GREEN incl.
the two negative assertions; FULL active E2E suite GREEN (rule 14); tsc GREEN; client regenerated; single
migration head + round-trip green; rule-11 conclusion recorded (none needed); ADR + steps; roadmap +
knowledge updated in the same commit.

## Exclusions
No context-aware definitions; no saving from own messages; no "save whole reply as note"; no save in
unbound chats; no streaming (8.3 → 4.8); no changes to practice/flashcards/cache/Learn–Test; no
folder-management UI; no 8.6 modes; no 7d quiz-highlight.
