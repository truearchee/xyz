## Build
- Add `time_management` as a conversation kind, migration 0044 chained onto 0043, with resume-or-create one active time-management conversation per student.
- Add the time-management strategy on the existing 8.6a coordinator seam. It must use the shared single gateway call, shared persist path, `feature="assistant"`, interactive priority, and create-then-poll.
- Compose deterministic structured context with no extra model call: next 14 days of `due_at` and `session_date`, overdue items, top-N weak topics, and grade/progress summary.
- Read only the current student's own schedule/progress through read-only `platform/query`. Another student's deadlines/progress must never enter the context.
- Route `time_management/v1` to Cerebras/V2 32k. Do not route to Think/Nvidia.
- Surface the grounding/source affordance as structured schedule/progress data, not transcript retrieval.
- Add empty-state support for no upcoming deadlines, no progress data yet, and partial progress data.
- Add workspace UI entry and conversation surface for time management; widget remains unchanged.
- Add rule-11 real-provider smoke for the V2 route with model-ID echo.
- Add a browser gate covering own-data-only sentinel, no saved artifact/calendar/ICS, no Stage 11 planner/analytics artifact creation, and no planner-domain import.
- Root-cause and fix the recurring Stage 6 screenshot PNG churn: full-suite runs must not rewrite committed `knowledge/steps/stage-06/screenshots/*.png` by default.

## Do not build
- No saved plan, calendar event, `.ics`, `WorkloadPlan`, `WorkloadPlanItem`, or `InternalCalendarEvent`.
- No Stage 11 analytics/planner/gamification domain changes or imports.
- No exact clock-time blocking. Advice is day-level only: today, tomorrow, this weekend, next week.
- No SSE; continue create-then-poll.
- No additional model call for compaction.
- No merge. All Stage 8 work lands in one PR at the end.

## Data model changes
- Migration 0044 only: widen assistant conversation kind CHECK to add `time_management`; add a partial unique index enforcing one active time-management conversation per student.

- Conversation DTOs may gain mode context fields only if needed for the time-management surface.

## Authz rules

- Schedule/progress context must be current-student only.
- Foreign student/module data must never be included in prompt context or surfaced in UI.
