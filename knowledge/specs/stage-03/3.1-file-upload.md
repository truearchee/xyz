## Goal

A lecturer uploads (and replaces) a PDF on a predefined section in an assigned module; the file lands in private object storage and its metadata lands in the DB as a `section_assets` row, with backend-enforced authorization and zero direct object-store SDK calls in domain code.

## Resolved decisions (read before building)

These were ambiguous in earlier drafts. They are now settled. Do not re-litigate or drift.

1. **`section_assets` is created in THIS session, not Stage 2.** Stage 2 owns identity, modules, memberships, and predefined `module_sections`. The asset table belongs to the upload feature and lands when upload behavior lands. → The system overview / Stage 2 spec must be corrected so the "DB spine" does **not** claim `section_assets`. Building it twice is the failure mode this prevents.
2. **Column is `storage_key`, never `file_url`.** It holds a private storage path, not a public URL. The DB describes reality; it does not preserve Slice 1's product-doc wording. The API never exposes `storage_key`.
3. **A section may have multiple assets. There is no "slot" model.** Slice 1 explicitly allows multiple PDFs per section. Do **not** add a unique constraint on `module_section_id`. If per-slot semantics (slides / reading / worksheet) are ever wanted, that's a deliberate `asset_slot_key` addition later — not smuggled in here.
4. **Replace is same-row update, in 3.1.** `PUT /assets/{assetId}` updates the existing row in place, keeping `assetId` stable. No version history, no `status`/`replaced_at`/`replaced_by_asset_id` lineage. Both write paths (create + replace) are proven together so the storage abstraction is fully exercised before 3.2.
5. **Bucket is private.** Signed read URLs are a 3.3 concern, but the `StorageProvider` interface is shaped for them now.

## Build

1. **`StorageProvider` abstraction first — with a defined contract.** Interface in `platform/storage/base.py`; concrete `SupabaseStorageProvider`. Domain code depends on the interface only. This is the load-bearing deliverable: if skipped "just for now," Stage 4 transcript storage inherits the leak.

   ```python
   @dataclass(frozen=True)
   class StoredObject:
       key: str
       size: int
       content_type: str

   class StorageProvider(Protocol):
       def put_object(
           self,
           *,
           key: str,
           content: BinaryIO,
           content_type: str,
           content_length: int,
           metadata: dict[str, str] | None = None,
           overwrite: bool = False,   # default False; upload must not clobber
       ) -> StoredObject: ...

       def delete_object(self, *, key: str) -> None: ...

       def create_signed_read_url(
           self,
           *,
           key: str,
           expires_in_seconds: int,
       ) -> str: ...
   ```

   `StoredObject` is a frozen dataclass (value object, not a behavior contract — `Protocol` is for behaviors). `StorageProvider` performs storage operations only; it never performs domain authorization. `create_signed_read_url` is unused until 3.3, but the interface is shaped for the Stage 3 gate now so 3.3 adds no new storage surface.

   **`delete_object` is idempotent.** Deleting a key that does not exist is treated as success. Cleanup code, replace races, and the future orphan-reconciliation job must all rely on this — none of them should branch on "was it already gone?"

   **Async/blocking note.** The contract is written with sync signatures. If `SupabaseStorageProvider` is called from async FastAPI handlers, blocking SDK calls must not block the event loop. Either define `StorageProvider` with `async def` methods, or execute blocking provider calls in a threadpool (`asyncio.to_thread` / `run_in_executor`). A 25 MB upload blocking the event loop freezes unrelated requests. Decide the async strategy before writing the first handler; retrofitting it later is messier than choosing upfront.

2. **`section_assets` model + migration** per the schema below.
3. **Upload endpoint** (`POST`): authz → validate → generate `assetId` + `storage_key` → put object → insert row → compensate on failure. `processing_status = 'completed'` synchronously (no pipeline in MVP); the column stays for Stage 4.
4. **Replace endpoint** (`PUT`): authz → spool/validate/hash new file → put new object → begin DB transaction → lock existing asset row → update same row → commit → best-effort delete old object.
5. **Backend PDF validation:** magic-byte check, server-normalized MIME, size ceiling, filename sanitization. Frontend validation is UX only.
6. **Storage key generation** (`keys.py`): deterministic shape, random nonce, never the raw filename.
7. **Read model** in `platform/query`: fetch a section + its assets for an assigned lecturer (renders existing files in the upload UI). No business decisions in this layer.
8. **Allowed section types:** PDF assets may be uploaded to **any** predefined `module_section` type — `lecture`, `lab`, `assignment`, `supplementary`. Do **not** apply the transcript `lecture/lab`-only restriction here; that constraint belongs to the `Transcript` model in Stage 4, not to `section_assets`.
9. **Checksum computation rule:** `checksum_sha256` is computed by the backend over the exact validated byte stream that is uploaded to storage. Validation and checksum calculation must not consume the stream without rewinding or spooling before upload — that produces an empty or partial upload with a valid-looking hash, which is deeply annoying to debug. MVP approach: spool into a `SpooledTemporaryFile` or bounded bytes buffer at the configured size ceiling, then validate, hash, rewind, upload. No need to stream multipart yet.

## Data model changes

### `section_assets` (NEW — owned by this session)

```
section_assets
- id                       uuid pk
- module_section_id        uuid  fk → module_sections(id)  NOT NULL   (ON DELETE RESTRICT)
- storage_key              text  NOT NULL  UNIQUE
- file_name                text  NOT NULL                 ← sanitized original, display only
- mime_type                text  NOT NULL                 ← server-normalized to application/pdf
- file_size                bigint NOT NULL                ← bytes
- checksum_sha256          text  NOT NULL                 ← of stored bytes; NOT globally unique
- processing_status        text  NOT NULL default 'completed'
                                   enum: uploaded | processing | completed | failed
- uploaded_by_user_id      uuid  fk → app_users(id)  NOT NULL
- created_at               timestamptz NOT NULL default now()
- updated_at               timestamptz NOT NULL default now()
```

Notes on the field choices:

- **`storage_key` not `file_url`** — private path; never exposed by the API; signed URLs come in 3.3.
- **`uploaded_by_user_id`** represents the user who uploaded the **current stored object**. On replace, it is updated to the replacing lecturer — the original uploader is not preserved. This is intentional: the field tracks current object provenance, not creation history. If original-creator tracking is ever needed, add a separate `created_by_user_id` column at that point; do not retrofit it here.
- **`app_users(id)` not `users(id)`** — the app authorization table, not the Supabase Auth table. Supabase Auth is identity, not the app's role/membership source of truth.
- **`uploaded_by_user_id` not `uploaded_by_lecturer_id`** — role lives in `CurrentUserContext`, not baked into the column name.
- **`checksum_sha256`** — verifies stored object matches upload; supports duplicate/debug checks and replacement logs. **Not** globally unique: the same PDF can validly appear in multiple sections/modules.

### Constraints

```sql
ALTER TABLE section_assets ADD CONSTRAINT uq_section_assets_storage_key UNIQUE (storage_key);
ALTER TABLE section_assets ADD CONSTRAINT ck_section_assets_file_size CHECK (file_size > 0);
ALTER TABLE section_assets ADD CONSTRAINT ck_section_assets_processing_status
    CHECK (processing_status IN ('uploaded','processing','completed','failed'));
```

- `processing_status` is a stable internal enum → DB constraint **now**.
- No DB CHECK on `mime_type` yet → service-layer validation (consistent with the roadmap's "service layer first, DB constraints once stable" stance). Harden later.

### Indexes

```sql
CREATE INDEX ix_section_assets_section  ON section_assets (module_section_id);
CREATE INDEX ix_section_assets_uploader ON section_assets (uploaded_by_user_id);
```

### Critical separation (do not merge)

`processing_status` (this table, technical file state) and `publish_status` (on `module_sections`, student visibility) are different axes. A file can be `completed` while its section is `draft`. Honoring this now is why 3.3 stays clean.

## Storage / DB failure behavior (two systems, no shared transaction)

Object storage and Postgres do not commit together. Compensation is explicit.

### Upload sequence

```
1. Authz passes.
2. Spool file into bounded buffer (SpooledTemporaryFile up to size ceiling).
3. Validate: magic bytes, size, MIME normalize, filename sanitize.
4. Compute checksum_sha256 over validated bytes.
5. Rewind buffer.
6. Generate assetId and storage_key.
7. put_object(overwrite=False) → storage.
8. INSERT section_assets row.
9. If DB commit fails → best-effort delete_object(storage_key); surface 5xx.
```

**Checksum stream rule:** steps 3–5 must not consume the stream without rewinding before upload. Computing the hash and then uploading from an already-exhausted buffer produces an empty or partial object with a plausible-looking hash — deeply annoying to debug. Spool → validate → hash → rewind → upload is the required order.

### Replace sequence

The previous draft held the DB transaction open across the storage call (lock → upload → update → commit). Revised to lock only at the DB update point, avoiding a held transaction during external I/O:

```
1. Authz passes (including asset↔section↔module chain).
2. Spool + validate + hash new file; rewind. (Same as upload steps 2–5.)
3. Generate a NEW storage_key.
4. put_object(overwrite=False) → storage with new key.
5. BEGIN DB transaction.
6. SELECT section_assets WHERE id = assetId FOR UPDATE.
   Capture current storage_key as old_storage_key.
7. UPDATE row: storage_key, file_name, mime_type, file_size,
   checksum_sha256, uploaded_by_user_id, updated_at.
8. COMMIT.
9. After commit: best-effort delete_object(old_storage_key).
10. If DB UPDATE fails → best-effort delete the NEW object; old row + old
    object remain intact and authoritative.
```

The `FOR UPDATE` at step 6 serializes concurrent replaces of the same `assetId` without holding a transaction open across an external storage call. Row lock over optimistic concurrency — simpler for MVP.

### Storage failure mapping

| Condition | Behavior | HTTP |
|---|---|---|
| `put_object` fails — provider outage / timeout | No DB row created | 503 Service Unavailable |
| `put_object` fails — unexpected provider error | No DB row created | 502 Bad Gateway |
| Storage succeeds, DB commit fails | Compensating `delete_object` attempted; warning logged on cleanup failure | 5xx |
| Old-object `delete_object` fails after replace commit | Request succeeds; orphaned key logged; reconciliation job (future) | — (200/201) |
| New-object `delete_object` fails after replace DB-fail | Old row + old object intact; warning logged | 5xx |

**Residual risk, stated not hidden:** a cleanup-delete failure after a replace commit leaves an orphaned object in the bucket. MVP = structured warning log including the orphaned `storage_key`. Future hardening = orphan-reconciliation job (object in bucket with no referencing `section_assets` row). Out of scope for 3.1.

## Storage key generation

```
modules/{moduleId}/sections/{sectionId}/assets/{assetId}/{randomNonce}.pdf
```

Rules:

- Never use the raw filename in the key.
- Generate `assetId` before upload (it's part of the key).
- Include a random nonce / UUID component.
- `storage_key` is `UNIQUE` in the DB.
- Provider `put_object` runs with `overwrite=False` — never clobber an existing object.
- Optional cosmetic human-readable suffix is allowed but never trusted:
  `.../{assetId}/{randomNonce}-lecture-3-slides.pdf`.

## Validation rules

### PDF content

- **Primary:** PDF magic header `%PDF-` at byte 0 (strict; acceptable for MVP).
- **Secondary:** declared MIME / extension may be inspected but **client MIME is not trusted**.
- **Stored `mime_type` is server-normalized** to `application/pdf`.

### Size

- `MAX_SECTION_ASSET_UPLOAD_BYTES` — configurable; **MVP default 25 MB** (10 MB is too tight for image-heavy slide PDFs).
- Enforce in layers: app-level max body size; ASGI/server/proxy body limit where available; `Content-Length` pre-check where available; validator still verifies actual bytes read.

### Filename (display-only)

- Strip path components.
- Trim whitespace; cap length.
- Reject / normalize null bytes and control characters.
- Never use the raw filename in the storage key.
- Never write the raw filename into response headers (e.g. `Content-Disposition`) without escaping.

## API changes

> Endpoint naming is provisional (roadmap: "exact endpoint naming can come later"). Shapes matter more than paths.

### `GET /api/modules/{moduleId}/sections/{sectionId}/assets`

Returns the list of assets for a section. Used by the lecturer upload UI to render existing files.

- **Auth:** lecturer assigned to `moduleId`; section belongs to that module.
- **Response 200:**
  ```json
  {
    "assets": [
      {
        "id": "uuid",
        "moduleSectionId": "uuid",
        "fileName": "lecture-3-slides.pdf",
        "mimeType": "application/pdf",
        "fileSize": 824133,
        "checksumSha256": "…",
        "processingStatus": "completed",
        "uploadedByUserId": "uuid",
        "createdAt": "2026-05-30T10:00:00Z",
        "updatedAt": "2026-05-30T10:00:00Z"
      }
    ]
  }
  ```
  Default ordering: `created_at ASC`. No `storageKey` exposed. Student access to asset content is 3.3's concern.

### `POST /api/modules/{moduleId}/sections/{sectionId}/assets`

Multipart upload, single field `file` (PDF). Creates a new `section_assets` row.

**Response 201:**
```json
{
  "id": "uuid",
  "moduleSectionId": "uuid",
  "fileName": "lecture-3-slides.pdf",
  "mimeType": "application/pdf",
  "fileSize": 824133,
  "checksumSha256": "…",
  "processingStatus": "completed",
  "uploadedByUserId": "uuid",
  "createdAt": "2026-05-30T10:00:00Z",
  "updatedAt": "2026-05-30T10:00:00Z"
}
```
Response never includes `storageKey`. Content access is 3.3's signed-URL concern.

### `PUT /api/modules/{moduleId}/sections/{sectionId}/assets/{assetId}`

Same multipart body. Updates the existing row in place (same `assetId`), supersedes the stored object per the replace sequence.

### Error contract

| Condition | Status |
|---|---|
| Not a PDF (magic-byte fail) | 422 |
| Over size limit | 413 |
| Role is student | 403 |
| Role is admin attempting upload | 403 |
| Lecturer not assigned to module | 403 |
| Section not in module / not found | 404 |
| `assetId` exists but belongs to a different section/module (replace) | 404 |
| `put_object` fails — provider outage / timeout | 503 |
| `put_object` fails — unexpected provider error | 502 |
| Storage succeeded, DB commit failed | 5xx (after compensating delete) |

**Ordering:** authorize **before** reading/streaming the body where the framework allows, so an unauthorized caller never pushes 25 MB through the system. Never persist or store anything before authz passes.

## Authz rules

Enforced in the service layer, consuming the resolved `CurrentUserContext` from Stage 2 — **no domain-local auth lookups** (P0 hard rule).

```
upload:
  role == lecturer                                       else 403
  moduleId ∈ current_user.module_memberships             else 403
  section.course_module_id == moduleId                   else 404
  (student | admin) → reject                             403

replace (additionally):
  asset.id == assetId                                    else 404
  asset.module_section_id == sectionId                   else 404
  section.course_module_id == moduleId                   else 404
  lecturer assigned to moduleId                           else 403
```

**Multiple lecturers per module:** any **assigned** lecturer (not "owning" lecturer) may upload and replace section assets in MVP. If a narrower rule is ever wanted, gate on `can_manage_content` from `CurrentUserContext` — but the MVP rule is explicit: assigned ⇒ may manage content. The term is "assigned lecturer," never "owning lecturer" (modules have multiple lecturers; "owner" is not a concept here).

**Inactive-user dependency:** inactive users are rejected by the request context / auth dependency (Stage 2) **before** the upload service runs. 3.1 depends on this and must not re-implement or bypass `active_state` checks at the feature layer.

### Negative authorization tests — built HERE, not later (Stage 3 mandate)

```
student attempts upload                       → 403
admin attempts upload                         → 403   (admins don't manage content)
lecturer uploads to unassigned module         → 403
lecturer uploads to section in another module → 404
replace assetId belonging to another section  → 404
non-PDF rejected by backend (not just FE)     → 422
oversize file rejected                        → 413
```

## Tests required

### Unit
- `StorageProvider` interface honored: domain service calls the interface, not the SDK (assert via an in-memory fake provider).
- PDF validator: accepts valid `%PDF-`-at-byte-0; rejects a renamed non-PDF (e.g. PNG with `.pdf` extension); rejects oversize; normalizes stored MIME to `application/pdf`.

- Storage key generator: correct shape, random nonce present, raw filename never embedded, collision-safe.

### Integration (DB + fake storage)
- List assets: returns empty array for a section with no uploads; returns all assets after two uploads to the same section.
- Upload happy path: object stored, row created, `processing_status = completed`, checksum persisted.
- Upload compensation: storage succeeds, DB commit fails → new object cleanup attempted; no row persists.
- Replace happy path: same `assetId`, row updated, old object deleted best-effort.
- Replace DB-fail: DB update fails → old row still points to old object; new object cleaned up.
- Replace cleanup-fail: old-object delete fails → request still succeeds, warning logged.
- Concurrent replace on same `assetId`: row lock serializes; no interleaved key/cleanup corruption.
- Storage provider outage: `put_object` raises → 503, no row created.
- FK integrity: upload to non-existent section → 404, nothing stored.
- Multiple assets per section: two uploads to the same section both succeed (no unique-constraint block).
- Checksum integrity: computed checksum over spooled bytes matches independent SHA-256 of the same test file.
- All seven negative-authz cases above.

## Done means

- A lecturer can upload and replace a PDF on a predefined section of **any type** (`lecture`, `lab`, `assignment`, `supplementary`) in an **assigned** module; metadata persists in `section_assets`, the file persists in a **private** bucket via the `StorageProvider` abstraction.
- A lecturer can list existing assets for a section via `GET /assets`.
- No domain or feature code imports the Supabase storage SDK directly.
- Backend rejects non-PDF and oversize files independently of the frontend; stored MIME is normalized; filenames are sanitized; checksum is computed after validation, before upload.
- A section can hold multiple assets; no slot constraint exists.
- Replace updates the same row (stable `assetId`), with lock deferred until the DB transaction, explicit storage/DB compensation, and storage failure mapped to 502/503.
- All seven negative-authz cases return the correct status and persist nothing.
- `processing_status` and `publish_status` live on separate tables/axes and are never conflated.
- OpenAPI → TS generation works for multipart; the frontend compiles against generated types.
- `.env.example` updated; bucket name comes from config, never hardcoded.
- `knowledge/` updated in the same commit, including the corrected Stage 2 schema claim.

> This session does **not** satisfy the Stage 3 phase gate on its own. The gate ("lecturer publishes → student sees published → student does NOT see draft → signed URL authorization enforced") completes at the end of 3.3. 3.1 lands the storage abstraction and the authorized write path the gate is built on.
