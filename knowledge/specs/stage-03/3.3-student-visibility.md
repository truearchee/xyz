## Goal

A student in an assigned module sees **only** published, active sections (title, lecturer notes, completed-asset metadata) and can open a published asset **solely** through a freshly minted, short-lived signed read URL. `storage_key` is never returned as a structured API field, the asset's authorization is **re-validated live at mint time**, and storage keys are non-semantic so the opaque URL leaks nothing meaningful.

## Resolved / inherited decisions (read before building — do not re-litigate)

Inherited from 3.1/3.2 and Stage 2. Settled; stay consistent.

1. **Module authz is already solved — consume it, don't rebuild it.** `require_module_access(moduleId)` (2.4) returns `ModuleAccessContext { module_id, module_status, global_role, can_publish, membership_id }` only when membership **and** module are both active. Non-member / archived module / admin → `404` (no existence leak). 3.3 reuses this guard verbatim; no domain-local membership lookups (P0 hard rule).
2. **Identity vs authorization stays split.** `CurrentUserContext` (2.2) carries identity only; inactive → `403` before any feature code runs. 3.3 never re-checks `active_state` at the feature layer.
3. **Empty published sections are valid** (ADR-009). A student viewing an empty published section sees title + notes (if any) + zero assets. Render harmlessly.
4. **Any active assigned lecturer may manage content** (ADR-011); students may not. 3.3 reaffirms the student-side `403` for writes and builds none.
5. **DTO casing is camelCase; the error envelope from 3.1/3.2 is reused.** No second API style.

## Storage-key and signed-URL invariants (the load-bearing security model)

These replace the earlier (incorrect) "storage_key never crosses the API boundary" claim. Signed URLs are not invisibility cloaks; they are bearer tokens wearing a trench coat.

- **`storage_key` is never returned as a structured API field**, and is never user-derivable. The student projection does not select it; no DTO carries it.
- **A signed URL is an opaque, short-lived bearer capability.** It may contain provider-specific object identifiers (Supabase embeds the object path in the URL). Anyone holding the URL can fetch the object until it expires. This is inherent and accepted; the mitigation is a short TTL + non-semantic keys.
- **Storage keys must be random / non-semantic** so that even if the key appears inside the opaque URL it leaks nothing — no module name, lecturer/student name, or section title. Confirm 3.1's key generator already satisfies this (3.1's test asserts "raw filename never embedded, collision-safe"); the shape is `section-assets/{assetId}/{random}.pdf` or whatever 3.1 settled. Do not change the scheme; only verify the property holds, since 3.3 now depends on it.
- **`publish_status` (section) and `processing_status` (asset) are independent axes** (3.1/3.2). They are never merged into one field and never derive one another. The student gate applies **both** independently (see Build #1).

## Build

1. **Student read projections in `platform/query/content_read.py`** — the visibility rule is encoded **in the query**, not in a branch:
   - `list_published_sections_for_student(db, module_id) -> list[StudentSectionRow]`
   - `get_published_section_for_student(db, module_id, section_id) -> StudentSectionDetailRow | None`
   - Filter: `publish_status = 'published' AND status = 'active'`. Joined assets are filtered to `processing_status = 'completed'`. A draft / unpublished / archived section returns **no row**; there is no code path that can return one. Projection rows carry only student-visible fields (no `storage_key`, no `processing_status`). Pure read model — no business decisions (rule 5).
   - **Deterministic ordering (required):** sections `ORDER BY order_index ASC, id ASC`; assets `ORDER BY created_at ASC, id ASC` (match 3.1's asset ordering). Without this, tests flicker in CI.
2. **Internal download-ref fetch (privileged, never serialized)** in `content_read.py`:
   - `get_asset_download_ref(db, module_id, section_id, asset_id) -> AssetDownloadRefRow | None`
   - Returns the **raw joined fields** the service needs to authorize and mint: `{ asset_id, section_id, course_module_id, section_publish_status, section_status, asset_processing_status, storage_key, file_name, mime_type }`.
   - This is the **only** read that returns `storage_key`. It exists so no one "temporarily" adds `storage_key` to the student projection (temporary fields become architecture — usually the bad kind). It makes **no authorization decision** (rule 5); it only fetches. Its result must never be wired to a response DTO.
3. **Role-aware read selection at the service boundary** — the section list/detail endpoints pick the projection by `ModuleAccessContext.global_role`:
   - `lecturer` → existing all-state projection from 3.1/3.2 (sees draft/unpublished).
   - `student` → the published-only projection above.
   - This is a **projection selection**, not a business rule; it lives in the service, never in `platform/query`.
4. **Download-authz decision + URL mint (service layer).** After `require_module_access`, the download service calls `get_asset_download_ref`, then decides:
   - `student`: `section_publish_status == published` AND `section_status == active` AND `asset_processing_status == completed` AND `asset.section/module` match the path → else `404`.
   - `lecturer (assigned)`: `section_status == active` AND `asset.section/module` match → else `409 SECTION_ARCHIVED` for archived (consistent with 3.2), `404` for wrong section/module.
   - On pass: mint a **fresh** URL via `StorageProvider.create_signed_read_url(storage_key, ttl)`. **Re-validated live every call** — never trust that the client once saw the section published; the lecturer may have unpublished it since. Return `{ url, expiresAt }`. Never return `storage_key`. Never persist or cache the URL. Set `Cache-Control: no-store` on the response so a stale minted URL is not served from an HTTP cache.
5. **`SupabaseStorageProvider.create_signed_read_url`** — implement against the real Supabase signed-URL call, honoring the async/threadpool decision from 3.1 (blocking SDK calls must not block the event loop). The in-memory fake provider returns a deterministic stub URL + expiry so tests assert the call shape without network. Minting is synchronous, request-scoped, and cheap — no worker queue (the "no direct calls from handlers" rule is about AI, not storage).
6. **Frontend (presentational only)** — a student section list and section view (title, lecturer notes, asset list with a download action). The download action calls the URL endpoint, then opens the returned URL; the URL is opaque and is **not** cached beyond the click. **Expired-link handling:** if opening/downloading fails due to an expired/invalid signed URL (browser PDF viewers can issue delayed range requests that outlive the TTL), re-mint on the next click and show "This file link expired — try opening it again." Render an empty published section harmlessly; handle `404` (gone / unpublished) without crashing. No business rules in the component.

## Do not build (scope guards)

- **Any student write path** — upload / replace / publish / unpublish / notes. Students are read-only; reaffirm `403` in tests, build nothing.
- **Asset-level publish flags or per-asset visibility.** Visibility is **section-level only**: an asset is visible iff its parent section is published. Do not add an asset publish column.
- **Coupling `publish_status` to `processing_status`.** Independent; the student gate applies both separately, never merged.
- **Caching/persisting signed URLs, making the bucket public, returning `storage_key` as a field, or download proxying/streaming through the API.** The 3.1 private-bucket posture holds; instant revocation of issued URLs is explicitly out of scope (see ADR-013).
- **Transcript / summary student views (Stage 4), any event emission (Stage 5), download audit logging.** Out of scope.
- **`published_at` / first-published audit** — still not added (3.2 Decision 5).
- **Pagination** on the student section list. Section count per module is bounded (mirrors 2.4's `/modules` reasoning). Contrast glossary, which *will* need it.
- **A `/student/...` or `/lecturer/...` route fork.** Endpoints are shared and role-aware (Locked decision 5).

## API changes

```
GET /modules/{moduleId}/sections                       (role-aware via require_module_access)
  lecturer → all active sections        (rows)
  student  → published + active only     (rows)
  → 200 SectionListItem[]               ← ONE shared DTO for both roles (no oneOf union)
  → 401 unauthenticated · 403 inactive
  → 404 module not accessible (non-member / archived module / admin — no special-casing)

GET /modules/{moduleId}/sections/{sectionId}           (path shape MUST match 3.1/3.2)
  lecturer → SectionDetail (existing 3.2 DTO, carries publishStatus + lecturer fields)
  student  → StudentSectionDetail
  → 200
  → 404 if (student AND section not published-and-active)   ← treated as nonexistent; no existence leak
  → 409 SECTION_ARCHIVED if (lecturer AND section archived)  ← consistent with 3.2
  → 404 if section not in an accessible module

GET /modules/{moduleId}/sections/{sectionId}/assets/{assetId}/download-url
  → 200 AssetDownloadUrl { url, expiresAt }    ← freshly minted, short-lived, opaque; Cache-Control: no-store
  → 404 if (student AND asset/section not visible — re-validated live; covers post-publish unpublish)
  → 409 SECTION_ARCHIVED if (lecturer AND section archived)
  → 404 if asset does not belong to the addressed section / module
  (student non-visible is NEVER signalled with 403 — that would leak existence)
```

DTOs (camelCase; error envelope reused):

```
SectionListItem      { id, title, type, orderIndex, hasAssets, hasNotes }   ← shared list DTO; NO publishStatus
StudentSectionDetail { id, title, type, orderIndex, lecturerNotes, assets: StudentAssetMeta[] }
StudentAssetMeta     { id, fileName, mimeType, fileSize }     ← NO storageKey, NO processingStatus
AssetDownloadUrl     { url, expiresAt }                        ← expiresAt = absolute UTC ISO-8601
```

DTO rules:
- **One shared `SectionListItem` for the list endpoint** (avoids an ugly `StudentSectionSummary[] | SectionSummary[]` union in the generated client). Rows differ by role; the schema does not. Richer lecturer fields (incl. `publishStatus`) live on the lecturer **detail** view, which lecturers already open to drive the 3.2 `PublishToggle`.
- **No `publishStatus` on the list DTO for either role.** Returning it to students teaches the frontend it can reason about visibility — it must not. The backend already guarantees the rows.
- **`hasAssets` / `hasNotes` are caller-relative**, defined as "the caller's *detail* view would return a non-empty result":
  - `hasAssets` = the caller's detail would return ≥1 asset → for students this means ≥1 **completed, visible** asset (so the list never claims `hasAssets=true` while detail returns `[]`); for lecturers it means ≥1 asset they can see.
  - `hasNotes` = `lecturer_notes` is non-empty after trim.
- `lecturerNotes` is included in **student** detail (Slice 1: notes are shown to students).
- `StudentAssetMeta` omits `storageKey` and `processingStatus` by construction.
- **Lecturer-list reconciliation:** if 3.1/3.2 already shipped a lecturer section-list DTO carrying `publishStatus`, 3.3 reconciles it to `SectionListItem` (publishStatus moves to detail) and records it as a "Modified prior sessions" deviation — mirroring 3.1's Stage-2 placeholder reconciliation. If no lecturer list endpoint exists yet, 3.3 introduces the shared one. (Residual confirmation below.)

## Authz rules

Enforced in the service layer, consuming the resolved contexts from Stage 2 — no domain-local auth lookups (P0 hard rule). Authz precedes business validation (3.2 validation-timing discipline).

```
read section list / detail / download-url:
  authenticated + active                         (get_current_user)            else 401 / 403
  require_module_access(moduleId)                 (active membership + module)  else 404
  role branch (global_role from ModuleAccessContext):
    student:
      section.publish_status == published                                       else 404
      section.status         == active                                          else 404
      (download) asset.processing_status == completed                           else 404
      (download) asset.module_section_id == section.id                          else 404
    lecturer (assigned):
      section.status == active                                                  else 409 SECTION_ARCHIVED
      any publish_status (existing 3.1/3.2 read behavior)
      (download) asset.module_section_id == section.id                          else 404
      (download) section.course_module_id == moduleId                           else 404
    admin:
      no membership → require_module_access already returned 404 (no branch)

  student attempting any write (upload/replace/publish/unpublish/notes) → 403 CONTENT_FORBIDDEN
    (valid member, role-forbidden action — reaffirm 3.1/3.2; build nothing)
```

## Tests required

### Unit (projection + service)
- Student projection returns **only** `published + active` sections; draft / unpublished / archived yield no row.
- Student asset projection returns **only** `processing_status = completed` assets (seed a non-completed asset under a published section; assert it is hidden — forward-compat for Stage 4).
- Projection rows never carry `storage_key` or `processing_status`.
- `get_asset_download_ref` returns `storage_key` (it is the privileged fetch) and is not referenced by any response DTO (grep/assert no `storageKey` field in generated schemas).
- `hasAssets` is `false` when the only asset under a published section is non-completed; `true` only with a completed visible asset. `hasNotes` reflects trimmed emptiness.
- Deterministic ordering: sections by `(order_index, id)`, assets by `(created_at, id)`.
- Signed-URL mint calls `create_signed_read_url` with the correct `storage_key` and the configured TTL; the fake provider's stub URL + expiry round-trip into `AssetDownloadUrl` with `expiresAt` as UTC ISO-8601.

### Integration (router + auth + fake storage)
- Student lists sections in an assigned module → sees only published + active; response is `SectionListItem[]` with no `publishStatus`/`storageKey`.
- Student gets a published section detail → title + `lecturerNotes` + completed-asset metadata; **no** `storageKey`.
- Student gets an **empty** published section → `200`, zero assets, `hasAssets=false` in the list (ADR-009 path).
- Student gets a draft / unpublished / archived section → `404` (assert body does not distinguish "absent" from "hidden").
- Student gets a section in an unassigned module → `404`.
- Student requests download-url under a published section → `200 { url, expiresAt }`, response carries `Cache-Control: no-store`.
- **Live re-validation:** publish → student mints URL `200`; lecturer unpublishes; student mints again → `404`.
- Student download-url under draft/unpublished section → `404`; under another module → `404`.
- Student write attempts (upload/replace/publish/unpublish/notes) → `403`.
- Lecturer download-url under an **active** own-module section → `200`; under an **archived** section → `409 SECTION_ARCHIVED`.
- Admin student-read → `404` (no membership).
- Lecturer read of own-module sections still returns all states (3.1/3.2 regression guard).

### E2E (the Stage 3 phase-gate vertical)
```
admin creates module + assigns lecturer + student (2.x)
→ lecturer uploads PDF (3.1)
→ lecturer publishes section (3.2)
→ student sees the published section + lecturer notes (3.3)
→ student opens the asset via a freshly minted signed read URL (3.3)
→ lecturer unpublishes
→ student no longer sees the section; download-url now 404 (3.3, live re-validation)
```

## Locked decisions (ratified in review)

1. **`404`, not `403`, for a student requesting non-visible (draft/unpublished/archived) content.** No existence leak; uniform with the cross-tenant `404` discipline. → ADR-012.
2. **Signed read-URL TTL: short, env-configurable `SIGNED_READ_URL_TTL_SECONDS`.** Already-issued URLs remain valid until expiry (bearer capability) — explicit residual risk in ADR-013. *(Value confirmation below.)*
3. **Download endpoint is role-aware now (lecturer + student)**, not students-only — the mint path is shared, only the gate differs; avoids a later retrofit. → ADR-013.
4. **Student-visible assets are filtered by `processing_status = 'completed'`.** No behavior change in MVP (everything is completed on upload), but it protects Stage 4 from surfacing half-processed assets, applied independently of `publish_status`. → ADR-012.
5. **Shared role-aware endpoints (contract decision, not a soft note).** `/modules/{moduleId}/sections...` with projection by `global_role`; no `/student` route fork; one `SectionListItem` DTO. → ADR-014.
