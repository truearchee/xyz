After a chunk job completes successfully, and while Stage 4.4 embeddings and Stage 4.5 summaries are not implemented, `Transcript.status` must become `completed`.

Rules:
- Do not introduce a new `chunked` status.
- Do not enqueue embeddings.
- Do not generate summaries.
- Do not add AI infrastructure.
- Preserve parse/chunk idempotency and ingestion job result metadata.

`chunking` is non-terminal and cannot be treated as success by the browser gate.

## Async worker-driven browser gate
The browser gate must prove a real vertical path:
- lecturer logs in through the real app shell;
- lecturer opens an assigned module detail page;
- lecturer uploads VTT/TXT on lecture/lab sections only;
- backend stores the raw private transcript file;
- backend enqueues parse work through Redis/RQ;
- worker parses and persists transcript segments;
- worker chunks and persists transcript chunks;
- product-visible transcript status reaches `completed`;
- test-level DB reads prove parse job completion, chunk job completion, segment count > 0, and chunk count > 0;
- student upload/read attempts return 403 without clearing the session.

## UI status separation
Transcript status is a separate status surface from:
- section `publishStatus`;
- asset `processingStatus`.

The transcript UI must not reuse section publish test ids or asset processing test ids. It must not show transcript text, parsed segments, chunks, raw storage keys, or checksums.

## Do not build
- frontend transcript UI before Part 2 prerequisite repair is verified;
- embeddings or embedding jobs;
- summaries or summary jobs;
- AIRequestLog or AI call infrastructure;
- retry/recovery UI or worker recovery engine;
- transcript replacement/supersession;
- new transcript statuses;
- new product endpoints;
- raw transcript, segment, or chunk exposure;
- student-facing transcript or summary viewer;
- Playwright tests before Checkpoint E scope is approved;
- `/tracer` deletion before the approved tracer-removal checkpoint.
