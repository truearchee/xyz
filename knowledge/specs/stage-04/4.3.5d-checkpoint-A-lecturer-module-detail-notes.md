## Goal
Implement the first real Stage 3 lecturer content UI slice after B1 fixed module section generation.

Checkpoint A proves:

```text
Lecturer opens assigned module
-> sees generated backend sections
-> edits lecturer notes per section
-> notes save through backend
-> UI re-fetches and shows persisted notes
```

This checkpoint does not prove upload, publish, or student visibility.

## Scope
In scope:
- lecturer module detail route/page
- section list from real backend data
- section title/type/order rendering
- section `publishStatus` display if available
- lecturer notes editor per section

- re-fetch after successful notes save

Out of scope:

Checkpoint A is notes + section visibility only.

## Product rules to preserve
The lecturer sees the generated fixed structure. Lecturers do not manage the structure.

Forbidden UI:
- Create section
- Delete section
- Reorder section

If the UI needs sections, it must read the real sections generated by B1. No hardcoded section arrays.

## Implementation requirements

### 1. Lecturer route/page
Expected route, adapted to repo conventions:

```text
frontend/src/app/(app)/lecturer/modules/[moduleId]/page.tsx
```

This page should be thin and render a feature component such as:

```text
frontend/src/features/content/lecturer/LecturerModuleDetail.tsx
```

Do not put business logic directly in the route if existing frontend structure separates pages and features.

### 2. Section list
The lecturer module detail must load sections from the real backend read path. Each section should render:
- title
- type
- order, if available
- publishStatus, if available
- current lecturerNotes

Forbidden:
- hardcoded section arrays
- fake default sections
- local-only section mocks

### 3. Notes editor
Add a notes editor per section.

Expected component:

```text
frontend/src/features/content/lecturer/SectionNotesEditor.tsx
```

Behavior:
- initial value comes from backend response
- lecturer edits notes
- save calls wrapper/generated client
- saving state shown
- success triggers re-fetch
- visible notes update from re-fetched backend data
- failure renders `role="alert"`

No save is considered successful from local state alone.

### 4. API usage
Use generated client through the approved wrapper. Do not call `fetch()` directly from app/components/features.

Allowed direct fetch exceptions remain only:
- `frontend/src/lib/api/core/request.ts`
- `frontend/src/lib/api/upload.ts`

Checkpoint A should not need `upload.ts`.

### 5. Auth and role rules
Role/access must continue using app shell and `/me`.

Forbidden:
- JWT role decoding
- `session.user.role`
- `app_metadata` role reads
- `user_metadata` role reads

401/403 behavior must remain unchanged:
- 401 -> sign out / login
- 403 -> keep session, render unauthorized/error state
