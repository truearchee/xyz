## Purpose
Close the remaining Stage 3 verification gap found after the Checkpoint E audit.

Checkpoint E proved:

```text
student can access signed URL for a published asset -> HTTP 200
```

It did not prove:

```text
after lecturer unpublishes the section
-> student no longer receives that section in the server response
-> a fresh signed-URL request for that asset returns 403
```

This session adds that missing access-revocation proof and records cleanup evidence for E2E-created module/assets/storage objects where possible.

## Outcome
PASS after 4.3.5d-E2-B1 on 2026-06-08 16:28 +04.

The initial E2 gate proved that after unpublish the student section-list server response excludes the unpublished section, but the fresh student signed-URL request returned `404 {"detail":"SECTION_NOT_FOUND"}` instead of the required authenticated `403`.

4.3.5d-E2-B1 repaired the denial status contract. The E2 rerun returned `403 {"detail":"CONTENT_FORBIDDEN"}` for the fresh post-unpublish signed URL request, and `/me` still returned role `student`.

Stage 3 returned to `FULLY VERIFIED`.

## Important signed URL rule
Do not assert that a previously issued signed URL immediately stops working. Object-storage signed URLs may remain valid until expiry.

The required proof is:

```text
after unpublish
-> backend refuses to issue a NEW signed URL to the student
-> response is 403
```

This is the correct authorization boundary.

## Required proof steps

### Part 1 - Published access still works
Using student context:

```text
student opens assigned module
student section-list server response includes Lecture 1
student UI shows Lecture 1
student sees lecturer notes
student sees asset filename
student requests signed URL for Lecture 1 asset
signed URL request returns HTTP 200
student /me returns student
```

Record:

```text
Published response titles: [...]
Published signed URL status: 200
```

### Part 2 - Lecturer unpublishes section
Using lecturer context:

```text
lecturer opens same module
lecturer clicks Unpublish for Lecture 1
UI re-fetches
Lecture 1 publishStatus becomes Unpublished or Draft, depending on backend semantics
asset processingStatus remains completed
```

Record:

```text
Post-unpublish lecturer publishStatus:
Asset processingStatus:
```

### Part 3 - Student loses section in server response
Using the same student context after unpublish:

```text
student refreshes module page
wait for student section-list response
parse JSON
assert Lecture 1 id/title is absent
assert Lecture 2 remains absent
```

Record:

```text
Post-unpublish student response titles: [...]
Lecture 1 absent from server response: yes
Lecture 1 absent from UI: yes
```

DOM absence alone is not sufficient. The report must include server-response proof.

### Part 4 - Fresh signed URL request is denied
Using authenticated student token/context:

```text
request signed URL for the same asset after Lecture 1 is unpublished
```

Expected:

```text
HTTP 403
```

Also verify:

```text
GET /me still returns student after 403
```

Record:

```text
Post-unpublish signed URL request status: 403
Student /me after 403 role: student
```

A `401` is failure for this assertion.
