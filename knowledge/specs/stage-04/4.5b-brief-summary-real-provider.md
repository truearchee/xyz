## 5. Pipeline change — brief-only, detailed deferred cleanly

4.5a's after-embed hook enqueues both jobs. 4.5b gates the detailed enqueue:

```
after-embed hook:
  always enqueue generate_brief_summary           (real provider, if LLM_PROVIDER=k2think)
  enqueue generate_detailed_summary ONLY IF ENABLE_DETAILED_SUMMARY   (default false in 4.5b)

⇒ no detailed IngestionJob row exists; no detailed AIRequestLog exists.
   steps.summary_detailed = 'not_started' (established no-job sentinel; NOT 'failed').
   overallState rests at 'summarizing' (brief done, detailed not started).
   overallState reaches 'summarized' only in 4.5c.
```

Gating the **enqueue** (not no-op'ing the handler) means no orphan job row for the 4.6 sweeper to misread, and Think-v0 (no access) is never called.

**Testable assertions (when `ENABLE_DETAILED_SUMMARY=false`):**
```
- no generate_detailed_summary IngestionJob row
- no detailed AIRequestLog row
- steps.summary_detailed == 'not_started'   (established no-job sentinel; NOT 'pending' or 'failed')
- overallState == 'summarizing' after brief success
```

## 6. Provenance honesty — `backendUsed` + `backendRouteSource`

The provider does not echo the serving backend. We control the *request* route; we cannot prove the *serving* backend.

```
backendUsed          = the REQUESTED route ("cerebras" | "nvidia")
backendRouteSource   = requested | provider_echoed     -- ALWAYS 'requested' in 4.5b
```

`backendRouteSource` puts the honesty in the schema, not just a comment: when (if) the provider ever echoes the route, it flips to `provider_echoed` without a contract change. Until then, "default route → cerebras budget" is a config mapping, **not** a verified fact. ADR-025 + column comments state this. Rule 15's per-backend budget rests on this assumption; acceptable at single-model, single-lecturer, brief-only scale; recorded as an assumption (F-4.5-28).

## 7. Brief output contract — structured JSON (blocking edit 3)

Students must see a clean paragraph, never provider chatter or model reasoning (Slice 2). The connectivity smoke returning explanatory text to a bare "reply with only X" is expected — no JSON mode was used there. The real brief path is hardened three ways:

```
1. Provider-side (primary): K2ThinkProvider sets response_format={"type":"json_object"} IF the
   smoke confirms K2-Think-v2 honors it. This forces a JSON body and sidesteps preamble entirely.

2. Prompt (brief_summary/v1.yaml): request EXACTLY
       {"text": "<one concise student-facing paragraph>"}
   explicitly: no preamble, no reasoning, no code fences, no extra keys.

3. Validator (tolerant extract → strict shape):
   - locate the JSON object even if wrapped in ```json fences or preceded by reasoning text
     (a reasoning-lineage model may still prefix despite instructions)
   - validate the SHAPE strictly: single key `text`, non-empty, within length bounds
   - store ONLY the extracted `text` → chatter never reaches the student
   - if no parseable {text} object can be extracted → invalid_output → bounded retry
```

**Why tolerant extraction, not literal whole-response rejection:** rejecting any preamble outright risks every call failing on a model that stubbornly prefixes reasoning. Extraction + strict shape gives clean student-facing text without brittle failure. If `response_format` proves reliable, extraction tolerance is rarely exercised; it remains the safety net.

**Real-provider smoke must prove the actual brief path returns a parseable `BriefSummary`** — not merely that the API returned 200.

## 8. Provider HTTP error classification (blocking edit 2)

`K2ThinkProvider.send()` maps every outcome. An invalid model ID or bad key must **not** look transient — retrying a bad key is a denial-of-wallet strategy.

| Provider outcome | Disposition | Retry | Notes |
|---|---|---|---|
| 200 + valid body | `succeeded` | — | |
| 400 invalid model / bad request | `provider_config_error` | **terminal, none** | misconfiguration; redacted body |
| 401 / 403 | `provider_auth_error` | **terminal, none** | **redacted body only** (no key, no headers) |
| 408 / network timeout | `provider_transient` | bounded RQ retry | not a retry-storm |
| 429 | `rate_limited` | in-call bounded backoff under limiter; terminal on exhaustion | NOT RQ-retried (rule 15) |
| 5xx | `provider_transient` | bounded RQ retry | |
| malformed JSON / missing choices/content | `invalid_output` | bounded retry | |
| valid HTTP but validation fails | `invalid_output` | bounded retry | §7 |

`provider_config_error` and `provider_auth_error` are added to `IngestionJob.failure_category` and `AIRequestLog.errorClass`. (If `failure_category` is a DB enum, migration 0009 adds the values; if a validated string, update the validator.) Under the deviation, pointing at an inaccessible model yields 400/403 — these must terminate cleanly, never retry-storm.

## 9. AIRequestLog — one row per gateway attempt; transport retries in-row (blocking edit 1)

**Restores the 4.5a lock.** v1 wrongly said "each provider attempt is its own AIRequestLog row." A single `gateway.complete()` that issues a 429, backs off, retries, and succeeds is **one** gateway completion attempt with multiple transport attempts inside it.

```
AIRequestLog = ONE row per LLMGateway.complete() attempt (unchanged 4.5a semantics).
Gateway-level retries (RQ re-runs the job) open a NEW row (attemptNumber++).
Provider transport retries WITHIN one gateway attempt are recorded IN THE SAME ROW:

  providerAttemptCount       int     -- total transport POSTs in this gateway attempt
  rateLimitBackoffCount      int     -- 429 backoffs in this gateway attempt
  lastProviderStatusCode     int     -- last HTTP status seen
  retryEventsJson            jsonb   -- [{attempt, statusCode, errorClass, latencyMs, atOffsetMs}, ...]
                                        NO headers, NO key, NO prompt text, NO transcript text
```

Migration 0009 adds these columns + `backendRouteSource` (§6) + any new `failure_category`/`errorClass` values (§8). 0008 is already applied (4.5a head), so this is additive.

## 10. `rate_limited` backoff (rule 15)

In-call, under the limiter, recorded in the single gateway-attempt row (§9). Two backpressure sources:

```
Source 1 — limiter has no free slot (our Redis budget):
  blocking acquire with bounded wait budget; slot frees → proceed; budget exhausted → terminal rate_limited.
Source 2 — provider HTTP 429:
  bounded exponential backoff (jittered) under the limiter, capped attempts / max elapsed;
  succeeds → proceed; exhausted → terminal rate_limited.

Terminal rate_limited: distinct status; failure_category='rate_limited'; NOT RQ-retried (rule 15);
recovery is Stage 4.6 lecturer retry. Backoff sequence captured in retryEventsJson (§9).
```

Single-lecturer scale makes persistent rate-limiting unlikely. *Revisit trigger:* swap terminal-on-exhaustion for delayed re-enqueue if rate-limiting becomes common (post-MVP).

## 11. Provider config (no accidental real calls)

```
LLM_PROVIDER=deterministic | k2think          default: deterministic
LLM_API_KEY                                   required iff LLM_PROVIDER=k2think  (4.5a name; provider-agnostic)
LLM_PROVIDER_BASE_URL=https://api.k2think.ai  provider APPENDS /v1/chat/completions (callers never pass full endpoint)
LLM_PROVIDER_JSON_MODE=false                  flip true ONLY after the smoke confirms response_format is honored (§7)
LLM_CONTEXT_FALLBACK_ENABLED=false            single-model deviation (§12, ADR-025)
ENABLE_DETAILED_SUMMARY=false                 (§5)
LLM_INTERACTIVE_HEADROOM_PERCENT=20           (4.5a)
```

**Naming standard:** provider-agnostic `LLM_*` namespace (matches 4.5a). Earlier `IFM_*`/`K2THINK_*` names are retired — provider-specific env names leak the vendor into config and contradict the swappable provider boundary.

- CI/test always uses `DeterministicTestProvider`. The real `K2ThinkProvider` is reachable only with `LLM_PROVIDER=k2think` **and** a key present.
- No local/dev/test run calls the real provider accidentally — a worker reaching `embedded` locally must not start spending requests.

## 12. Context-window behavior under the deviation (improvement 4)

We know only that both routes *accepted* the request and echoed the same model — **not** that they share a context window. So:

```
Under the single-model deviation, context-window fallback is DISABLED unless route-specific
context limits are confirmed. If the prompt exceeds the CONFIGURED K2-Think-v2 context limit
(a config value, model-driven — not hardcoded 32k/128k) → invalid_input (no truncation, D3).
The fallback MECHANISM stays in code, dormant, and reactivates when the dual-model split returns.
```

This avoids encoding the false assumption that the two routes share a window.
