## Goal
Close the carried recovery debt — rows stuck by a worker crash or an enqueue-after-commit miss, and
storage objects orphaned by failed deletes/replacements — with two idempotent, singleton-locked,
observable units, so the pipeline self-heals and every recovery action is queryable via `MaintenanceRun`.

## Why now
4.6a/4.6b made replacement and retry safe. The roadmap has tracked stuck-row recovery + storage orphan
cleanup as deferred since Stage 4.3. 4.6c owns them. Stage 12 verifies recovery from `MaintenanceRun`
rows, not log-scraping.

## Build
- **MaintenanceRun** table + migration 0012 (run_type/mode/status/triggered_by/started/completed/summary_json/error).
- **Singleton advisory lock** (`pg_try_advisory_lock` on a dedicated connection) — a worker that can't take it skips.
- **Stuck-row reaper** (ADR-46-C): never-enqueued parse → re-enqueue; queued-not-live-in-RQ → re-enqueue;
  running-past-step-threshold-not-live → mark `failed`+`crashed` (fenced). Step-aware thresholds;
  RQ-registry + age liveness; action-capped; MaintenanceRun-logged. Runs at worker startup + admin trigger.
- **Storage reconciliation** (ADR-46-D): list objects (recursive, scoped, capped) vs DB keys; orphans
  (older than grace) reported (deleted only in cleanup mode + flag, capped); missing refs reported loudly,
  never auto-fixed; superseded files referenced. Admin trigger (+ optional startup).
- **Storage `list_objects`** on the Protocol + Supabase (recursive) + FakeStorageProvider.
- **Admin endpoints** `POST /admin/maintenance/reap-stuck-rows`, `…/reconcile-storage` (admin-only).

## Do not build
- Lecturer replace/retry UI + active-summary preview endpoint + browser gate → **4.6d**.
- Student surface → 4.7. Cron/scheduler pointing at these entrypoints → 11.1.
- Heartbeat columns (rely on RQ registry + age). MaintenanceRun browser UI (§5) — Stage 12 queries the table.

## Data model changes
- New `maintenance_runs` table (migration 0012). No IngestionJob changes (no heartbeat columns).

## API changes
- Two admin-only maintenance trigger endpoints returning `MaintenanceRunRead`.

## Worker / job changes
- Worker startup runs the reaper (`REAPER_RUN_AT_STARTUP`, default on) + optional reconciliation
  (`RECONCILE_AT_STARTUP`, default off), best-effort (never blocks boot). `scripts/reenqueue_summaries.py`
  removed (obsolete after the 4.6b DAG decouple; the reaper subsumes it).

## Authz rules
- Maintenance triggers are admin-only (`require_role("admin")` → 403).
