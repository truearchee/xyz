## 0. Why this stage exists (context, not negotiable)

Stages 5–8 are the heavy UI of the product: quiz attempt flow, glossary, assistant workspace. Every surface so far (login, admin, lecturer content, the 4.5d–4.7 summary/status/retry panels) was built thin and inline-styled by design — the correct call under rule 2 (*thin, never fake*). It stops being correct the moment we build three stages of complex, reused UI on top of nothing.

4.9 is the conversion point. We adopt a styling system, build the shared component library that Stage 5+ consumes, stand up the first unit-test harness, and clear the platform debt that will otherwise detonate later. **No new product feature ships in this stage.** Its gate proves the conversion broke nothing *and* that the result matches the design plan.

The trap this stage defuses: repainting and retro-testing the quiz/glossary/assistant UI at Stage 12, after it's been built three times over on inline styles with zero test coverage.

## 4. Design-system contract

### 4.1 Token contract

**Values** come from the design plan; **structure and naming** are locked here.

**The two-layer rule (the part that matters):**
```
Layer 1 — raw palette: the literal values from the design plan
          (e.g. --palette-indigo-600: #4f46e5). Components NEVER touch these.
Layer 2 — semantic roles: reference the raw layer and carry MEANING. Components
          consume ONLY these.
            --color-surface, --color-surface-raised, --color-border
            --color-text, --color-text-muted
            --color-primary, --color-on-primary
            --color-success / -warning / -danger / -info  (+ their -on- pair)
            --color-focus-ring
```
Why: a future theme (or dark mode) becomes a swap of the semantic→raw mapping, not a rewrite of every component. A component needing a color the semantic set lacks is a **finding** (the plan is missing a role), not a reason to reach into the raw layer.

**Token categories to define:** Color (two-layer), Type (display/body/utility families + a scale: size, line-height, weight per step), Spacing (one scale), Radius, Elevation (shadow scale), Motion (duration + easing + the reduced-motion convention), Layering (z-index tokens: shell < dropdown < modal < toast, so portal stacking is deterministic).

**Mechanics (precise — the ADR picks v4 or v3; this assumes v4, the recommendation):**
- CSS custom properties are the source of truth, surfaced into Tailwind. In **Tailwind v4**, `@theme` variables ARE the utility classes — CSS variable and token are one object.
- **Disable the default color namespace** with `--color-*: initial`, then define only the semantic color utilities product components use. This turns "components use semantic tokens" from discipline into a *compile-impossible-to-violate* constraint (same philosophy as the backend's CHECK constraints / grep gates).
- Do **not** use the global `--*: initial` (which clears every namespace) unless 4.9a also defines all required spacing, font, radius, shadow, and breakpoint tokens — otherwise you strip scales you still need.
- If automatic source detection misses component directories, use the v4 `@source` directive — do not carry v3-style `content`-glob assumptions into a v4 setup (and vice-versa). The ADR's chosen version dictates the terminology.
- **Contrast validated at the token level, here.** Every semantic text-on-surface pair meets WCAG AA (4.5:1 body, 3:1 large/UI). A failing pair is a finding against the design plan — caught now, not deferred whole to Stage 12 (Stage 12 audits the *composed* result; 4.9 guarantees the *base* pairs).

**Font loading (close the loop between design-plan typography and tokens):**
```
- use next/font in the root layout (self-hosted, zero layout shift, no external request)
- expose each font family as a CSS variable
- map those variables into the Tailwind/theme type tokens
- no external font <link> / @import CSS
```

### 4.2 Component contract

Build these public components (matches the roadmap — no additional *public* components; internal helpers as needed per §3). Contract (variants, states, a11y) locked here; values come from tokens. Build code is 4.9b.

| Component | Variants | States | Non-negotiable a11y |
|---|---|---|---|
| **Button** | primary, secondary, ghost, destructive | default, hover, active, focus-visible, disabled, loading | real `<button>`; visible focus ring; `aria-busy` when loading; disabled not the only signal |
| **Input** | text, textarea, select | default, focus, error, disabled, read-only | label association (`htmlFor`/`id`); error linked via `aria-describedby`; error not color-only |
| **Card** | default, interactive (clickable) | default, hover (interactive) | if clickable, a real control with focus — not a `div` + onClick |
| **Badge** | neutral, info, success, warning, danger | static | status by **text label**, never color alone (the quiz/risk accessibility rule) |
| **Modal** | default, confirm (destructive) | open, closing | focus trap; `Esc` closes; focus returns to trigger; `role="dialog"` + label; renders via the §2.1 portal |
| **Table** | default, with row-emphasis (later risk rows) | empty, loaded, loading | semantic `<table>`; sortable headers are buttons; row emphasis uses border+tint, not color alone |
| **Toast** | info, success, error | enter, auto-dismiss, manual-dismiss | polite live region; errors do **not** auto-dismiss; keyboard-dismissible; renders via the §2.1 provider |
| **Empty State** | default | — | a heading + one clear action (an empty screen is an invitation to act) |
| **Progress / Step** | linear, stepped (pipeline) | per-step: pending, active, completed, **failed** | the **failed** state is mandatory and visually distinct — what the 4.5d/4.6 status badge needs; a failed step is NEVER rendered as merely "not completed" |

**Contract stability (stable, not concrete — it must survive meeting Stage 5 traffic):**
```
Breaking changes are frozen after 4.9 (require an ADR):
  - prop rename, variant rename, removal
Additive changes are allowed (update the design-system.md changelog, no ADR):
  - additive optional props, additive variants, additive slots
```
Stage 5 is the first real consumer; additive growth is expected and healthy. Renames/removals are what cause churn across call sites — those are what we freeze.

**State carried by explicit props**, never inferred from display position/order (same principle as quiz option identity — locked early so the habit is uniform).

**Primitive strategy — build vs. wrap (ADR, §11).** Modal and Toast are not harmless: focus trapping, portal stacking, `Esc`, focus restoration, live regions, reduced motion. Owning that from scratch is rarely where a product team creates value.
```
Option A: build Modal/Toast ourselves, own all behavior.
Option B (recommended): use a HEADLESS primitive (behavior only — e.g. Radix /
  React Aria / Ariakit / Headless UI) and style it through OUR tokens.
Distinction that keeps this compatible with §4.4: headless = behavior, no styles
  → fine. A STYLED component kit (MUI/Chakra/AntD) brings its own styling and
  bypasses our token system → NOT allowed.
```

**App Router boundary:** interactive primitives are **Client Components** (`"use client"`): Button, Input, Modal, Toast, interactive Card, sortable Table. Presentational ones (Badge, Empty State, linear Progress, static Card) need no directive. Push `"use client"` to the leaf primitive, not whole pages, so page-level Server Components stay server-rendered.

**Tailwind purge footgun (prod-only break):** variant→class lookups use **complete literal class strings**, never constructed ones. `{ primary: "bg-primary text-on-primary" }` is correct; `` `bg-${variant}` `` is purged in the production build (works in dev, vanishes in prod). Ensure source detection (`@source` in v4 / `content` glob in v3) covers every component directory.

### 4.3 Async, loading & error conventions

Stage 5+ is full of async surfaces — quiz generation (minutes-long under the rule-15 limiter), glossary definition jobs, assistant streaming, summary processing. Set the convention **here**, promoting what 4.5d established:
```
Pending / processing  passive "generating…" / "processing…" state, NO hard timeout
                      (4.5d removed the 60s timeout — long completions + 429 backoff
                      routinely exceed it). Backoff polling, not a frozen spinner.
Loading (content)     skeletons for content regions; inline spinner for button-level
                      actions. One per context, used consistently.
Error                 transient/action → Toast; field/form → inline on the Input; failed
                      load of a region → Empty State with a retry action. Errors explain
                      what happened and how to fix it (the interface's voice, never a
                      raw stack trace to the user).
```
**These ship as concrete primitives in 4.9b** (skeleton, inline spinner, Empty-State-with-retry, the styled error boundary) — so 4.9c restyles *consume* them rather than re-inventing a loading treatment per surface. Inherited by Stage 5 quiz-gen waits, Stage 7 definition jobs, Stage 8 streaming.

### 4.4 App Router, migration & dependency constraints

The current frontend is **inline-styled**. Adopting Tailwind on top is not greenfield:
```
INLINE-STYLE SPECIFICITY: an inline style={} attribute BEATS a Tailwind utility class.
  A half-migrated element silently IGNORES its new Tailwind classes while looking "done."
  Rule: restyling a surface REMOVES its inline styles — never layers Tailwind underneath.
  Enforced by the inline-style gate (§8), blocking, not by eye.

NO HARDCODED VALUES: components/restyled surfaces reference tokens only — no raw hex, no
  magic pixel paddings. Enforced by the design-token gate (§8). Exceptions are logged findings.

THE upload.ts EXCEPTION STAYS: the one sanctioned direct-fetch path
  (frontend/src/lib/api/upload.ts, rule 3) is untouched. The client-regen alias (§7.3)
  regenerates the GENERATED client; it does not touch the upload path.
```
**Dependency policy** (this stage adds Tailwind, Vitest, RTL, maybe a headless primitive lib, maybe an icon lib):
```
- every new runtime dependency is listed in the 4.9 ADR / session report with a one-line reason
- ONE icon source; icons imported PER ICON (tree-shakeable) — no full icon-pack barrel imports
- no STYLED component library that bypasses our token system (headless behavior libs are fine)
- build output reviewed after 4.9b and 4.9e for unexpected growth (sanity check, not a hard KB budget)
```

## 5. The restyle boundary rule (prevents scope creep)

**The rule: componentize-or-leave.**
```
A surface gets restyled in 4.9 IF AND ONLY IF either:
  (a) it is the app shell / shared chrome, OR
  (b) Stage 5+ will reuse the markup/component (the summary panel, the status badge, a
      form pattern, a card layout the quiz/glossary screens inherit).

Everything else — one-off admin forms, lecturer screens Stage 5+ never touches — STAYS
as-is until the Stage 12 consistency sweep. Leaving it ugly is correct.
```

**Migration inventory (required 4.9c artifact — `knowledge/steps/stage-04/4.9-restyle-inventory.md`).** "Left for Stage 12" becomes "forgotten forever" without a tracked list:

| Surface | Route / file | Restyled in 4.9? | Reason | E2E hooks preserved? | Inline styles removed? | Stage 12 backlog? |
|---|---|---|---|---|---|---|
| Login | … | yes | Design Plan §2.0 | yes | yes | no |
| Admin module list | … | partial | reused table/card pattern | yes | yes | no |
| One-off admin detail | … | no | not reused by Stage 5+ | n/a | no | yes |
| … | … | … | … | … | … | … |

**Anti-regression sub-rule (the part that actually bites).** The restyle changes DOM; E2E selectors and the `window.__xyzE2E` bridge live in that DOM:
```
- Every data-testid / E2E hook / typed bridge selector is PRESERVED byte-for-byte, OR
  the affected spec is updated in the SAME commit.
- A restyle that silently changes a selector and "fixes" the test by loosening it is a
  rule-10 stop-and-escalate, not a pass.
- ROLLBACK STANCE: if a restyled surface cannot be made honestly green, the RESTYLE
  reverts — the test is never loosened to pass. Each sub-session is its own commit so a
  bad restyle is revertible without losing the component work.
```

**Restyle adds no behavior (keeps "no new feature" honest):** the restyle applies the new *visual* system to states that already exist. If the design plan's screen spec shows a state the thin v1 never implemented (an error/empty state it skipped), *adding* it is a **finding** — decided explicitly (small + in-scope, or deferred) — not silently bundled into "restyling."

## 6. Unit + a11y tests (Vitest)

Pure logic where a silent regression is most expensive and an E2E test is the wrong tool. No real backend, no network. The command surface and CI-reality decision are §8.

### 6.1 `wrapper.ts` auth recovery — the 401/403 mapping
```
- 401 → session cleared + redirect-to-/login signal produced.
- 403 → session KEPT + unauthorized state surfaced to the caller.
- 403 NEVER produces a login redirect (the classic, dangerous bug).
- Network/5xx → neither; retryable error, session intact.
```
The executable form of the 401/403 boundary (rule 5). If it goes red, a security-relevant behavior changed.

### 6.2 Status badge step mapping
```
- Each step state (pending / active / completed / failed) maps to its intended badge.
- "embeddings succeeded, summary failed" renders a FAILED summary step while earlier
  steps stay completed (exactly what Stage 4.5's prereq #3 demanded).
- A terminal failure surfaces failureCategory + retryable correctly (4.6b).
- No combination of step states produces a blank or contradictory badge.
```

### 6.3 SessionProvider state transitions
```
- unauthenticated → authenticating → authenticated (getSession + GET /me resolved).
- Any auth-context resolution failure lands in a clean error state, not a hang.
- signOut returns to unauthenticated and clears resolved app context.
- Active-memberships-only context is exposed — no JWT-claim role leakage (the GET /me
  authority rule, asserted at the client edge).
```

### 6.4 Component static-a11y smoke — **REQUIRED**
Without this the component contract is mostly prose. Minimum set (static assertions; jsdom):
```
- Button: renders a real <button>; loading sets aria-busy
- Input: label association present; error wired via aria-describedby
- Badge: a visible text label exists for each status (not color-only)
- Modal: role=dialog, labelled, an initial focus target present
- Table: semantic <table>; sortable headers are buttons
- Toast: live region present; error toast does not auto-dismiss
- Progress/Step: failed step renders explicit text, not color alone
```
Dynamic keyboard behavior (focus trap / `Esc` / focus-return / skip-link / tab order) stays a **manual pass**, recorded in a concrete artifact: `knowledge/steps/stage-04/4.9-keyboard-checklist.md`. Full automated keyboard + axe audit is Stage 12.

**Why these and not coverage %:** each encodes a standing invariant (auth boundary, pipeline status truth, session authority, component a11y contract). Coverage targets are deferred.
