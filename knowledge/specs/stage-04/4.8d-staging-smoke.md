## Goal

read both AI summaries **only because** they are an ACTIVE member of the module AND the section is
PUBLISHED — and cannot reach a raw transcript, an unpublished section, or another module's content.

## 1. THE LOAD-BEARING SEQUENCE — membership + publish drive REAL visibility (not weak auth)

**This is the part that, if wrong, passes the whole gate on weak auth instead of the real visibility
rules — the exact trap this entire stage exists to avoid.** A green smoke means nothing if the student
sees content for ANY reason other than *(active membership in THIS module) AND (section published)*.
So the smoke does not merely "log in as a student and see summaries"; it drives the membership/publish
**state machine through the real admin + lecturer surfaces** and asserts the student's visibility
**flips at each real gate**, with negative controls that fail the gate if visibility leaks early.

**Real surfaces (no backdoors, no privileged shortcuts):**
- admin assigns via `POST /admin/modules/{moduleId}/members` (role must equal the user's global role).
- lecturer publishes via `POST /modules/{moduleId}/sections/{sectionId}/publish`.
- student reads via the 4.7 Option-B endpoints in `student_summaries.py` — **with the student's own
  real token**, never an admin/lecturer token, never a hook-forced token.

**The state machine + negative controls (each NC must 404 before the gate it tests is opened):**
```
S0  admin creates the smoke module (runId in title) + the lecture section (draft, no transcript)
NC1 STUDENT (real login) GETs the section  ── NOT a member yet ──────────────→ 404  (row P)
S1  admin assigns the seeded LECTURER (role=lecturer) to the module
S2  admin assigns the seeded STUDENT  (role=student)  to the module
NC2 STUDENT GETs the section  ── now a member, but section is DRAFT ─────────→ 404  (row D)
        └─ ASSERT NC1 body == NC2 body  (byte-identical 404 — 4.7 D≡P; the 404 must not leak
           WHICH gate failed: "not a member" and "unpublished" are indistinguishable)
S3  lecturer uploads the FIXTURE transcript → pipeline runs on hosted workers
S4  lecturer PUBLISHES the section
NC3 STUDENT GETs the section + /summaries  ── member AND published ──────────→ 200, both summaries
NC4 raw transcript NEVER visible to the student: no transcript text/segment/chunk in any student
        response, no transcript-viewer/file affordance in the UI, no student-mintable signed URL
```
**Why these NCs are the proof, not decoration:**
- NC1 fails if a non-member can see the section → membership is not enforced (weak auth).
- NC2 fails if a draft section is visible to a member → publish is not enforced.
- NC2-body-equality fails if the 404s differ → an **existence leak** (the student can tell "exists but
  unpublished" from "not my module").
- NC3 is the demo; NC4 is the transcript-non-exposure invariant (4.7 §9 S4).
- **Role isolation:** each role runs in a SEPARATE browser context with its OWN real Supabase login
  (rule 9). The student context never holds an admin/lecturer session — so a pass cannot come from
  context bleed.

## 3. Budgets & post-smoke assertion
- **Timeout:** hard cap **15 min**; poll with backoff, interval **≤10 s**; pipeline-not-complete-in-
  budget = hard failure (no infinite "still processing").

  internals): the transcript's derived `overallState` is terminal/`summarized` with **all steps
  completed and none failed/retryable**; both summary rows present and student-visible.
- **DB cleanup — Option 2 (locked):** smoke data is runId-scoped (module title / storage key); runs
  accumulate intentionally in staging. Run-scoped DB cleanup / periodic reset deferred to Stage 12.
