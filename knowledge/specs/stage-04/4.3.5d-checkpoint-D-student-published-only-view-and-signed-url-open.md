## Purpose
Implement the student-facing module detail view for Stage 3 content visibility.

This checkpoint proves:

```text
Student opens assigned module
-> sees only published sections
-> does not see draft/unpublished sections
-> sees lecturer notes for published sections
-> sees published section assets
-> opens/downloads asset through signed URL
-> has no upload/replace/publish/unpublish controls
```

This is still not the final Stage 3 gate. The final cross-role Playwright/browser gate comes in Checkpoint E.

## Scope

### In scope
- Add student module detail route/page.
- Render only sections returned by the student read path.
- Render published section title/type/order.
- Render lecturer notes for published sections.
- Render published section assets.
- Request signed URL through wrapper/generated client.
- Open asset via signed URL.
- Prove draft/unpublished section is absent from the student UI.
- Prove draft/unpublished section is absent from the student server response, not just hidden in React.
- Ensure student has no upload/replace/publish/unpublish controls.
- Render load/error/unauthorized states.
- Preserve 401/403 behavior.

### Out of scope
- Lecturer upload/replace/publish work.
- Student upload attempt test.
- Final Playwright Stage 3 gate.
- Transcript UI.
- AI/embeddings/summaries.
- Backend endpoint changes.
- Hardcoded student section filtering.
- Any frontend-only hiding of forbidden content.

## Required behavior

### 1. Student route
Add or complete:

```text
frontend/src/app/(app)/student/modules/[moduleId]/page.tsx
```

Expected component direction:

```text
frontend/src/features/content/student/StudentModuleDetail.tsx
frontend/src/features/content/student/StudentSectionView.tsx
frontend/src/features/content/student/StudentAssetRow.tsx
```

Use current repo conventions if names differ.

### 2. Published-only content
The student page must use the real backend student read path.

Expected behavior:

```text
student requests module sections
backend returns only published sections
UI renders only returned sections
```

Do not fetch all sections and filter on the frontend. Do not hide draft sections with CSS or conditional UI if the backend sent them. The server must be the authority.

### 3. Required visibility scenario
Use the module from Checkpoint C or a fresh module with at least two sections:

```text
Section 1: published, has uploaded PDF asset, has lecturer notes
Section 2: draft or unpublished
```

Student result:

```text
sees Section 1
does not see Section 2
sees notes for Section 1
sees asset for Section 1
```

### 4. Server-response proof
Checkpoint D must prove that draft/unpublished sections are absent from the server response.

Recommended method:

```text
Use Playwright page.waitForResponse() for the student section-list request.
Parse JSON.
Assert the unpublished section id/title is absent from the response payload.
```

If not using Playwright yet in D, use an authenticated student API probe and record it in the report. The proof must hit the real backend with the student token.

### 5. Signed URL open
For a visible asset, student can click `Open file`.

Expected flow:
- student clicks open
- frontend requests signed URL through wrapper/generated client
- backend authorizes published-section access
- frontend opens signed URL in new tab/window or same tab according to current UI convention

Do not expose raw storage keys. Do not construct storage URLs manually. Do not call object storage directly from the UI without the backend signed URL endpoint.

### 6. Student controls forbidden
The student page must not render:
- upload PDF
- replace asset
- publish section
- unpublish section
- edit lecturer notes
- create section
- delete section
- reorder section

Students can only view published content.

## API / wrapper requirements
Use wrapper/generated client methods only.

Expected wrapper methods may be:

```ts
api.modules.get(...)
api.content.listSections(...)
api.content.getSection(...)
api.content.getAssetDownloadUrl(...)
```

Use actual project names. If a generated client method exists but wrapper does not expose it, expose it through the wrapper. Do not call `fetch()` directly from app/components/features.

## Error behavior
Render visible errors with `role="alert"`.

Minimum cases:
- module not found
- student not assigned to module
- signed URL request fails
- 403 forbidden
- 401 unauthenticated
- network error

For `403`, preserve the session and show unauthorized/error state. Do not redirect to login unless the backend returns `401`.
