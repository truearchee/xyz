## Goal
Build real, thin, keepable Stage 2 admin UI so an admin can create lecturer/student users, create modules, assign/remove module memberships, and prove lecturer/student users see only assigned modules entirely through browser UI.

## Why now
4.3.5a and 4.3.5b proved the auth spine, routing shell, wrapper behavior, and E2E hooks. Stage 2 remains UI PENDING until the admin product flows are browser-proven against real backend data.

This session is approved with Option A selected: add a narrow read-only admin module-membership projection so the assign/remove UI can be real and Stage 2 can reach FULLY VERIFIED.

## Build
- Add `GET /admin/modules/{module_id}/members`, admin-only and read-only.
- Return active module memberships only, joined with user display fields.
- Do not return admin users from the membership projection.
- Include inactive users if they still have an active membership, with `userIsActive=false`, so admins can see and remove stale memberships.
- Sort module members deterministically by role ascending, then email ascending.
- Build `/admin` user management: user list, create lecturer, create student, deactivate user, and reset password.
- Build `/admin` module management: module list, create module with owner lecturer, assign lecturer, assign student, real member list, and remove/archive membership from the member list.
- Build `/lecturer` and `/student` assigned-module lists using `GET /modules` through the wrapper.
- Add Playwright browser gate proving all Stage 2 flows using real browser, real backend, real DB, and real Supabase auth.
- Use subagents only as quality gates after primary implementation checkpoints.

## Do not build
- No migration.
- No assignment/removal write-path changes.
- No auth redesign.
- No backend login endpoint.
- No broad admin refactor.
- No content, PDF, section, upload, module detail, or transcript UI.
- No self-registration, SSO, bulk import, audit log, forced password change, or email invitations.
- Do not delete `/tracer`; that remains for 4.3.5e.
- Do not use direct product-flow API calls or `window.__xyzE2E` except for the non-admin 403 probe.

## Fixed contract assumptions
- `create-user` accepts `password`.
- `reset-password` accepts `newPassword`.
- `create-module` requires `ownerId`.
- `create-module` auto-creates an active lecturer membership for the owner.
- No module-member listing endpoint exists before this session.
- Human selected Option A, so the read-only membership projection is approved.

## API changes
- Add `GET /admin/modules/{module_id}/members`.

Response shape:

```ts
ModuleMemberResponse {
  membershipId: string
  userId: string
  moduleId: string
  email: string
  fullName: string
  role: "lecturer" | "student"
  membershipStatus: "active"
  userIsActive: boolean
  createdAt: string
}
```
