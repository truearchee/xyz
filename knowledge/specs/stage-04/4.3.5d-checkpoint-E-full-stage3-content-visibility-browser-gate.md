## Purpose
Run the full Stage 3 browser-verifiable gate.

This checkpoint proves the complete lecturer-to-student content path:

```text
Lecturer logs in
-> opens assigned module
-> sees generated sections
-> uploads real PDF
-> replaces PDF asset
-> adds lecturer notes
-> publishes one section
-> leaves another section draft/unpublished

Student logs in separately
-> sees only published section
-> does not see draft/unpublished section
-> sees lecturer notes
-> opens PDF through backend signed URL
-> cannot upload content
```

If this passes, Stage 3 may move from `UI PENDING` to `FULLY VERIFIED`.

The content slice requires lecturers to fill predefined sections and students to see only published content in assigned modules. It also requires backend authorization, PDF upload, replace, publish/unpublish, and separate `processingStatus` / `publishStatus` handling.

## Full browser gate steps

### Part 1 - Lecturer flow
1. Lecturer logs in through `/login`, redirects to `/lecturer`, and `/me` returns lecturer role.
2. Lecturer opens assigned module at `/lecturer/modules/[moduleId]`.
3. Generated sections render: `Lecture 1`, `Lecture 2`, `Lab 1`, and `Assignment 1`.
4. Each section shows order/type/title/publishStatus and no create/delete/reorder controls.
5. Lecturer adds notes to `Lecture 1`: `Stage 3 final gate notes 2026-06-08`.
6. Lecturer uploads a real PDF to `Lecture 1`.
7. Asset row appears from backend re-fetch; filename and asset `processingStatus` render separately from section `publishStatus`.
8. Lecturer replaces the uploaded asset at the specific asset row / asset id.
9. Invalid non-PDF upload is rejected by the backend and renders `role="alert"` without ending the session.
10. Lecturer publishes `Lecture 1`; backend re-fetch renders `publishStatus = Published` and asset `processingStatus` remains completed.
11. `Lecture 2` remains draft or unpublished.

### Part 2 - Student flow
Use a separate browser context.

1. Student logs in through `/login`, redirects to `/student`, and `/me` returns student role.
2. Student opens the same assigned module.
3. Student sees `Lecture 1`, lecturer notes, asset filename, and `Open file`.
4. Student does not see `Lecture 2`, `Lab 1`, or `Assignment 1` unless published.
5. Student does not see upload, replace, publish, unpublish, edit-notes, create, delete, or reorder controls.
6. Server-response proof captures the student section-list request and confirms unpublished section title/id is absent.
7. Student clicks `Open file`; frontend requests the signed URL from the backend and the returned URL opens/downloads with HTTP 200.
8. Authenticated student upload attempt against the upload endpoint returns 403, not 401, and the student session remains active.
