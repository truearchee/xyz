## 0. One line

Expose the two AI-generated summaries to students inside a published lecture/lab section — and **prove**, in a real browser against the real backend, that a student can read them while being unable to reach a raw transcript, an unpublished section, or content from a module they are not assigned to.

---

## 1. Why this is not a thin stage

4.7 adds **no new generation, no new AI calls, no new pipeline writes**. It is a read surface plus an authorization boundary. Its entire difficulty is that it is the **first point where three prior guarantees are enforced simultaneously on the student read path**:

```
Stage 3   — students never see unpublished content
4.5 split — brief lands before detailed; "being generated" is a real, common state
4.6       — student-visible state never mixes old and new transcript content
```

Get the boundary wrong and you ship an existence leak, a stale-content bug, or an XSS vector — each more expensive after 4.8 puts this on a hosted URL. The rendering is trivial; the contract is not.

## 4. Domain resolution chain & the active-transcript identity guard

Resolution path:

```
student (GET /me → role + identity)
  → role == student?                     ── else 403 (§5 row R, short-circuit)
  → SCOPED QUERY (membership + publish)  ── zero rows ⇒ 404 (§5 rows D/P/I, identical body)
    → section.type ∈ {lecture, lab}?     ── else both slots not_applicable (§5 row T)
      → ACTIVE Transcript for section     ── §4 resolution below
        → GeneratedLectureSummary rows    ── §4 identity guard + §6 precedence
          → per-slot student state
```

**Active-transcript identity guard (REVISED from v0.2 — primary change of v1.0).**
Transcripts are immutable, so a given `transcriptId` has a fixed checksum. Therefore:

```
A summary's CONTENT is eligible only if:
    summary.transcriptId == activeTranscript.id            ← PRIMARY (record identity)
    AND summary.sourceTranscriptChecksum == activeTranscript.checksum   ← TRIPWIRE
    AND summary.status == generated
```

`transcriptId` is the selector. The checksum is a **corruption tripwire**: because immutability means a matching ID *already implies* a matching checksum, an ID-match-with-checksum-mismatch can only mean corruption (mis-recorded checksum, or an immutability violation). In that case the projection **fails safe** to `unavailable` and logs — it never renders content of uncertain provenance. This closes the "but it passed the checksum check" class of bug and the identical-file-re-upload collision (same checksum, different record) that checksum-alone would mishandle.

**Active-transcript resolution determinism.** Resolve the single active transcript per P2c. If resolution ever yields **more than one active transcript** for a section (data-integrity bug), do NOT pick one — return the `being_generated`/`unavailable` fallback and log the violation. A read surface fails safe; it does not guess what content a student is entitled to.

**Multiple generated rows for the same active transcript + slot.** The 4.5 one-active partial-unique index should make this at-most-one, but defense-in-depth: select the **latest by `generatedAt`**. A `failed` older row never overrides a `generated` newer row; a `generated` older row never overrides a `generated` newer row. Deterministic, never order-of-insertion dependent.

## 5. Access × availability decision table — **RATIFIED (§19 confirms D3/D4)**

| # | Role / publish | Membership reaches section? | Section type | Summary state | Response |
|---|----|----|----|----|----|
| 1 | student / published | yes (active) | lecture/lab | both eligible (§4) | **200** — brief + detailed |
| 2 | student / published | yes | lecture/lab | brief eligible, detailed in-flight | **200** — brief shown, detailed `generating` |
| 3 | student / published | yes | lecture/lab | both in-flight, pipeline progressing | **200** — both `generating` |
| 4 | student / published | yes | lecture/lab | summary failed OR upstream terminally failed OR completed-but-missing | **200** — `unavailable` |
| 5 | student / published | yes | lecture/lab | no transcript at all | **200** — `unavailable` |
| 6 | student / published | yes | lecture/lab | active-transcript mismatch (mid-regeneration, §7) | **200** — `generating` |
| T | student / published | yes | assignment/supplementary | n/a | **200** — both slots `not_applicable` |
| D | student / draft·unpublished | yes | any | any | **404** — pinned body |
| P | student / any | **no** (other module) | any | any | **404** — body IDENTICAL to D |
| I | student / any | membership inactive | any | any | **404** — body IDENTICAL to D |
| R | **lecturer/admin** hitting student endpoint | — | — | — | **403** (wrong surface; short-circuits before resource lookup → no existence info) |
| A | unauthenticated | — | — | — | **401** (rule 5 → signOut + redirect) |

**Load-bearing distinctions:**
- **Rows 4/5 → 200, not 404.** "Visible to you but not ready/failed" is an availability state, not a denial.
- **Rows D/P/I → byte-identical 404.** Unpublished, other-module, and inactive-membership are **indistinguishable**. Asserted by body comparison, not just status (§9 S2).
- **Row R → 403, not 404.** A lecturer/admin on the student endpoint is using the wrong surface; the role gate fires **before** any section lookup, so 403 leaks nothing about specific resources. (This supersedes v0.2's uniform-404-for-everyone.) Lecturers view summaries via the 4.5d lecturer surface; no "preview as student" here.
- **Row T → `not_applicable`, not `unavailable`.** "Unavailable" implies a summary that could exist; an assignment never has one.

Student failure vocabulary is binary on the failure axis: `generating` vs `unavailable`. Never `failed`/`rate_limited`/`invalid_output` (lecturer-only, 4.6).

## 6. Per-slot state precedence — **RATIFIED** (first match wins)

Request-level gates already applied (401 unauth; 403 non-student; 404 if the scoped query returns no visible section). Then, for EACH slot (`brief`, `detailed`), evaluate top-down — first match wins:

```
0. section.type ∉ {lecture, lab}                                      → NOT_APPLICABLE
1. no active transcript for the section                               → UNAVAILABLE
2. a row exists with transcriptId == active.id AND status == generated:
     2a. checksum != active.checksum  (corruption tripwire, §4)       → UNAVAILABLE + log
     2b. content empty/blank after trim (anomaly)                     → UNAVAILABLE + log
     2c. otherwise (latest by generatedAt if several)                 → READY
3. a row exists with transcriptId == active.id AND status == failed   → UNAVAILABLE
4. no generated/failed row for the ACTIVE transcript yet:
     4a. this summary job queued/running, OR upstream steps still
         progressing toward it (NOT terminally failed)                → GENERATING
     4b. upstream pipeline terminally failed (job will never run)     → UNAVAILABLE
     4c. pipeline reports complete/terminal but no summary row exists
         (completed-but-missing-artifact)                            → UNAVAILABLE
         + raise lecturer-side inconsistency (retry/debug state, 4.6)
```

**Notes.**
- Superseded summaries (transcriptId != active.id) are never matched by steps 2/3 → a section mid-replacement falls to step 4a → `GENERATING`. That IS the §7 no-mixing behavior.
- Step 4c is the classic "row stuck in generating forever" trap. For students it resolves to `UNAVAILABLE` (refreshing won't help); the inconsistency surfaces on the lecturer side, not the student's.
- `GENERATING` is shown **only when generation is still actually possible** (step 4a). A terminally-failed-upstream pipeline → `UNAVAILABLE`, never a forever-spinner. This is why the derivation consults `overallState`, not just the summary row.

**Pinned student copy (Slice 2):** `generating` → "Summary is being generated." · `unavailable` → "Summary is currently unavailable."

**Mechanism reconciliation (code-wins, recorded at 4.7a start):** `GeneratedLectureSummary` is a
success-only table with **no `status` column** — a row's existence *is* "generated". So §4/§6
"status == generated" maps to "an eligible row exists"; "status == failed" and the GENERATING vs
UNAVAILABLE split are derived from the `overallState`/`steps` projection (`transcript_status.py`), never
from a summary-row status. Intent preserved; mechanism corrected. **Corruption (transcriptId match +
checksum mismatch → UNAVAILABLE+log) MUST stay distinct from supersession (no row for the active
transcript → GENERATING)** — `is_summary_eligible` returns False for both, so precedence must inspect
identity and checksum separately and never collapse onto `is_summary_eligible == False`.

## 7. Supersession / regeneration read contract — **LOCKED**

4.7 is the read side where 4.6's *"student-visible state never mixes old and new"* is enforced, via the §4 identity guard:

```
A summary's content reaches a student only if it belongs to the section's CURRENTLY
ACTIVE transcript (transcriptId match) AND is content-consistent (checksum tripwire)
AND status == generated.

Replacement (4.6): new active transcript has no matching summaries yet → students see
GENERATING (row 6) until regeneration completes. Never the prior transcript's summary
attributed to the new one; never a half-updated pair (old brief + new detailed).
Retry (same transcript): transcriptId unchanged → GENERATING while re-running, then READY.
Republish (unchanged section): no regeneration → existing summaries reappear as-is.
```

A filter in the read model, not new write logic (rule 8). 4.6 decides what is active; 4.7 reads the marker. Superseded-row unreachability is asserted via seeded rows if the in-browser window is too narrow to force (§14; §17 `accepted with rationale`).

## 8. API contract

### 8.1 List vs detail — do not fan out
- **Section LIST** (module view): publish-filtered sections + a **coarse per-section flag** (`summaries_state: ready | partial | generating | none | not_applicable`), computed in **one batched query**. No content.
- **Section DETAIL**: full brief + detailed content + per-slot state (§6).

### 8.2 Endpoint shape — **LOCKED: Option B**
```
GET …/student/sections/{sectionId}
   → title, lecturerNotes, materials, type, per-slot summary STATE (no content)
GET …/student/sections/{sectionId}/summaries
   → per-slot content + state ONLY
```
Rationale: brief-first is meant to be observable; the sub-resource is a clean poll target (§11); and the security-sensitive blob sits behind its own hardened handler that 4.8's smoke reuses. For `assignment`/`supplementary` the sub-resource returns **200 with both slots `not_applicable`** — NOT 404 (404 is reserved for access/existence; overloading it for "type N/A" muddies the leak semantics).

### 8.3 Response schema (codegen-friendly; flat, not a union)
```jsonc
// GET …/sections/{id}/summaries
{
  "sectionId": "uuid",
  "summaries": {
    "brief":    { "state": "ready|generating|unavailable|not_applicable", "content": "markdown|null" },
    "detailed": { "state": "ready|generating|unavailable|not_applicable", "content": "markdown|null" }
  }
}
```
`content` is non-null only when `state == ready`. Flat `{state, content:nullable}` is deliberate over a discriminated union: rule 3's generated TS client handles it cleanly, whereas OpenAPI `oneOf` produces awkward client types. The client derives "should I poll?" from the states (poll iff any rendered slot is `generating`); the server does NOT return polling hints — backoff cadence is the reused 4.5d mechanism, not a response field.

**MUST NOT serialize (any student-reachable response):**
```
transcriptId · ingestionJobId/jobId · errorMessage · modelName · promptVersion
tokenUsage · sourceTranscriptChecksum · segment/chunk IDs or text · storageKey
raw transcript filename · overallState/steps internals · stack traces
```
Provenance exists for audit/debug, not for the student page.

### 8.4 Status codes, caching, error body
```
200 — rows 1–6, T
401 — unauthenticated (rule 5)
403 — non-student caller (row R)
404 — rows D/P/I → ONE pinned, identical body (no "not found" vs "forbidden" variance)
Cache-Control: private, no-store   on BOTH student section and summary responses.
  User-specific + access-sensitive; once 4.8 puts a proxy in front, no shared/browser
  cache may preserve a stale 200 after unpublish or membership removal. ETag/304 polling
  optimization DEFERRED (§17).
Pre-Stage-12: no global exception handlers (raw 500 bodies). The frontend treats ANY
  non-200/401/403/404 as a generic error and NEVER surfaces the raw body.
```

### 8.5 Contract rules
```
OpenAPI updated → regenerate TS client → commit → consume via generated client (rule 3).
Section-scoped entry ONLY. No by-summary-id and no by-transcript-id student route
  (IDOR closure, §9 S3).
```

### 8.6 The scoped query (illustrative)
```sql
SELECT s.*, <active transcript join>, <summary rows>
FROM module_sections s
JOIN course_modules m       ON m.id = s.course_module_id
JOIN <membership> cm        ON cm.<module-or-course fk> = <m or course>.id   -- per P2d
WHERE s.id = :section_id
  AND s.publish_status = 'published'
  AND cm.user_id   = :student_id
  AND cm.role      = 'student'
  AND cm.status    = 'active';
-- Zero rows ⇒ 404 (rows D/P/I). Never "fetch section, then branch to 403/404."
-- D4 CONFIRMED: membership is MODULE-LEVEL → cm.module_id = s.course_module_id.
```

## 9. Security (consolidated)
```
S1  AI-CONTENT SANITIZATION (XSS) — non-negotiable. Treat summary markdown as
    UNTRUSTED even though it is ours (AI output can emit raw HTML; prompts change;
    content becomes editable post-MVP). Render via react-markdown with raw-HTML
    DISABLED (or sanitize through a known sanitizer if HTML is ever enabled). Block:
    script tags, inline event handlers, iframes, remote image URLs, unsafe link
    protocols (javascript:, data:). This is the one place generated text reaches the
    browser as markup.
S2  EXISTENCE-LEAK / IDENTICAL 404. Rows D/P/I return BYTE-IDENTICAL 404 bodies.
    Asserted by comparing the two responses, not just status codes.
S3  IDOR. Section-scoped only; membership re-checked on every read; no direct
    summary/transcript-id student route (§8.5).
S4  TRANSCRIPT NON-EXPOSURE — three surfaces:
      (a) schema cannot serialize transcript/segment/chunk text or provenance (§8.3);
      (b) every text-bearing endpoint from P3 rejects the student token;
      (c) NO student path can mint a signed URL for a raw transcript object, and
          transcript objects are not public. (Storage leak ≠ DB-API leak.)
S5  NO ROLE BYPASS (row R). Membership — not role — grants the student surface;
    lecturer/admin without student membership → 403. "Preview as student" is a
    deliberate non-feature (§19 D3).
S6  SIGNED-URL-AFTER-UNPUBLISH. Known carried debt OWNED BY STAGE 12 — 4.7 must not
    REGRESS it: mint material signed URLs only for PUBLISHED sections; never pre-sign
    unpublished content. The TTL window itself stays a Stage 12 decision.
S7  READ THROTTLE. Polling is throttled only by 4.5d backoff (the AI Redis limiter
    governs AI calls, not reads). Acceptable for MVP — recorded, not overlooked.
```

## 10. Frontend states & polling lifecycle

**Four distinct states per slot — never collapsed:**
```
LOADING          HTTP in flight                       → spinner
GENERATING       request OK, transient                → "Summary is being generated." + poll
UNAVAILABLE      request OK, terminal no-summary       → "Summary is currently unavailable." (no poll)
NOT_APPLICABLE   non-lecture/lab section               → block absent
READ_ERROR       non-200/401/403/404                   → generic "couldn't load, retry" (never raw body)
```
The page is **never in one global state**: brief and detailed render independently (row 2). Title/notes/materials render immediately regardless of summary state.

**Polling (Option B sub-resource):**
```
start  on mount IFF any rendered slot == generating
poll   reuse 4.5d backoff (no 60s hard timeout)
stop   when all rendered slots are terminal (ready | unavailable | not_applicable)
stop   on unmount — clear timers (no leaked intervals)
cap    a generous wall-clock ceiling (detailed via K2-Think under the limiter is
       legitimately slow). On cap: stop and show "still being generated — refresh to check."
```
