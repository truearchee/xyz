# Session 4.5.1b — Brief-from-detailed DAG + gating + backfill + fencing (BACKEND VERIFIED target)

> Slice 2 of [[specs/stage-04/4.5.1-map-reduce-summarization]]. Builds on the 4.5.1a engine: rewire the
> brief to derive from the completed detailed (not the raw transcript), establish the full-coverage gating
> predicate downstream consumers (Stage 5) read, build the backfill command that regenerates stale
> truncated summaries, complete the stale-write fencing audit, and confirm the AIRequestLog chain. No
> real-provider run or browser gate (that is 4.5.1c). **No source edits until this spec + plan are approved.**

## Goal
After 4.5.1a, the DETAILED summary is full-coverage (map_reduce). 4.5.1b makes the BRIEF derive from that
completed detailed in one small call, exposes the STATE that gates downstream generation (§0.1), and provides
the backfill that turns the existing truncated 4.7 summaries into full-coverage ones (run in 4.5.1c).

## Build
**1 — Brief-from-detailed DAG (ADR-052).** Today `insert_summary_jobs` ([parse_service.py:254](backend/app/domains/transcripts/parse_service.py#L254))
creates brief + detailed at embed-complete and both enqueue independently; the brief reads the (truncated)
transcript. Rewire so:
- the brief is derived from the COMPLETED detailed summary in ONE call on the BRIEF route (V2/Cerebras,
  unchanged) — a small writing task ("compress this detailed summary to a short paragraph"), NOT a
  re-summarization of the transcript. New prompt `brief_from_detailed/v1` (input = the detailed
  `content_json`); feature `brief_from_detailed`; `SummaryFeature` + the AIRequestLog feature check extended.
- DAG: brief forks from DETAILED, not from the embed leaf. The detailed handler, on success, enqueues the
  brief job; the brief handler reads the latest completed detailed (via `get_latest_transcript_summaries`)
  rather than the transcript. Lost brief-first ordering is accepted (ADR-052).
- the brief's persisted provenance: `generation_strategy='derived_from_detailed'` (a new strategy value);
  `input_hash` derived from the source detailed row (so a regenerated detailed ⇒ a new brief). It records the
  source detailed summary id in `generation_metadata`.

**2 — Full-coverage gating predicate (§0.1).** A pure predicate `is_full_coverage_detailed(summary) ->
bool = (generation_strategy == 'map_reduce' AND truncated is False)` in the summary domain, unit-tested,
that Stage 5 (quiz), 7 (glossary), 8 (assistant) consume to decide whether a detailed summary may seed
generation. 4.5.1b builds + tests the predicate and surfaces the state on the read projection where a
consumer/admin can query it; it does NOT build a Stage-5 consumer (none exists yet) — it provides the gate
the §0.1 rule names. `truncated_fallback` and `single_call` are NOT full-coverage.

**3 — Backfill re-enqueue command.** A guarded dev/admin command (function + a worker/CLI entrypoint) that
finds every ACTIVE-transcript detailed summary that is stale (`generation_strategy != 'map_reduce' OR
truncated = true`) and re-enqueues its detailed IngestionJob (which now runs map-reduce), then the brief
follows via the new DAG. Idempotent, bounded (a per-run cap), dry-run mode (report what WOULD be
re-enqueued). 4.5.1b builds + unit-tests it on the deterministic adapter; 4.5.1c RUNS it against the real
Checkpoint A + 4.7 data (the §0.1 backfill).

**4 — Stale-write fencing audit (§6.1).** 4.5.1a reused the existing superseded fence at claim + final
persist. 4.5.1b completes the audit: before EACH map-unit write, the reduce write, and the brief write,
verify the transcript is lifecycle-eligible (not superseded) AND the source checksum / partition hash still
match — abort clean (write no stale artifact) on mismatch. Reuse the 4.6 fencing helpers. A test forces a
mid-flight supersede and asserts no partial/summary is written for the superseded transcript.

**5 — AIRequestLog chain by job id.** The map/reduce/brief calls already share the orchestrating
`ingestion_job_id` (via `ContextRefs`). Add a read helper that returns the correlated chain (map units +
reduce + brief, ordered) for one detailed job, and a test asserting the chain is complete + correlated — the
queryable provenance the 4.5.1c real-provider proof (count successful rows, per-phase model echo) relies on.

## Do not build (4.5.1b)
Real-provider smoke / browser gate / running the backfill / the 4.7 re-gate (all 4.5.1c). No Stage-5 quiz
consumer. No new student-facing UI (the truncated label already ships; once backfilled, `truncated=false`
hides it). No tool-calling.

## Open questions (resolve in the plan / with the developer)
- **OB1 — brief when detailed is DISABLED.** Brief-from-detailed requires a detailed summary. With
  `ENABLE_DETAILED_SUMMARY=false` (the 4.5b cost-control mode) there is no detailed source. Proposed: when
  detailed is disabled, the brief FALLS BACK to the existing transcript-based single-call (Option-A
  truncated) path — preserving the cost-control mode — and is labeled `single_call`+`truncated`. Confirm, vs
  "detailed off ⇒ no AI summaries at all."
- **OB2 — new strategy/constraint values.** `derived_from_detailed` (brief) — add to the
  `ck_gen_summaries_generation_strategy` check (migration 0016) or treat brief strategy as informational
  only. Proposed: extend the check (one migration), so the column is always a known value.
- **OB3 — retry/reaper paths.** `retry.py` + `reaper.py` re-enqueue summary jobs by job_type. Under the new
  DAG a brief re-enqueue must still find a completed detailed (or trigger detailed first). Confirm the retry
  of a brief-only failure reads the existing completed detailed (no transcript re-read).

## Amendments — approved 2026-06-13 (the OB resolutions are LAW for the build)
- **OB1 RESOLVED + TIGHTENED.** The `ENABLE_DETAILED_SUMMARY=false` fallback brief (transcript-based,
  Option-A) MUST inherit the FULL §0.1 treatment, not just the generation path: it is written `truncated=
  true`, `generation_strategy='single_call'`, it does NOT satisfy `is_full_coverage_detailed`, and it carries
  the "based on the first portion" UI label. The detailed-off mode is a DEGRADED escape hatch that produces
  HONESTLY-degraded (truncated, labeled, non-quiz-eligible) summaries — it must NOT be a back door that puts
  truncated content into a state Stage 5 would read as full-coverage. ADR-052 names whether the mode is
  intended-for-prod or a 4.5b leftover (it is a cost-control escape hatch, default OFF→ actually default
  `ENABLE_DETAILED_SUMMARY=true`; flipping it false in prod silently makes those lectures non-quiz-eligible —
  a named product consequence, not a silent code path).
- **OB2 RESOLVED.** `derived_from_detailed` is a distinct `generation_strategy` value (migration 0016,
  additive). The gating predicate `is_full_coverage_detailed` keys off the **DETAILED** summary's strategy —
  quiz-eligibility is a property of the detailed the brief derived from; the brief's `derived_from_detailed`
  label is INFORMATIONAL provenance only. Stage 5 reads the DETAILED state, never the brief's label.
- **OB3 RESOLVED.** A brief retry reads the EXISTING completed detailed, never re-summarizes the transcript
  (the brief is a pure function of the detailed). The brief-claim's §6.1 stale-write fence MUST verify the
  detailed it reads still matches the ACTIVE transcript checksum — a brief retry firing after a transcript
  replacement reads the NEW detailed or aborts clean, never staples an old detailed's brief onto a replaced
  transcript. A stuck-detailed reaper re-fork re-triggers the brief on detailed completion.
