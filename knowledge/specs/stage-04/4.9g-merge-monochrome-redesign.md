## Goal
Merge `stage/4.9f` into `main` so the Stage 4.9f monochrome Apple x Linear frontend design system lands without regressing Stage 5-9 backend, schema, generated-client, or frontend functionality.

## Why now
Stage 4.9f was branched before Stages 5, 5.5, 6, 7, 8.1, 8.2, and 9 landed. It contains the intended UI restyle, but it is stale for backend domains and newer frontend routes. This session reconciles the stale design branch with current `main`.

## Developer-approved resolution principles
- For design/styling files such as globals, Tailwind/PostCSS setup, design tokens, component styles, and layout, take Stage 4.9f's design system.
- For backend files such as routers, models, migrations, services, prompts, gateway, and validation, take `main`; 4.9f backend is stale and must not revert newer domains.
- For frontend components that exist in both branches, keep `main` functionality and apply 4.9f design tokens/classes.
- For `frontend/src/features/content/student/StudentSectionDetail.tsx`, keep `main` despite deletion in 4.9f, then apply 4.9f design tokens locally.
- For knowledge docs, union both histories while letting code/current `main` win when docs conflict.
- For `.conductor/**`, take `main`.
- For migrations, keep `main` chain exactly as-is through `0039`; 4.9f adds no accepted migrations.
