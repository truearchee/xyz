## Purpose
Add or restore the approved frontend multipart helper:

```text
frontend/src/lib/api/upload.ts
```

This helper is required before Checkpoint B because PDF upload/replace uses browser multipart form data, and the generated JSON client/wrapper may not handle file uploads cleanly.

This session creates the upload helper only. It does not build the upload UI yet.

## Authoritative rules
The roadmap says pages and components must not call `fetch()` directly. The controlled exception for browser uploads is `frontend/src/lib/api/upload.ts`.

Everything else must go through the generated client or centralized wrapper.

The content slice says lecturers can upload and replace PDF files attached to predefined module sections, file metadata is stored in the DB, actual files are stored in object storage, and file `processingStatus` remains separate from section `publishStatus`.

## Goal
Provide a centralized upload helper for Stage 3 lecturer PDF upload and replace flows.

Required helper capabilities:
- upload PDF to a section
- replace an existing section asset
- attach Supabase access token
- send `multipart/form-data` correctly
- return typed backend response
- surface backend errors consistently
- preserve 401/403 behavior

## Non-goals
Do not implement upload UI, replace UI, publish/unpublish UI, student view, signed URL open, Playwright Stage 3 gate, backend endpoint changes, backend upload behavior changes, storage cleanup changes, transcript upload, AI, or embeddings.

This is a prerequisite helper only.

## Required public helper API
Prefer this shape unless current project conventions strongly suggest different names:

```ts
export async function uploadSectionAsset(input: {
  moduleId: string;
  sectionId: string;
  file: File;
  signal?: AbortSignal;
}): Promise<SectionAssetResponse>;

export async function replaceSectionAsset(input: {
  moduleId: string;
  sectionId: string;
  assetId: string;
  file: File;
  signal?: AbortSignal;
}): Promise<SectionAssetResponse>;
```

Use the actual generated response DTO name if different. Do not invent a parallel frontend-only asset type if a generated type already exists.

## Backend contract discovery
Before writing `upload.ts`, inspect the generated client and backend router. Confirm upload endpoint path, replace endpoint path, methods, multipart field name, response DTO shape, error response shape, and auth requirements.

Do not assume the form field is named `file`; confirm it from backend source.

If upload or replace backend endpoints are missing, stop and write a finding. Do not add backend write paths in this session.

## Implementation requirements
- Create or restore `frontend/src/lib/api/upload.ts`.
- Attach current Supabase access token as `Authorization: Bearer <access_token>`.
- Do not decode JWT role claims, read Supabase metadata for role, use service-role keys, or hardcode tokens.
- Use `FormData`.
- Do not manually set `Content-Type`; the browser must set the multipart boundary.
- Use the same backend base URL convention as the existing wrapper/generated client. Do not hardcode `http://localhost:8000`.
- Align error behavior with the existing wrapper: 401 signs out/redirects to login, 403 preserves session and throws a forbidden error, validation/network errors are renderable failures.
- Return parsed JSON as the generated response type.
