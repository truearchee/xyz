## Why this session exists
4.3.5d Checkpoint 0 found:

```text
Admin creates module
-> CourseModule is created
-> 0 ModuleSection records are created
```

That blocks the lecturer content UI because the lecturer is supposed to open a module and see predefined sections such as Lecture 1, Lecture 2, Lab 1, and Assignment 1. The lecturer fills those sections with PDFs and notes and must not create the structure manually.

This session repairs the backend creation path:

```text
Admin creates module
-> CourseModule is created
-> predefined ModuleSection records are created in the same transaction
```

After this session passes, 4.3.5d may resume at Checkpoint A. This session does not mark Stage 3 FULLY VERIFIED.

## Goal
Add a real backend/product path that generates predefined `ModuleSection` records when an admin creates a module.

Minimum successful flow:

```text
POST /admin/modules
-> creates CourseModule
-> creates at least two ModuleSection records
-> sections default to draft
-> lecturer can later fill those sections
```

Recommended MVP default structure:

| order | title | type |
|---:|---|---|
| 1 | Lecture 1 | lecture |
| 2 | Lecture 2 | lecture |
| 3 | Lab 1 | lab |
| 4 | Assignment 1 | assignment |

At least two sections are required so the later Stage 3 browser gate can publish one section and leave another draft.

## Non-goals
Do not implement lecturer create-section UI, lecturer delete-section UI, lecturer reorder-section UI, section template tables, full course schedule builder, lecturer/student content UI, upload UI, publish/unpublish UI, Playwright Stage 3 browser gate, transcript UI, embeddings, summaries, AI features, or any 4.3.5e work.

This is a blocker repair. Keep it narrow.

## Design decision
Use a temporary MVP default section-generation policy. Do not build schedule-driven or template-driven generation yet.

Shape the code so future policies can replace the default:

```text
mvp_default now
schedule_based later
template_based later
```

Recommended service shape:

```text
AdminModuleService.create_module(...)
  -> create CourseModule
  -> SectionGenerationService.generate_initial_sections(module, policy="mvp_default")
  -> commit module + sections together
```

Do not scatter default section creation inside a route handler.

## Functional requirements
1. Admin module creation through `POST /admin/modules` creates `CourseModule` plus `ModuleSection[]` in one transaction. Expected section count is at least two; preferred count is four.
2. Generated sections use the existing `ModuleSection` model and schema fields only: `courseModuleId`, `title`, `type`, `order`, `publishStatus`, `lecturerNotes`, `createdAt`, and `updatedAt`; `status` is active if present.
3. Defaults: `publishStatus=draft`, `lecturerNotes=null` or empty, and `status=active`.
4. Allowed section types are the existing values: `lecture`, `lab`, `assignment`, `supplementary`; do not introduce new types.
5. Generation is atomic: module + sections commit together or nothing commits. If section creation fails, module creation fails and rolls back.
6. Lecturers still cannot create/delete/reorder sections. Add no endpoints or UI for structure management.
7. Existing content APIs must work with generated sections without changing write-path semantics.
8. Student visibility remains unchanged: generated sections default to draft, students do not see draft sections, and after one generated section is published students see only that section.

## Data/model requirements
Use the default generator equivalent to:

```python
DEFAULT_MODULE_SECTIONS = [
    {"title": "Lecture 1", "type": "lecture", "order": 1},
    {"title": "Lecture 2", "type": "lecture", "order": 2},
    {"title": "Lab 1", "type": "lab", "order": 3},
    {"title": "Assignment 1", "type": "assignment", "order": 4},
]
```

Adapt naming/style to backend conventions.

Optional DB constraint: consider `unique(course_module_id, order)` only if current schema/data allows it safely. If risky, skip and enforce stable order in service tests. Do not add a section-template table.

## API contract requirements
Prefer no public API contract change. Existing admin module creation request should continue working:

```text
same POST /admin/modules payload
new backend side effect: default sections are generated
```

Only change DTOs if absolutely required. If API contract changes, run OpenAPI generation, commit generated client changes, and document why. Expected path is no DTO change and no frontend changes.

## Authorization requirements
Existing authorization remains unchanged:
- Admin can create modules.
- Lecturer can access and manage content only in assigned modules and cannot create/delete/reorder module sections.
- Student can access only published sections in assigned modules and cannot upload or modify content.
- Backend remains the authority.
