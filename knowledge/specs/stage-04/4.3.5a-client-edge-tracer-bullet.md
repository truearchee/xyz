## Why this session exists

Stages 1–4.3 are backend-complete and committed at baseline `8144529`. Stage 1 is FULLY VERIFIED. Stages 2, 3, 4.1–4.3 are UI PENDING — their backends pass tests but **no browser has ever authenticated against them**. The entire client-side auth model (Supabase JWT acquire → attach → call backend → refresh → 401/403 recovery) has never executed in a real browser.

This session proves that spine — once — before any product UI is built on top of it. If the spine is broken, we find out now, against an ugly throwaway page, not after three stages of polished screens.

**This is not a Stage 2 UI session.** It builds no dashboards, no admin screens, no shell navigation. It builds the keepable auth infrastructure plus one ugly `/tracer` page that exercises the full path.

## Checkpoint A — Required backend change: GET /me

This is the **one** allowed backend change in this session. The frontend must not infer product role or memberships from JWT claims.

```
GET /me  →  200
{
  userId: string
  email: string
  fullName: string
  role: "admin" | "lecturer" | "student"
  timezone: string
  activeModuleMemberships: Array<{ moduleId: string, role: "lecturer" | "student" }>
}
```

**Rules:**
- **Active memberships only.** Archived membership rows must never reach app bootstrap. Route guards must not need to understand `status: "archived"`.
- **Admin returns `activeModuleMemberships: []`.** Admins are never assigned to modules. If an admin membership row exists due to bad data, exclude it silently and log a server-side warning.
- **No `isActive` field.** A successful `GET /me` response is itself proof of active status — inactive users are rejected by `get_current_user` (Session 2.2) before this response is produced.
- Reuse existing `CurrentUserContext` (2.2) and `course_memberships` query (2.4). This endpoint composes existing pieces; it does not invent new authz logic.

**Backend tests must cover:**
```
admin    → role="admin", activeModuleMemberships=[]
lecturer → role="lecturer", one active lecturer membership
student  → role="student", one active student membership
archived membership → excluded from response
inactive user       → rejected before response (assert existing behavior from 2.2; do not change the status code or semantics in this session)
```

**Build order — do not reverse:**
```
1. Implement GET /me in the backend.
2. Add backend tests covering all five cases above.
3. Regenerate the OpenAPI TypeScript client (scripts/generate-api-client.sh).
4. Verify generated-client freshness (see below).
5. Include the regenerated client files in the session diff — do not commit partial checkpoint work separately.
   The final session commit must include: code + generated client + knowledge updates together.
6. ONLY THEN write frontend code that consumes /me.
```
Building frontend against a missing endpoint or stale generated client wastes a full cycle.

**Router registration:** after implementing `backend/app/api/me.py`, register the `/me` router in `backend/app/main.py` (or the existing API router aggregation path used by Sessions 2.3/2.4). Verify the route appears in the OpenAPI schema at `http://localhost:8000/openapi.json` before regenerating the frontend client. The classic failure mode here is a route file that exists but was never mounted.

**Generated-client freshness check after regenerating:**

Run:
```bash
git diff -- frontend/src/lib/api
```

Expected diff:
- New `/me` service method
- New response type for `GET /me`

Unexpected diff (stop and report if seen):
- Unrelated DTO renames
- Removed services
- Wholesale formatting churn across unrelated files

If unexpected diff appears: stop and report before committing.

## Checkpoint B — Frontend auth infrastructure

Build these files:
- `frontend/src/lib/supabase/client.ts` — real Supabase browser client
- `frontend/src/lib/session/SessionProvider.tsx` — session acquisition, persistence, exposure
- `frontend/src/lib/api/wrapper.ts` — wraps the generated client, retrieves access token per request
- `frontend/src/app/login/page.tsx` — throwaway login/logout UI

**Rules:**
- The API wrapper retrieves the current access token on every request. Do not cache the token at module initialization — a cached token will not refresh correctly.
- Login calls Supabase browser auth directly. Do not add `/auth/login` or any custom auth endpoint.
- Do not build an app shell, role dashboards, or navigation. `SessionProvider` may wrap the app only as needed to expose session state — it is not the app shell.
- No Redux, Zustand, TanStack Query, or other state management libraries.

**Minimum verifiable behavior before moving to Checkpoint C:**
- User can sign in via Supabase browser auth.
- User can sign out.
- Current session state is visible on `/login` or `/tracer` (logged in / logged out).
- After sign-in, calling `GET /me` through the API wrapper returns 200.

**Verification:**
```bash
docker compose exec frontend npx tsc --noEmit
```
Also confirm: no direct `fetch()` calls exist in page components or feature code. The API wrapper and the dedicated upload helper (`frontend/src/lib/api/upload.ts`) are the only permitted call sites.

## Checkpoint D — /tracer page minimum controls

The page is ugly. That is correct. It must expose at minimum:

```
- show current session state (logged in / logged out, role if known)
- button to call GET /me and display raw response
- button to call GET /modules and display raw response

- file input + upload button (non-PDF, for negative test)
```

## Checkpoint E — The 15 Playwright assertions (the actual deliverable)

**Isolation rule:** lecturer and student tests run in **separate browser contexts** — no shared cookies, localStorage, or session state. A test that passes because state leaked proves nothing about role separation.

**CORS rule:** Playwright must call the backend directly from the frontend origin (`http://localhost:3000` → `http://localhost:8000`). No Next.js rewrite or proxy may hide this. If the call looks same-origin, CORS was never exercised.

Each step is action → asserted observable outcome:
```
1.  Lecturer logs in via real Supabase            → supabase.auth.getSession() returns a session with non-null access_token
2.  Frontend calls GET /me with that token         → 200, role="lecturer", activeModuleMemberships present
3.  Browser calls GET /modules with that token     → 200, list contains ONLY the assigned module
4.  CORS on that cross-origin call                 → authenticated Authorization request succeeds with no browser console CORS error.
                                                     If an OPTIONS preflight is observable, assert it returns CORS-allowed headers.
                                                     Use a fresh browser context and unique query param if needed to bypass preflight cache.
                                                     Do not fail solely because OPTIONS was not visible — the passing authenticated call is the primary assertion.
5.  Lecturer uploads a real PDF                    → 200, asset record returned
6.  Lecturer uploads a non-PDF (negative)          → 4xx (backend validation fires on real browser payload)
7.  Lecturer publishes the section                 → publish_status = published
8.  [NEW browser context] Student logs in          → getSession() returns a DIFFERENT access_token
9.  Frontend calls GET /me for student             → 200, role="student"
10. Student calls GET sections                     → published section visible
11. Student requests draft section (negative)      → 403/404 — UI renders unauthorized state, does NOT redirect to /login
12. Student requests signed download URL           → 200, URL returned from backend
13. Backend /download-url response header          → Cache-Control: no-store (assert on the backend endpoint response, not on object storage)
14. Browser opens the signed URL (NO auth header)  → asset bytes load
15. Student attempts a direct upload               → 403 — UI renders unauthorized state, does NOT redirect to /login
```

The 401/403 split in assertions 11 and 15 is load-bearing: a 403 must keep the session and render an unauthorized state. A student hitting a forbidden resource must never appear logged out.
