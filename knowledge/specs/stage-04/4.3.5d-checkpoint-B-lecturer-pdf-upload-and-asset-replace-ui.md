## Purpose
Implement the lecturer-side UI for uploading and replacing PDF assets inside predefined module sections.

This checkpoint proves:

```text
Lecturer opens an assigned module
-> sees generated sections
-> uploads a real PDF to a section
-> uploaded asset appears from backend data
-> asset processingStatus renders
-> lecturer replaces that specific asset
-> updated asset appears after re-fetch
-> invalid upload renders a visible error
```

This is not the full Stage 3 browser gate yet. Student visibility, publish/unpublish, and signed URL access come later.

## Authoritative behavior
The content-management slice says lecturers can upload and replace PDFs attached to predefined sections, while students cannot upload or modify content. It also says PDF validation must happen on the backend.

It also defines an important separation:

```text
SectionAsset.processingStatus = technical file processing state
ModuleSection.publishStatus  = student visibility state
```

Those must remain separate in the UI. A file can be completed while the section remains draft.

## Scope

### In scope
- Add upload control per lecturer section.
- Add asset list per section.
- Add asset row per uploaded asset.
- Render asset filename.
- Render asset `processingStatus`.
- Upload PDF through `frontend/src/lib/api/upload.ts`.
- Replace a specific asset through `frontend/src/lib/api/upload.ts`.
- Re-fetch backend data after upload/replace.
- Show loading/submitting state.
- Show mutation failures as `role="alert"`.
- Backend rejection of non-PDF renders clear UI error.
- Keep section `publishStatus` visible and separate from asset `processingStatus`.

### Out of scope
- Publish/unpublish controls.
- Student module page.
- Signed URL open/download.
- Student upload rejection.
- Full Playwright Stage 3 gate.
- Backend endpoint changes.
- Transcript upload.
- AI, embeddings, summaries.
- Section create/delete/reorder.
- Storage cleanup changes.

## Contract investigation

### 1. Upload helper
Confirm `frontend/src/lib/api/upload.ts` exports `uploadSectionAsset(...)` and `replaceSectionAsset(...)`.

Confirm it uses:
- FormData field: `file`
- `Authorization: Bearer <token>`
- no manual `Content-Type`
- typed `SectionAssetResponse`

If missing or wrong, stop and report. Do not duplicate upload logic elsewhere.

### 2. Asset read path
Checkpoint B needs a way to render uploaded assets after re-fetch.

Confirm one of these exists:
- section detail/list response includes section assets
- lecturer-only asset list endpoint exists: `GET /modules/{moduleId}/sections/{sectionId}/assets`

If assets are not included in the section read model, use the existing lecturer asset-list endpoint.

If no asset read path exists, stop and write a finding. Do not add backend read projections unless explicitly approved.

### 3. Replace path
Confirm `replaceSectionAsset(...)` maps to the existing backend replace endpoint and replaces a specific asset, not "the section file."

Expected UI model:

```text
section
  asset row
    Replace button for this asset
```

Replace must be asset-level.

## UI behavior

### Section upload control
Each section should show an upload control.

Required behavior:
- select PDF
- click Upload
- call `uploadSectionAsset`
- show uploading state
- on success: re-fetch section/module/asset data from backend
- on failure: render `role="alert"`

Do not consider local selected file state as success.

### Asset list
Each section should show existing uploaded assets.

For each asset render:
- filename
- processingStatus
- replace control

If no assets: `No files uploaded`.

### Asset replace
Each asset row has its own replace control.

Required behavior:
- select replacement PDF
- click Replace
- call `replaceSectionAsset` with `assetId`
- show replacing state
- on success: re-fetch backend data
- on failure: render `role="alert"`

The UI must not replace "the section's file" generically. It replaces a specific `assetId`.

## Status rendering
Keep two status concepts separate:

### Section publish status
Already from Checkpoint A:

```text
Draft / Published / Unpublished
```

This belongs to the section.

### Asset processing status
New in Checkpoint B:

```text
uploaded / processing / completed / failed
```

This belongs to each asset.

Do not merge these into one badge.

Suggested test IDs:

```text
data-testid="section-publish-status-{sectionKey}"
data-testid="section-asset-processing-status-{assetId}"
data-testid="section-asset-row-{assetId}"
data-testid="section-asset-replace-{assetId}"
data-testid="section-upload-control-{sectionKey}"
```

Important: asset processing status should use `assetId`, not `sectionKey`, because a section can have multiple assets.

## Error behavior
All upload/replace failures must render `role="alert"`.

Cases to handle:
- non-PDF file rejected by backend
- oversized file, if limit is enforced
- 403 forbidden
- 401 unauthenticated
- network failure

For this checkpoint, at minimum prove the non-PDF rejection path in the browser.

Do not fake client-only PDF validation as the proof. Frontend validation is allowed for UX, but backend rejection must still be observable.

## Direct fetch rule
No direct `fetch()` in app/components/features.

Allowed locations only:
- `frontend/src/lib/api/core/request.ts`
- `frontend/src/lib/api/upload.ts`

Run scan after implementation.
