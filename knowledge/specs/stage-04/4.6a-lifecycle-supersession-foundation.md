## Goal
A replacement transcript processes as `pending` alongside the still-`active` old one and atomically
swaps in only once it reaches `summarized` with eligible brief + detailed summaries — with no data
loss, no duplicate active/pending, and no student surface. This is the foundation the 4.6 retry
(4.6b) and recovery (4.6c) sit on.

## Why now
Stage 4.6 is the first stage where the pipeline may be *re-run*. Before retry/recovery can exist, the
state machine must be safe: a replacement cannot degrade working content, and "which summary belongs
to the active transcript" must be a first-class, owned decision. Stage 4.5 is FULLY VERIFIED, so all
five job types and the per-step projection (the things retry keys off) already exist.

## Build
- **State model & supersession (ADR-46-A / adr-029):** migrate `Transcript.is_active` → `lifecycle_state`
  (active|pending|superseded); add lineage (`replacement_of`/`superseded_by`/`supersession_reason`,
  keep `superseded_at`); one-active AND one-pending partial-unique indexes; section-locked
  pending-creation (replacement no longer 409s) with prior-pending discard; `try_activate_pending_transcript`
  atomic swap, triggered post-summary-completion.
- **Eligibility split (ADR-46-E / adr-030):** `transcripts/domain/summary_eligibility` owns the
  predicate (identity + provenance) + write-side activation readiness; `platform/query/
  ActiveTranscriptSummaryResolver` wraps the SAME predicate for reads only.
- **Provenance stamps:** `created_by_ingestion_job_id` on segments/chunks/summaries + a SEPARATE
  `embedding_created_by_ingestion_job_id` on chunks; embed never clobbers the chunk creator.
- **Fault-injection harness:** env-gated (`PIPELINE_FAULT_INJECTION_ENABLED`), no-op when off, refuses
  outside non-prod; forces any of the five steps to fail + seeds failed-`IngestionJob` records.
- **Contract:** expose `lifecycleState` on `TranscriptMeta`; regenerate the frontend client.

## Do not build
- Retry / resume-from-failed-step + fenced destructive deletes → **4.6b**
- Stuck-row reaper, storage reconciliation, `MaintenanceRun`, heartbeat fields → **4.6c**
- Lecturer replace/retry UI, active-summary preview HTTP endpoint, full browser gate → **4.6d**
- Student summary HTTP surface / content-visibility / 404-not-403 → **4.7**
- Scheduler/cron for recovery jobs → **11.1**
- parse/chunk one-active "current job" pointer indexes → **4.6b** (would break the existing
  two-chunk-jobs-coexist replacement path; designed with the retry that consumes it)

## Data model changes
- `transcripts`: `+lifecycle_state` (replaces `is_active`), `+replacement_of_transcript_id`,
  `+superseded_by_transcript_id`, `+supersession_reason` (keep `superseded_at`); lifecycle CHECKs;
  `uq_active_transcript_per_section` re-pointed to `lifecycle_state='active'`; new
  `uq_pending_transcript_per_section`.
- `transcript_segments` / `generated_lecture_summaries`: `+created_by_ingestion_job_id` (FK SET NULL).
- `transcript_chunks`: `+created_by_ingestion_job_id` AND `+embedding_created_by_ingestion_job_id`.
- Migration `0010`. (No `MaintenanceRun`, no heartbeat — those are 4.6c.)

## API changes
- `TranscriptMeta` gains `lifecycleState: str`. The existing upload endpoint now performs replacement
  (creates `pending`) instead of returning 409. No new route in 4.6a.

## Worker / job changes
- Each step stamps its provenance column on write. Summary-completion path opportunistically calls
  `try_activate_pending_transcript` (no-op unless a ready pending exists).

## Authz rules
- Unchanged surface: lecturer-only + assigned + lecture/lab on the (re)used upload endpoint.

## Done means
lifecycleState migrated (`is_active` removed); one-active AND one-pending enforced at DB level; lineage
populated; replacement creates pending alongside active; section-locked atomic swap; failed/incomplete
replacement leaves old active; double replacement discards prior pending; activation requires
`summarized` + exactly one eligible brief + one detailed; eligibility predicate + write-side in the
domain, resolver is a read-only wrapper; provenance stamped (embed never clobbers chunk creator);
fault harness env-gated + no-op when off; backend suite + fresh-DB migration round-trip + frontend
type-check all green. (Browser gate is 4.6d.)
