## 3. Terminal-state and status model — C-lite (locked, pending §3.1)

`Transcript.status` is demoted to a low-fidelity breadcrumb. **`IngestionJob` rows are the source of truth** for step outcomes. The **processing-status read projection** (§7) is the sole doneness authority for both UI and the browser gate.

```
Transcript.status           = furthest coarse pipeline phase (breadcrumb; not asserted on by gate)
IngestionJob rows           = source of truth for per-step outcomes
Processing-status endpoint  = derived read projection consumed by UI and Playwright
```

**Rationale:** `Transcript.status = completed` already shipped meaning "chunked" in 4.3.5e. Keeping `status` as the doneness signal forces `completed` to either lie (work continues after completed) or shift meaning across stages. The projection decouples doneness from the coarse enum so 4.5+ inherit no status-field debt.

## 4. Data model changes

### 4.1 Embedding column (`TranscriptChunk`)

Replace the existing `UserDefinedType` with:

```python
from pgvector.sqlalchemy import Vector

embedding: Mapped[list[float] | None] = mapped_column(Vector(384), nullable=True)
```

`all-MiniLM-L6-v2` emits **384-dim** vectors. Dimension is fixed at schema level; a different-dim model later is a migration (acceptable, recorded in ADR).

**Migration safety:** fail fast if any non-null embedding is found in the existing column unless explicitly nulled. Guards against test-era writes by the old type.

### 4.2 Embedding provenance fields (`TranscriptChunk`)

| Field | Type | Purpose |
|---|---|---|
| `embedding` | `vector(384)`, nullable | The vector |
| `embeddingModel` | `text` | Full HF id: `sentence-transformers/all-MiniLM-L6-v2` |
| `embeddingModelRevision` | `text` | Pinned HF commit SHA — no floating "latest" |
| `embeddingDimension` | `integer` | `384` |
| `embeddingNormalization` | `text` | `l2` |
| `embeddingVersion` | `text` | Human-readable policy bucket, e.g. `minilm-l6-v2@<sha>/chunker-v1/l2` — for bulk regen targeting |
| `embeddingInputHash` | `text` | sha256 per-row staleness selector (§4.3) |

`embeddingVersion` and `embeddingInputHash` are **not redundant**:
- `embeddingVersion` answers "which policy produced this" — used to select a family for bulk regeneration.
- `embeddingInputHash` answers "is this specific row stale" — the per-row selection predicate.

Both are stored. Neither replaces the other.

### 4.3 `embeddingInputHash`

Computed as `sha256(canonical_json(payload))` where `payload` is:

```json
{
  "chunkText":              "<TranscriptChunk.text exactly as passed to the encoder>",
  "embeddingModel":         "sentence-transformers/all-MiniLM-L6-v2",
  "embeddingModelRevision": "<pinned sha>",
  "embeddingNormalization": "l2",
  "chunkerVersion":         "chunker-v1"
}
```

Use canonical JSON (deterministic key order, no whitespace variation) — not raw string concatenation. Hash the **text actually passed to the encoder**, not the stored chunk text, so the hash tracks the real input.

### 4.4 `chunkerVersion` on `TranscriptChunk`

Add `chunker_version text not null` to `TranscriptChunk`. Backfill all existing 4.3-era rows as `chunker-v1`. Required for `embeddingInputHash` correctness and future provenance when the chunker changes.

### 4.5 DB provenance constraint

```sql
CHECK (
  embedding IS NULL
  OR (
    embedding_model IS NOT NULL
    AND embedding_model_revision IS NOT NULL
    AND embedding_dimension = 384
    AND embedding_normalization = 'l2'
    AND embedding_version IS NOT NULL
    AND embedding_input_hash IS NOT NULL
  )
)
```

Adapt to repo naming conventions. No vector without provenance — enforced at the DB level, not only in done criteria.

### 4.6 `IngestionJob` — embed job type

Confirm `jobType = embed` exists. Standard fields: `transcriptId`, `jobType`, `status`, `attempts`, `errorMessage`, `startedAt`, `completedAt`, `createdAt`.

**Retry row semantics:** queue-level automatic retry reuses the same `IngestionJob` row. `attempts` increments; `status` transitions through `queued → running → failed/completed`; `errorMessage` is overwritten per existing job conventions. No new `IngestionJob` row is created for automatic retry.

### 4.7 Vector index — deferred, intentionally

No search/retrieval in 4.4. Index lands with retrieval (Stage 8). Recorded deferral, not oversight.

## 5. Worker design

### 5.1 `EmbeddingEncoder` abstraction (internal — NOT AI infra)

```python
class EmbeddingEncoder(Protocol):
    def encode(self, texts: list[str]) -> list[list[float]]: ...
```

Implementations:
- `SentenceTransformersEmbeddingEncoder` — real local model; production and browser gate.
- `DeterministicEmbeddingEncoder` — unit/integration tests only.

**No feature-level mock in the browser gate.** The deterministic encoder is permitted only in unit/integration tests, never in the stage gate. Mirrors the provider-boundary adapter rule for LLMs.

### 5.2 Model loading, startup checks, and offline behavior

On worker startup (before accepting any jobs), verify:
- `EMBEDDING_MODEL_PATH` exists on disk.
- Model revision in the config/tokenizer files matches `EMBEDDING_MODEL_REVISION`.
- Required model files (config, tokenizer, weights) are present.

If any check fails: **fail fast with a safe error** — not a hang, not a silent fallback. This prevents the "revision A in config, revision B on disk" class of corruption, and catches the "it downloaded locally but not in CI" class of environment drift.

Load the model **once per worker process** (lazy singleton after startup checks pass), never per job. Pre-download/vendor the model in the image build. Document first-load size (~80 MB) in the runbook.

### 5.3 Chunk selection and per-row staleness

Chunk selection is a two-phase operation to avoid a race between staleness evaluation and row locking:

**Phase 1 — coarse DB filter (no lock)**

Load candidate chunk IDs for the transcript where the coarse DB predicates are likely stale:
```sql
WHERE transcript_id = :transcript_id
  AND (
    embedding IS NULL
    OR embedding_version != :current_embedding_version
  )
```

**Phase 2 — lock and per-row hash check**

For each candidate row:
```sql
SELECT ... FROM transcript_chunks WHERE id = :chunk_id FOR UPDATE SKIP LOCKED
```

Once locked, compute `expectedEmbeddingInputHash` from the chunk text that will actually be passed to the encoder. Keep (and encode) only rows where:
- `embedding IS NULL`, or
- `embedding_version != current_embedding_version`, or
- `embedding_input_hash != expectedEmbeddingInputHash`, or
- any required provenance field is missing or mismatched.

This two-phase approach ensures the staleness decision is made inside the row lock, not before it.

### 5.4 Normalization

Embeddings stored **L2-normalized** (`normalize_embeddings=True`). Future retrieval uses cosine similarity / compatible pgvector operator. Data-compatibility decision — changing it post-storage means re-embedding everything. Any change requires an explicit ADR.

### 5.5 Partial-failure semantics

```
Embedding writes commit per batch.
A failed embed job may leave some chunks embedded.
Retry resumes from chunks that fail the per-row staleness check (§5.3 Phase 2).
Job reaches `completed` ONLY when embeddedChunkCount === chunkCount (non-empty chunks).
```

### 5.6 Truncation caveat — measurable assessment required before merge

`all-MiniLM-L6-v2` truncates at ~256 word-pieces. Chunks exceeding this are silently embedded on their first 256 tokens only.

Before merge, run a chunk distribution check on seeded/E2E transcripts and record:
- Max estimated word-piece count across all chunks.
- p95 estimated word-piece count.
- Count of chunks above 256 word-pieces.
- Resolution: `accepted with rationale` / `finding filed against 4.3` / `chunker adjusted`.

Result recorded in `knowledge/steps/4.4-truncation-assessment.md`. "Probably fine" is not a resolution.

### 5.7 Empty-chunk handling

```
Zero non-empty chunks for transcript  → embed job FAILS with safe error.
Any empty/whitespace chunk text       → embed job FAILS + finding filed against 4.3 chunking.
```

Skipping empty chunks hides a 4.3 data-quality bug. Fail loudly.

### 5.8 Concurrency guard

Partial-unique index scoped to `embed` only — 4.4 does not broaden the invariant to all job types:

```sql
CREATE UNIQUE INDEX ingestion_jobs_one_active_embed_per_transcript
  ON ingestion_jobs (transcript_id, job_type)
  WHERE job_type = 'embed'
    AND status IN ('queued', 'running');
```

If the broader invariant (all job types) is wanted, that is a separate cross-pipeline decision with its own ADR and preflight duplicate check.

### 5.9 Resource configuration

```
EMBEDDING_MODEL_PATH          path to vendored model (required)
EMBEDDING_MODEL_REVISION      pinned HF commit SHA (required; validated at startup)
EMBEDDING_BATCH_SIZE          tunable; default determined by worker author
EMBEDDING_WORKER_CONCURRENCY  cap separately from lightweight jobs
EMBEDDING_DEVICE              cpu (default for MVP)
```

Embedding worker concurrency must be capped separately from the parsing/chunking lane to prevent one CPU-heavy job from starving lightweight workers.

## 7. Processing-status API / read projection

Lives in `platform/query` (read-models-only rule — no business decisions here).

```
GET /lecturer/module-sections/{sectionId}/transcript-processing-status
```

### `overallState` enum (full set)

```
uploaded | queued | parsing | parsed | chunking | chunked | embedding | embedded | failed
```

`parsed`, `chunked`, `embedded` are clean terminal states per phase. `failed` is the catch-all failure terminal.

### `currentPhase` and `failedStep`

```
currentPhase:  upload | parse | chunk | embed | null
failedStep:    upload | parse | chunk | embed | null
```

For all terminal states (`chunked`, `embedded`, `failed`): `currentPhase = null`. `failedStep` identifies which step caused `overallState = failed`; it is `null` for non-failure terminals.

A failed embed response:

```json
{
  "overallState": "failed",
  "currentPhase": null,
  "failedStep":   "embed"
}
```

### `steps` shape — keyed object

`steps` is a **keyed object**, not an array. This avoids `.find()` gymnastics in Playwright and client code.

```json
{
  "steps": {
    "upload": { "status": "completed", "startedAt": "…", "completedAt": "…" },
    "parse":  { "status": "completed", "startedAt": "…", "completedAt": "…" },
    "chunk":  { "status": "completed", "startedAt": "…", "completedAt": "…" },
    "embed":  { "status": "completed", "startedAt": "…", "completedAt": "…" }
  }
}
```

`steps[key].status` values: `not_started | queued | running | completed | failed`.

### Full response shape

```json
{
  "activeTranscriptId":  "…",
  "transcriptStatus":    "…",
  "overallState":        "embedded",
  "currentPhase":        null,
  "failedStep":          null,
  "steps": {
    "upload": { "status": "completed", "startedAt": "…", "completedAt": "…" },
    "parse":  { "status": "completed", "startedAt": "…", "completedAt": "…" },
    "chunk":  { "status": "completed", "startedAt": "…", "completedAt": "…" },
    "embed":  { "status": "completed", "startedAt": "…", "completedAt": "…" }
  },
  "segmentCount":        42,
  "chunkCount":          12,
  "embeddedChunkCount":  12,
  "safeFailureMessage":  null,
  "updatedAt":           "…"
}
```

**Never returns:** raw transcript text · chunk text · embedding vectors · stack traces · raw job errors.

Spec uses camelCase for readability. DB/model fields use snake_case per repo convention. OpenAPI DTO maps between them via the project's alias generator (`populate_by_name=True`).
