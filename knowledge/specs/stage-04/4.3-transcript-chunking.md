## Goal
Group immutable parsed `transcript_segments` into normalized, ordered, persisted `transcript_chunks`
ready for embedding and summarization, without mutating segment text, without stalling on queue failure,
and without duplicating on re-run.

## Versioned policy constants (this session)
```
NORMALIZATION_VERSION = "norm-v1-structural"
CHUNKING_VERSION      = "chunk-v1-no-overlap-180w"   # hard cap is the 180 in the name
TOKEN_COUNT_METHOD    = "heuristic_word_count_v1"    # change => CHUNKING_VERSION bump (by policy)

TARGET_CHUNK_WORDS    = 160
HARD_CHUNK_WORDS      = 180   # single segment exceeding this => oversized singleton
```
All constants live in version-controlled code, not environment variables. Any change to the normalizer
boilerplate list bumps `NORMALIZATION_VERSION`. Any change to grouping, token policy, or word caps bumps
`CHUNKING_VERSION`. Both changes are independently re-chunkable without affecting the other.

## Build
1. `TranscriptChunk` model + Alembic `0006` (see Data model changes). Migration runs
   `CREATE EXTENSION IF NOT EXISTS vector` per project migration convention.

2. **Normalizer** (`normalize_segment_text` → chunk-layer text): deterministic and structural only —
   collapse whitespace, strip residual VTT artifacts, drop empty and adjacent-duplicate segments, plus a
   small structural pattern list stored in version-controlled code. Never writes `transcript_segments`.
   See Resolved decision D1.
   - **Skipped segments:** segments dropped by normalization may create text gaps inside a chunk range.
     For MVP, a chunk's `start/end_segment_id` spans the raw segment range inclusive of any skipped
     segments; skipped segments contribute no text. Exact per-segment membership is deferred post-MVP.

3. **Chunker**: accumulate **contiguous** segments (ordered by `sequence_number`) until the
   `TARGET_CHUNK_WORDS` budget is reached; emit a chunk; continue. Chunks are non-overlapping (MVP).
   - Each chunk records: `start_segment_id`, `end_segment_id`, `start_sequence_number`,
     `end_sequence_number`, `start_time`, `end_time` (mirroring ADR-018), normalized `text`,
     `token_count`, `token_count_method`, `chunk_index`, `normalization_version`, `chunking_version`.
   - **Medium-large singleton rule:** if one segment exceeds `TARGET_CHUNK_WORDS` but is below
     `HARD_CHUNK_WORDS`, emit it as a normal singleton chunk with `is_oversized = false`.
   - **Oversized rule:** a single normalized segment >= `HARD_CHUNK_WORDS` emits a singleton chunk with
     `is_oversized = true`. Never split a segment. Never loop. Count these in `oversized_segment_count`
     on the job report; 4.4 must report embedding truncation if the embedding model truncates it.
   - **Zero chunks:** no usable text after normalization → chunker produces an empty set. Chunk job
     completes with `chunk_count = 0` and must **not** enqueue the embed step. 4.4 treats zero chunks
     as a clean controlled state.

4. **Parse → chunk hand-off (transactional outbox):** at the tail of a successful parse, **in the same
   DB transaction that records parse completion**, insert the `chunk` `ingestion_jobs` row at
   `status = 'queued'` with the idempotency key below. **After commit**, enqueue the RQ worker job
   referencing `ingestion_job.id`. If the RQ enqueue fails after commit, the DB still holds a
   recoverable `queued` chunk job for 4.6.

5. **Chunk job handler** (worker-internal):
   - Before relying on idempotency, confirm `ingestion_jobs` enforces uniqueness for the idempotency key
     or equivalent job identity. If not, add the required constraint/index before implementing chunk
     concurrency.
   - Acquire a **row-level lock (`FOR UPDATE`) on the `ingestion_jobs` row** for this job.
   - **Idempotency:** if a `completed` chunk job for the same idempotency key exists → no-op. If another
     worker is already processing the same job row (lock contention) → return/defer according to the
     existing worker locking convention; do not clear or insert chunks.
   - Acquire a **row-level lock (`FOR UPDATE`) on the owning `transcripts` row** before clearing or
     inserting any chunks. Chunk writes are serialized per transcript, not only per ingestion job. This
     prevents two chunk jobs with different idempotency keys (e.g., a version bump mid-flight) from
     racing on the same transcript's chunk set.
   - **Precondition gate (defensive):** verify the `parse` `ingestion_jobs` row for this transcript is
     `completed`. Because the chunk job is created only inside the parse-completion transaction, the gate
     firing means a stale or replayed job. If the precondition is not met: **leave the transcript
     unchanged**; mark the chunk job `failed` with sanitized error code
     `precondition_not_met_parse_not_completed`; do not add a new job-status state.
   - In a **single DB transaction**: delete any existing chunks for the transcript, insert the fresh
     ordered set, advance transcript `parsing → chunking`, record chunk-job `completed` with `chunk_count`
     and `oversized_segment_count`. **Rollback is the primary cleanup mechanism.** If the transaction
     fails and rolls back, the prior committed chunk set (if any) remains intact. The delete at
     transaction start also eliminates any partial state left by a prior crashed attempt.
   - **Failure path (genuine processing error):** chunk replacement must be atomic. If chunk insertion
     fails, rollback preserves any prior completed chunk set. Failure status/error is recorded in a
     separate cleanup transaction after rollback. The sanitized error message must not contain storage
     keys, URLs, credentials, stack traces, raw segment text, or chunk text. This path applies only to
     processing failures, never to the precondition gate.

## Data model changes
New table `transcript_chunks`:
```
transcript_chunks
- id                      uuid pk
- transcript_id           uuid not null  fk -> transcripts(id)            (cascade-delete with transcript)
- chunk_index             integer not null                                 (0-based order within transcript)
- start_segment_id        uuid not null  fk -> transcript_segments(id)
- end_segment_id          uuid not null  fk -> transcript_segments(id)
- start_sequence_number   integer not null
- end_sequence_number     integer not null
- start_time              bigint null                                      (null for TXT; ms per ADR-018)
- end_time                bigint null                                      (null for TXT; ms per ADR-018)
- text                    text not null
- token_count             integer not null
- token_count_method      text not null
- normalization_version   text not null
- chunking_version        text not null
- is_oversized            boolean not null default false
- embedding               vector(384) null            (D3: nullable now, populated by 4.4)
- embedding_model         text null
- embedding_version       text null
- embedding_generated_at  timestamptz null            (null = never embedded; set by 4.4)
- created_at              timestamptz not null default now()
- updated_at              timestamptz not null default now()
```
Constraints / indexes (follow existing `uq_/ix_/ck_/fk_` conventions):
```
uq (transcript_id, chunk_index)
ck token_count > 0
ck length(btrim(text)) > 0
ck chunk_index >= 0
ck end_sequence_number >= start_sequence_number
fk transcript_id     -> transcripts(id)
fk start_segment_id  -> transcript_segments(id)
fk end_segment_id    -> transcript_segments(id)
```
Notes:
- **No standalone `ix(transcript_id)`** — the unique btree on `(transcript_id, chunk_index)` covers the
  prefix.
- **Cross-transcript integrity:** plain FKs don't guarantee start/end segments belong to this chunk's
  transcript. Enforce via a **service-level assertion** (start/end segments' `transcript_id` matches the
  chunk's) and a **reject integration test**. Composite FK is optional hardening deferred post-MVP.
- `transcript_segments` are **not** altered (rule 6). The chunker reads them read-only.
- `updated_at` is needed because 4.4 will write the embedding fields on existing rows.

Add nullable `result_metadata jsonb` to `ingestion_jobs` for structured worker result counts such as
`chunk_count` and `oversized_segment_count`.

## Idempotency key (chunk job)
```
transcript_id + 'chunk' + source_checksum + parser_version + normalization_version + chunking_version
```
- `source_checksum` — raw uploaded file hash (from 4.1).
- `parser_version` — **must be read from the completed parse job's persisted metadata**, i.e., the version
  that actually produced the current `transcript_segments`. Do not use the current parser code constant
  unless the transcript has been re-parsed with that version. Using the code constant on stale segments
  would create a false new key while operating on old output.
- `normalization_version`, `chunking_version` — independent re-chunk axes.
- ADR-019 / the parse job stores this as `ingestion_jobs.processor_version`.

## API changes
None. Chunking is worker-internal. `chunking` status surfaces through the existing transcript `status` DTO.

## Worker/job changes
- New `chunk` job entrypoint on the `ingestion` queue.
- Parse-completion transaction creates the `queued` chunk job row; RQ enqueue (by `ingestion_job.id`)
  after commit. Job chain: upload → parse → **chunk** → [embed, 4.4].
- Handler takes `FOR UPDATE` on the job row, then on the `transcripts` row, before any chunk write.

## Authz rules
- No user-facing surface added; assert chunk text never enters any DTO or response (negative test).
- Worker reads segments and writes chunks within the transcript's existing ownership boundary.

## Status semantics (record in db-spine)
`transcripts.status = 'chunking'` means the transcript has **reached** the chunking stage. Actual
completion is determined by the `chunk` `ingestion_jobs` row. Do not add a `chunked` status. Session 4.4
advances `chunking → embedding`.

## Tests required
**Unit**
- Chunker: contiguous accumulation to `TARGET_CHUNK_WORDS`; never exceeds `HARD_CHUNK_WORDS` except for
  oversized singletons; correct `chunk_index`, `start/end_segment_id`, `start/end_sequence_number`,
  `start/end_time` (incl. TXT null-time path); single-segment transcript → one chunk; empty segment set
  → zero chunks, no infinite loop.
- Medium-large singleton: single segment above `TARGET_CHUNK_WORDS` and below `HARD_CHUNK_WORDS` → one
  normal singleton chunk.
- Oversized: single segment >= `HARD_CHUNK_WORDS` → one `is_oversized = true` singleton chunk.
- Normalizer: deterministic and idempotent; chunk text differs from segment text where rules applied;
  `transcript_segments.text` byte-identical afterward (rule 6 assertion).
- `token_count` matches `TOKEN_COUNT_METHOD`; empty normalized output yields no chunk row.

**Integration**
- Parse success creates a `queued` chunk job inside the parse-completion transaction, then enqueues by id.
- Chunk handler: persists ordered chunks, advances `parsing → chunking`, records `completed` with
  `chunk_count` and `oversized_segment_count`.
- Idempotent re-run (same key) → no-op, no duplicate chunks.
- **Transcript-row lock:** two chunk jobs with different idempotency keys for the same transcript do not
  race; second writer blocks until first commits, then sees a completed state and no-ops.
- **Job-row concurrency:** a second worker for the same job row defers without clearing or inserting.
- **Gate:** parse job not `completed` → transcript unchanged, chunk job `failed` with
  `precondition_not_met_parse_not_completed`, no transcript status change.
- **Atomic replacement:** a transaction failure rolls back cleanly; prior committed chunks remain intact.
- **Cross-transcript:** start/end segment belonging to another transcript is rejected at service level.
- **Versioning:** `chunking_version` bump → re-chunk permitted even when `normalization_version` unchanged;
  `parser_version` change → new idempotency key (no false no-op on old segments).
- **Zero chunks:** chunk job completes `chunk_count = 0`; embed is **not** enqueued.
- Failure path: transaction rolled back, transcript + job `failed`, error sanitized.

**DB-spine**
- Table, columns, types, `uq/ck/fk` all present; `token_count > 0`, `length(btrim(text)) > 0`,
  `chunk_index >= 0`, `end_sequence_number >= start_sequence_number` enforced; no redundant
  `ix(transcript_id)`.

## Done means
A successfully parsed transcript automatically produces persisted, ordered `transcript_chunks` with
normalized text, segment linkage (ids + sequence numbers), time bounds (or nulls for TXT), token counts,
and recorded `normalization_version` / `chunking_version`; `transcript_segments` remain byte-identical;
the chunk job row is created in the parse-completion transaction and DB-recoverable if the RQ enqueue
fails; concurrent chunk jobs for the same transcript serialize via the transcript-row lock and do not
corrupt the chunk set; oversized segments produce singleton chunks; zero-chunk transcripts complete
cleanly without enqueuing embed; a precondition gate miss leaves the transcript unchanged; genuine
failures roll back atomically leaving prior committed data intact; the transcript rests at
`status = 'chunking'`; full Docker verification (build, migrate to `0006 (head)`, targeted + full pytest,
`tsc`, `git diff --check`, wikilink resolution) passes.
