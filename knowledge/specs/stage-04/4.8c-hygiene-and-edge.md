## Goal
Close the security-hygiene + edge gaps: prove the **§8 two-surface hygiene** against the
**Dockerfile.prod** artifact (frontend E2E hook build-time ABSENT; backend fault injection
fail-loud + no reachable surface), ship the gated **`/internal/sse-probe`** validated over the **D1
direct transport** (the SSE-buffering de-risk the whole stage exists for), set the **staging CORS
origin**, and make **`check-staging-env` a HARD deploy gate** (a bad env fails the deploy, not warns).

## Why now
This is the umbrella's third sub-session (§10): the security-sensitive deliverable + the SSE forcing
function. 4.8a/4.8b stood the deployment up; 4.8c proves it is safe to put on a public URL and that
the streaming transport survives the hosted proxy before Stage 8.3 depends on it.

## Reconciliations (evidence in context)
1. **No backend fault-injection HTTP route exists** — `rg` finds no fault/test/e2e route and no
   `StreamingResponse` in `backend/app/api`. Fault injection is env-flag + worker-side
   (`maybe_fail_step`, `pipeline_faults.py`) only; `seed_failed_ingestion_job` is a non-prod Python
   helper, not an endpoint. ⇒ the umbrella §8 "the fault-injection **route/action** is UNAVAILABLE →
   404/403" has **no such route to 404**. The real backend proof is **(a) flags absent** in staging
   (`check-staging-env`), **(b) fail-loud boot** if a flag is set with `ENVIRONMENT=staging` (the 4.8a
   boot check), **(c) no HTTP surface** that can trigger fault injection at all. Reframed honestly.
2. **Frontend hook is registered unconditionally; the gate is inside the function** —
   `SessionProvider.tsx:90 registerE2ETestHooks()`; `testHooks.ts:35` early-returns when
   `process.env.NEXT_PUBLIC_E2E_TEST_HOOKS !== 'true'`; `window.__xyzE2E` is assigned at
   `testHooks.ts:43`. ⇒ build-time absence (so the §8 bundle grep is meaningful) needs a **call-site
   guard** (`if (process.env.NEXT_PUBLIC_E2E_TEST_HOOKS === 'true') registerE2ETestHooks()`) that the
   prod minifier statically eliminates. The proof binds to the **Dockerfile.prod** output, NOT the dev
   image (F-4.8a-9). The dev `next dev` build still registers hooks at runtime (env true).
3. **No existing SSE** in the tree — the probe is the codebase's first `text/event-stream`
   (deliberately: Stage 8.3 will be the first product SSE; C1 de-risks the transport now). Reuse
   `require_role("admin")` (`auth/guards.py`) for probe auth.

## Build
- **B1 — `/internal/sse-probe` (C1, §7.C2 = acceptance criteria).** A `StreamingResponse`
  (`text/event-stream`) emitting **3–5 chunks**, closing **< 10 s**, **no compression**, with
  anti-buffering headers (`Cache-Control: no-cache`, `X-Accel-Buffering: no`, `Connection: keep-alive`).
  **Admin-authenticated** (`require_role("admin")`). Registered in `create_app()` **ONLY when**
  `ENABLE_INTERNAL_SSE_PROBE=true` (new config; default false → route 404s). Validated from the
  **staging browser over the D1 direct transport** (browser → FastAPI) asserting chunks arrive
  **progressively, not buffered into one flush**.
- **B2 — Staging CORS origin (D1, §7.G).** Set the staging frontend origin in `CORS_ORIGINS`
  (backend secret/env); confirm preflight passes for a non-simple request (`OPTIONS` + headers
  `Authorization`/`Content-Type`/`Idempotency-Key` — already `allow_headers=["*"]`). `allow_credentials`
  removal stays **4.9**.
- **B3 — §8 frontend hook build-time ABSENCE (binds to Dockerfile.prod, F-4.8a-9).** Guard the
  call-site so the prod build tree-shakes the hook. **Proof:** in the **Dockerfile.prod** bundle,
  `window.__xyzE2E` is `undefined` (definitive — the bridge is not installed) AND a grep of the built
  bundle finds no reachable token-override hook (if the minifier leaves inert strings, resolve
  accept-with-rationale: build-false ⇒ unreachable — see O1). **Regression:** the dev/e2e build still
  registers hooks (re-run the active suite).
- **B4 — §8 backend hygiene proof (reframed per reconciliation #1).** Assert: the fault flags are
  ABSENT in the staging env (`check-staging-env`); a boot with a fault flag + `ENVIRONMENT=staging`
  **fails loud** (the 4.8a `assert_fault_injection_safe`); there is **no HTTP surface** that triggers
  fault injection (env+worker-only). No "404 on a fault route" claim — there is no such route.
- **B5 — `check-staging-env` as a HARD deploy gate.** `deploy-staging.sh` already runs it under
  `set -e` (4.8b); 4.8c **proves it is hard, not warn-only**: a rehearsal asserts
  `deploy-staging.sh --dry-run` exits **non-zero** on a stale-name and on a dev-Supabase env (the
  deploy aborts before any `fly deploy`).
