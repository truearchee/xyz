## Purpose
Implement lecturer-side publish/unpublish controls for module sections.

This checkpoint proves:

```text
Lecturer opens assigned module
-> section has uploaded asset
-> asset processingStatus is visible
-> section publishStatus is visible separately
-> lecturer publishes section
-> publishStatus updates after backend re-fetch
-> asset processingStatus remains unchanged
-> lecturer unpublishes section
-> publishStatus updates again after backend re-fetch
```

Checkpoint C is specifically about visibility state control and status separation. The 4.3.5d spec requires publish/unpublish controls through the wrapper, with `processingStatus` and `publishStatus` rendered as two separate elements that never collapse into one badge.

## Scope

### In scope
- Add publish/unpublish control per section.
- Use existing backend publish/unpublish endpoints through wrapper/generated client.
- Re-fetch backend data after each toggle.
- Render section `publishStatus`.
- Render asset `processingStatus`.
- Ensure both status values are visible at the same time.
- Ensure both status values are separate DOM elements.
- Ensure status labels use visible text, not color-only indicators.
- Render mutation failures as `role="alert"`.
- Keep upload/replace behavior from Checkpoint B working.

### Out of scope
- Student module page.
- Student published-only visibility.
- Signed URL open/download.
- Student upload rejection.
- Full Playwright Stage 3 gate.
- Backend endpoint changes.
- Transcript upload.
- AI, embeddings, summaries.
- Section create/delete/reorder.
- Storage cleanup changes.

## Required behavior

### 1. Publish/unpublish control
Each lecturer section must show a control:

```text
If section is draft/unpublished -> show "Publish <section title>"
If section is published -> show "Unpublish <section title>"
```

Accessible name must include the section title, for example `Publish Lecture 1` and `Unpublish Lecture 1`. Do not use an unlabeled icon-only toggle.

### 2. Publish flow
When lecturer clicks publish:
- call backend publish endpoint through wrapper/generated client
- show pending state
- on success: re-fetch module/section data
- render `publishStatus = published`
- keep asset `processingStatus` unchanged

Do not update only local state and call that success.

### 3. Unpublish flow
When lecturer clicks unpublish:
- call backend unpublish endpoint through wrapper/generated client
- show pending state
- on success: re-fetch module/section data
- render backend `publishStatus` value (`unpublished` or `draft`, depending on backend semantics)
- keep asset `processingStatus` unchanged

Use the actual backend status value. Do not invent a frontend-only state.

### 4. Two separate status elements
`processingStatus` belongs to the asset. `publishStatus` belongs to the section. They must render separately.

Required DOM intent:

```text
section-publish-status-{sectionKey}
section-asset-processing-status-{assetId}
```

Do not use one generic `Status: ready` badge. The content slice explicitly separates technical file processing state from student visibility state.

### 5. Required visible status scenario
Checkpoint C must prove these states:

```text
uploaded PDF asset:
  processingStatus = completed

section:
  publishStatus = draft
```

After publish:

```text
asset processingStatus = completed
section publishStatus = published
```

After unpublish:

```text
asset processingStatus = completed
section publishStatus = unpublished/draft
```

Publishing controls visibility. It does not reprocess the file.

## UI requirements
Expected component direction, adapted to existing structure:

```text
frontend/src/features/content/lecturer/SectionPublishControl.tsx
frontend/src/features/content/lecturer/SectionStatusBadges.tsx
frontend/src/features/content/lecturer/LecturerModuleDetail.tsx
```

Recommended ownership:
- `SectionPublishControl` renders section-level `publishStatus` control.
- `SectionAssetRow` continues to render asset-level `processingStatus`.

A shared badge component is allowed, but ownership must not blur.

## API / wrapper requirements
Use existing wrapper/generated client methods. Expected wrapper methods:

```ts
api.content.publishSection(moduleId, sectionId)
api.content.unpublishSection(moduleId, sectionId)
```

If wrapper methods do not exist but generated client methods do, expose them through the wrapper. Do not call `fetch()` from components. Direct fetch remains allowed only in generated API core and `upload.ts`.

## Error handling
All publish/unpublish failures must render `role="alert"`.

Minimum handled cases:
- 403 forbidden
- 401 unauthenticated
- network failure
- backend validation failure

For `403`, preserve the session and surface the error. Do not redirect to login unless it is actually `401`.
