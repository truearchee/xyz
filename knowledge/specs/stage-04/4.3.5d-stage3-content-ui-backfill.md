## Why this session exists

Stage 3 backend (`module_sections`, `section_assets`, PDF upload/replace, storage abstraction, signed URLs, publish/unpublish, student published-only reads) was built and is marked BACKEND VERIFIED → UI PENDING. The raw multipart upload and signed-URL consumption were already proven in 4.3.5a against the tracer. This session wires the real lecturer content-management UI and the real student published-content view, and — when its browser gate passes — flips **Stage 3 from UI PENDING to FULLY VERIFIED.**

The Stage 3 UI proof obligation:

> A lecturer uploads a real PDF from the browser and publishes it; a student in a *separate* session opens that PDF — and cannot see any unpublished section.

## The Stage 3 nuance that governs this whole session

`processingStatus` and `publishStatus` are **two distinct fields** and must **never** be merged in the UI:

```
processingStatus = technical file processing state   (uploaded | processing | completed | failed)
publishStatus    = student visibility state          (draft | published | unpublished)
```

A file can be `completed` (fully processed) while its section is still `draft` (hidden from students). In the MVP there is no background pipeline yet, so `processingStatus` is set to `completed` immediately on upload — but the UI must still render the two states **separately**. Showing one merged "status" badge is a spec violation, not a simplification. The browser gate explicitly asserts both states render distinctly.

## Thin, not fake — but real and keepable

This UI ships. Acceptable: plain forms, ugly tables, basic status labels, raw-but-real data. Not acceptable: mocked state, fake upload success, hardcoded section lists. Every value rendered comes from a real backend response through the wrapper (or the approved upload helper for multipart).

## Checkpoint 0 — Contract investigation (report before building anything)

The entire UI design depends on the Stage 3 backend surface. **Read the generated client and backend source — do not assume.** Report all findings before writing any UI.

### 0.1 Section auto-generation — where do sections come from?

Slice 1 states sections are auto-generated from the course schedule at module creation; lecturers cannot create, delete, or reorder them. Confirm empirically:
- Does creating a module (the 4.3.5c flow) auto-generate `module_sections`?
- If yes: what drives it (schedule fields, lecture/lab/assignment/supplementary counts)? What fields does each section carry (`id`, `title`, `type`, `order`, `publishStatus`, `lecturerNotes`)?
- If no auto-generation exists: this is a **hard blocker.** Lecturers cannot create sections (MVP forbids it), so there is nothing to attach content to. Stop and escalate — a backend section-generation path is needed before 4.3.5d can proceed. Do not work around it by adding a lecturer "create section" UI.

**The E2E fixture must not directly seed sections as a substitute for the product path.** Directly seeding sections would bypass the auto-generation mechanism and prove nothing about the real content structure. If auto-generation is missing, log the finding, name the required backend session, and stop. Stage 3 cannot be marked FULLY VERIFIED without real section auto-generation. A tech-lead decision to accept a direct-seed workaround must be explicitly recorded in `findings-4.3.5d.md` as a deferral, and in that case Stage 3 remains UI PENDING until the named backend session closes.

### 0.2 Map the full Stage 3 backend write-path surface

Enumerate the generated client methods and confirm each of these exists:
- Upload PDF to a section (already exercised in 4.3.5a via `upload.ts`)
- Replace PDF in a section
- Update lecturer notes
- Publish a section
- Unpublish a section

**If any write-path endpoint is missing** (publish, unpublish, replace, notes), this is a **hard blocker** — not a read-only projection like 4.3.5c. The recovery block's narrow backend allowance covers read-only projections only. A missing write-path means the Stage 3 backend is not actually complete. Stop and escalate:
```
A missing Stage 3 write-path endpoint cannot be added under the recovery block.
It requires a dedicated Stage 3 backend session before 4.3.5d resumes.
Log the finding, name the required session, and stop.
```

### 0.3 Section read contracts — lecturer AND student

Two distinct read paths must be confirmed; they have different visibility:

**Lecturer read path** (Checkpoint A depends on this):
- How does an assigned lecturer fetch ALL sections of a module, including drafts/unpublished?
- Confirm the response includes `publishStatus` and the asset's `processingStatus` so the lecturer UI can render both states and the publish controls.
- If no lecturer "all sections" read path exists, this is a gap — flag it. (It is a read path, so the recovery-block read-only-projection allowance may apply with tech-lead approval; it is not a write-path hard blocker.)

**Student read path** (Checkpoint D depends on this):
- Is it `/modules/{id}/sections` filtered server-side to published, or a separate student endpoint?
- Confirm draft/unpublished sections are **excluded server-side** — the UI must not be the only thing hiding them.
- Confirm lecturer notes are included on published sections.

### 0.4 Signed-URL contract

Confirm the endpoint that issues a signed URL for an asset (reuse the 4.3.5a mechanism and ADR-022 origin rewrite). Confirm:
- Signed URLs are time-limited.
- The student-facing read path issues signed URLs only for assets in published sections of assigned modules.

### 0.5 The two status fields in the DTOs

Confirm `processingStatus` (on the asset) and `publishStatus` (on the section) are **separate fields** in the generated client models. Confirm `processingStatus` is set to `completed` immediately on upload in the MVP (no pipeline). If they are merged into one field anywhere, flag it as a finding.

### 0.6 Report before proceeding

Stop after 0.1–0.5. Report: section auto-generation mechanism, full write-path surface (or hard-blocker gap), lecturer and student read contracts, signed-URL contract, the two status fields. Do not begin Checkpoint A until the report is filed and any hard blocker is resolved by a human decision.

## Scope boundary

**In scope — real, keepable, thin product UI:**

Lecturer module-detail view (`/lecturer/modules/{id}` or equivalent):
- Open an assigned module → see its auto-generated section structure (read-only structure; lecturer fills, never creates/deletes/reorders)
- Upload PDF to a section (real multipart via `upload.ts`)
- Replace PDF in a section (real multipart via `upload.ts`)
- Add/edit lecturer notes on a section (JSON via wrapper)
- Publish / unpublish a section (JSON via wrapper)
- Render `processingStatus` AND `publishStatus` as two distinct labels

Student module-detail view (`/student/modules/{id}` or equivalent):
- Open an assigned module → see ONLY published sections
- Draft/unpublished sections hidden (and absent from the server response)
- See lecturer notes on published sections
- Open/download a PDF via signed URL
- No upload UI (and backend rejects student uploads)

**Not in scope:**
- Transcript upload / parsing / summaries UI (4.3.5e and Stage 4.4+)
- AI processing, the real background pipeline (processingStatus is immediate-completed in MVP)
- Rich-text editor, file version history, PDF annotation (Stage 3 exclusions)
- Lecturer-created sections, section reordering, section deletion (MVP forbids — do not build, even if it seems convenient)
- Lecturer-created custom sections (Stage 3 exclusion)
- Deleting `/tracer` (4.3.5e only)
- Any backend write-path endpoint — a gap is a hard blocker, not a silent addition

## Frontend file structure

```
frontend/src/
  app/(app)/
    lecturer/modules/[moduleId]/page.tsx    # thin; renders LecturerModuleDetail
    student/modules/[moduleId]/page.tsx     # thin; renders StudentModuleDetail
  features/
    content/
      lecturer/
        LecturerModuleDetail.tsx            # section list + per-section controls
        SectionUploadControl.tsx            # ADD a new asset to a section (multipart via upload.ts)
        SectionAssetList.tsx                # renders the section's assets
        SectionAssetRow.tsx                 # one asset: filename + processingStatus + asset-level Replace (upload.ts)
        SectionNotesEditor.tsx              # notes via wrapper
        SectionPublishControl.tsx           # publish/unpublish + status display
        SectionStatusBadges.tsx             # the TWO distinct badges (processingStatus + publishStatus)
      student/
        StudentModuleDetail.tsx             # published-only section list
        StudentSectionView.tsx              # notes + signed-URL open
  lib/
    api/upload.ts                           # existing multipart helper (the fetch exception)
    api/wrapper.ts                          # unchanged
```

Upload (add new asset, section-level) and replace (asset-level, on `SectionAssetRow`) both use `upload.ts`. Everything else (notes, publish, unpublish, section reads, signed-URL request) goes through the generated client wrapper. No other direct `fetch()`.

## Playwright selectors + slug rules

Primary selectors are accessible labels and visible button text. Add `data-testid` where text would be brittle:

```
data-testid="lecturer-section-list"
data-testid="lecturer-section-row-{sectionKey}"
data-testid="section-upload-control-{sectionKey}"
data-testid="section-publish-toggle-{sectionKey}"
data-testid="section-processing-status-{sectionKey}"    # processingStatus badge
data-testid="section-publish-status-{sectionKey}"        # publishStatus badge — SEPARATE element
data-testid="section-notes-editor-{sectionKey}"
data-testid="section-asset-row-{assetId}"                # per-asset row
data-testid="section-asset-replace-{assetId}"            # per-asset replace control
data-testid="student-section-list"
data-testid="student-section-row-{sectionKey}"
data-testid="student-asset-open-{assetId}"
```

The two status badges must be **separate DOM elements** with separate test-ids — the gate asserts both independently.

Slug rule (consistent in component and test):
```
sectionKey  = section.order + "-" + slugify(section.title) + "-" + section.id.slice(0, 8)
              e.g. "01-lecture-3-requirements-engineering-a1b2c3d4"
```

Section test-ids use `sectionKey` rather than title alone — titles can duplicate, change, or contain unexpected characters. Stability beats readability in selectors. (In the assertions below, `{s1}`/`{s2}` are shorthand for section 1's and section 2's `sectionKey`.)

## Error rendering requirement

All mutation failures render a visible `role="alert"` message using the backend error where safe. Upload/replace show a submitting/uploading state. After a successful mutation, the UI re-fetches from the backend — no mutation is considered successful from local state alone. An upload that the backend rejects (wrong type, too large, unauthorized) must render a clear error, not a fake success.

---

## Locked behaviors (inherited, do not deviate)

- **Role from `GET /me`, never JWT claims.**
- **401 vs 403 split:** 401 → `signOut()` + `/login`; 403 → surface to caller, session kept.
- **Token per request.**
- **Product flow via real UI; auth-edge probes via the bridge only.**
- **All API calls through the wrapper, except multipart upload/replace via `upload.ts`.** No other direct `fetch()`.
- **Backend remains the visibility authority.** The student UI hides drafts, but the backend must also exclude them — proven by the server response containing no draft sections.

## Checkpoint E — The Stage 3 browser gate (Playwright)

**This gate, when green, flips Stage 3 to FULLY VERIFIED.** Stage 3 cannot be marked FULLY VERIFIED while `findings-4.3.5d.md` has any unresolved finding (four-state resolution rule).

### E2E fixture strategy

Stage 3's proof obligation is the lecturer upload→publish and student view flow — **not** the admin creation flow (that is Stage 2, already verified). Fixture setup may seed the lecturer, the student, memberships, and any required auth records **directly via backend/DB** (setup is not the thing being proven). The module must be created through the confirmed backend path that generates sections — sections must not be inserted directly. If a tech-lead-approved finding explicitly defers section auto-generation, Stage 3 remains UI PENDING. The product proof is the content flow through real UI.

**Run-scoped identities** (use the approved E2E domain — `.test` is rejected by `EmailStr`, per F-001):
```
d_lecturer_{runId}@xyz-lms-e2e.dev
d_student_{runId}@xyz-lms-e2e.dev
module title: "Module D {runId}"
```

- The seeded module must have **at least two sections** (one to publish, one to leave draft — so "student does not see draft" is provable, not trivial).
- Section auto-generation must have been confirmed in Checkpoint 0.1. The fixture creates the module through the backend so sections are generated the real way. Direct section seeding is not an allowed substitute (see Checkpoint 0.1).
- `runId` generated before cleanup; cleanup deletes only identities matching `runId` and the `d_` prefix; removes Supabase Auth users and app DB records.
- **Storage cleanup must delete the objects by their actual backend-assigned keys.** Uploads in this session go through the real product UI, so the **backend** (StorageProvider, Session 3.1) assigns the storage key — the test does not control it and it will not contain `runId`. Read each created asset's `storageKey`/`fileUrl` from its DB record (before deleting the DB records) and delete those specific storage objects. Do **not** rely on an `e2e/{runId}/` prefix for UI-uploaded files, and do **not** attempt to force `runId` into the storage key (that would be an out-of-scope backend change). If a prior run's storage objects cannot be enumerated, scope storage deletion to the run's created module/section/asset records.
- Cleanup may use direct Supabase Admin API, DB, and storage operations — product proof through UI, setup/teardown direct.
- A small real PDF fixture lives in the tests directory for the upload.

**Isolation:** lecturer and student in **separate browser contexts**.

### Assertions

```
1. Lecturer logs in → opens the assigned module → sees its section structure
   (both sections visible to the lecturer regardless of publish state)

1b. Lecturer adds notes to section 1 THROUGH THE UI (notes editor, save via wrapper)
    → notes persist after re-fetch of the section

2. Lecturer uploads a real PDF to section 1 THROUGH THE UI (real multipart)
   → after upload, section 1 shows asset file name
   → section-processing-status-{s1} shows "completed"
   → section-publish-status-{s1} shows "draft"
   → BOTH badges are present as separate elements (the two-state proof)

3. Lecturer publishes section 1 THROUGH THE UI
   → section-publish-status-{s1} shows "published"
   → section-processing-status-{s1} still shows "completed"

4. Lecturer adds a note (or uploads a PDF) to section 2 but leaves it UNPUBLISHED (draft)
   → giving section 2 real content while keeping it draft proves it is publishStatus — not emptiness —
     that hides it from the student in assertion 6

5. Lecturer replaces section 1's uploaded asset THROUGH THE UI (asset-level Replace, real multipart)
   → the asset row shows the new file; its processingStatus is completed; the section's publishStatus
     is unchanged (still published)

6. [separate context] Student logs in → opens the assigned module
   → sees section 1 (published)
   → does NOT see section 2 (draft)
   → student sees the lecturer notes saved in assertion 1b on section 1
   → server response authority: use `page.waitForResponse()` for the student section-list request,
     parse the JSON body, and assert the draft section's id is absent from the response payload.
     DOM absence alone is insufficient — the backend must be the authority, not the UI filter.

7. Student opens section 1's PDF via signed URL
   → the UI requests the signed URL through the wrapper (assert the request succeeds)
   → verify the signed URL resolves using a Playwright APIRequestContext GET on the returned URL
     (test-level request, not app fetch()) → assert 200 and a PDF content-type
   → the ADR-022 origin rewrite must make the URL reachable from the test/browser in E2E

8. Lecturer unpublishes section 1 THROUGH THE UI
   → student reloads → section 1 no longer visible (assert via waitForResponse + JSON, same as assertion 6)
   → student attempts to request a NEW signed URL for the asset via the backend → request is rejected (403)
     (do not assert that a previously issued signed URL expires immediately — that is a storage TTL issue,
     not a product requirement; the product requirement is that the backend stops issuing new ones)

9. Student cannot upload
   → no upload control rendered in the student view (UI assertion)
   → server-authority probe: the E2E bridge has no upload method, so do NOT use it here.
     Obtain the student access token via window.__xyzE2E.getSession(), then issue a Playwright
     APIRequestContext POST (multipart) to the real upload endpoint with that bearer token.
     This is test-level code, not the app upload helper.
   → assert the response is **403** (not 401)
   → assert the student's Supabase session is still active afterward (getSession non-null; no /login redirect)
   → 401 here is a failure: an authenticated student receiving 401 indicates an auth model bug, not
     an authorization rejection. The roadmap's 401/403 split is explicit and must hold.

10. Invalid file upload rejection
    → capture the current asset IDs on section 1 before the attempt
    → lecturer attempts to upload a non-PDF file (e.g., a .txt file) to section 1 THROUGH THE UI
    → backend rejects it
    → UI renders role="alert"
    → re-fetch and assert asset IDs on section 1 are unchanged (same set, no addition)
      (comparing IDs is more precise than "no new asset appeared", especially after replacement)
```
