## Goal
The thin vertical UI slice that closes Stage 4.6: a lecturer can replace and retry a transcript on the
4.5d status surface (per-step states + sanitized reasons + a "new version processing" badge), backed by
the active-summary preview endpoint 4.6a deferred — and the browser gate proves it on real infra (a
crashed transcript retried to "Summaries ready"; a replacement swapping in without a content gap).

## Why now
4.6a/b/c are backend-verified. The roadmap's permanent rule: no backend slice merges without a thin UI
and a browser-verifiable gate. 4.6d is that gate for the whole of Stage 4.6.

## Build
- **Active-summary preview endpoint** (lecturer; over `ActiveTranscriptSummaryResolver`):
  `GET .../transcript-active-summary-preview` → `activeTranscriptId`, brief/detailed content, `briefEligible`,
  `detailedEligible`, `hasPendingReplacement`. NO student surface.
- **Lecturer UI** on the 4.5d surface: Replace button (file input + inline confirm; double-replacement
  warning when a pending exists), Retry button (when `failed` && `retryable`; targets the active transcript),
  per-step states strip + sanitized reason, "new version processing" badge (preview poll), swap-aware refresh.
- **Browser gate** (`tests/e2e/4.6d-replace-retry.spec.ts`): RETRY FLOW (forced step failure → lecturer
  retries → summarized, no duplicates; embed failure does not block summaries) + REPLACEMENT CONTINUITY
  (preview stays on the active while pending, flips to v2 on the atomic swap; old superseded with lineage).
  Fencing is the deterministic pytest companion (test_recovery.py).
- **Fix the cross-stage e2e breaks** rule 14 requires: 4.3.5e duplicate-upload (409 → 201 pending);
  db.mjs `is_active` → `lifecycle_state` + add `getTranscriptsBySection`.

## Do not build

- Cron for recovery → 11.1. Fuller pending-replacement progress/retry UI. Superseded-history / MaintenanceRun UI.
