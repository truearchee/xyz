## Goal
A fresh hosted DB goes from empty → schema-at-head → first admin + seed identities present, driven
by a **release-phase command** that runs `alembic upgrade head` over the DIRECT connection then an
idempotent identity bootstrap, **aborts the deploy on any non-zero exit**, and **gates every process
group**. After it, `/health/ready` is green (`alembic_version == head`) and all three workers process
jobs — the **first ready boot**. Plus a repeatable, committed deploy script + runbook with rollback.

## Why now
4.8a left readiness legitimately red (no migration had run). 4.8b is the umbrella's second
sub-session (§10): the one step that turns the provisioned-but-empty topology into a working
deployment. It is the first time the dual-URL wiring, the extension bootstrap, and a fresh-DB
identity bootstrap are exercised against real hosted infra — the things that "work on the laptop"
precisely because the laptop DB is never empty.

## Reconciliations (evidence in context)
1. **`create_user()` forbids `admin`** — `backend/app/domains/admin/service.py:81-82`
   (`if payload.role == "admin": raise "Admin users cannot be created"`). ⇒ the first-admin bootstrap
   is a **dedicated path**, NOT the admin API. The sanctioned pattern already exists in the e2e seed:
   ensure the Supabase auth user, then **upsert `app_users` with `role='admin'`** (`seed.mjs:159,191-197`,
   `ON CONFLICT (email) DO UPDATE`). 4.8b mirrors that in Python.
2. **No new migration needed for the extension** — `CREATE EXTENSION IF NOT EXISTS vector` already
   runs in `alembic/versions/0006:30` + `0007:115` (and `docker/postgres/init/001-create-vector.sql`
   locally). ⇒ F006's residual gap is only the Supabase **migration-role privilege** to create it
   over the direct connection — verified on the first deploy; SQL-console fallback documented. No new
   migration, no pgcrypto (F-4.8a-4).
3. **Alembic reads `DATABASE_URL`** — `alembic/env.py:19-20` (`os.environ.get("DATABASE_URL", …)`).
   ⇒ it must prefer `DIRECT_DATABASE_URL` so migrations (DDL, advisory locks) run over the session
   endpoint, not the transaction pooler (adr-041, F-4.8a-3).
4. **Reusable bootstrap primitives** — `get_supabase_admin_client()` (service-role `AsyncClient`,
   `supabase_client.py`) + `supabase.auth.admin.create_user({email, password, email_confirm: True})`
   (CONFIRMED users) + the `AppUser` model. The e2e `ensureAuthUser` (`seed.mjs:138-160`) is the
   idempotency reference (list/find-or-create).

## Build
- **B1 — Alembic over the DIRECT connection.** `env.py::get_database_url()` returns
  `DIRECT_DATABASE_URL or DATABASE_URL or <ini url>`. Local single-URL unchanged.
- **B2 — Extension bootstrap = verification + fallback (no new migration).** Confirm the existing
  `CREATE EXTENSION IF NOT EXISTS vector` succeeds over the Supabase direct role on the first deploy;
  if the role lacks privilege, the runbook documents a one-time `CREATE EXTENSION vector` via the
  Supabase SQL console (then the migration `IF NOT EXISTS` no-ops). Close F006 on this.
- **B3 — Idempotent first-admin + seed-identity bootstrap** (`app/cli/bootstrap_identities.py`,
  runnable `python -m app.cli.bootstrap_identities`). For each configured identity
  (admin / lecturer / student): ensure a CONFIRMED Supabase auth user via the service role
  (create-or-fetch, never duplicate), then **upsert** the `AppUser` row (admin via direct insert —
  the API guard is bypassed only here, the sanctioned bootstrap path). **USERS ONLY — no module
  memberships** (created in-smoke, §9 / prereq 4). Fresh-DB-safe and safe to re-run every deploy.
  Secrets (passwords) read from env, **never logged**.
- **B4 — Staging gate on the seed identities.** The bootstrap creates known-credential lecturer/
  student users — it MUST refuse to auto-create those outside a non-prod/staging env (a future prod
  must never get known-password users). Gate via an explicit flag + the `ENVIRONMENT`/`IS_NON_PROD`
  predicate (see Open question O1). The first ADMIN (real secret password) is the one identity that
  could be prod-safe; for 4.8 (staging only) all three run.
- **B5 — Release wrapper** (`scripts/release.sh`): `set -euo pipefail; alembic upgrade head;
  python -m app.cli.bootstrap_identities`. One command, non-zero exit aborts the release.
- **B6 — fly.toml release_command.** Uncomment on the **backend** app → `scripts/release.sh`; confirm
  Fly aborts the deploy on non-zero and that api + all three worker groups start only after it
  succeeds (start-ordering; workers crashloop against a schema-less DB if they precede it). Frontend
  app has no release command.
- **B7 — Repeatable deploy script** (`scripts/deploy-staging.sh`): `check-staging-env` gate → build +
  push immutable-tagged/digest-pinned images → `fly deploy` both apps (backend runs the release) →
  poll `/health/ready` until 200. Re-runnable from a clean commit.
- **B8 — Committed runbook** (`deploy/staging-runbook.md`): ordered steps; start-ordering; the
  extension-privilege fallback; the seed-identity env-gate; **rollback** (app = redeploy the prior
  immutable tag; DB = forward-fix, migrations backward-compatible one version where feasible).

## Do not build
- The **actual hosted deploy execution** and the first-ready-boot verification → **human handoff**
  ("you provision, I build"): the developer runs `deploy-staging.sh`, returns `/health/ready` + the
  workers-process evidence for the report.
- `/internal/sse-probe`, staging origin into `CORS_ORIGINS`, the §8 proof harness → **4.8c**.
- The hook-free staging smoke, module memberships, the rich fixture → **4.8d**.
- `allow_credentials` removal → **4.9**. Any new schema/migration or pgcrypto (F-4.8a-4).

## Data model changes
None. No new migration — the `vector` extension `CREATE` already exists (0006/0007). The bootstrap
only INSERTs/UPSERTs `app_users` rows via the existing model.

## API changes
None. The bootstrap is a CLI run in the release phase, not an endpoint. The admin API keeps its
"admins cannot be created" guard; the bootstrap is the sole sanctioned admin-creation path.

## Worker / job changes
None at runtime. Workers gain only the property that they start **after** the release gate.

## Authz rules
- The identity bootstrap requires the **service-role key** and runs only in the release phase, gated
  to non-prod/staging (B4). It is the one place an `admin` row is created (the API forbids it).

## Verification
**Local (I run):**
- `pytest` holds 409+ and adds: bootstrap idempotency (run the bootstrap twice against local Supabase
  + DB → admin/lecturer/student present, CONFIRMED, `role='admin'` on the admin, no duplicates,
  exit 0 both times; second run is a no-op); `env.py` prefers `DIRECT_DATABASE_URL` (unit); the
  staging-gate refuses seed identities when `ENVIRONMENT` is prod-like.
- `bash scripts/release.sh` against the local stack: `alembic upgrade head` (no-op at head) +
  bootstrap → exit 0; passwords absent from logs.
- `scripts/deploy-staging.sh --dry-run` (or a `check-staging-env`-only path) exits non-zero on a bad
  env, 0 on good.
- Full active Playwright suite still 11/11 (the bootstrap/env.py changes are deploy-time; no runtime
  regression).

**Hosted (developer runs; fed into the report):**
- First deploy: `release_command` runs, migration + bootstrap succeed, deploy promotes.
- `/health/ready` → **200**, `alembic_version == head`; all three workers process a job.
- A forced non-zero release (e.g. a deliberately bad migration on a throwaway run) **aborts** the deploy.

  - **O1:** explicit env triples (`BOOTSTRAP_{ADMIN,LECTURER,STUDENT}_EMAIL/PASSWORD/NAME`) + a

  - **MF1 (critical) — release-abort PROOF, not assertion.** (a) Local rehearsal
    (`test_release_abort.py`): prove `release.sh`'s `set -e` propagates a non-zero `alembic` exit AND
    that the bootstrap is gated behind migrate success — a fake `alembic` exits non-zero, assert
    `release.sh` returncode≠0 and the bootstrap step **never ran**; plus a real isolated poison revision
    (temp script_location) → `alembic upgrade head` exits non-zero, `alembic_version` unchanged.
    (b) **Hosted (developer, both directions, captured):** deploy with a poison migration → release
    phase exits non-zero, the **new version never serves** (old stays up / nothing promoted),
    `/health/ready` never green on the new release → revert → clean deploy completes. **Reported FIRST.**
  - **MF2 — pooler verifications are now 4.8b-discoverable** (a real pooler exists). Developer-run
    pass/fail the moment the staging DB is up: (a) the reaper advisory lock survives a
    connection-handback cycle on the real pooler; (b) `statement_cache_size=0` actually suppresses the
    asyncpg prepared-statement error on the real pooler. Re-classify F-4.8a-2/3 from 4.8d-provable →
    4.8b-discoverable.
  - **MF3 — connection-budget integers + runbook table.** Set small per-process pool defaults
    (api=5, each worker=2, overflow=0; direct = migrator 1 transient during release + reaper lock 1
    transient, singleton-serialized); record the budget table (api + 3 workers + migrator + reaper vs
    the plan `max_connections`, accounting for the pooler multiplexing app connections) in the runbook
    (failure-catalog #4).
  - **MF4 — "workers process" signal defined.** First-ready-boot floor: all three Fly process groups
    `Up` **AND** registered in the RQ worker registry (ingestion/embedding/ai each ≥1). `/health/ready`
    proves only the API. A functional pipeline round-trip stays 4.8d; "booted + registered" is the floor.
    `deploy-staging.sh` includes an RQ-registry check; the runbook gives the `fly ssh … rq info` fallback.
  - **MF5 (confirm) — immutable artifact.** `deploy-staging.sh` builds+pushes once, captures the image
    **digest**, and deploys that digest (`fly deploy --image …@sha256:…`) — never a rebuild/re-resolved
    tag — so the deployed artifact is the verified one. Confirmed in the report.
  - **MF6 (confirm + tighten) — idempotency = never rotate an existing user's password.** The ensure
    step sets the password only on CREATE; an existing user is fetched, password untouched (asserted:
    run twice with a changed password var → existing secret unchanged — distinct from "no duplicate
    rows"). The **seed gate lives in the bootstrap CLI** (env-driven refusal), not only in
    `check-staging-env` (which the operator can skip).

- 2026-06-12 — **DUAL-GATE PREDICATE DISTINCTION (do not collapse).** 4.8b deliberately uses TWO
  different environment predicates that LOOK similar but guard DIFFERENT risks. A future "cleanup" that
  merges them onto one predicate reintroduces a bug either way — this entry exists to prevent that.
  - **Seed-identity gate** = `BOOTSTRAP_SEED_IDENTITIES` **AND `not IS_PRODUCTION`**. The seeded
    lecturer/student MUST exist in **staging** (4.8d's smoke logs in as them) but must NEVER auto-exist
    in **production** (known-credential users). So this predicate **allows staging**, blocks only
    production.
  - **Fault-injection gate** = **`not IS_NON_PROD`** (`IS_NON_PROD` excludes `{production, staging}`).
    Fault hooks must be **refused in BOTH staging and production** — they may never fire in any hosted
    env.
  - They differ **precisely on staging**: seeds allowed there, fault hooks refused there. Collapsing
    onto `IS_NON_PROD` would block seeds in staging (breaks the smoke); collapsing onto `IS_PRODUCTION`
    would let fault hooks into staging (security hole). Keep `IS_PRODUCTION` and `IS_NON_PROD` as
    **distinct** predicates with distinct call sites.
