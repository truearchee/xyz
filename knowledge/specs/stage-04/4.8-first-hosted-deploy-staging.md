## 1. Goal

Take the walking skeleton from local-only to a repeatable **staging** deploy behind a public URL, and prove the full lecturer → student summary path runs there against real hosted infrastructure — managed Postgres + pgvector, Supabase auth/storage, Redis, **three** worker types, the real K2Think provider, and a release-phase migration.

This is a **constraint-discovery stage**, not a hardening stage. We pay to learn "what breaks when this leaves the laptop?" while the bill is small. The forcing function the roadmap names: **SSE dies under buffering proxies, and finding that out in Stage 8.3 is expensive.** We surface that and the other first-deploy constraints here (§4, §9).

**Non-goal, restated:** throwaway-grade staging. Autoscaling, CDN, observability stack, multi-region, CI/CD = Stage 12, excluded (§14). Every urge to make staging production-shaped is out of scope. *But "no observability stack" ≠ "no logs" — basic log access is in scope; you cannot debug a hosted gate failure without it (§7.I).*

---

## 2. The reframe that drives this spec

> **The staging browser gate is NOT the local E2E suite.**

The local Playwright suite authenticates via the `NEXT_PUBLIC_E2E_TEST_HOOKS` token-override path and drives failures through env-gated fault injection. Stage 4.8's headline security requirement is that **those hooks are absent from the hosted artifacts** (§8). Therefore:

- The hosted artifact has no `window.__xyzE2E` bridge and rejects all fault-injection requests. The local suite **cannot** run against it — by design.
- The staging gate is a **separate, hook-free smoke** (`4.8-staging-smoke.spec.ts`) that logs in as **real seeded Supabase users** (rule 9 allowlist) and exercises the real product flow — including membership assignment and publishing — using only surfaces the hosted build exposes. It is authored *without* the `__xyzE2E` helpers.
- Rule 14 (full active suite at every stage close) is still satisfied — it runs **locally**, unchanged, as the regression guard. It is not redirected at staging.

Two suites, two purposes. Conflating them is the trap; everything below assumes the split.

### Decision B — App database → **dedicated STAGING Supabase project** (Postgres + auth + storage)

- **Two DB URLs (§7.B):** app → transaction pooler (prepared statements off, `ssl=require`); Alembic → direct/session connection (DDL, advisory locks, multi-statement need it).

### Decision C — SSE proxy validation → **C1: ship the probe now** *(escape hatch: SSE-safe proxy → C2, accepted-with-rationale)*

- Tiny env-gated `/internal/sse-probe` (constraints in §7.C2), hit from the staging browser, **probed over the same transport the app uses** (Decision D), no compression on `text/event-stream`.

### Decision D — Browser→backend transport → **D1: direct cross-origin (browser → FastAPI)**

Deciding factor is 8.3: a same-origin Next proxy (D2) inserts a Node buffering hop **into the SSE path** we're de-risking. Pay the CORS + env-specific-build cost to keep `browser → FastAPI` clean. Baked URL handled by a staging-tagged frontend image. *(D2 viable only if you accept the Node hop and make C1 probe through the proxy.)*

## 5. Backend / infra scope (summary; details in §7)

- Full topology hosted: managed Postgres (Supabase), Redis, backend API, **three separate worker groups** — pipeline/default (parse, chunk), `embed` (isolated), `ai` (isolated) — and the frontend. Collapsing workers is not a faithful test.
- **Extension bootstrap:** idempotent Alembic migration `CREATE EXTENSION IF NOT EXISTS vector; …pgcrypto;` (closes F006); privilege caveat in §7.B. *Justify pgcrypto:* confirm a real consumer (e.g. `digest()`); if unused, don't bootstrap it.
- **Release-phase migration:** never on boot; one release step runs `alembic upgrade head` over the **direct** connection, gating all api/worker starts; deploy aborts on non-zero exit.
- **First-admin + seed-identity bootstrap:** idempotent, fresh-DB-safe (prereqs 3–4).
- **Secrets** in the platform store (matrix §7.H); never in images/repo; never logged at boot.
- **`scripts/check-staging-env`** (§7.H) — executable env matrix; required pre-deploy and pre-smoke gate.
- **Health:** `/health` (liveness, cheap, **unauthenticated**, **no DB/Redis dependency**) and `/health/ready` (DB + Redis reachable + `alembic_version == head`).
- **CORS** (D1): allow staging frontend origin, methods, headers incl. `Authorization`, `Content-Type`, `Idempotency-Key`. *Out of scope:* removing `allow_credentials=True` (4.9).
- **Environment hygiene (security, §8):** frontend E2E hook build-time absent; backend fault injection env-gated + **fail-boot-loud** if enabled outside test/e2e; SSE probe gated.
- **Repeatable deploy script + runbook**, committed: immutable-tagged images → release migration → rolling start; rollback note (§7.K).
- **(C1) `/internal/sse-probe`** — constrained per §7.C2.

## 7. Detailed infra requirements (the nuances)

### A. Networking & ingress
- **Upload path — confirm first; it flips the failure mode.** Through-FastAPI → proxy/app body-size default (~1 MB) **413s a 10 MB transcript in staging while passing locally** → raise the limit. Direct-to-Supabase via signed URL → failure moves to storage-bucket CORS + signed-URL reachability. Confirm which, configure accordingly.
- **Request timeouts:** confirm no API request exceeds the platform hard timeout (pipeline/AI are async via enqueue-after-commit, so OK; verify polling/readiness).
- **TLS / mixed-content:** backend must be https (D1).
- **noindex:** `X-Robots-Tag: noindex` / robots disallow on the public staging URL.
- **Edge access:** self-registration is closed (admins create users), so the closed-auth model already gates casual visitors. Optional edge basic-auth/IP allowlist; relying on closed-auth is defensible — state it, don't leave it implicit.

### B. Database & connections
- **Two URLs:** `DATABASE_URL` (pooler, `statement_cache_size=0`, unique `prepared_statement_name_func` or `NullPool`, `ssl=require`) for app+workers; `DIRECT_DATABASE_URL` (session/direct) for Alembic.
- **Connection budget — do the math.** API pool + 3 worker pools + migrator + reaper vs a small managed PG's `max_connections` (low tens, minus reserved). Cap per-process pools small; route app through the pooler; avoid direct connections from workers. Wrong → "remaining connection slots reserved" on warm-up.
- **Vector round-trip through the pooler:** confirm pgvector inserts/selects over the staging pooler with prepared statements off (and verify whether 4.4's `UserDefinedType`→pgvector-lib carry-forward landed; whichever is in tree must adapt over asyncpg+pooler).
- **Extension privilege:** migration role needs `CREATE EXTENSION` rights; if not, enable once via Supabase SQL console and document it. Prefer the migration path.
- **Migration safety:** single release instance; optional `pg_advisory_lock` against a rolling-deploy race.
- **Clock/timezone:** hosted PG is UTC; confirm the reaper age liveness + reconciliation grace window behave on the hosted clock with tz-aware timestamps.

### C. Redis & queues
1. **RQ needs the Redis protocol (TCP), persistent connections, blocking pops** (`BRPOP`). REST-only / heavily-metered serverless Redis is a poor fit. Choose a protocol Redis with persistent blocking connections.
2. **Eviction:** managed Redis often defaults to `allkeys-lru`, which can **silently evict RQ job hashes** → lost jobs. Set `noeviction` (limiter keys are TTL'd and safe; job data is not). Set sane `result_ttl`/failed-registry trimming.
3. **Enqueue-after-commit + hosting:** the app↔Redis network adds enqueue-failure surface (commit ok, enqueue blips → stuck row). The reaper is the net and **must run in staging** — this is why 4.6c recovery precedes deploy.

### D. Workers & memory
- **Three groups, separate scaling.** Pipeline + ai are light. **Embed worker carries torch + MiniLM (hundreds of MB resident)** — a 256 MB instance OOMs; size it larger. **CPU-only torch** in the image.
- **Singleton/startup hooks across replicas:** the reaper runs on startup + admin trigger, singleton-locked. With multiple replicas + rolling deploys, "startup" fires per replica — the singleton lock MUST be a cross-process (Redis) lock, not in-process. Hosting is the first multi-replica env; verify.
- **Warmth vs cold start:** scale-to-zero workers / sleeping PG hang the pipeline and fail the gate for infra reasons. Keep the three workers + PG warm for the smoke window. (4.5d backoff-polling / no-60s-hard-timeout badge tolerates hosted latency.)

### E. Auth / JWT / Supabase
- **JWKS + issuer are per-project.** Point JWKS URL / issuer at the **staging** project, cache the JWKS, ensure egress to Supabase auth. Staging-minted tokens won't validate against dev keys.
- **Confirm ECC P-256 asymmetric signing** on the staging project — the backend assumes it; new projects may not default to it.
- **Seed users via service role**, CONFIRMED, password sign-in on; Site URL / redirect allowlist set for the staging frontend.
- **Service role key backend-only**, never `NEXT_PUBLIC_`, never in the bundle.
- **Token refresh during a long smoke** (minutes): Supabase refresh must be transparent — the **401→signOut/redirect path must NOT fire on a normal expiry/refresh** (only on genuine auth failure). Verify headless.

### F. Frontend build / runtime
- **`NEXT_PUBLIC_*` inlined at build** (Decision D1): staging-tagged image with staging API URL + Supabase keys; `NEXT_PUBLIC_E2E_TEST_HOOKS` absent.
- **`output: 'standalone'`** for a small image; pin the Node base by digest.

### G. CORS (D1 only)
- Preflight for non-simple requests must pass: allow staging origin, `OPTIONS`, headers `Authorization`/`Content-Type`/`Idempotency-Key`. Moot under D2.

## 8. Security — hygiene proof (frontend build-time absence; backend runtime gate + fail-loud)

Most security-sensitive deliverable; precise per surface.

```
Frontend E2E hook:
  - build-time ABSENT from the hosted bundle.
  - Proof: window.__xyzE2E undefined in the staging browser; grep of the built
    bundle finds no token-override hook.

Backend fault injection (single image across envs; code may exist):
  - env flags ABSENT in staging.
  - the fault-injection route/action is UNAVAILABLE → request returns 404/403.
  - STARTUP FAILS LOUDLY if any fault-injection flag is enabled while APP_ENV is
    not test/e2e. Do not merely ignore a bad flag — refuse to boot.
  - Proof: a request that triggers fault injection in e2e returns 404/403 against
    staging; a boot with a fault flag set + APP_ENV=staging fails fast.

SSE probe: covered by §7.C2 (gated, admin-auth, absent in prod).
```
Runtime-toggle-without-fail-loud is one misconfiguration from shipping the override to a real user. The fail-loud boot check is the backstop.

> **Code-wins note:** `APP_ENV` → `ENVIRONMENT`; `IS_NON_PROD` already excludes `{production,staging}`
> (config.py:144–146), so "not test/e2e" is implemented as `not IS_NON_PROD`. The fail-loud boot
> check covers `PIPELINE_FAULT_INJECTION_ENABLED`/`PIPELINE_FAULT_INJECTION` and the LLM-transport
> fault double. This boot-time refusal is NEW (today the guard fires only at step-call time).

## 9. Browser gate (hook-free staging smoke)

`4.8-staging-smoke.spec.ts` — separate Playwright config (staging `baseURL`, no `webServer`, real Supabase creds from env, **no `__xyzE2E` helpers**), real seeded users, runId-scoped data:

```
Pre-flight: scripts/check-staging-env passes.

admin logs in (real Supabase)
→ creates smoke module (runId in title)        # auto-generates the section template
→ assigns seeded LECTURER to the module        # real admin assignment flow
→ assigns seeded STUDENT to the module
→ lecturer logs in
→ uploads the FIXTURE transcript to a lecture/lab section
→ pipeline completes on hosted workers (parse → chunk → embed → summaries)
→ ONE real K2Think summary; detailed summary passes the section validator (prereq 6)
→ lecturer PUBLISHES the section
→ student logs in
→ sees brief + detailed summaries
→ does NOT see the raw transcript (4.7 invariant)
→ /health and /health/ready green (incl. alembic_version == head)
→ window.__xyzE2E undefined; fault-injection endpoint returns 404/403 (§8)
→ (C1) SSE probe streams progressively, not buffered, over the app's transport
→ smoke cleans its e2e/{runId}/ STORAGE prefix (rule 9)
```

**Timeout budget:** hard cap **15 min**; poll with backoff, interval **≤10 s**; pipeline-not-complete-in-budget = hard failure (no infinite "still processing").

**Post-smoke app-data assertion** (via real status-projection / summary endpoints — not RQ internals): the transcript's derived `overallState` is terminal/`summarized` with **all steps completed and none failed/retryable**; both summary rows present and student-visible. *(Source-checksum integrity is enforced by 4.6 fencing; asserted here only if an endpoint exposes it — not faked through a hook.)*

**Real-provider consolidation (rule 11):** the smoke's brief (Cerebras/V2) + detailed (Nvidia/Think) calls **are** the Stage 4.8 real-provider smoke — both routes' model-ID echo recorded. **Do not run a second real-provider call** unless the first failed to capture model/provenance.

**DB cleanup policy — LOCKED: Option 2.** Smoke data is runId-scoped by module title/storage key; runs **accumulate intentionally** in staging. A run-scoped DB cleanup script (Option 1) or a periodic staging-reset (Option 3) is **deferred to Stage 12** unless staging gets noisy. Documented, not accidental.

Plus, **locally and unchanged**, the full active Playwright suite (rule 14) at the close commit.

## 11. Works-local / breaks-hosted failure catalog

| # | Symptom in staging | Root cause | Fix |
|---|---|---|---|
| 1 | 10 MB upload 413s | proxy/app body-size default (~1 MB) | raise limit (API path) / storage CORS (signed-URL) — §7.A |
| 2 | Frontend hits localhost | `NEXT_PUBLIC_*` inlined at build | staging-tagged build (D1) — §4 D |
| 3 | Migrations hang/error | run over transaction pooler | run over direct connection — §7.B |
| 4 | "remaining connection slots reserved" | api+3 workers+migrator > max_connections | small pools + pooler — §7.B |
| 5 | Queued jobs vanish | Redis `allkeys-lru` evicts job hashes | `noeviction` — §7.C |
| 6 | Workers can't dequeue | REST-only/metered serverless Redis | protocol Redis w/ blocking pops — §7.C |
| 7 | All tokens 401 | JWKS/issuer point at dev project | staging JWKS/issuer + ECC confirm — §7.E |
| 8 | Embed worker OOM / huge image | torch+model on tiny instance / CUDA wheel | size embed worker + CPU-only torch — §7.D |
| 9 | Concurrent reaps | in-process singleton across replicas | cross-process Redis lock — §7.D |
| 10 | Unexplained 429s | shared IFM key, per-deployment limiter | separate staging key — §7.J |
| 11 | Restart loops on a DB blip | liveness depends on DB/Redis | cheap liveness, strict readiness — §7.K |
| 12 | SSE never streams | proxy/compression buffering | SSE-safe proxy + no gzip on event-stream — §4 C |
| 13 | Pipeline hangs at gate | worker/PG scaled to zero | keep warm for smoke window — §7.D |
| 14 | Token expires mid-smoke → logout | 401 path fires on normal refresh | transparent refresh; 401 only on real fail — §7.E |
| 15 | Detailed summary rejected | fixture too thin for section validator | richer fixture — prereq 6 |
| 16 | Lecturer can't upload / student can't read | missing membership / unpublished section | assign + publish in-smoke — §9 |
| 17 | "Passed" but failed jobs linger | completion ≠ clean queue | post-smoke app-data assertion — §9 |

> **Failure #9 is already mitigated** (Filing note): the reaper lock is cross-process via Postgres
> `pg_try_advisory_lock`, not in-process. The 4.8 nuance is routing that lock connection over the
> **direct** URL (session-level advisory locks are unreliable through a transaction pooler).
