### 4. Parsers (pure functions — no DB, no IO — unit-testable in isolation)

**`timestamps.parse(ts: str) -> int`** → integer milliseconds. Handle `HH:MM:SS.mmm` and `MM:SS.mmm`. Reject malformed input. Reject end ≤ start at the call site (the handler checks the pair after both are parsed).

**`vtt.parse(raw: bytes) -> list[ParsedSegment]`**

Use `webvtt-py` for cue extraction — it handles BOM, `\r\n`, `NOTE`/`STYLE`/`REGION` blocks, cue settings after the timestamp arrow, and multi-line payloads. Join multi-line payloads with a single space.

**Implementation note on voice spans:** `webvtt-py` may expose cleaned cue text (tags stripped) or raw payload depending on the attribute accessed. Use the attribute that preserves `<v Name>` markup so the speaker extractor sees it. Add a unit test that proves `<v Dr Smith>Hello</v>` produces `speaker_name="Dr Smith"`, `text="Hello"` — this test will catch a wrong attribute choice immediately.

**Content sniff / routing — decode order matters:**

Routing happens before full strict decoding, so use a permissive decode for the sniff only:

```
# Step 1 — sniff (permissive, for routing only)
sniff_text = raw_bytes.decode("utf-8-sig", errors="replace")
first_line  = next((l.strip() for l in sniff_text.splitlines() if l.strip()), "")
is_vtt      = (first_line == "WEBVTT" or first_line.startswith("WEBVTT "))

# Step 2 — full decode (strict for VTT, permissive for TXT)
if is_vtt:
    text = raw_bytes.decode("utf-8-sig")        # strict — raises on bad bytes
else:
    text = raw_bytes.decode("utf-8-sig", errors="replace")   # forgiving fallback
```

If strict VTT decoding fails, fail the parse with a sanitized error — a structurally invalid VTT file is not recoverable. TXT is the fallback format and should tolerate a few bad bytes rather than fail entirely. Document this asymmetry in ADR-019. Use stored `mime_type` only as a secondary signal; sniff always wins.

**`txt.parse(raw: bytes) -> list[ParsedSegment]`**

Split on blank-line paragraphs. If no blank lines exist, split on non-empty lines. Trim each candidate; drop empties. `start_ms=end_ms=speaker_name=None`. Decode with `utf-8-sig`, `errors="replace"`.

**`speaker.extract(payload: str) -> tuple[str | None, str]`**

Two recognition tiers:

*Tier 1 — VTT `<v>` voice spans (unambiguous).* Extract from `<v Name>text</v>` or `<v Name>text`. Speaker is the voice span name; text is the payload with the span tags removed.

*Tier 2 — Zoom-style leading prefix (conditional).* Match `^[A-Z][A-Za-z .'\-]{0,39}:\s` at the start of the payload. Before extracting, check the candidate against an **educational-label stoplist**:

```python
STOPLIST = {
    # general educational labels
    "definition", "example", "note", "question", "answer",
    "important", "summary", "objective", "recall", "exercise",
    "task", "step", "warning", "hint",
    # maths / CS lecture labels
    "theorem", "proof", "lemma", "corollary", "remark",
    "intuition", "formula", "algorithm", "solution",
    "case", "observation",
}
```

Reject the match if `candidate.strip().lower()` is in the stoplist. Use `.strip().lower()`, not just `.lower()`, to avoid whitespace masking the match. Accept only when the candidate passes. On any doubt, return `(None, full_payload)`.

**The invariant:** when in doubt, preserve the text intact. A null speaker is recoverable. Stripped content (cross-cutting rule #6) is not.

### 5. Enqueue-on-upload
Extend the 4.1 upload service: after the `transcripts` row commits at `uploaded`, enqueue `parse_transcript(transcript_id)` via RQ. Then issue a **conditional status update**:

```sql
UPDATE transcripts
SET    status     = 'queued'
WHERE  id         = :transcript_id
  AND  status     = 'uploaded';
```

The conditional guard prevents the upload service from downgrading a transcript that the worker has already advanced to `parsing` or beyond. If enqueue fails (Redis unavailable), log the failure and do not set `queued` — the transcript stays at `uploaded`, which is the recoverable state the 4.6 sweeper will target.

### 6. Parse handler — `parse_transcript(transcript_id)`

**Treat RQ as at-least-once delivery.** The database idempotency key and job-claim logic are the source of truth, not RQ's execution guarantee.

**Phase 1 — Claim (short DB transaction)**
```
compute idempotency_key
INSERT INTO ingestion_jobs (...) VALUES (...)
ON CONFLICT (idempotency_key) DO NOTHING;

SELECT * FROM ingestion_jobs
WHERE  idempotency_key = :key
FOR UPDATE;

if job.status == 'completed':
    return  # already parsed this exact content — no-op

if job.status == 'running':
    return  # another worker is live on this job — no-op for now
    # 4.6 will add stale-running recovery

# status in (queued, failed): claim it
UPDATE ingestion_jobs
SET    status     = 'running',
       attempts   = attempts + 1,
       started_at = now(),
       updated_at = now()
WHERE  idempotency_key = :key;

# Store the attempt number as a claim token for Phase 3 and the failure path.
claimed_attempt = job.attempts + 1

UPDATE transcripts
SET    status = 'parsing'
WHERE  id     = :transcript_id;

COMMIT;
```

**Phase 2 — Parse (outside any DB transaction)**
```
raw_bytes  = storage.get_object(transcript.storage_key)
# raises StorageUnavailableError / StorageProviderError on failure
segments   = route_and_parse(raw_bytes)   # VTT or TXT
# zero segments → raise ParseError("no parsable content")
validate each segment (non-empty text, paired timestamps, end_ms > start_ms)
```

`sequence_number` is assigned **after** filtering skipped/empty cues, using `enumerate(segments)`. This guarantees persisted sequence numbers are always `0..n-1` with no gaps — the DB cannot enforce contiguity, so the service is the sole enforcement point.

**Phase 3 — Persist (short DB transaction)**
```
# Re-lock the job and verify the attempt token before writing anything.
SELECT * FROM ingestion_jobs
WHERE  idempotency_key = :key
FOR UPDATE;

if job.status != 'running' OR job.attempts != claimed_attempt:
    # A recovery worker has reclaimed this job while we were parsing.
    # Abort without touching transcript_segments or transcript status.
    ROLLBACK; return

DELETE FROM transcript_segments WHERE transcript_id = :id;
INSERT INTO transcript_segments (...) VALUES [...bulk...];
UPDATE ingestion_jobs
SET    status       = 'completed',
       completed_at = now(),
       updated_at   = now()
WHERE  idempotency_key = :key;
-- transcript.status remains 'parsing' — see D2 note below
COMMIT;
```

**Failure path (any exception in Phase 2 or 3)**
```
open short DB transaction

# Same attempt guard — do not overwrite a newer recovery attempt's state.
SELECT * FROM ingestion_jobs WHERE idempotency_key = :key FOR UPDATE;
if job.status != 'running' OR job.attempts != claimed_attempt:
    ROLLBACK; return

DELETE FROM transcript_segments WHERE transcript_id = :id;   -- ensure no partial write
UPDATE transcripts     SET status = 'failed'      WHERE id  = :transcript_id;
UPDATE ingestion_jobs  SET status = 'failed',
                           error_message = :sanitized_message,
                           updated_at    = now()
WHERE  idempotency_key = :key;
COMMIT;
```

`error_message` must not contain: storage keys, signed URLs, provider credentials, raw stack traces, **or raw transcript text / cue payload content**. Transcripts may contain personal information, student names, or class discussion. Worker logs must identify `transcript_id`, `job_id`, `job_type`, and a sanitized error reason — not transcript content.

**D2 — locked: rest at `parsing`.** A successfully-parsed transcript rests at `transcript.status = 'parsing'`. The authoritative parse-complete signal is `ingestion_jobs` where `job_type='parse'` and `status='completed'`. The canonical enum has no per-stage "done" marker between `parsing` and `chunking`; adding a transient `parsed` status would pollute the enum permanently or require an ADR-in then ADR-out within two sessions. Nothing in 4.2 or 4.3 reads `transcript.status='parsing'` as an action trigger — the IngestionJob ledger is the correctness surface. 4.3 wires the parse→chunk handoff and the transcript naturally advances to `chunking`.

## Do not build
- **No normalization.** No greeting/mic-check/admin-chatter removal. That is `TranscriptChunk.text` in 4.3. `transcript_segments.text` is parsed-from-raw and never mutated (cross-cutting rule #6).
- **No chunking, embeddings, summaries, or any K2Think call.** `AIRequestLog` is still 4.5's gate.
- **No retry/backoff, no replacement/supersession.** `attempts` and `error_message` exist; auto-retry wiring is 4.6. A crashed mid-parse transcript stuck at `parsing`/`running` is an accepted known gap until 4.6's recovery.
- **No new endpoints.** The existing 4.1 GET surfaces the new statuses. No segment-viewing API — students never see raw transcripts (Slice 2, permanent), and lecturers see status + summaries, not segments.
- **No frontend code.** Status strings already render; copy refinement is 4.7.
- **No second job type wired.** `ingestion_jobs` supports the full enum; only `parse` runs.
- **No `0004` edits.** Status index goes in `0005`.
- **No `processor_version` in the idempotency key.** A `processor_version` column is stored on `ingestion_jobs` as queryable metadata (see Data model), but it is not part of the key. Including it in the key means every parser bug-fix requires a coordinated version bump and re-queue decision, which is operational overhead with no live students yet. Manual recovery for MVP (delete completed job, re-enqueue) is sufficient. Record this deferral explicitly in ADR-019.

## Data model changes (migration `0005`)

### `transcript_segments`
```
- id              uuid        pk
- transcript_id   uuid        not null  FK -> transcripts(id) ON DELETE CASCADE
- sequence_number int         not null  -- 0-based, contiguous, gap-free, file order
- start_ms        bigint      null      -- integer milliseconds; null for TXT segments
- end_ms          bigint      null      -- integer milliseconds; null for TXT segments
- speaker_name    text        null      -- best-effort; null when unknown or ambiguous
- text            text        not null  -- parsed-from-raw; NEVER normalized (rule #6)
- created_at      timestamptz not null  default now()

UNIQUE  (transcript_id, sequence_number)   -- covers (transcript_id, …) prefix lookups
CHECK   (sequence_number >= 0)
CHECK   ((start_ms IS NULL AND end_ms IS NULL)
          OR (start_ms IS NOT NULL AND end_ms IS NOT NULL))
CHECK   (start_ms IS NULL OR start_ms >= 0)
CHECK   (end_ms   IS NULL OR end_ms > start_ms)
CHECK   (length(trim(text)) > 0)
```

The paired-null CHECK enforces the VTT-has-timestamps / TXT-has-none invariant at the DB layer. `end_ms > start_ms` prevents impossible timestamp ranges from persisting even if the parser validator misses them. The service validates first; the DB is the final guardrail. No separate `(transcript_id)` index needed: the unique constraint's leading column covers it.

Refines Slice 2's generic `startTime`/`endTime` into explicit integer-millisecond columns — see ADR-018.

### `ingestion_jobs`
```
- id                uuid        pk
- transcript_id     uuid        not null  FK -> transcripts(id) ON DELETE CASCADE
- job_type          text        not null  CHECK in (parse|chunk|embed|generate_brief_summary|generate_detailed_summary)
- status            text        not null  CHECK in (queued|running|completed|failed)  default 'queued'
- idempotency_key   text        not null
- processor_version text        null      -- e.g. "parse:v1"; stored for debugging/re-queue decisions, not part of key
- attempts          int         not null  default 0
- error_message     text        null      -- sanitized; no storage keys, credentials, or stack traces
- started_at        timestamptz null
- completed_at      timestamptz null
- updated_at        timestamptz not null  default now()  -- updated on every status transition; supports 4.6 staleness queries
- created_at        timestamptz not null  default now()

UNIQUE  (idempotency_key)
INDEX   (transcript_id, job_type)
```

`text + CHECK` per repo convention (4.1 P3). FK target `transcripts(id)` consistent with 4.1. `updated_at` is required: the 4.6 recovery sweeper will query `WHERE status='running' AND updated_at < now()-interval` to detect stale jobs; without it, stale detection has no anchor.

`processor_version` is `null`-allowed in this migration; 4.2's parse handler writes `"parse:v1"`. This gives a queryable audit trail without coupling the idempotency semantics to version strings.

> **D1 — locked.** `ingestion_jobs` is introduced in this session. Rationale: the first worker session is the correct home for the job ledger; retrofitting it across parse+chunk+embed+summary simultaneously in 4.6 creates the coupling-on-live-tables problem the roadmap repeatedly warns against. See ADR-017.

### `transcripts` status index (the 4.1-deferred one — P4)
```sql
CREATE INDEX ix_transcripts_status_created_at
ON transcripts (status, created_at);
```

A single-column `(status)` index is weak for recovery queries because status is low-cardinality. The composite `(status, created_at)` directly serves the 4.6 sweeper pattern: `WHERE status IN ('uploaded','queued','parsing') AND created_at < now()-interval`. A `(status, updated_at)` composite would be even better for staleness detection, but `transcripts` does not have `updated_at` yet — adding it is out of scope for 4.2. Upgrade path: add `updated_at` to `transcripts` in whichever session first needs staleness-by-update-time; at that point, replace this index.

### Idempotency key construction (cross-cutting rule #3)
```
parse / chunk / embed (non-AI):  "{job_type}:{transcript_id}:{source_checksum}"
AI jobs (4.5+):                  "{job_type}:{transcript_id}:{source_checksum}:{prompt_version}"
```

`source_checksum = transcripts.checksum` (stored at upload, P6). Replacement (4.6) changes the checksum → new key → legitimate re-parse without touching the old completed job's record. `prompt_version` is absent for non-AI jobs; `processor_version` is stored as a column but not in the key (see Do not build).

## Tests required

**Unit (pure parsers — no DB, no IO):**

*Timestamps:*
- `HH:MM:SS.mmm` and `MM:SS.mmm` → correct integer ms.
- Malformed string → raises.
- `end_ms <= start_ms` pair → validation rejects at call site.
- Negative timestamp → raises.

*VTT parser:*
- Well-formed Zoom VTT (cue number + `Name:` prefix) → correct sequence, timestamps, speaker.
- `<v Name>` voice spans → `speaker_name` set, tags removed from text; test `<v Dr Smith>Hello</v>` → `speaker_name="Dr Smith"`, `text="Hello"`.
- No speaker → `speaker_name=None`, full text preserved.
- Multi-line payload → joined with space.
- `NOTE` / `STYLE` / `REGION` blocks → ignored, not emitted as segments.
- BOM + `\r\n` line endings → parsed correctly.
- Cue settings after timestamp arrow (e.g., `align:start`) → ignored.
- Whitespace-only cue payload → segment skipped.
- Failed UTF-8 decoding → parse failure with sanitized error.

*TXT parser:*
- Blank-line paragraph split → multiple segments with null timestamps.
- Single line (no blank lines) → one segment.
- No trailing newline → handled cleanly.
- Multiple consecutive blank lines → collapsed, not creating empty segments.
- All-whitespace input → zero segments (which the handler treats as failure).
- Bad bytes → decoded with replacement characters, not a parse failure.

*Speaker extraction:*
- `<v Dr Smith>text` → `("Dr Smith", "text")`.
- `Dr Smith: Let's begin` → `("Dr Smith", "Let's begin")` (passes stoplist check).
- `Definition: A loss function…` → `(None, "Definition: A loss function…")` — **stoplist blocks extraction**.
- `Example: Suppose x = 2` → `(None, "Example: Suppose x = 2")`.
- `Question: Why does this happen?` → `(None, "Question: Why does this happen?")`.
- `Important: This will be on the exam.` → `(None, "Important: This will be on the exam.")`.
- `Note: We will revisit this later.` → `(None, "Note: We will revisit this later.")`.
- Sentence containing a mid-sentence colon, no capital-word prefix → `(None, full text)`.

*Routing / sniff:*
- `WEBVTT\n…` → VTT parser.
- `WEBVTT - Zoom transcript\n…` → VTT parser (startsWith match).
- BOM + `WEBVTT\n…` → VTT parser.
- Blank lines before `WEBVTT` → VTT parser.
- Plain text with no WEBVTT header → TXT parser.
- `mime_type` disagreeing with content → sniff wins.

**Integration (DB + fake storage + handler invoked directly):**

- Upload VTT → status `uploaded`→`queued`→`parsing`; segments persisted with correct order, timestamps (ms), speakers; `parse` IngestionJob `completed`.
- Upload TXT → segments with null `start_ms`/`end_ms`, null `speaker_name`.
- **Idempotency** (Slice 4 hardening test): invoke `parse_transcript` twice on the same transcript → identical segment set, stable row count, second invocation returns no-op (completed job short-circuits). No duplicate segments.
- **Sequence contiguity:** after any successful parse, `persisted sequence_numbers == list(range(len(segments)))` — no gaps, no skips, 0-based.
- Malformed VTT → transcript `failed`, job `failed` with non-empty `error_message`, zero segments.
- Zero usable segments (empty file) → `failed`, not silent success.
- **Concurrent claim:** two handler calls on the same job simultaneously → only one proceeds; second sees `running` and returns no-op. Segment count is correct after both return.
- **Storage read failure:** `fake_storage.get_object` raises → transcript `failed`, job `failed`, `error_message` is sanitized (no storage key, no URL, no credentials, **no transcript text**), zero segments persisted.
- **Stale worker guard:** Phase 3 executes after another worker has already incremented `attempts` beyond `claimed_attempt` → the stale worker aborts without writing segments or updating status.
- Conditional `queued` update does not downgrade a transcript already at `parsing`.
- Enqueue on failed upload does not occur (rolled-back DB transaction does not enqueue).
- Enqueue on successful upload enqueues exactly one parse job.
- DB-spine assertions: both new tables exist; all CHECK constraints enforced; unique constraints enforced; `ix_transcripts_status_created_at` index exists.

**Not in scope this session:** retry, backoff, replacement, supersession, AI, chunking — those are 4.3–4.6 surfaces.

## Done means
A lecturer-uploaded VTT lands `uploaded`→`queued`→`parsing` with correct, ordered `transcript_segments` (timestamps in ms, best-effort speakers, un-normalized text per rule #6) and a `completed` parse `IngestionJob`. A TXT upload produces segments with null timestamps. Re-running parse on the same content is a verifiable no-op (no duplicate segments; `completed` job short-circuits). Malformed or empty input lands `failed` with a sanitized `error_message` and zero persisted segments. Storage-read failure follows the same failure path. The conditional `queued` update cannot downgrade a transcript already advanced by the worker. Migration `0005` applies cleanly on `0004 (head)`. Full `pytest` green. `tsc --noEmit` green (no FE change, kept green).

## Risks introduced
- Enqueue-after-commit can leave a transcript stuck at `uploaded` if the process dies between commit and enqueue. A mid-parse crash that fails before Phase 3 leaves the transcript at `parsing` with the job at `running`. Both are accepted gaps until the 4.6 recovery sweeper; `ix_transcripts_status_created_at` is laid now to serve it.
- Best-effort speaker extraction may under-attribute (null) on unusual formats; this is the preferred failure mode over corrupting `text`.
- `webvtt-py` is a new dependency; pin the version. Confirm via unit tests that it exposes voice-span markup in the attribute used by the speaker extractor.
- The stoplist is not exhaustive. Lecturer speech that begins with an unlisted educational label may be mis-attributed. The stoplist now covers common ML/math terms (`theorem`, `proof`, `lemma`, `algorithm`, etc.) but is not exhaustive. Extend it as new patterns emerge; it is a plain constant in `speaker.py`, not buried in config.

## Locked decisions

- **D1 — locked:** `ingestion_jobs` introduced in 4.2 (first worker session). See ADR-017.
- **D2 — locked:** transcript rests at `parsing` after successful parse. `ingestion_jobs.status='completed'` for `job_type='parse'` is the authoritative parse-complete signal. See the D2 note in the Build section and ADR-017.
- **D3 — locked:** parse only; normalization is the chunk layer's responsibility. Cross-cutting rule #6 is non-negotiable.

**Manual reprocess procedure (before 4.6 auto-retry exists):** to reprocess a transcript after a parser bug fix, delete or set `failed` the completed `parse` ingestion_job for that transcript/checksum, then re-enqueue. The new job gets a fresh idempotency key because no `completed` job blocks it. No DB surgery beyond that one row.
