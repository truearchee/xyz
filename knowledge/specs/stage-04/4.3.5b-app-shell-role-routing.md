## Why this session exists

4.3.5a proved the raw spine: JWT acquire, cross-origin API calls, multipart upload, signed-URL consumption. But it proved this against one ugly `/tracer` page with manual buttons. There is still no **app**: no shell, no role-based routing, no route guards, and — critically — **token refresh on expiry has never been proven**.

This session builds the navigable frame and proves the two auth behaviors the tracer did not:
1. A token that expires gets **refreshed**, and subsequent calls use the rotated token.
2. A token that is genuinely invalid triggers **401 recovery** — session cleared, redirect to `/login`.

It also wires role routing so a logged-in user lands in the correct area based on their **`GET /me` role** — never on a JWT claim.

**This is still recovery, not product UI.** It builds the routing skeleton and a minimal shell. It does **not** build the admin, lecturer, or student dashboards — those are 4.3.5c, 4.3.5d, 4.3.5e.

## Locked auth model

```
Supabase browser login → access token → Authorization: Bearer <token>
→ backend validates JWT → GET /me returns role + memberships
→ SessionProvider exposes session state → guards read session state
```

**Role comes from `GET /me`, never from the JWT.** Enforce with a static check:
```bash
rg -n "jwt-decode|decodeJwt|app_metadata|user_metadata|session\.user.*role" frontend/src
# expected: no matches
```

**401 vs 403 split — the wrapper must enforce both sides:**
```
401 (unauthenticated / invalid token):
  wrapper calls supabase.auth.signOut()
  wrapper redirects to /login
  wrapper throws AuthRequiredError or equivalent

403 (authenticated but not permitted):
  wrapper does NOT call signOut()
  wrapper does NOT redirect to /login
  wrapper surfaces ForbiddenError / status=403 to caller
```

**Token retrieved per request** (from 4.3.5a) — do not regress.

## SessionProvider state machine

Replace or extend the existing `SessionProvider` with this explicit state type:

```ts
type SessionState =
  | { status: "loading" }
  | { status: "unauthenticated" }
  | { status: "authenticated"; user: MeResponse }
  | { status: "forbidden"; reason?: string; email?: string }
```

`email` in the `forbidden` state allows the AccessDenied UI to show who is logged into Supabase even though they have no app account.

**Transition rules:**

```
On mount:
  call supabase.auth.getSession()
  if no session          → "unauthenticated"
  if session exists      → call GET /me
    if /me 200           → "authenticated", user: response
    if /me 401           → supabase.auth.signOut() → "unauthenticated"
    if /me 403           → "forbidden", email: session.user.email (session kept)

On SIGNED_IN (supabase onAuthStateChange):
  call GET /me
    if /me 200           → "authenticated", user: response
    if /me 401           → supabase.auth.signOut() → "unauthenticated"
    if /me 403           → "forbidden", email: session.user.email

On TOKEN_REFRESHED (supabase onAuthStateChange):
  do not re-route; future API calls use the latest token via per-request retrieval

On SIGNED_OUT (supabase onAuthStateChange):
  → "unauthenticated"
  → redirect to /login
```

The `SIGNED_IN` transition is required. Without it, a successful Supabase login may not reload `/me` correctly until a full page reload — the login page calls `signInWithPassword`, Supabase fires `SIGNED_IN`, and the provider must respond.

**The guard must treat `"loading"` as a real state, not as "not logged in."** Show a neutral loading indicator. Do not redirect during loading.

## Role-prefix guard in ProtectedAppLayout

**Do not implement this in Next.js middleware.** Middleware runs before session state resolves and would require JWT claim decoding or a backend roundtrip — both violate the auth model. Keep it client-side, keyed off `SessionProvider` state from `/me`.

Implement directly in `ProtectedAppLayout`:

```ts
const roleRouteMap: Record<string, "admin" | "lecturer" | "student"> = {
  "/admin":    "admin",
  "/lecturer": "lecturer",
  "/student":  "student",
} as const
```

Use a segment-safe helper — `pathname.startsWith("/admin")` would also match `/administrator`:

```ts
function matchesRoutePrefix(pathname: string, prefix: string): boolean {
  return pathname === prefix || pathname.startsWith(`${prefix}/`)
}
```

Guard logic:
```
if pathname starts with /unauthorized:
  render children (exempt — prevents redirect loop)

for each prefix in roleRouteMap:
  if matchesRoutePrefix(pathname, prefix) AND user.role !== roleRouteMap[prefix]:
    replace /unauthorized
    return

render AppShell + children
```

**Do not call `router.replace()` during render.** Compute a derived boolean and redirect inside `useEffect`:

```ts
const shouldShowUnauthorized =
  session.status === "authenticated" &&
  isWrongRole(pathname, session.user.role)

useEffect(() => {
  if (shouldShowUnauthorized) router.replace("/unauthorized")
}, [shouldShowUnauthorized, router])

if (shouldShowUnauthorized) return <LoadingState />
```

This avoids React warnings and flaky route behavior from state mutations during render.

`/unauthorized` **must be exempt** from the role-prefix check, or a student navigating to `/admin` will be sent to `/unauthorized`, which re-triggers the guard, which sends them back to `/unauthorized` — an infinite loop.

**`/unauthorized` page** renders inside the AppShell (a valid `MeResponse` exists at this point) and shows a clear unauthorized message with a link to the user's correct role home.

## Forbidden bootstrap state (no AppShell)

When `GET /me` returns 403 during bootstrap, there is **no `MeResponse`** — the user exists in Supabase but has no app account. `AppShell` cannot safely render user identity or role nav.

Render a standalone `AccessDenied` component (no AppShell) that:
- Shows a clear "you don't have access to this application" message
- Shows the Supabase email if available from the `forbidden` state
- Provides a logout button (calls `supabase.auth.signOut()`)
- Does not attempt to render role nav or role-home links

If this state is reached in development, emit `console.warn` only — this is a client-side warning, not a backend call.

## E2E test hook bridge

Playwright cannot call your app's Supabase client or API wrapper unless you expose a test-only bridge. Without this, tests either use raw `fetch()` (bypasses the wrapper, defeats the assertions) or manipulate localStorage (brittle, as noted in 4.3.5a).

Create `frontend/src/lib/e2e/testHooks.ts`. Compile and register it **only** when `NEXT_PUBLIC_E2E_TEST_HOOKS=true`:

```ts
// Only loaded in E2E builds — gate at import site in layout.tsx or SessionProvider
if (
  process.env.NEXT_PUBLIC_E2E_TEST_HOOKS === "true" &&
  typeof window !== "undefined"
) {
  window.__xyzE2E = {
    refreshSession:     async () => supabase.auth.refreshSession(),
    getSession:         async () => supabase.auth.getSession(),
    forceNextBearerToken: (token: string) => { /* single-use override in wrapper */ },
    callMe:             async () => apiWrapper.get("/me"),
    callAdminUsers:     async () => apiWrapper.get("/admin/users"),
  }
}
```

Add the TypeScript global declaration (e.g., in `frontend/src/types/e2e.d.ts`):

```ts
declare global {
  interface Window {
    __xyzE2E?: {
      refreshSession:       () => Promise<unknown>
      getSession:           () => Promise<unknown>
      forceNextBearerToken: (token: string) => void
      callMe:               () => Promise<unknown>
      callAdminUsers:       () => Promise<unknown>
    }
  }
}
export {}
```

**Avoid circular imports — use a shared override module.**

Create `frontend/src/lib/e2e/e2eAuthOverride.ts`:

```ts
let forcedToken: string | null = null

export function forceNextBearerToken(token: string) { forcedToken = token }
export function consumeForcedBearerToken(): string | null {
  const t = forcedToken; forcedToken = null; return t
}
```

Then:
- `testHooks.ts` calls `forceNextBearerToken()` (write side)
- `wrapper.ts` calls `consumeForcedBearerToken()` (read + clear side)

No circular import, single-use behavior is obvious, the wrapper stays clean.

**`forceNextBearerToken` behavior in the wrapper:**
- On the next request, the wrapper calls `consumeForcedBearerToken()` first; if non-null, uses that value instead of `supabase.auth.getSession()`.
- Clears immediately after use (fires once).
- Only accessible when `NEXT_PUBLIC_E2E_TEST_HOOKS=true`.
- Must not exist in production builds — gate with compile-time check.

**Hook return values must be serializable** — `page.evaluate()` rejects if the bridge throws an unserializable error, making Playwright assertions fragile.

Define a result envelope:

```ts
type E2EApiResult<T = unknown> =
  | { ok: true;  status: number; data: T }
  | { ok: false; status?: number; errorName: string; message?: string }
```

Implement hooks with try/catch:

```ts
callAdminUsers: async () => {
  try {
    const data = await apiWrapper.get("/admin/users")
    return { ok: true, status: 200, data }
  } catch (err) {
    return {
      ok: false,
      status: (err as any)?.status,
      errorName: (err as any)?.name ?? "UnknownError",
      message: (err as any)?.message,
    }
  }
}
```

This gives Playwright stable assertions:

```ts
const result = await page.evaluate(() => window.__xyzE2E!.callAdminUsers())
expect(result.ok).toBe(false)
expect(result.status).toBe(403)
```

Apply the same pattern to `callMe` and `callAdminUsers`. `refreshSession` and `getSession` can return Supabase's own serializable response shapes directly.

Add `NEXT_PUBLIC_E2E_TEST_HOOKS` to `.env.example` (default: unset) and `.env.e2e.example` (placeholder `true`).
