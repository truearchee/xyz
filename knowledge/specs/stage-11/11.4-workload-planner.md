## Goal
Ship a deterministic, read-only, student-facing workload planner that generates reproducible study items from
real deadlines, current 11.1 risk gaps, Stage 9 forecast context, and the student's availability.

## Why now
11.1 established deterministic risk snapshots, 11.2 recommendations, and 11.3 assessment insights. 11.4 moves
Stage 11 into the student planning sub-sessions without adding AI or editable scheduling behavior.

## Owner algorithm lock
The 11.4 prerequisite was resolved by owner decision on 2026-06-20.

Owner's three locked choices:
- **Priority:** nearest deadline first.
- **Overflow:** `max_study_minutes_per_day` is a soft cap. It may be exceeded only by a bounded, config-backed
  allowance when needed to meet a deadline.
- **Horizon:** plan the entire remaining course, including all remaining deadlines and current gaps through
  course end.
- **Legacy fallback:** triggers only when `CourseModule.ends_on` is genuinely unknown; it uses the later of the
  latest known remaining `due_at` and the config-backed fallback horizon, records fallback provenance, and must
  never override or truncate a resolvable course end.

The deterministic six phases are:

1. **GATHER.** Pin `sourceCutoffAt`, then collect from now through course end: all remaining assessed sections
   with `due_at` from Stage 5.5, all current weak-topic gaps from the latest 11.1 risk snapshot and
   `risk_reasons`, student availability with `availabilityVersion`, and Stage 9 forecast context. The resulting
   task set is finite and bounded by deadlines plus gaps.
2. **BUILD TASKS.** Create one task per upcoming deadline, such as "prepare for [assessment] due [date]", and one
   task per weak topic, such as "reinforce [topic]". Each task carries a stored config-backed effort estimate and
   a reason of `deadline` or `gap`. If a weak topic also has an upcoming assessment, merge it into one
   deadline-linked task. MVP scope creates exactly one reinforcement task per current gap; no spaced-repetition
   expansion.
3. **PRIORITIZE.** Order deadline-bound tasks by effective deadline ascending. Gap-only tasks come last, ordered
   by gap severity descending and then by a stable id. No tie is left to nondeterministic database or runtime
   ordering.
4. **LAY OUT.** Walk the priority order and place each task's estimated minutes only on selected study days, in
   the preferred window, up to `max_study_minutes_per_day` before overflow. Deadline-bound tasks must be scheduled
   before their deadline using configured window start/end times. Same-day windows that occur after `due_at` are
   not eligible. Large tasks may split across multiple available days.
5. **RESOLVE OVERFLOW.** If a deadline-bound task cannot fit before its deadline within the cap, allow relevant
   days to exceed the cap up to a config-backed allowance, for example `PLAN_DAILY_OVERFLOW_PERCENT` defaulting
   to 25. If even the allowance cannot fit it, schedule as much as fits before the deadline and mark affected
   items/tasks `tight=true` with an honest message such as "may not fully fit before [date]". Never fabricate
   capacity, silently drop a task, or schedule beyond the allowance. Gap-only tasks never trigger overflow; they
   fill remaining spare capacity and spill to later free days. If zero minutes fit before the deadline, still emit
   a tight unscheduled residual item so the task is visible and honest.
6. **EMIT.** Persist `WorkloadPlan` plus `WorkloadPlanItem[]`. Each item stores date/window, topic or section,
   stored estimate, reason (`deadline` or `gap`), and `tight` when set. Timed study items store start/end
   timestamps; a tight unscheduled residual item may have null start/end and a required tight message. The plan
   carries `algorithmVersion`, `inputHash`, `availabilityVersion`, and `sourceCutoffAt`. Regeneration supersedes
   the old active plan with `supersededAt` and marks the new one `isActive`.

## Build
- Add migration `0058_workload_planner.py` with `down_revision="0057"`.
- Add `StudentAvailability`:
  - `student_id`, `module_id`;
  - selected study days Monday through Sunday;
  - preferred window: `morning | afternoon | evening | no_preference`;
  - `max_study_minutes_per_day`;
  - `availabilityVersion`;
  - timestamps.
- Add `WorkloadPlan`:
  - `student_id`, `module_id`;
  - `algorithmVersion`, `inputHash`, `availabilityVersion`, `sourceCutoffAt`;
  - `isActive`, `supersededAt`;
  - timestamps.
- Add `WorkloadPlanItem`:
  - `workload_plan_id`;
  - `source_section_id` nullable;
  - stable task key;
  - scheduled date/window plus start/end timestamps, nullable only for tight unscheduled residual items;
  - topic or section label;
  - stored estimate minutes;
  - reason: `deadline | gap`;
  - `tight` boolean plus optional tight message;
  - source metadata needed to trace back to the deadline or risk reason.
- Implement the six-phase deterministic planner in the analytics domain, using read models for cross-domain data.
- Reuse 11.1 latest risk snapshot and its `risk_reasons` for weak-topic gaps. Do not recompute risk a different
  way.
- Treat weak-topic gap tasks as risk reasons with a stable topic/target label, currently `topic_deadline_gap`.
  Non-topic risk reasons remain input/provenance but do not become invented topic tasks.
- Include Stage 9 forecast context in the deterministic input hash and plan provenance. Do not add grade advice
  or new grade math.
- Add config-backed planner defaults and thresholds:
  - `WORKLOAD_PLAN_ALGORITHM_VERSION`;
  - task effort estimates;
  - daily overflow percent;
  - preferred-window start/end times;
  - legacy fallback horizon days for modules without `ends_on` or remaining deadlines;
  - valid min/max availability minutes.
- Add student-only APIs to get/update availability, generate/regenerate the plan, and read the active plan.
- Add read-only, list-first UI on student module/progress surfaces. Items show date, preferred window, topic or
  section, stored estimate, reason, and tight state when present.
- Regeneration supersedes the prior active plan and creates a new active plan.

## Worker / job changes
None. 11.4 plan generation runs synchronously through the student action and does not use scheduled triggers.

## Authz rules
- Student role only.
- Student must have an active student membership on the requested module.
- A student can read/update only their own availability and plan.
- Lecturer/admin access to student workload endpoints returns 403.
- A student querying a module they are not assigned to returns 403.
- Responses contain no other student's availability or plan data.

assigned module. Plan items trace to real deadlines or current 11.1 risk gaps, store estimates and reasons, honor
nearest-deadline-first priority, handle bounded overflow and physically impossible work honestly, supersede old
plans on regeneration, and pass the 11.4 browser gate plus the full active Playwright suite.
