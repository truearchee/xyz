## Goal
Ship Stage 11.6: a student-facing, AI-phrased explanation of the student's **deterministic** Stage 9
grade forecast, shown alongside the forecast on the My Progress surface. The AI **explains, never
calculates** — every number it cites is Stage 9's. When the target is mathematically unreachable, the
advice is honest and constructive. The deterministic template renders immediately; AI swaps in when
ready; if AI is down, the template is the shipped output.

## Why now
11.1–11.5 are FULLY VERIFIED. Stage 9 already computes the deterministic forecast (target grade,
current weighted score, required remaining average, forecast status incl. the "impossible" case). 11.2
already established the lazy/cached AI phrasing pattern with two backend OutputValidators, provenance,
and template fallback. 11.6 is the thin, final slice that joins them: reuse Stage 9's numbers, reuse
11.2's AI machinery, add only the advice phrasing layer and its student surface.

## THE HARD LINE — AI EXPLAINS, NEVER CALCULATES
- **Reuse Stage 9's deterministic forecast engine** (`calculate_forecast`,
  `requiredRemainingAverage = (target − current) / remainingWeight`, the six forecast states incl.
  `impossible`). **No new grade math** anywhere in 11.6. The numbers are Stage 9's; AI only phrases
  advice around them.
- If AI is unavailable, the deterministic advice still renders (template fallback). **No black-box
  scores.** AI never decides the forecast, the target, or any number.

## Dependencies
- **Stage 9 forecast engine** is the single source of forecast truth. 11.6 calls it through a
  `platform/query` read model — it does not reimplement the formula and does not duplicate the grade
  math (rule 8: the agent domain reads cross-domain inputs through `platform/query`, never imports the
  owning domain).
- **11.2 AI machinery** (`platform/llm` gateway, PromptRegistry, Redis limiter, `AIRequestLog`,
  `OutputValidator`, the numeric-consistency + student-copy-safety validators, the deterministic test
  provider) is reused — 11.6 adds a new prompt, a new output schema, a forecast-specific
  numeric/fact validator, and **reuses the student-copy safety guard unchanged** (same banned-idea set).
- 11.6 does **not** read Stage 10 gamification data and does **not** depend on `AgentRun` — advice is
  generated **lazily on read** when the student views their forecast (not by a scheduled run).

## Build

### Deterministic forecast read (reuse, no new math)
- Add a `platform/query` read model, e.g. `get_student_forecast(db, *, student_id, module_id)`, that
  assembles `ForecastInput` from the existing `progress_read.get_grade_scheme_bundle` +
  `get_active_target_goal` read functions and calls Stage 9's `calculate_forecast`. It returns the
  `ForecastResult` (or `None` when there is no target/scheme).
- The grade math stays solely in `calculate_forecast`. To guarantee zero drift from the My Progress
  forecast, a test asserts the read-model forecast **equals** the Stage 9 progress-domain forecast for
  the same seeded student/module. (See the plan for the preferred single-path option.)

### Advice record (the cache) — reproducible
- Add `student_forecast_advice` (migration `0059`): **one row per `(student, module)`** carrying:
  - reproducibility: `algorithmVersion`, `inputHash`, `sourceCutoffAt`, `forecastState`, and a
    `deterministicPayload` (the forecast numbers + allowed sets + the deterministic template advice);
  - AI cache + provenance: `aiStatus`, `aiText`, `aiModelId`, `aiPromptVersion`, `aiInputHash`,
    `aiGeneratedAt`, `aiRequestLogId`, `aiFailureMessageSanitized`.
- `inputHash` hashes the forecast-relevant fields + `algorithmVersion` (`forecastState`,
  `targetLetterGrade`, `targetPoints`, `earnedSoFar`, `remainingWeight`, `requiredRemainingAverage`,
  `currentLetterGrade`, `bestReachableLetterGrade`). The AI cache is reused only while
  `aiInputHash == inputHash` AND `aiPromptVersion == "v1"`; otherwise it regenerates.
- Widen the `ai_request_logs.feature` CHECK to include `grade_forecast_advice` (in `0059`).

### Deterministic advice + which states get AI (rule 15 frugality)
- Deterministic template advice is produced for every forecast state and **must pass both validators by
  construction** (it is the fallback — an unsafe template is a bug). The template is what renders
  immediately and what ships if AI is down.
- AI phrasing is generated **only for states where guidance is meaningful**: `on_track`, `at_risk`,
  `requires_high_score`, `impossible`. For `achieved` and `final_no_remaining` the advice is the
  deterministic line only — **no AI call** (the course outcome is settled; an AI request would waste
  the budget rule 15 protects).

### Impossible-target advice (owner-locked tone)
- When Stage 9 returns `state == "impossible"`, the advice states the target is unreachable
  **honestly and constructively**: pair the honest fact (`targetLetterGrade` is not reachable from the
  current scores) with what **is** still achievable (`bestReachableLetterGrade`), where to focus, and
  that the student may revisit the target. **No false hope, no sugarcoating, no shaming, no diagnosis.**

### AI contract (mirrors 11.2)
- AI is lazy, cached, asynchronous, and last-layer only; it runs through `platform/llm`,
  PromptRegistry, Redis limiter, and `AIRequestLog`, at **BACKGROUND** priority, **one call per
  advice**, regenerated only when `inputHash` or prompt version changes.
- The page renders the deterministic template **immediately**; AI never blocks the initial render. The
  frontend polls `aiStatus` and swaps the AI text in when it becomes `succeeded`.
- The AI output passes the `GradeForecastAdvice` schema validator, then **both** domain validators
  (below). If AI is missing, pending, invalid, unsafe, or unavailable, the UI ships the deterministic
  template. A failed AI result is never persisted or served as partial copy.
- AI provenance is `{ modelId, promptVersion, inputHash, generatedAt }`.

### Two backend OutputValidators (the 11.2 machinery)
- **(a) Numeric / fact consistency** — a forecast-specific validator built on the **reused** generic
  helpers (number normalization/extraction, peer-comparison and diagnosis guards). The AI advice:
  - invents **no** numbers outside the forecast-derived `allowedNumbers` and **no** facts outside the
    forecast-derived `allowedFactPhrases` (target/current/best-reachable grade letters, the forecast
    status label, the module title);
  - **does not contradict** the deterministic `targetGrade` / `requiredRemainingAverage` /
    `forecastStatus` / `remainingWeight` / `currentWeightedScore`;
  - **includes** the state-relevant deterministic numbers (the actionable subset per state — see the
    plan for the exact per-state "must-include" set; the catch-all is no-invention + no-contradiction).
- **(b) Student-copy safety guard** — the **same** `StudentCopySafetyOutputValidator` and banned-idea
  set used in 11.2 (supportive, non-judgmental; "failing", "at risk", "behind the class", peer
  comparisons, diagnoses, etc. regenerate rather than ship). The advice is student-facing, so this
  guard always applies.
- On validator failure → regenerate once, then deterministic template fallback. Whole-result
  rejection: any failure rejects the entire AI result.

### Surface (My Progress forecast area; coordinate with Stage 10)
- On the Stage 9 My Progress surface (`ProgressDashboard.tsx`), add a `ForecastAdviceCard` in the
  **forecast area** (alongside `ForecastPanel`), **before** the Stage 10 gamification block. Changes
  are scoped to the forecast area; the gamification placeholder is **not** touched (Parallel
  Development item 5 — same page, different sections).
- The card shows the deterministic advice immediately, a quiet "preparing advice…" affordance while
  `aiStatus` is queued, then the AI text once ready. It is read-only and student-facing; the student
  sees only their own advice.

## API changes
- `GET /student/modules/{module_id}/forecast-advice` — returns the advice for the **authenticated
  student's** forecast in that module: `{ moduleId, forecastState, text, source ("ai"|"template"),
  aiStatus, … }`, with `Cache-Control: private, no-store`.
- Authz (mirror 11.2 / honor 11.4–11.5 precedent): role must be `student` → else **403**; the student
  must be enrolled in the module → else **404**; the advice is derived from the auth context, so a
  student can only ever fetch **their own** advice. The payload carries no other student's data.
- No lecturer endpoint and no Send/notification path (11.6 is student-only).

## Do not build
- No new grade math, no second forecast computation, no override of Stage 9's numbers.
- No `AgentRun`/scheduler dependency for advice (advice is lazy-on-read).
- No lecturer advice surface, no Send/notification/email/push/toast.
- No Stage 10 read; no change to the gamification block.
- No peer comparison, no diagnosis, no black-box score, no false-hope sugarcoating on impossible cases.
- No 11.1–11.5 rework beyond the additive forecast read model and the AI-machinery extensions listed.
