## Summary
Stage 11.7 is FULLY VERIFIED. It fixes the stuck-run handoff bug in the Stage 11.1 `AgentRun`
scheduler/manual-trigger path: a run can commit before the RQ enqueue attempt succeeds, and the old retry path
only enqueued when the DB row was newly created. Existing `queued` or `failed` runs are now recoverable on a
later scheduler tick or manual trigger.

No migration was added. Alembic remains `0059 (head)`. Rule 15 stays intact: scheduler and manual trigger paths
remain AI-free and enqueue only the deterministic AgentRun worker job on the ingestion/default queue.

## Implementation
- Added `enqueue_run_agent_if_needed(run_id)` in `backend/app/workers/queues.py`.
  - Uses the stable `agent-run-{run_id}` RQ job id.
  - Fetches the existing RQ job by stable id before enqueueing.
  - Suppresses enqueue when the RQ job is live: `queued`, `started`, `deferred`, or `scheduled`.
  - Deletes stale non-live RQ jobs before enqueueing the same stable job id again.
- Added `agent_run_status_is_requeueable(status)` with the DB status set `queued | failed`.
- Updated the scheduler tick to call the reconciliation helper for due existing runs in requeueable states, not
  only for newly-created rows.
- Updated the admin manual trigger route to do the same for the returned run, preserving DB idempotency while
  allowing stranded rows to recover.
- Tests now cover:
  - enqueue failure after `AgentRun` commit, followed by a later scheduler tick re-enqueueing the same run;
  - existing `failed` scheduled run re-enqueue;
  - manual trigger retry re-enqueueing the same run after a first enqueue failure;
  - genuinely live RQ job duplicate suppression;
  - stale RQ job replacement before re-enqueue;
  - `run_agent_run` idempotency so the same run executes compute exactly once after recovery.

## Environment notes
- One early full-suite attempt accidentally ran the already-started workers with the real provider because
  `LLM_PROVIDER=deterministic` was set only on the Playwright command, not at Compose startup. That produced AI
  queue backlog and was discarded. The green proof above recreates the stack with deterministic set at Compose
  startup.
- A later full-suite attempt failed Stage 7 setup because `.env.e2e` was not sourced for the Playwright process;
  the Node-side Supabase Auth admin helper therefore had no service-role key. The green proof above sources
  `.env.e2e`; no secret values are recorded here.
