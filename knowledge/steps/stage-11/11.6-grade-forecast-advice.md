## Summary
Stage 11.6 is FULLY VERIFIED. It adds a student-facing AI note that EXPLAINS the Stage 9 deterministic
grade forecast on the My-Progress surface, alongside the deterministic forecast. The AI phrases only —
every number is Stage 9's; the AI calculates nothing. The deterministic template renders immediately and
is the validator-safe fallback; lazy/cached AI swaps in when ready. The impossible-target case is honest
and constructive (names the best reachable grade, states the target is not possible, never shames). 11.6
is the LAST sub-session of Stage 11 and the second (final) AI sub-session.

No new grade math was added (Stage 9's `calculate_forecast` is reused through one shared input-assembly
path). No Stage 10 read, no scheduler/`AgentRun` dependency (advice is lazy-on-read), and no lecturer
surface. Migration `0059` adds the advice cache; Alembic head is `0059` (below the Stage-10 `0060` block).

## Implementation
- **Reuse Stage 9, single path:** extracted `build_forecast_input(...)` into
  `backend/app/domains/progress/forecast.py` and routed BOTH the progress dashboard
  (`progress/service._build_module_detail`) and the analytics agent
  (`analytics/service._forecast_result`, refactored from `_forecast_state`) through it — one forecast
  computation path, zero drift. A backend test asserts both paths agree.
- **Advice cache (migration `0059`):** `student_forecast_advice`, one row per `(student, module)`
  (unique constraint), carrying reproducibility (`algorithm_version="forecast-advice-v1"`, `input_hash`,
  `source_cutoff_at`, `forecast_state`, `deterministic_payload`) + AI cache/provenance (`ai_status`,
  `ai_text`, `ai_model_id`, `ai_prompt_version`, `ai_input_hash`, `ai_generated_at`,
  `ai_request_log_id`, `ai_failure_message_sanitized`). The same migration widens the
  `ai_request_logs.feature` CHECK and the model `AI_REQUEST_LOG_FEATURES` tuple to add
  `grade_forecast_advice`.
- **Deterministic advice + templates:** `analytics/forecast_advice.py` produces per-state template
  advice that passes BOTH validators by construction (the immediate render + fallback). AI is generated
  only for the states where guidance helps (`at_risk | requires_high_score | impossible`); the other
  states render a deterministic line with no AI call (rule-15).
- **Number contract:** `required_remaining_average` is quantized HALF_UP to 1 dp once and used for both
  the template and `allowedNumbers` (bare + `%` forms); `allowedNumbers` is derived from the rendered
  template (so the template's own numbers — including any digits in the module title — are always
  allowed). The impossible state cites no number (its allowed set is empty, so no ">100% needed" quote).
- **Two validators (reuse + extend 11.2):** `ForecastAdviceNumericConsistencyValidator` (invention +
  peer/diagnosis/unsupported + a **state-aware contradiction guard**) plus the **reused**
  `StudentCopySafetyOutputValidator` and an 11.6-specific impossible-case shaming lexicon (layered, the
  shared 11.2 sets are not mutated). The impossible guard requires an unreachable-framing phrase + the
  best-reachable grade and rejects any "still reachable" claim about the target; reachable states reject
  unreachable claims + overclaiming and must include the target + required average.
- **Lazy AI (mirrors 11.2):** `forecast_advice_ai.generate_forecast_advice_async` claims under
  `FOR UPDATE`, calls the gateway at BACKGROUND priority (one call per advice), validates schema + both
  domain validators, stores text + provenance on success, retries once on `InvalidOutput`, then falls
  back to the template. A terminal failure records the attempted `input_hash` so a re-view does not
  re-attempt the same forecast.
- **Service + endpoint:** `GET /student/modules/{module_id}/forecast-advice` (student-self; 403 for a
  non-student, 404 if not enrolled; advice derived from the auth context — no other student's data in
  the payload). The service computes the live forecast, upserts the cache via
  `on_conflict_do_update` (with a `where input_hash != excluded.input_hash` no-op guard), commits before
  enqueue (stable job id), and enqueues regeneration only when the cache is stale for an AI-eligible
  state, bounded by a per-(student,module) generation rate limit (throttled reads still serve the
  template — never a 429).
- **LLM wiring:** `GradeForecastAdvice { advice }` schema; `grade_forecast_advice` added to
  `GatewayFeature` + the gateway/validator union sites + the `OutputValidator` dispatch; deterministic
  test-provider fixture (echoes the validator-safe `templateAdvice`); prompt
  `grade_forecast_advice/v1.yaml` (`MBZUAI-IFM/K2-Think-v2` / `cerebras`, anchored on rewriting the
  template so the reasoning model stays brief) + `CHECKSUMS.json`.
- **Frontend:** `ForecastAdviceCard` in the My-Progress forecast area (after `ForecastPanel`, before the
  gamification placeholder). Tone-neutral for every state; renders the template immediately; backoff-poll
  with an `aria-live` height-stable swap when AI is ready; "preparing advice…" affordance + `aria-busy`
  while queued. Regenerated the OpenAPI client + wrapper.

## Browser gate details
The 11.6 gate (`tests/e2e/11.6-grade-forecast-advice.spec.ts`) seeds the standing E2E student into two
modules — an impossible-target module (4/5 components @ 82.50, target A → max 86 < 93 → impossible, best
B+) and a reachable module (4/5 @ 93.75, target A → required remaining average 90 → requires_high_score).
On the My-Progress dashboard it asserts:
- the advice card renders the deterministic template immediately, then swaps to AI (`data-source=ai`,
  `data-ai-status=succeeded`) via the gateway (deterministic E2E provider → both validators pass);
- the impossible advice names the best reachable grade (`B+`) and the unreachable framing, and contains
  no banned/shaming/peer phrase (`/failing|at risk|behind|give up|too late|hopeless|fallen|peer/i`);
- the reachable advice references the deterministic numbers (`90%` + target `A`) and invents none;
- forced AI-unavailable (DB-injected `template_fallback`) renders the template (no broken/empty card);
- the gamification block on the same page is untouched;
- a non-student (lecturer) token gets `403` on the advice endpoint.
The negative control (valid AI text actually rendering as `source=ai`, not just attacks that must fail)
is the AI-swap assertion itself, proving the AI layer is live and a too-strict validator hasn't silently
killed it. The real model + echo is proven separately by the rule-11 smoke.

## Rule-10 findings
- **E2E provider leak (recurring, same class as 11.5):** Docker Compose interpolates `${LLM_PROVIDER}`
  from the project `.env` (`k2think`), defeating the e2e compose's `:-deterministic` default, so an
  initial gate run hit real k2think and (via gateway cerebras routing) used `K2-V2-Instruct`, whose
  output failed the impossible contradiction guard → `template_fallback`. Re-upping the stack with
  `LLM_PROVIDER=deterministic` exported (shell env overrides `.env`) made the E2E hermetic and green.
  The real K2-Think-v2 route/echo is proven by the separate rule-11 smoke.
- **Stage 9 My-Progress AI-free assertion (cross-stage seam, rule 10):** the Stage 9 gate
  (`9-my-progress.spec.ts`) asserted `aiLogCount() == beforeAi` (viewing My Progress triggers no AI).
  11.6 intentionally adds lazy grade-forecast advice AI on that exact surface (master-spec
  coordination), so the assertion now legitimately sees `grade_forecast_advice` logs. Resolved by
  updating the Stage 9 gate's `aiLogCount()` to count only NON-advice AIRequestLog rows — it still
  proves My Progress triggers no summary/quiz/recommendation AI, while allowing the intended 11.6
  advice. This is the only prior-stage gate change; product code for Stage 9 is unchanged.
- **Reasoning-model truncation on the impossible case (smoke):** K2-Think reasons inline in `content`
  with no request-level reasoning control (F-4.5-04). At `max_tokens=2000` the impossible case
  intermittently truncated before the JSON. Resolved by anchoring the prompt on rewriting the
  deterministic `templateAdvice` (shorter reasoning), `max_tokens=4000` (under the 60s route timeout),
  and a bounded smoke retry (production already retries then falls back); both smoke cases passed on
  attempt 1.
