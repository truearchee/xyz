## Summary
Stage 11.5 is FULLY VERIFIED. It adds a student-facing `.ics` snapshot export for active workload plans.
Students download a calendar file from the read-only workload planner; the export contains scheduled study
blocks and timed deadline markers, with stable UIDs and UTC-normalized `Z` datetimes.

No Google Calendar OAuth, Apple Calendar API, external sync, subscription feed, plan regeneration, plan-item
editing, done state, drag/drop, accept/reject flow, AI, prompt, provider, `AIRequestLog`, Stage 10 read, or
11.6 grade-advice behavior was added. No migration was added; Alembic remains `0058`.

Empty active plans export a valid empty `VCALENDAR` with no `VEVENT` rows. Superseded/inactive plan exports
return `409 Workload plan is no longer active` instead of silently exporting stale data.

## Implementation
- Added `backend/app/domains/analytics/calendar_export.py`, a small deterministic serializer for workload
  calendars:
  - CRLF-delimited `VCALENDAR`;
  - `VERSION:2.0`;
  - `PRODID:XYZ LMS`;
  - `METHOD:PUBLISH`;
  - `X-XYZ-SNAPSHOT:true`;
  - stable `UID`s from `WorkloadPlanItem.id` and deadline `ModuleSection.id`;
  - `DTSTAMP` from export time;
  - UTC `DTSTART`/`DTEND` values formatted as `YYYYMMDDTHHMMSSZ`;
  - text escaping and line folding for commas, semicolons, backslashes, and newlines.
- Added `service.export_student_workload_calendar(...)` and
  `GET /student/workload/plans/{plan_id}/calendar.ics`.
- Authz is by direct `plan_id`: student role only, owned active plan only. Another same-module student gets 403
  for a known plan id; missing plans return 404.
- Deadlines are exported as timed 15-minute marker `VEVENT`s at `ModuleSection.due_at`, preserving due time
  instead of flattening deadlines to all-day dates.
- The frontend wrapper now has `downloadWorkloadPlanCalendar(planId)`, using authenticated `fetch` for the
  attachment response while preserving 401 redirect and 403 no-signout behavior.
- `StudentWorkloadPlanner` now shows `Download calendar snapshot` only when a plan exists. Plan items remain
  read-only; no edit/done/drag/accept/reject affordance was added.
- Regenerated the OpenAPI snapshot/client so the new calendar route is visible in `backend/openapi.json` and
  `AnalyticsService`, while the wrapper uses manual download handling for the blob attachment.
- Added the 11.5 browser gate `tests/e2e/11.5-calendar-ics-export.spec.ts`.

## Timezone contract
The export stores datetimes as absolute UTC instants, not floating local wall times. The browser gate seeds a
Europe/London DST-boundary study block intended for 2026-03-29 18:00 institution time. The file encodes:

```text
DTSTART:20260329T170000Z
DTEND:20260329T180000Z
```

The test parses the event and verifies the same instant renders as 18:00 in Europe/London and 21:00 in
Asia/Dubai, proving the event is not re-anchored to 18:00 in the viewer's timezone.

## Browser gate details
The 11.5 gate seeds:
- one Europe/London module spanning the DST transition;
- one active plan for student one;
- one `WorkloadPlanItem` scheduled at the intended 18:00 institution-local instant;
- one published assignment deadline with a timed `due_at`;
- a second same-module student.

The gate asserts:
- the student opens the workload planner and clicks the export button;
- Playwright captures a downloaded `.ics` file;
- the file parses as a `VCALENDAR`;
- `PRODID` is `XYZ LMS`;
- the plan-item event has `UID=workload-plan-item-{item.id}@xyz-lms`;
- `SUMMARY` is `Study: DST Focus Block`;
- `DESCRIPTION` carries `Reason: deadline` and `Estimate: 60 minutes`;
- `DTSTART`/`DTEND` encode the exact UTC instants;
- the parsed UTC instant renders as 18:00 in Europe/London and 21:00 in Asia/Dubai;
- the deadline event has a stable `module-deadline-{section.id}@xyz-lms` UID and timed `DTSTART`;
- re-exporting keeps the same event UIDs;
- student two gets 403 for student one's `plan_id` export.

  `.env.e2e` exported passed `1 passed (16.6s)`.
- **Fault command missing run manifest:** an initial fault-suite invocation omitted `E2E_RUN_ID`, so both tests
  failed before app behavior. It was discarded as invalid evidence.
- **Fault worker provider leak:** the first manifest-backed fault run recreated `ai_worker` with
  `LLM_PROVIDER=k2think`, so forced faults did not fire and summaries were generated. Re-running with
  `LLM_PROVIDER=deterministic` made the fault suite pass `2 passed (25.6s)`. The current Stage 11 worker was then
  restored with deterministic provider.
