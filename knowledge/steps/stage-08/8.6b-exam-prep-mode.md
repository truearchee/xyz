### Gate-ops findings (recorded for 8.6c + future gates)
The live gate surfaced THREE environment issues (no product-code bugs) before it went green — all the
documented "shared-stack contention" friction, amplified by Stage 10/11 running in parallel:
1. **`createAssessmentScope` returns 200, not 201** — a test-wiring assumption; fixed to accept 200/201.
2. **Shared `kyiv-backend` image tag race** — a sibling (Stage 10/11) workspace re-tags `kyiv-backend`
   on its own builds, and `ai_worker`/`embedding_worker` kept reverting to the sibling's image (which has
   no 8.6 modes → every homework/exam-prep turn `access_denied` → "No answer."). Fixed with a
   workspace-UNIQUE image tag (`dallas-stage86-gate`) in the override for backend + all workers, PLUS
   `docker tag dallas-stage86-gate kyiv-backend` (so even a reverted worker runs my code), then
   `up -d --force-recreate`.
3. **`next dev` OOM-killed mid-suite** — the Docker VM has ~7.7 GiB shared across many sibling stacks;
   the dev-mode frontend ballooned and was OOM-killed (`oomkilled=true`), failing the last specs on
   `ERR_CONNECTION_REFUSED`. Fixed by running the gate frontend in PRODUCTION mode (`next build` +
   `next start`, ~140 MiB) via the override `command`.
   Plus a clean DB was required (my many gate reruns had polluted the standing student's data → 8.5/5.5e
   data-accumulation false-fails).
