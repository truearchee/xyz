# Stage 8.2 — retrieval relevance-threshold calibration (review #8)

`RELEVANCE_MAX_DISTANCE` (cosine distance; a chunk is "relevant" iff its distance to the query is `<=`
this) lives in `backend/app/domains/assistant/config.py` alongside `RETRIEVAL_CONFIG_VERSION`. The final
value is stamped into every answer's server-side `context_snapshot`, so a later change is auditable.

## Final value
- **`RELEVANCE_MAX_DISTANCE = 0.35`** (cosine distance), `RETRIEVAL_CONFIG_VERSION = "retrieval-v1"`.
- `RETRIEVAL_TOP_K = 6`; per-summary cap 2500 chars; per-chunk cap 1500 chars; total combined
  summary+retrieved-context cap 6000 chars.

## Observed distances — real MiniLM-L6-v2 (the production encoder)
Measured with the real `sentence-transformers/all-MiniLM-L6-v2` snapshot baked into the backend image,
against a representative lecture chunk ("Photosynthesis converts light energy into chemical energy stored
in glucose, using carbon dioxide and water."):

| Query relationship to the chunk | cosine distance | vs threshold 0.35 |
|---|---|---|
| identical text | 0.0000 | far below (grounded) |
| in-lecture paraphrase ("How do plants turn sunlight into stored chemical energy as glucose?") | 0.2052 | below (grounded) |
| same-topic keywords ("photosynthesis glucose light energy") | 0.1744 | below (grounded) |
| off-lecture study question ("Explain the proof of the Pythagorean theorem.") | 0.8872 | far above (not grounded) |
| unrelated chit-chat ("What movie should I watch this weekend?") | 0.9490 | far above (not grounded) |

**Interpretation:** in-lecture queries cluster at **0.17–0.21** and off-lecture queries at **0.89–0.95**.
`0.35` sits in the wide empty band between them, with ~0.14 margin below (in-lecture headroom) and ~0.54
margin above (off-lecture rejection). It is a defensible MVP starting value; the per-answer snapshot lets
us recalibrate against real student traffic later without guessing.

## Approved summaries in the grounded context
The 2026-06-18 review follow-up mixes eligible generated summaries into the prompt only after the same
stored-section visibility gate and only when chunk retrieval found context under the threshold. This does
not shift the measured distances or the `0.35` cutoff: summaries are appended after the vector scan, while
the scan still runs solely over active-transcript normalized chunks with the same embedding model/version.

## CI / E2E does not depend on the real-MiniLM number (review #9)
The deterministic embedding encoder (`EMBEDDING_PROVIDER=deterministic`) maps identical text → distance
**0.0** and different text → **≈1.0** (near-orthogonal hash vectors). Both extremes are far from `0.35`,
so the gate's grounded vs general outcomes are stable regardless of the exact threshold — the threshold,
the pgvector scan, `decide_grounding`, and the gateway chain all run for real; only the vector *values*
are deterministic. Verified in `tests/test_embedding_platform.py` (distance contract) and the live
`tests/e2e/8.2-assistant-grounding.spec.ts` gate.
