## MF1 — release-abort proof (reported FIRST, as instructed)
A deploy that *succeeds* against a broken migration is the silent failure that defeats
migrate-as-release. **Local half — proven three independent ways (real, captured):**

| Proof | Result |
|---|---|
| `test_release_sh_aborts_and_skips_bootstrap_on_alembic_failure` | **PASSED** — fake `alembic` exits non-zero; `release.sh` returncode≠0 AND the bootstrap marker never appears (proves `set -e` actually *propagates* the failure + ordering, not just that the line is written) |
| `test_real_poison_revision_makes_alembic_exit_nonzero` | **PASSED** — a real poison revision (upgrade raises) at head in an isolated alembic tree → `alembic upgrade head` exits non-zero; the real `alembic/versions` tree is never touched |
| Real `release.sh`, unreachable DB | `release.sh exit=1`; `alembic upgrade head` line present; **`bootstrap identities` line ABSENT** (gated behind the failed migration) |

**Hosted half — developer-run, both directions** (procedure in `deploy/staging-runbook.md` §"Poison-
migration abort rehearsal"): deploy a poison migration → release exits non-zero, deploy **aborts**, the
new version never serves, `/health/ready` never green on the new release → revert → clean deploy
completes. To be captured in the 4.8b deploy run.

## Summary
A fresh hosted DB can now go empty → schema-at-head → first-admin + seed identities, driven by a
release-phase command that migrates over the DIRECT connection then runs an idempotent identity
bootstrap, aborting the deploy on any failure. In-repo: `env.py` prefers `DIRECT_DATABASE_URL`; a
bootstrap CLI (Supabase service-role ensure + `AppUser` upsert, admin via the sanctioned direct path,
seed gate IN the CLI); `release.sh` (the Fly `release_command`); `deploy-staging.sh` (digest-pinned,
worker-registry + readiness verify); a committed runbook with rollback; tests. The hosted first-ready-
boot is the developer's run.

## Key reconciliation (surfaced, not silent)
**The seed gate is `not IS_PRODUCTION`, NOT `IS_NON_PROD`.** `IS_NON_PROD` excludes staging (it's the
fault-injection predicate); the seed identities are *required* in staging for the 4.8d smoke. Gating
them on `IS_NON_PROD` would block them exactly where they're needed. Added `IS_PRODUCTION`
(`ENVIRONMENT == "production"`); seeds run when `BOOTSTRAP_SEED_IDENTITIES` AND not production.
