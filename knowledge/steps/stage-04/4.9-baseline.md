## Environment / run protocol (recorded so 4.9c's re-run is reproducible)
- Stack: `docker compose -f docker-compose.yml -f docker-compose.e2e.yml up -d --build` (the e2e
  overlay applies `.env.e2e`; base `up` points the frontend at prod Supabase and breaks login).
- Local Supabase on `127.0.0.1:54321` must be up (the `supabase_*_test2` containers); `ENVIRONMENT=
  development` (so the fault harness is allowed to fire; a `{test,e2e}`-only predicate would crash-loop
  the faulted `ai_worker`).
- `E2E_RUN_ID` must be exported before `seed.mjs`; the full suite runs **`--workers=1`** (serial — a
  single `embedding_worker` is overwhelmed by parallel pipeline specs; accepted-with-rationale, 4.7-R2/§17).
- Fault specs each need the `ai_worker` recreated with the matching `LLM_FAULT_INJECTION` (one fault
  per stack run); restore by recreating `ai_worker` without `docker-compose.fault.yml`.
- Always run `teardown.mjs <runId>` after the suite (see F-4.9-1 — skipping it accumulates e2e data).

## F-4.9-1 (baseline contamination) — RESOLVED: fixed in this block
First (naive) single-invocation attempts showed `4.3.5c` failing (`admin-module-row` not visible) and
the 2 fault specs failing. Diagnosed: **not a code regression** — `POST /admin/modules` returned
**201**, but the admin list is `GET /admin/modules?limit=50` and the local DB had **60 `course_modules`**
(58 owned by e2e-domain users — orphaned artifacts from prior runs that skipped teardown, oldest
2026-06-03), so a new module sorted past 50 and never rendered. Per the developer's decision, the 58
e2e-owned modules were purged in a scoped, ordered transaction (transcripts → section_assets →
module_sections → course_memberships → course_modules; cascades handled segments/chunks/ingestion_
jobs/summaries), keeping the 2 non-e2e dev modules (`Machine Learning`, `Checkpoint A Smoke Module`,
owner `demolecture@xyzlms.dev`). After the purge the documented orchestration is 11/11 green.
