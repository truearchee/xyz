## Worker Design
Added an internal encoder abstraction:

- `EmbeddingEncoder` protocol
- `SentenceTransformersEmbeddingEncoder`
- `DeterministicEmbeddingEncoder` for unit/integration tests

Embed job behavior:

- Created after chunk transaction commits.
- Uses `IngestionJob` lifecycle with `job_type = 'embed'`.
- Uses the embed-only active-job partial unique index.
- Enqueues to the separate `embedding` RQ queue.
- Uses deterministic RQ job IDs: `embed-{ingestion_job_id}`.
- Uses embed-only RQ retry config; parse/chunk retry behavior is unchanged.
- Records safe DB failure before re-raising from the RQ wrapper so RQ retry sees a failed queue execution.
- Allows a failed or previously running embed job row to be reclaimed by retry using the same `IngestionJob` row.
- Marks embed enqueue failure as a safe failed embed job instead of leaving a queued DB row with no Redis job.

Staleness/provenance behavior:

- Computes a per-row canonical hash from exact encoder input, model, revision, normalization, and `transcript_chunks.chunking_version`.
- Treats a chunk as stale if any of these are missing or mismatched:
  - vector
  - model
  - revision
  - dimension
  - normalization
  - embedding version
  - input hash
- Completion requires current vectors and matching provenance for every chunk.
- Batch commits are used, so partial failure can leave some chunks embedded; retry resumes from stale/missing rows.

## Projection Behavior
Backend projection now supports embed outcomes:

- completed embed job with current vectors reports `overallState = embedded`
- failed embed job reports `overallState = failed`, `failedStep = embed`, and a safe failure message
- projection does not expose vectors, raw chunk text, model internals, or stack traces

## Review Findings and Disposition
Initial read-only worker review found blockers:

- embed jobs could be left queued in DB if Redis enqueue failed
- RQ retry was absent for embed jobs
- a crashed worker could leave an embed job `running`
- RQ job IDs were not deterministic
- concurrency config is validated but not process-count-enforced
- production encoder loads per job

Fixed:

- enqueue failure now marks the embed job failed with a safe message
- embed queue uses deterministic RQ job ID and embed-only retry config
- RQ wrapper records safe DB failure then re-raises a safe exception
- failed/running embed rows can be reclaimed by retry
- tests cover retry-safe failure, running-row reclaim, deterministic RQ job ID, and enqueue-failure visibility

- `EMBEDDING_WORKER_CONCURRENCY` is validated but deployment still controls process count; one worker service is the current default.

## Checkpoint E.1 Model Lifecycle Addendum
Checkpoint E.1 resolved the production encoder lifecycle ambiguity.

Investigation result before E.1:

- `SentenceTransformersEmbeddingEncoder.__init__` called `SentenceTransformer(...)` directly.
- Because `embed_transcript_async(...)` constructs a production wrapper when no encoder is injected, the underlying model would load once per embed job.
- That failed the E.1 decision rule.

Fix:

- Added a process-local lazy model cache in `backend/app/domains/transcripts/embedding_encoder.py`.
- Cache key: resolved model path, expected revision, and device.
- `SentenceTransformersEmbeddingEncoder` wrappers can still be constructed per job, but they retrieve the cached model.
- `validate_model_snapshot(...)` still validates model path, revision marker, required files, and required weight/tokenizer alternatives before cache access.
- `DeterministicEmbeddingEncoder` and `embed_transcript_async(..., encoder=...)` test injection are unchanged.

Operational confirmations:

- `EMBEDDING_WORKER_CONCURRENCY` is validated but deployment/Compose controls actual process count.
- Compose keeps embedding work separate from ingestion work with separate `worker` and `embedding_worker` services.
- No global retry behavior was added to parse/chunk jobs; only embed enqueue uses embed-specific RQ retry.
