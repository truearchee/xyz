# Session 4.5.1b — Brief-from-detailed + gating + backfill + fencing — REPORT (BACKEND VERIFIED)

Inverts the summary DAG (brief derives from the completed detailed), establishes the §0.1 full-coverage
gating predicate, builds the backfill command, completes the stale-write fencing audit, and adds the
AIRequestLog chain helper. Real-provider smoke + browser gate + the backfill RUN + the 4.7 re-gate are
4.5.1c. **ADR-052.**

## Built
- **DAG inversion (ADR-052).** ``insert_summary_jobs`` creates both job rows but enqueues ONLY the detailed
  ([summary_service.py](backend/app/domains/transcripts/summary_service.py)); the detailed handler forks the
  brief on success (``_fork_brief_after_detailed``, best-effort). The brief claim reads the latest completed
  detailed and DEFERS (no attempt consumed) if none exists. New prompt ``brief_from_detailed/v1`` (input =
  the detailed ``content_json``); ``SummaryFeature`` + ``brief_from_detailed`` feature; the brief persists
  ``generation_strategy='derived_from_detailed'`` + ``sourceDetailedSummaryId``, inheriting the detailed's
  ``truncated``.
- **OB1 fallback (detailed disabled), with the FULL §0.1 treatment.** When ``ENABLE_DETAILED_SUMMARY=false``
  the brief falls back to the transcript-based single-call path — ``truncated=true``, ``single_call``, NOT
  ``is_full_coverage_detailed``, "based on the first portion" label. Honestly degraded; never a back door.
- **Gating predicate (§0.1).** ``is_full_coverage_detailed`` ([summary_eligibility.py](backend/app/domains/transcripts/summary_eligibility.py))
  = ``generation_strategy=='map_reduce' AND not truncated``, keyed off the DETAILED. Stage 5/7/8 read this;
  the brief's ``derived_from_detailed`` label is informational (OB2).
- **Accept-set eligibility.** ``EXPECTED_PROMPT_VERSION_BY_SUMMARY_TYPE`` is now a tuple of accepted versions
  per type, so both the derived brief (``brief_from_detailed/v1``) and the fallback brief
  (``brief_summary/v2``) are recognized as current while ``prompt_version`` stays the true producing prompt.
- **Backfill command** ([backfill.py](backend/app/domains/transcripts/backfill.py)) — selects active
  transcripts whose latest detailed is stale (not ``map_reduce``+¬truncated), re-enqueues the detailed job
  (regenerates map-reduce → brief re-forks). ``dry_run`` (default) enqueues nothing; per-run ``cap``;
  idempotent. BUILT — RUN in 4.5.1c.
- **Stale-write fencing audit (§6.1).** ``map_reduce.py`` fences (not-superseded + checksum match) EARLY
  (before any map call) and before every map-unit write + the reduce — aborts clean as ``MapReduceFenced``
  (caught as a no-op, not a failure). The brief claim fences on the source detailed matching the ACTIVE
  transcript checksum (a brief firing after a replacement reads the NEW detailed or defers — OB3).
- **AIRequestLog chain helper** ([summary_read.py](backend/app/platform/query/summary_read.py)) — the
  correlated map+reduce(+brief) chain for one job id, the queryable provenance the 4.5.1c proof uses.
- **Migration 0016** — additive: ``brief_from_detailed`` feature + ``derived_from_detailed`` strategy check
  widenings. Existence-safe; down→up round-trip green.

## Design refinement found during the build
The §6.1 fence is now also applied EARLY in ``MapReduceRunner.run`` (before partition/map), not only before
writes — so a superseded transcript aborts with zero map calls (no wasted provider calls, no AIRequestLog
rows). Strengthens the audit beyond the spec's "before each write."
