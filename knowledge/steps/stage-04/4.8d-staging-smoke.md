## How the load-bearing assertions are coded (spec §1)
- **NC1** (non-member → 404) and **NC2** (member+draft → 404) hit the SAME student endpoint as the
  SAME student (O2 — one subject, one token); only the membership/publish state changes.
- **NC1≡NC2** is `expect(nc2.body).toEqual(nc1.body)` — the two captured 404 bodies compared to EACH
  OTHER (the existence-leak proof), NOT against a hardcoded string.
- **NC3** (member+published → 200) asserts both summaries `ready`; detailed contains `Overview`.
- **NC4 — three independent surfaces (MF1):** (1) UI DOM-absence (weakest), (2) API response shape
  carries no raw transcript / sentinel (the real one), (3) **no signed storage URL** in any student
  payload + every transcript text-bearing endpoint returns 403 to the student (the most dangerous,
  easiest-missed surface). A best-effort non-vacuous canary: the LECTURER confirms the sentinel is in
  the raw transcript while the student never sees it.
- **Carry-ins coded:** SSE inter-chunk timing measured from a real browser `fetch`+reader over D1
  (`maxDeltaMs > 200`, not chunk count); hosted `window.__xyzE2E === undefined`; 15-min bounded poll
  (≤10 s) that fails loud; `/health` + `/health/ready`; the single real K2Think run's model IDs pulled
  from the lecturer provenance into the artifact JSON.
