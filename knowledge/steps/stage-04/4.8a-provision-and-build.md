## Deviations from spec
- **B5:** added `frontend/Dockerfile.prod` instead of *rewriting* `frontend/Dockerfile`. A single
  image cannot serve both the dev e2e suite (runtime `NEXT_PUBLIC_E2E_TEST_HOOKS` injection) and the
  prod standalone build (build-time `NEXT_PUBLIC_*` inlining). Two Dockerfiles, two purposes
  (finding F-4.8a-9). Verified: the dev image still runs the e2e suite (11/11); the prod build emits
  standalone output.
- **Node digest pin:** version-pinned (`node:20.18.1-alpine`), digest captured at provisioning
  (F-4.8a-10) — an offline build cannot resolve a live `sha256:`.

## Decisions made
- **O1 (developer-ruled):** boot check uses `IS_NON_PROD` (refuse in production/staging), consistent
  with the step guard; `check-staging-env` separately asserts `ENVIRONMENT=staging`.
- **MF3:** pooler discriminator is the explicit `DATABASE_POOLER` flag, never a port/host sniff.
- **MF4:** two Fly apps (`backend/fly.toml` + `frontend/fly.toml`).
- **pgcrypto:** dropped entirely (no consumer; vector `CREATE` already in migrations 0006/0007).

## Risks introduced
- The exact asyncpg pooler `connect_args` combo and per-process pool sizes are only validatable
  against the real Supabase pooler — finalized in 4.8b (F-4.8a-2, F-4.8a-6). Off-pooler behavior is
  byte-identical (409 + 11/11 green prove no local regression).
- The advisory-lock-over-pooler *handback* mode is provable only on a real pooler (4.8d); the unit
  test proves singleton-under-concurrency on a direct connection (F-4.8a-3).
