## Executive summary

Session 4.3.5e was the final Client Edge Recovery session for the implemented Stage 4.1-4.3 transcript pipeline. Its job was to prove, through a real browser path, that a lecturer can upload transcript files, the backend and worker parse/chunk them, persisted segment/chunk rows exist, and students cannot upload or read transcript data.

Before 4.3.5e, transcript upload, parse, and chunk persistence existed in backend code and tests, but no real lecturer UI drove the vertical path. The browser had no way to observe a worker-driven terminal state. The temporary `/tracer` route also still existed from earlier recovery work, the old transcript frontend code was unmounted and bypassed wrapper/upload-helper conventions, and E2E teardown did not have run-scoped cleanup for transcript artifacts.

4.3.5e fixed the prerequisite worker terminal-state bug, added the lecturer transcript UI and status polling, removed `/tracer`, added safe run-scoped E2E teardown, and passed the final transcript Playwright browser gate. Stage 4.1, Stage 4.2, and Stage 4.3 are now FULLY VERIFIED. The Client Edge Recovery Block 4.3.5 is COMPLETE. The next roadmap stage is Stage 4.4 - Embeddings.

## Part 1 - Contract investigation

Part 1 was read-only contract investigation. It mapped backend routes, services, frontend integration points, E2E tooling, and the worker path before any UI work.

Preflight:
- Working tree was expected to be clean except for the 4.3.5e findings file.
- Docker services were checked for backend, frontend, db, redis, and worker.
- Generated OpenAPI client freshness was required before implementation.
- `NEXT_PUBLIC_E2E_TEST_HOOKS=true` was preserved.

Transcript upload contract:
- Endpoint: `POST /modules/{module_id}/sections/{section_id}/transcript`.
- Multipart field: `file`.
- Accepted extensions: `.vtt`, `.txt`.
- Accepted content: UTF-8; VTT must start with `WEBVTT`; TXT must be non-empty.
- Effective MIME values: `text/vtt`, `text/plain`.
- Max size: `MAX_TRANSCRIPT_UPLOAD_BYTES`, default `10485760`.
- Accepted section types: `lecture`, `lab`.
- Rejected section types: `assignment`, `supplementary`, with `422 SECTION_TYPE_UNSUPPORTED`.
- Duplicate active transcript behavior: `409 TRANSCRIPT_ALREADY_EXISTS`.

Status read contract:
- Endpoint: `GET /modules/{module_id}/sections/{section_id}/transcript`.
- Response DTO: `TranscriptMeta` metadata only.
- Product response does not expose raw transcript text, parsed segments, chunks, storage key, checksum, or counts.
- Counts must be proven with test-level DB reads unless a later approved product projection is added.

Student denial and worker behavior:
- Student upload returns `403 TRANSCRIPT_FORBIDDEN`.
- Student status read returns `403 TRANSCRIPT_FORBIDDEN`.
- Student `/me` remains valid after the 403.
- No student-facing raw transcript read path was found.
- Backend enqueues RQ jobs to the `ingestion` queue.
- Worker runs `python -m app.workers.worker` and consumes the same queue via `REDIS_URL` and `INGESTION_QUEUE_NAME = "ingestion"`.

Key findings:
- F-4.3.5e-001: no persisted 4.3.5e session spec file existed.
- F-4.3.5e-002: successful parse+chunk left transcript status at `chunking`.
- F-4.3.5e-003: product status endpoint exposes no segment/chunk counts.
- F-4.3.5e-004: transcript frontend code was unmounted and bypassed wrapper/upload-helper auth recovery.
- F-4.3.5e-005: E2E lacked reusable run manifest, safe teardown, and DB proof helpers.

## Part 2 - Terminal-state repair

Part 2 fixed the approved prerequisite blockers only.

Observed bug:
- A controlled upload probe reached completed parse and chunk jobs.
- Segment rows persisted.
- Chunk rows persisted.
- Product-visible transcript status still remained `chunking`.

Why `chunking` was not acceptable:
- `chunking` describes work in progress.
- The browser gate could not treat an in-progress label as success.
- Checkpoint A/B polling needed a real terminal status.

Decision:
- After successful chunk completion, and while Stage 4.4 embeddings and Stage 4.5 summaries are not implemented, `Transcript.status` becomes `completed`.
- No new `chunked` status was added.

- Polling runs every 2.5 seconds while status is non-terminal.
- Polling stops on `completed`, `failed`, timeout, and unmount.
- Timeout renders `Processing is taking longer than expected. Refresh to check again.`
- Errors render with `role="alert"` and no stack traces.
- No replacement, retry, supersession, raw text, segment, chunk, storage key, checksum, or count UI was added.

## Architecture notes

- Transcript status is separate from section publish status and asset processing status.
- The product transcript endpoint exposes `TranscriptMeta` metadata only.
- Segment and chunk counts are test-level DB proof, not product API fields.
- Raw transcript text, parsed segments, chunks, normalized text, storage keys, checksums, and counts are not exposed through product UI/API responses.
- Students cannot upload transcripts.
- Students cannot read transcript status or raw transcript data in the MVP.
- Only one active transcript is supported per section before Stage 4.6.
- Replacement, retry, recovery, and supersession remain Stage 4.6 scope.
- Embeddings remain Stage 4.4 scope.
- AIRequestLog and summary generation remain Stage 4.5 scope.
- Student-facing summaries remain Stage 4.7 scope.
- Successful chunk completion sets `Transcript.status = "completed"` until Stage 4.4 introduces downstream embedding transitions.
- 403 keeps the browser session; 401 clears session and redirects to `/login`.
- Role still comes from backend `GET /me`, never JWT claims.
- Multipart uploads are allowed only through the approved upload helper exception; product components must not call `fetch()` directly.
