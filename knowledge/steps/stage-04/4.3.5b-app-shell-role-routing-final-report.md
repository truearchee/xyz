## Executive summary
4.3.5b replaced the tracer-only navigation proof with a real app shell and role-routed frontend skeleton. It proved that browser sessions bootstrap through `GET /me`, role decisions come from backend user context rather than JWT claims, 401 redirects to login, and 403 keeps the session active.

No backend product code changed in this session.

## Why this session existed
- 4.3.5a proved the client edge through `/tracer`, but users still lacked a real shell and role-specific app routes.
- The roadmap required app-shell and role-routing proof before admin/content/transcript UI backfills.
- Auth behavior had to preserve the recovery rule: 401 clears session and redirects, while 403 surfaces unauthorized state without destroying a valid session.

## Starting state
- Temporary `/tracer` and throwaway login existed from 4.3.5a.
- Product route groups, shell layout, and role dashboards were not yet proven.
- The frontend needed a stable E2E bridge and session state machine for later browser gates.

## What was implemented
- Route groups for `(auth)/login` and `(app)`.
- Admin, lecturer, and student placeholder routes.
- Root role routing.
- `AppShell` and `AccessDenied` components.
- `SessionProvider` state machine.
- `ProtectedAppLayout` with role-prefix guard.
- API wrapper 401/403 handling.
- E2E bridge and test hooks for browser assertions.
- Tracer gate behind `NEXT_PUBLIC_TRACER_ENABLED` until later removal in 4.3.5e.

## What was proven
- Admin, lecturer, and student can log in in separate browser contexts.
- Each role lands on its correct role route.
- Role is confirmed by backend `/me`, not decoded from JWT claims.
- Token refresh rotates the lecturer token and subsequent wrapper calls use the new Authorization value.
- Malformed bearer returns backend 401; wrapper signs out and redirects to `/login`.
- Student access to `/admin` redirects to `/unauthorized`, not `/login`.
- Student admin endpoint call returns 403 and keeps the session active.
- Tracer disabled returns 404 and exposes no tracer controls.
- Logout clears session and routes to `/login`.

## Auth and routing behavior
- Role source: `GET /me`.
- 401 behavior: session clears and browser routes to `/login`.
- 403 behavior: session remains valid and the UI/API surfaces unauthorized state.
- Route protection: app route prefixes are checked against backend-sourced role state.
- Tracer behavior: disabled by flag after app shell proof; deleted later in 4.3.5e.

## Findings and repairs

- Stale frontend image caused type-check noise; rebuild resolved it.
- Tracer `ForbiddenError` display regression was repaired.
- No durable ADR was added because the auth/routing rules were already part of the approved recovery constraints.
