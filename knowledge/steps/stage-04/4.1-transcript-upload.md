## Summary
Session 4.1 is complete. Assigned lecturers can upload raw VTT/TXT transcripts to lecture/lab sections through module-scoped transcript endpoints; the backend stores raw bytes unchanged through `StorageProvider`, persists a `transcripts` row at `status='uploaded'`, and enforces one active transcript per section with both a pre-check and the partial unique index.

No worker, queue, parser, chunker, embedding, summary, replacement, signed transcript URL, student transcript view, language detection, K2Think call, or status transition beyond `uploaded` was added.

The 2026-06-01 review failure around Supabase cleanup observability is fixed. `SupabaseStorageProvider.delete_object()` no longer silently returns success when provider deletion fails; it raises `StorageUnavailableError` for timeout/connection failures and `StorageProviderError` for other delete failures or provider error responses. Transcript cleanup still catches those failures in `_best_effort_delete()` and logs the orphan cleanup warning while preserving the handled API response.

## Review failure resolution
- Fixed: real Supabase delete failures are now observable to transcript cleanup callers.
- Final contract: `StorageProvider.delete_object()` remains a raising `None` contract. Successful deletes return `None`; failed provider deletes raise a `StorageProviderError` subtype.
- Tests added: `backend/tests/test_storage_supabase.py` for provider delete failures, plus transcript integration tests for DB-failure cleanup logging and lost-race cleanup logging when delete fails.
- No ADR added: the storage abstraction already declared a raising delete contract; this follow-up makes the Supabase implementation conform to it.

## Deviations from spec
- The pasted spec named `/sections/{section_id}/transcript`; the approved plan amendment uses the repo's existing module-scoped route convention: `/modules/{module_id}/sections/{section_id}/transcript`.
- P2 could not prove live row population because the local database had no `module_sections` rows during preflight. The schema check exists and tests create/populate all relevant section types.
- The storage key follows the Stage 3 module/section convention, documented in ADR-016 and `storage.md`: `modules/{moduleId}/sections/{sectionId}/transcripts/{transcriptId}/{safeFileName}`.
