  redirect (`location.assign('/login')`) + `AuthRequiredError`. **403 → `ForbiddenError` (with body),
  session KEPT, and NEVER a login redirect** — asserted as a NEGATIVE (`signOut` not called, `assign`

  propagate unchanged, no signOut/redirect.

  summarized → "Summaries ready"; **"embed ok, summaryBrief failed" → FAILED summary step with explicit
  text while earlier steps stay completed**, retryable terminal failure shows Retry, non-retryable shows
  none; every in-flight state yields non-blank status text.
- **§6.3 `SessionProvider.test.tsx` (5) — session authority.** loading→authenticated; **role comes from
  GET /me, NOT the JWT claim on the session** (active-membership authority); a generic /me failure →
  clean `unauthenticated` (not a hang); ForbiddenError → `forbidden` (no leaked role); SIGNED_OUT →
  `unauthenticated` + cleared context.

## CI enforcement — local PROVEN; remote handed off (developer decision: "local gate proven; you own remote")
**GitHub Actions `.github/workflows/ci.yml`:** `frontend` job (the 7 fast gates as required) + `backend`
job (pytest with pgvector + redis service containers) on push(main, stage/**) + PR(main); `e2e` job is
nightly/`workflow_dispatch` only (the dockerized suite is too heavy per-PR; rule-14 full suite stays the
local stage-close gate).

**Husky pre-commit (`.husky/pre-commit`) — local enforcement, PROVEN BLOCKING:** runs the fast subset
(typecheck + check:design-tokens + check:inline-styles). **Demonstrated:** a probe with `style={{color:
"#ff0000"}}`, staged + `git commit` → `check:design-tokens FAILED … _probe.tsx` → `husky - pre-commit
script failed (code 1)` → **HEAD unchanged, no commit created**. A deliberately-failing check actually
blocks a commit — not folklore.

**Remote branch-protection — HANDED OFF (owner: developer).** `gh` is not installed and `gh auth login`
is interactive, so it can't be configured/proven headlessly; the remote `main` is also 20 commits behind
(4.8 + 4.9a-d unpushed). To make the Actions checks ENFORCED (the only way a CI gate is load-bearing),
the developer runs, after pushing + one workflow run:
```bash
brew install gh && gh auth login            # interactive (developer's GitHub identity)
git push origin stage/4.9-frontend-foundation   # so Actions runs once and the check names register
# then mark the checks required on main (classic branch protection):
gh api -X PUT repos/truearchee/xyz/branches/main/protection \
  -H "Accept: application/vnd.github+json" \
  --input - <<'JSON'
{ "required_status_checks": { "strict": true, "contexts": ["frontend gates", "backend pytest"] },
  "enforce_admins": true, "required_pull_request_reviews": null, "restrictions": null }
JSON
# PROVE it: open a PR whose frontend gate fails (e.g. a raw hex) → GitHub blocks merge ("Required check failing").
```
Until then: **local pre-commit enforcement is proven; remote enforcement is the developer's documented
step** (this is the prereq-5 "real gate, not folklore" requirement — the remote half is owned, not faked).
