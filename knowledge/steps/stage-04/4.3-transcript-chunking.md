## Summary
Session 4.3 is complete. Successful parse now creates a recoverable queued chunk job in the parse-completion transaction and enqueues the chunk worker by `ingestion_job.id` after commit.

The chunk worker normalizes immutable `transcript_segments` into ordered `transcript_chunks`, records token counts and version strings, stores result counts in `ingestion_jobs.result_metadata`, and advances transcripts to `status='chunking'`. No HTTP endpoint, frontend UI, OpenAPI regeneration, embedding population, summaries, retry/recovery sweeper, or user-facing chunk view was added.

## Deviations from spec
- The implementation used `result_metadata jsonb` for chunk result counts, per the approved plan clarification.
- `embedding vector(384)` is represented with a local SQLAlchemy `UserDefinedType`; no pgvector Python dependency was added.
- The first targeted test run failed because duplicated test text was correctly skipped by adjacent-duplicate normalization. The test data was corrected and the targeted suite passed.
- Review follow-up used deterministic worker integration and source-order assertions for row-lock coverage rather than a blocking concurrent PostgreSQL test. A draft `asyncio.gather` concurrency shape was attempted during review follow-up and removed because it could hang while waiting on row locks without a controlled lock timeout.
