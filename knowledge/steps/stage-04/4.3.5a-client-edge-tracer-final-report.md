## Executive summary
4.3.5a created the temporary browser tracer bullet that proved the client could authenticate with local Supabase, call the backend with a real bearer token, upload private storage-backed content, and preserve the 401/403 session distinction through a browser path.

The session was not intended to build product UI. It deliberately used `/tracer` as recovery scaffolding so the project could prove the edge before backfilling real product screens. `/tracer` was later removed in 4.3.5e after the real UI browser gates replaced it.

## Why this session existed
- Backend stages already existed, but the roadmap required browser proof rather than backend-only verification.
- The frontend needed a thin, inspectable path through Supabase auth, backend authorization, database rows, and storage.
- The tracer route created a controlled browser surface before the app shell, role routing, admin UI, content UI, and transcript UI were available.

## Starting state
- Local Supabase E2E wiring and seeded actors were not yet proven end to end.
- The client edge had no durable evidence for browser login, bearer propagation, or authenticated backend calls.
- The product app shell and role-routed dashboards were still future 4.3.5b work.
- Health proof details were part of the broader Client Edge Recovery context; a distinct 4.3.5a health-only assertion was not recorded in the available source reports.

## What was implemented
- `GET /me` backend current-user endpoint.
- Supabase browser client and session provider.
- Token-per-request frontend API wrapper behavior.
- Temporary `/login` and `/tracer` routes.
- Local Supabase E2E fixture setup and deterministic E2E actors.
- `SUPABASE_PUBLIC_URL` handling so signed URLs returned browser-reachable `127.0.0.1` origins while backend containers used `host.docker.internal`.
- Private `section-assets` bucket smoke setup.
- Playwright tracer spec for browser proof.

## What was proven
- Real Supabase login for E2E lecturer and student actors.
- Real backend calls with bearer tokens from the browser.
- Real database rows for users/modules/sections/assets.
- Private Supabase storage upload and exact-key cleanup.
- PDF upload through the tracer path.
- Non-PDF upload rejection.
- Publish path and student published visibility.
- Draft denial without login redirect.
- Signed URL response with `Cache-Control: no-store`.
- Browser consumption of signed URL without forwarding Authorization.
- Student upload rejection with HTTP `403` while the session stayed active.

## Browser/test evidence
- Command: `set -a; . ./.env.e2e; set +a; npx playwright test tests/e2e/4.3.5a-tracer.spec.ts`
- Result: `1 passed (4.5s)`.
- Source report recorded 15 assertions across auth, backend, storage, upload, publish, student visibility, denial behavior, and cleanup.
- Post-run asset residue check returned zero `section_assets` rows.

## Findings and repairs
- Storage signed URL origin mismatch was resolved with `SUPABASE_PUBLIC_URL=http://127.0.0.1:54321` for browser-facing URLs while backend containers continued to use `SUPABASE_URL=http://host.docker.internal:54321`.
- The durable decision was recorded in ADR-022.
- Token refresh, app shell, role routing, and product navigation were explicitly deferred to 4.3.5b.

## Files changed by category
- Backend/auth: current-user route and associated auth surface.
- Frontend/auth: Supabase browser client, session provider, and token-aware request wrapper.
- Temporary recovery UI: `/login` and `/tracer`.
- E2E fixtures: local Supabase users, app users, module, memberships, draft/published sections, storage smoke checks.
- Knowledge: spec, plan, report, ADR, and recovery status/log updates.

## Final status
- 4.3.5a status: complete.
- Recovery value: proved the first browser-to-backend authenticated tracer path.
- Temporary tracer status: removed later in 4.3.5e after real UI gates replaced it.
