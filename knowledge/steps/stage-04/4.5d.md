**Must run `--workers=1`** — the manifest is a read-modify-write JSON file, so parallel specs race and
orphan teardown (F-4.5-46). The suite passed in parallel too, but only serial tears down cleanly.

**Gate 2 — Forced-fault browser coverage (deterministic, no key): ✅ DONE.**
```

LLM_FAULT_INJECTION=invalid_output → tests/e2e/4.5d-summary-fault.spec.ts --grep invalid_output → 1 passed
  proof: AIRequestLog status=invalid_output; failure_category=invalid_output (RETRYABLE); 0 summaries
  stored; panel shows "…Retry available."
LLM_FAULT_INJECTION=invalid_input  → --grep invalid_input → 1 passed

  retried); 0 summaries stored; panel shows "…too long…" (no detail leaked)
```

Caveat (F-4.5-47): the AI worker runs `worker.work()` **without** `with_scheduler=True`, so the bounded
RQ retry for `invalid_output` is *scheduled* but never executes — the reliable, observable proof is the
**retryable vs terminal classification** + the badge copy, not a 2nd attempt firing. Actual re-drive is
a Stage 4.6 concern (matches the runbook's per-request-fault-injection-→-4.6 note).
