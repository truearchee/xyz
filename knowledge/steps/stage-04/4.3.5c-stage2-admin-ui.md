## Backend
- Added `ModuleMemberResponse`.
- Added admin-only `GET /admin/modules/{module_id}/members`.
- The projection is read-only, returns active non-admin memberships, includes inactive users with `userIsActive=false`, and sorts by role then email.
- No migration, no assignment/removal write-path change, and no auth redesign.

## Frontend
- `/admin` now renders user and module management as feature components rather than one large page.
- Admin user UI supports user listing, lecturer/student creation, deactivation, and password reset.
- Admin module UI supports module listing, module creation with owner lecturer, assignment, real member listing, and removal from the member list.
- Assignment pickers are derived from real users and exclude inactive users/admins by role.
- `/lecturer` and `/student` render a shared read-only assigned-module list through `GET /modules`.
- Mutations re-fetch after success; mutation failures render `role="alert"`; forms expose submitting state.

## E2E browser proof
The Playwright gate proved:
- Admin logs in.
- Admin creates owner lecturer A, owner lecturer B, main lecturer, and student through UI.
- Admin creates Module A with owner A and Module B with owner B through UI.
- Admin assigns main lecturer and student to Module A.
- Admin verifies Module B does not contain main lecturer or student.
- Admin assigns student to Module B, removes student from Module B, and sees the re-fetched member list without the student.
- Lecturer logs in in a separate context and sees Module A, not Module B.
- Student logs in in a separate context and sees Module A, not Module B.
- Admin deactivates main lecturer through UI.
- Fresh login as the deactivated lecturer is denied app access.
- Existing lecturer session reload is denied app access.
- Student receives 403 from the admin endpoint through the allowed E2E probe.
- Student session remains present and is not redirected to `/login`.
