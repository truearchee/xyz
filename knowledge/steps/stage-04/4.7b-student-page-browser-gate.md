brief + detailed summaries INDEPENDENTLY, each in one of four per-slot states (LOADING / GENERATING /

wall-clock cap). Summary content is server-rendered markdown shown via `react-markdown` with raw HTML
disabled (S1). The module page shows a coarse per-section summary flag (one batched call, no fan-out) +
a link to each section page. Browser gate G1–G9 green against the real backend.

## Post-review closure (R1–R3 + minors)

### R1 — sentinel canary validity (verified by strengthening; G3 stands)
**Trace:** G3's section = A1 (Lecture 1). The gate uploads `sentinel-lecture.vtt` to A1, derives
`transcriptId` from A1's active transcript, and waits for summaries on **that** transcriptId — so the
sentinel rode the exact transcript backing the summary G3 opens. To make this provable (not assumed), the
gate now asserts, right after summaries settle:
- `getActiveTranscriptForSection(a1.id).id === transcriptId` AND `.originalFileName === 'sentinel-lecture.vtt'`,
- `countTranscriptSegmentsContaining(transcriptId, SENTINEL) > 0` (the sentinel IS in the raw transcript segments),
- `getGeneratedSummariesForTranscript(transcriptId)` == `['brief','detailed_study']` (both summaries the
  student reads were generated FROM that transcript).
Then G3 asserts the sentinel is absent from the page AND all 3 student API responses. **Re-run GREEN
(`4.7-student-summaries.spec.ts`, 1 passed, 10.0s).** The canary is live, not vacuous. (Strengthening only.)

### R2 — `--workers=1` root cause: CAPACITY (accept-with-rationale §17; GAP ruled out)
**Classification: CAPACITY** — a transient embed failure under single-worker saturation that the product
SELF-HEALS; not a terminal failure and not a concurrency logic gap.
Evidence (terminal-vs-transient):
- **Embed jobs carry an RQ retry policy: `EMBEDDING_RQ_RETRY_MAX = 3` at `[30, 120, 300]`s**
  (`app/workers/queues.py`). A failed embed is re-enqueued and re-embeds (the job re-reads and re-processes
  since status ≠ 'completed'). So an embed failure is **non-terminal by design**.
- **The embed path is fully concurrency-fenced**: `with_for_update()` throughout +
  `with_for_update(skip_locked=True)` (`embedding_service.py`). The FOR UPDATE SKIP LOCKED primitive IS present.
- **Serial (`--workers=1`) → 100% pass** (full suite 11/11). Parallel → a **random, single, non-deterministic**
  spec fails (full-suite: several; fresh-worker repro: 1×4.3.5e) — the signature of resource contention, not a
  deterministic logic bug.
- **Every parallel run ends with all surviving transcripts `completed`** (DB: `transcript_status=completed`,
  `embed_status=completed`, no lingering `failed`) — the embeds that momentarily showed `failed` recovered.
- **Fresh-worker repro logged NO embed exception** and failed only 1 spec (vs 4 originally) — the original
  4-failure cascade was aggravated by a stale/mixed worker image state, on top of the underlying saturation.
- **Why the test trips:** `waitForTranscriptEmbedded` fails FAST on `transcript.status==='failed'` with a 90s
  wait, which is SHORTER than the 30s+ RQ retry interval — so under parallel saturation it catches the
  momentary `failed` before the retry heals it. Serial avoids the saturation and is the correct harness config.
- **GAP ruled out:** no embed reaches *terminal* failed (retry policy + all-completed end-state prove it);
  the row-locking the GAP scenario worries about is present and not bypassed.
**Honest caveat:** I did not capture the exact transient exception class — the failures occurred on a prior
worker instance whose logs rotated, and the fresh-worker repro produced no embed exception. The
classification rests on the retry-policy design + the non-terminal self-healing end-state + serial-passes,
which is sufficient to distinguish CAPACITY from a terminal-failure GAP. Recorded in open-questions.md as a
non-blocking note (NOT a §17 blocking finding).
