## Summary
Checkpoints A, B, and C are complete. `GET /me` is implemented, mounted, covered by backend tests, visible in OpenAPI, and included in the regenerated TypeScript client. Frontend auth infrastructure is in place with a Supabase browser client, session provider, token-per-request API wrapper, and throwaway `/login` page.

Checkpoint C now has a local Supabase target, local-only `.env.e2e`, E2E compose override, and deterministic fixture seed script. The seed script creates four Supabase Auth users, four matching `app_users`, one `e2e_module`, two active memberships, and draft/published lecture sections. It was run twice with stable counts.

Checkpoint C.5 storage readiness is complete after adding `SUPABASE_PUBLIC_URL`. Backend storage calls keep using `SUPABASE_URL=http://host.docker.internal:54321`, while returned signed URLs use the browser-facing public origin `http://127.0.0.1:54321`. The local private storage bucket exists, the host/browser side can open a backend-generated signed URL, and the exact smoke object was deleted.

Checkpoint D is complete. The throwaway `/tracer` page exists with raw controls for session state, `GET /me`, `GET /modules`, module/section selection, PDF and non-PDF upload actions, publish, signed URL request/open, and 403 display without redirecting to `/login`.

Checkpoint E is complete. The Playwright tracer spec passed all 15 required assertions through real Supabase browser login, real backend, real app DB rows, and real local Supabase Storage. Lecturer and student used separate browser contexts with distinct access tokens. The test proved JWT acquisition, cross-origin authenticated API calls without CORS console errors, browser multipart PDF upload, non-PDF validation, publish, student published-section visibility, draft denial without redirect, backend `Cache-Control: no-store` on `/download-url`, browser consumption of the signed URL without an Authorization header, and student upload 403 while preserving session state.

App shell, role routing, dashboards, admin UI, lecturer UI, and student UI were not started. Token refresh on expiry remains deferred to 4.3.5b.

- C.5 added `SUPABASE_PUBLIC_URL=http://127.0.0.1:54321` for browser-openable storage signed URLs. This does not change auth/JWT issuer behavior.
- Checkpoint D did not add `frontend/src/lib/api/upload.ts`; generated client multipart methods accept browser `File`/`Blob`.

## Decisions made
- `/me` follows the locked recovery plan: Supabase proves identity; backend provides application context.
- ADR-022 was added for the approved `SUPABASE_PUBLIC_URL` signed URL public-origin fix. This backend change beyond `GET /me` was required to prove browser integration against local Supabase Storage: backend containers need the internal `SUPABASE_URL`, while browsers need a reachable public origin in returned signed URLs. The change is limited to signed URL origin rewriting, preserves the path, query string, and signed token parameters, and does not change storage keys, auth/JWT issuer behavior, or bucket visibility.

## Risks introduced
- Admin users with bad-data active module memberships are logged and returned with `activeModuleMemberships: []`. This intentionally avoids teaching frontend route guards about admin memberships.
- NPM reported `2 vulnerabilities (1 moderate, 1 critical)` after adding `@supabase/supabase-js`; no audit fix was run because dependency remediation is outside Checkpoint B.
- Existing backend storage-key generation writes under `modules/{moduleId}/...`; run-scoped `e2e/{runId}/...` prefixes are not supported without a future backend/storage test-infrastructure change. Later upload tests must record and clean exact returned keys only.
- Local Supabase bucket `section-assets` now exists and is private.
- `/tracer` is throwaway and intentionally not an app shell or product UI.
- Token refresh on expiry is intentionally deferred to 4.3.5b.
