# Session 4.5.1a — Partition + Map + Reduce — REPORT (BACKEND VERIFIED)

The map-reduce ENGINE for detailed summaries, removing the single-call transcript-size ceiling
(F-4.5-50/51). Detailed generation is now **partition → map (per unit) → reduce**, fully exercised on
the deterministic adapter. Brief-from-detailed, downstream gating, backfill, and the real-provider/browser
gate are 4.5.1b/c.

## Decisions honored (the four)
1. Distinct `DetailedSummaryPartial` for lenient dispatch (validator dispatches on schema identity).
2. Input-aware adapter → content flow-through proof, not just a manifest count.
3. `job_timeout` scales with `MAX_MAP_UNITS` from one `settings` source (no drift).
4. Detailed `input_hash` folds `partition_config_hash` → distinct provenance per budget, no collision with
   the prior truncated single-call row (and doubles as the C3 stale-partial invalidator).

## Findings
- **F-4.5.1a-1 (pre-existing HEAD debt, FIXED in passing).** The sacred-rule reality check found the backend
  pytest suite was RED on HEAD: the 4.5 Option-A v2 prompt bump (commits 630e118/23d738e) was incompletely
  propagated — `prompts/CHECKSUMS.json` never recorded `brief_summary/v2`+`detailed_summary/v2` (drift guard
  red), and the lifecycle test helpers stamped a hardcoded `"v1"` that no longer matched the v2 expectation
  (5 activation tests red). The E2E "11/11" close did not cover pytest. Fixed: recorded all six prompts in
  CHECKSUMS; made `_make_summarized`/`_create_summary_row` track `EXPECTED_PROMPT_VERSION_BY_SUMMARY_TYPE`
  per type instead of a literal. **This means "4.5 FULLY VERIFIED" was true for the browser gate + the v1-era
  pytest, but pytest silently reddened at the v2 bump — flag for the developer.**
- **F-4.5.1a-2 (eligibility expectation, RESOLVED here).** A map-reduce detailed summary persists
  `prompt_version` = the REDUCE call's version; eligibility (ADR-46-A §3.3) now expects the reduce version
  (single source of truth: `MAP_PROMPT_KEY`/`REDUCE_PROMPT_KEY` live in `summary_specs`, imported by the
  engine). LIMITATION: a map-prompt-only version change is NOT caught by the flat `prompt_version`
  eligibility check (the map version lives in `generationMetadata`); full strategy/version-aware gating is
  4.5.1b §0.1.
- **F-4.5.1a-3 (4.5.1c risk, NOT a 4.5.1a fix).** The stuck-row reaper's flat `REAPER_THRESHOLD_SUMMARY_SECONDS`
  (900s/15min) will prematurely reap a legitimately-running map-reduce detailed job (~20min for a real
  ~60-min lecture, ~9 units sequential). The deterministic adapter returns instantly so the gate is
  unaffected — but before the 4.5.1c real-provider run, the reaper needs a per-job-type (detailed-vs-brief)
  threshold, else it reaps healthy in-flight map-reduce jobs (the resume would recover, but it is wrong).
