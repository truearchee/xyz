## Finding
Checkpoint G was blocked by a baseline-regression expectation conflict:

```text
4.3.5e still requires overallState === "chunked".
With the Stage 4.4 embedding worker enabled, the full stack can advance to overallState === "embedded" before the browser observes chunked.
```

G.1 disposition before continuing Checkpoint G:

- Do not define or use a pre-embedding mode for Checkpoint G.
- Supersede the 4.3.5e exact terminal assertion.
- The 4.3.5e regression now verifies parse/chunk step completion under the projection while allowing `overallState` to be `chunked`, `embedding`, or `embedded`.
- The 4.4 browser gate owns the terminal `overallState === "embedded"` assertion.

This disposition is also recorded in [[steps/stage-04/4.3.5e-regression-under-projection]] and in the Stage 4.4 spec amendments for the pending C-lite ADR.

## Duplicate-Delivery Finding Disposition
The worker-review duplicate-delivery gap is resolved for Checkpoint G with existing and added evidence:

- Existing DB test proves the embed-only active partial unique index blocks duplicate active embed jobs.
- Existing worker test proves partial failure resumes using the same `IngestionJob` row and only stale/missing rows are embedded on retry.
- Added backend source-level guard proves stale batch selection uses `FOR UPDATE SKIP LOCKED`.
- Added backend source-level guard proves embedding batch persistence rechecks current embedding provenance and input hash before writing a row.

Targeted checks containing this evidence passed in the `80 passed` backend run above.

Residual note:

- This is not a two-process RQ stress test. The accepted Checkpoint G rationale is that DB uniqueness prevents duplicate active embed jobs, row selection uses `SKIP LOCKED`, and write persistence rechecks staleness/hash before mutation; the remaining duplicate-encode risk would be wasted work, not corrupt persisted vectors.
