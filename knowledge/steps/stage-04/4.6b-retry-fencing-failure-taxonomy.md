## Summary
Built the lecturer retry surface and its correctness machinery. A failed transcript can now be retried
(`POST …/transcript/{transcriptId}/retry`): the backend resolves the earliest failed step over the true
DAG, resets those jobs, and re-enqueues after commit; the workers do the fenced delete-and-regenerate.
Summaries were **decoupled to fork from parse** (an embed failure no longer blocks summaries), every
destructive write is **fenced** against superseded/stale attempts, and each step records a **sanitized
failure category** surfaced by the projection (+ `retryable`). **No reaper/reconciliation/MaintenanceRun
(4.6c), no UI/browser gate (4.6d).** Backend **329 passed** (+24); migration 0011 round-trips fresh;
frontend `tsc` clean.

## Deviations from spec
1. **Endpoint path** is `…/transcript/{transcriptId}/retry` (section-scoped namespace), not the spec's
   literal `/transcripts/{transcriptId}/retry` — reuses `get_authorized_lecturer_section_context` and
   supports targeting a failed pending replacement. Developer-approved; recorded as a spec amendment.
2. **`unsupported_file`** has no parse-job producer (file type is rejected at upload with 422 before a
   parse job exists); parse emits `parse_failed` for content/format failures and `storage_missing` for a
   missing object. The category stays defined/mapped. Spec amendment recorded.

## Risks introduced
- **DAG decouple** changes the order summaries appear relative to embed (now concurrent with chunk/embed
  off parse output). Covered by explicit decouple tests; activation still gates on embed + both summaries.
- **Dev DB drift** (carried from 4.6a): dev `xyz_lms` is at 0009; apply 0010 + 0011 together at the next
  backend deploy (running pre-4.6 code expects `is_active` and the old enum).
- **Re-parse-after-success** edge (a completed chunk job whose chunks parse deleted) is unreachable via the
  retry endpoint (only failed steps are retried); forced re-runs are a 4.6c reaper concern.
