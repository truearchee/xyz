## Deviations from spec
- The spec said Session 3.1 creates `section_assets`; the repo already had a Stage 2 placeholder table. Per approved plan, Session 3.1 preserves migration history and uses Alembic `0003` to migrate the placeholder into the final upload-owned shape.
- The pasted spec referenced non-existent roadmap/schema paths. The implementation used the current stage-scoped knowledge files and updated `knowledge/architecture/db-spine.md`.
- The generated OpenAPI client now emits multipart file fields as `Blob`; the feature helper passes `File` values directly into the generated service methods. There is no `File as unknown as string` cast.

## Risks introduced
- Supabase Storage calls are covered behind a provider boundary, but the real Supabase SDK path is not live-tested in this session; Docker tests use fake storage.
- A replace commit followed by old-object delete failure can leave an orphaned object. The service logs the orphan key; reconciliation remains future work.
- `SUPABASE_STORAGE_BUCKET` must be present in real backend environments before using the upload endpoints with the real provider.

- A future storage reconciliation job should scan for orphaned objects with no `section_assets` row.
