## Summary
Built and ran the Stage 5.5d dev reseed. No authoritative per-module schedule list exists in the repo
or attached Stage 5.5 context, so the reseed intentionally applies the reference course schedule to
every recreated dev module: 11 May to 26 Jun 2026, Monday/Tuesday/Wednesday lecture, Thursday lab,
Friday quiz day.

The dev DB reseed replaced 16 existing module rows, deleted 962 old sections, recreated 16 modules
through schedule-driven generation, generated 448 stamped sections, restored memberships, and seeded
one published lab with a processable PDF and an attachment `.ipynb`.

## Guardrails proved
- The CLI refused to proceed unless `--confirm-dev-reseed` is present, the environment is non-prod, the
  database host is local/container-scoped, and `alembic_version` is exactly `0021`.
- Storage failures happened before commit and rolled back the DB twice; direct counts after each failure
  stayed at 16 modules, zero reference-schedule modules, and zero seeded assets.
- The successful run used a one-off backend container with no published ports, so it did not contend
  with `kyiv` on `:8000`.
- Stage 5.5d stopped only this workspace's `stage-55` workers for the destructive run and restarted them
  after the DB proof.

## Deviations / notes
- The dev DB had 16 modules at run time, not the roadmap's historical 29-row snapshot. Code wins; the
  report records the actual count.
- The normal `.env` Supabase hostname/key were not valid from the one-off backend container. The
  successful command used the host-published local Supabase URL and mapped `.env.e2e`'s service-role key
  into `SUPABASE_SECRET_KEY` without printing the secret value.
- No frontend/browser proof was run. 5.5e owns the UI/browser gate and full active E2E suite after this
  reseed.
