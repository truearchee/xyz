## Summary
Replaced the fixed 4-section template with schedule-driven generation. Module creation now requires a
`schedule` (weekday × date-range) and generates lecture+lab sections with `week_number`/`session_date`
synchronously inside the creation transaction; a missing schedule returns 422. Added migration 0020
(nullable schedule config on `course_modules`), a pure `generate_sections` function with the 28-section
reference oracle, the schedule request/response contract, and the API-path call-site audit. Verified in
an isolated Docker stack: migration to 0020 (single head) + round-trip, **406 backend tests passing**,
OpenAPI client regenerated, **tsc exit 0**. Not committed — stopped for review before 5.5b.

## Deviations from spec
- **E2E section-selection rework is a 5.5b fix-now task (corrected after review — was wrongly deferred
  to 5.5d/5.5e).** The specs select generated sections by hardcoded title (`'Lecture 1'`, `'Lab 1'`,
  `'Assignment 1'`), which schedule-driven titles no longer match. These specs **seed their own modules
  per run** (`e2e/{runId}`) — they do NOT depend on the 5.5d dev reseed, and **none of them consume the
  5.5b week→sections resolver** (that resolver is a Stage 6 consumer). New titles are deterministic
  (`"Lecture — Week 1 (Mon 11 May)"`), so selection can move to title-pattern or `type` against the
  spec's own schedule-generated module; `4.3.5e`'s "unsupported type rejected" assertion can directly
  seed a non-generated `assignment`/`supplementary` section (enum still exists, D12). So the expected
  quarantine set is **empty** — all 7 are fixable in 5.5b without the resolver or the reseed. 5.5a
  delivered the audit + contract-valid payloads only; the red list below is **PREDICTED, not observed**
  (the full Playwright suite was not run — it needs the heavy e2e stack). **Directive accepted: 5.5b
  opens by running the suite once for an OBSERVED failure list (each red traced to a cause), fixing all
  fixable, quarantining only genuinely resolver-blocked specs (`fixme` + named owner + restore-by-5.5e),
  with a close-out grep gate asserting no quarantine tag survives.**
- **CreateModuleForm.tsx received a minimal UI touch** (out of the strict 5.5a backend scope) because
  the contract change orphaned its sole consumer and tsc-green is a 5.5a acceptance criterion. It sends
  a fixed default pattern AND **labels that interim state visibly to the admin** (an on-form note: the
  weekly pattern is fixed Mon–Wed/Thu pending the 5.5e picker) — thin and evident, not silently fake.
- **seed.mjs left unchanged** — it inserts `course_modules` via direct SQL (nullable columns → still
  valid) and its module isn't schedule-consumed by the shell/routing specs. Real reseed is 5.5d.

## Merge / enforcement note
"No module without a schedule" is **application-enforced only** — the create-module service + the
required `schedule` field on `CreateModuleRequest` produce the 422. The 0020 columns are nullable, so
there is **no DB-level guarantee**: a direct ORM/SQL insert can still create a schedule-less module
(intentional — it keeps ORM-direct test fixtures and the SQL seed working). Do not assume a DB
constraint that isn't there.
