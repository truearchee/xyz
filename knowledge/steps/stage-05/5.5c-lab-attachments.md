## Summary
Built the 5.5c backend feature work. Added `section_assets.asset_kind` with `processable` /
`attachment`, migration `0021` (`down_revision = "0020"`) with existing-row backfill to
`processable`, `.ipynb` lab attachment validation, upload-time lab `dueAt`, and a backend streaming
download endpoint for attachments with `Content-Disposition: attachment` and
`X-Content-Type-Options: nosniff`. PDFs remain processable and use the existing signed URL path.

The deterministic backend proof is a DB assertion: after uploading a notebook attachment through the
API, the test asserts zero rows for `transcripts`, `ingestion_jobs`, `transcript_segments`,
`transcript_chunks`, and `generated_lecture_summaries` for that section.

## Guardrails proved
- **Migration chain/backfill:** `0021_section_asset_kind.py` has `down_revision = "0020"` and updates
  legacy rows to `asset_kind='processable'`; `test_section_asset_kind_migration_backfills_existing_rows`
  proves it by inserting a pre-0021 row at revision `0020`.
- **No pipeline DB proof:** `test_lab_notebook_attachment_upload_sets_due_at_and_creates_no_pipeline_rows`
  uploads an `.ipynb` through the API, then asserts zero matching rows in `transcripts`,
  `ingestion_jobs`, `transcript_segments`, `transcript_chunks`, and `generated_lecture_summaries`.
- **Download-only serving:** attachment downloads go through the backend streaming route and return
  `Content-Disposition: attachment`, `X-Content-Type-Options: nosniff`, and `Cache-Control: no-store`.
- **Signed URL split:** processable PDFs keep signed URLs; attachment signed URL requests return
  `404 SECTION_NOT_FOUND`.
- **Lab-only semantics:** `.ipynb` attachments and upload-time `dueAt` are rejected on lecture sections
  with `422 SECTION_DUE_AT_LAB_ONLY`.
