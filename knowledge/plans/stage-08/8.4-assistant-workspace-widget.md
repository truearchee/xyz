# Session 8.4 — Implementation Plan — Assistant Workspace + Floating Widget (Option A)

## Linked documents
- Spec: [[specs/stage-08/8.4-assistant-workspace-widget]]

## Scope confirmation
**Delivers (Option A, student-role only):** a ChatGPT-style Workspace (list/open/rename/soft-delete
the student's own conversations) and a floating assistant button on every student page, both reaching
the **existing 8.2 grounded lecture chat** through new entry points. Conversation-management
correctness (invariants A–E) + single-source-of-truth between the inline panel and the floating
widget on a lecture page.

**Does NOT build:** SSE (8.3); cross-lecture/cross-week retrieval (8.6); general ungrounded chat
(Appendix B); slide/PDF lecture-review workspace; many-chats-per-lecture; **AI-generated titles
(forbidden, rule 15)**; save-to-glossary (8.5); homework/exam/time modes (8.6); folders/tags/pinning;
sharing/export; lecturer monitoring; voice/video; hard delete (soft-delete only). No new AI surface:
no new prompt, model route, or grounding mode.

## Approach
The hard part is conversation-management correctness, not AI. Key decisions:
1. **Reuse, don't fork.** Get-or-create + send + retry + the 8.2 grounding path are unchanged. The
   chat surface (`AssistantPanel`) is extracted into a shared hook + presentational view so the inline
   panel, Workspace, and widget share one implementation.
2. **Soft-delete + index rebuild (invariant A).** 8.1 already enforces one `lecture_default` per
   (student, section); 0040 adds `deleted_at` and rebuilds the partial-unique index to
   `WHERE conversation_kind='lecture_default' AND deleted_at IS NULL`, so delete-then-reopen yields a
   fresh row. `_existing_lecture_default` + `_resolve_owned_conversation` learn `deleted_at`.
3. **Current-access-wins (invariant C) is structural.** The list query joins through the **same** 4.7
   visibility predicate as the per-row gate (reuse `get_visible_student_section_list`'s pattern), so a
   revoked conversation simply produces no row — and direct open already 404s via
   `_resolve_owned_conversation`. List filter ≡ per-row gate, by construction.
4. **Keyset for messages = the spec's anticipated rule-10 escalation.** The shared offset
   `PaginatedResponse` cannot express a cursor; add a `KeysetPage[T]` **sibling** envelope (leave the
   offset envelope untouched; the conversation list keeps using it). Opaque base64 `(created_at,id)`
   cursor; the `(created_at,id)` tiebreak is load-bearing because 8.1 stamps the user+assistant rows
   with the same timestamp.
5. **Derive-on-read title, no AI, no backfill.** Display title = manual title when
   `title_source='manual'`, else the section title (always present in Option A). Old null-title 8.1
   rows render correctly with no data migration.
6. **`last_activity_at` bumps on user-message creation + successful assistant completion only.** The
   worker bump is guarded by `deleted_at IS NULL` → a worker finishing after a delete can never
   resurrect the conversation (invariant E); rename/delete/failure do not reorder.
7. **Single-source-of-truth store.** A client-side `AssistantStoreProvider` (one poll loop per pending
   conversation, per-conversation drafts) mounted in a new `(app)/student/layout.tsx` makes "a message
   sent in one view is visible in the other" true and avoids double-polling — the browser gate tests
   this explicitly.

## Risks & mitigations
- **Changing the messages endpoint shape** breaks the client → regenerate in-session; both envelopes
  expose `items` so the chat hook is unaffected; full suite re-run catches regressions.
- **Shared store complexity** → keep it thin (message arrays + drafts + poll lifecycle only; access
  stays in the lecture-page adapter); justified by the explicit cross-view gate.
- **Widget over Submit on quiz pages** → read an explicit `data-assistant-safe-area` anchor rather than
  hard-coding quiz DOM.
- **e2e wall-clock** (multiple embedded transcripts + two contexts) → generous `test.setTimeout`; split
  cross-view / workspace / widget-a11y into separate `test()`s.
- **Stage 9 a11y is not actually a reusable pattern** (claimed but thin) → 8.4 establishes
  `useFocusTrap` + aria-live as the pioneer pattern.

## Design decisions (/plan-design-review, D3:B — focused gap-fix, no mockups)
Classified **App UI** (data-dense workspace) → App-UI rules: calm surface, few colors, minimal chrome,
cards only when the card IS the interaction. All surfaces reuse the shipped inline idiom (AssistantPanel
`styles`: block border `#d7dde8`/radius 8/pad 16, primary `#174a63`, muted text `#4b5563`, user bubble
`#eef1f5`, card `#fff`) — NO new palette; the monochrome repaint is the Stage 4.9 shared sweep (debt by
design, not a new divergence).

**P1 Information architecture (4→9).** Workspace `/student/assistant`:
```
[ Assistant ]                                   [ + New chat ]
Your lecture chats.
─────────────────────────────────────────────────────────────
 <Display title, 15/600, #111827>                    2h ago ›
 <Course → Lecture · 6 messages, 13px #4b5563>   [Lecture grounded]
 <last message preview, 1 line ellipsis, 13px #4b5563>
─────────────────────────────────────────────────────────────  (hairline rows, full-width tap)
```
Row scan order: title → course/lecture context → preview; activity time + grounded chip are tertiary.
**List, not cards** (hairline `#e5e7eb` separators, ~14px vertical pad). **Flat, newest-activity-first**;
course shown as a per-row context label, NOT hard group headers (MVP scale — "continue where I left off"
beats taxonomy; group headers deferred). Conversation view: sticky header = back + context pill
("Chatting about: <lecture> · Grounded in published lecture material") + "Open lecture" + overflow
(rename/delete); thread oldest→newest, newest at bottom, "load older" at top on scroll-up.

**P2 Interaction states (5→9).** Every surface gets the full triad:
```
FEATURE            | LOADING                 | EMPTY                                   | ERROR                         | SUCCESS / PARTIAL
Workspace list     | 3–4 skeleton rows       | "No lecture chats yet — open a          | inline card + "Try again"      | rows; "Load more" btn when more pages
                   |                         |  published lecture and tap Ask."+CTA    |  (refetch)                     |
Conversation thread| "Loading conversation…" | "No messages yet — ask below."(existing)| failed bubble + Retry (existing)| bubbles; "Thinking…" pending (4.5d)
Load older (keyset)| top spinner             | —                                       | silent retry next scroll       | prepend older; hasMore=false → none
Lecture picker     | "Loading your lectures…"| "No published lectures yet." /          | "Couldn't load — try again."   | Disabled="Processing…"; Unavail=
                   |                         |  "No modules assigned."                 |                                |  "No transcript yet"; Hidden=absent
Send               | btn "Sending…"(existing)| —                                       | "Couldn't send" + keep draft   | user bubble + pending assistant
Rename             | "Saving…"               | —                                       | "Couldn't rename — try again"  | title updates in place
Delete             | "Removing…"             | —                                       | "Couldn't remove — try again"  | row drops from list
Widget drawer      | list skeleton           | "No chats yet — start from a lecture."  | "Couldn't load — try again"    | recents; pending DOT on button
```
Empty states carry warmth + a primary action + context (not "No items found").

**P3 User journey (6→8).** Lecture page → floating button → THAT lecture's chat (feels: "it knows the
lecture") → ask → honest "Thinking…" (not a frozen spinner) → grounded answer + "where from" (trust) →
later, Assistant nav → chats newest-first (continuity). Emotional risk handled: access-revoked → a clean
"This conversation is no longer available" state with a link back to the list, never a scary error.

**P4 AI-slop guards (7→9).** NO card grid for the list (hairline list). NO icons-in-circles, NO emoji, NO
colored left-border cards, NO centered-everything, NO decorative blobs. "Lecture grounded" = small neutral
text pill (existing muted style), not a colored badge. Widget = plain circular labeled button, not a blob.
Copy is utility ("Ask about this lecture", "Your lecture chats"), never marketing. (Universal "real
typeface" rule is violated project-wide via the system stack — that's the 4.9 idiom, out of scope here.)

**P5 Design-system alignment (5→8).** Calibrated to the SHIPPED inline idiom (code wins), not the
not-in-code monochrome doc. New surfaces (list row, drawer, context pill, chip, picker row, widget button)
reuse AssistantPanel tokens; zero new palette → they ride the 4.9 monochrome sweep with everything else.

**P6 Responsive & a11y (4→9).** Responsive: list/conversation cap measure ~720px ≥768px, full-width <768px;
mobile rows single-line title + truncated preview. Floating button bottom-right with
`env(safe-area-inset-bottom)` and quiz-attempt pages offset off the sticky submit bar via a
`data-assistant-safe-area` anchor. The originally planned visualViewport keyboard-lift/mobile keyboard
gate was not implemented or proven in 8.4, so no keyboard-aware claim is made. Drawer: mobile =
full-height bottom sheet (≤92vh); ≥768px = right panel ~380px. A11y: button real `<button>` ≥44×44,
`aria-label`+`aria-expanded`,
focus ring; drawer `role="dialog"`/`aria-modal`/labelled, focus trap (`useFocusTrap`), ESC closes + returns
focus; message region keeps `aria-live="polite"` (extended to workspace view + drawer); list = `<ul>/<li>`
with real `<a>`/`<button>` rows; rename input has a (sr-only) label; delete confirm is focus-managed +
ESC-cancellable; contrast ≥4.5:1 on all body/preview/time text (keep `#4b5563`, not lighter).

**P7 Resolved decisions** (rationale): list grouping = flat newest-first + per-row course context (hard
headers deferred); pagination = "Load more" button (matches Stage 5 offset envelope; simpler than infinite
scroll); delete = inline two-tap confirm in the idiom (NO Modal component exists) with the exact retention
copy; rename = inline edit (Enter saves / Esc cancels / blur saves, optimistic + rollback); widget pending
= a dot on the button while that conversation has a pending turn.

**NOT in scope (deferred, design):** hard course-group headers; infinite scroll; monochrome repaint of
these surfaces (4.9); unread badges / per-message list timestamps (noise at MVP); motion beyond the
existing press/transition (4.9 motion pass).

**What already exists (reuse):** AssistantPanel `styles` tokens + 4.5d polling/"Thinking…" + `SummaryMarkdown`
(raw-HTML-off) + the `<details>` "where from" disclosure; SessionProvider/`useRole`; the offset
`PaginatedResponse` envelope. No `components/ui/` → build the list/drawer/pill in the same inline idiom.
