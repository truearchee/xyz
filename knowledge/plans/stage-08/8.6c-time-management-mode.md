## Scope confirmation
This delivers only the time-management assistant mode. It is conversational, one active chat per student, day-level only, and grounded in the student's own schedule/progress data. It does not create plans, calendars, `.ics`, Stage 11 planner entities, analytics/gamification records, or exact clock-time blocks. The screenshot-churn fix is included because the owner explicitly asked for it this session.

## Approach
Reuse the 8.6a/8.6b mode seam. `time_management` is another `conversation_kind` with a per-mode turn builder that prepares structured schedule/progress context, then hands off to the shared one-call gateway/persist flow. Add read-only platform/query helpers to gather visible upcoming sections/deadlines and progress summaries for the current student only. Route via `time_management/v1` on Cerebras/V2. Keep the UI in the existing inline idiom: workspace starter, mode label, context pill, starters, and a structured "where this came from" affordance.

The Stage 6 screenshot churn is fixed by making screenshot writes opt-in/redirected via env var instead of writing to committed knowledge PNGs during every rule-14 run.

## Test strategy
- Backend tests prove kind validation, one-active-per-student idempotency, no foreign student data in context, deterministic context ordering/caps, no planner/domain artifacts, route is Cerebras, and snapshot refs are structured.
- Component tests prove UI state and source affordance.
- Browser gate proves the live student path, own-data-only sentinel, empty/partial states where practical, no artifact creation, no planner/analytics import boundary, and general assistant regressions via the full suite.
- Rule-11 smoke proves V2/Cerebras model-ID echo for `time_management/v1`.

## Risks & mitigations
- Schedule/progress query could leak cross-student data -> platform/query takes explicit `student_id`; tests seed another student's sentinel deadline/progress and assert absence.
- Stage 11 boundary creep -> no planner tables/models/imports; browser gate and code scan assert no `WorkloadPlan`, `WorkloadPlanItem`, `InternalCalendarEvent`, `.ics`, or planner-domain import.
- Full-suite screenshot churn -> screenshot capture gated by env var; final `git status` must be clean except intentional files.
- Context gets too large -> deterministic caps: 14-day window, overdue cap, top-N weak topics, compact summaries.
