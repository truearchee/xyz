## Scope confirmation
This delivers the backend half of Sub-scope C: asset kind, migration/backfill, upload validation, lab upload `due_at`, attachment download headers, and deterministic DB proof that attachments do not create transcript or AI pipeline state. It does not build the product UI or Playwright browser gate; those remain in 5.5e, after reseed sequencing.

The developer approved proceeding into 5.5c on 2026-06-16 and called out two report-sensitive checks: migration `0021` must have `down_revision = "0020"` and must backfill existing `SectionAsset` rows to `processable`; the no-pipeline proof must be a backend DB assertion, not a UI label.

## Approach
Keep PDFs on the existing section-asset upload and signed-download path, adding only an explicit `asset_kind="processable"` default. Extend upload validation to classify assets: PDFs remain processable, `.ipynb` is accepted only for lab sections as an attachment after extension, content-type, size, and JSON notebook-shape validation.

Attachments get a dedicated backend streaming download endpoint because signed URLs cannot set `X-Content-Type-Options: nosniff`. The service will reuse the existing role-aware download authorization, reject attachments on the signed-URL endpoint, and set `Content-Disposition: attachment` plus `nosniff` on the streaming response.

## Order of operations

2. Add model and migration changes.
3. Extend validators and service upload classification.
4. Add the attachment streaming endpoint and signed-URL guard.
5. Add and run targeted tests until the backend invariant is locked.
6. Run migration/head checks, full backend pytest, ruff, and regenerate OpenAPI client if needed.
7. Write the report from `git diff` and command output; update STATUS/log/master links.

## Test strategy
Targeted content tests prove behavior at the API and database boundary: `.ipynb` validation, lab-only attachment uploads, `due_at` persistence, attachment download headers, and signed-URL exclusion. The no-pipeline test will query all relevant DB tables after attachment upload and assert zero transcript, ingestion job, segment, chunk, and generated summary rows. `test_db_spine.py` covers schema expectations and migration shape; full backend pytest guards cross-stage regressions.

## Risks & mitigations
- Attachment uploads could accidentally enter future processing paths → `asset_kind` is persisted and the deterministic DB test asserts no current transcript/AI rows are created.
- Signed URL path cannot set `nosniff` → attachments use backend streaming; PDFs keep the existing signed URL behavior.
- Browsers may send varied notebook MIME types → validator requires `.ipynb` extension plus an allowed content type and JSON notebook shape, keeping acceptance practical without trusting MIME alone.
- Upload-time `due_at` could drift from metadata edit rules → reuse the same lab-only validation semantics and cover it in tests.
