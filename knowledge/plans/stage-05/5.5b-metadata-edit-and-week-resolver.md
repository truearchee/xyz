## Scope confirmation
This delivers only backend Sub-scope B from the master spec: a metadata-edit endpoint, authorization,
D13 recompute behavior, and the read-only week resolver in `platform/query`. It does not build
attachments, upload due-date wiring, any UI, reseed, quiz generation, schedule editing, or structural
section mutation.

The developer approved proceeding into 5.5b on 2026-06-16 and added three explicit guardrails:
D13 must decide after reading the full payload, `includeUnstamped=false` must be documented as a Stage 6
safety boundary, and cross-module section access must return backend 404 rather than leaking existence.

## Approach
Reuse the existing content module/section boundary because it already owns section-level routes,
lecturer assignment checks, student 403 handling, and cross-module 404 behavior. Add a strict Pydantic
request model that rejects extra fields, then update only the stored metadata fields that were explicitly
provided.

For D13, compute the patch decision from `model_fields_set`: `weekNumber` present means explicit stored
week wins; `sessionDate` present without `weekNumber` recomputes from the module's stored
`starts_on`/`week_start_day`. The resolver will be a standalone read model in `platform/query` that reads
stored values only and never mutates or recomputes.

## Test strategy
Endpoint tests prove the user-visible contract and authorization split. Dedicated resolver tests prove
the Stage 6 read-model contract without coupling it to FastAPI. The targeted test run exercises only
the new/changed surfaces first; the full backend run is the regression gate.
