## Scope confirmation
This delivers dev reseed tooling only. It replaces legacy dev modules by recreating them with the Stage 5.5 reference schedule and seeds one published lab with PDF + notebook assets for 5.5e. It does not add migrations, product endpoints, UI, browser proof, quiz behavior, in-place stamping, or any production/staging data path.

The schedule decision is explicit: no repo-authored module-by-module schedule map exists, so every recreated dev module uses the reference course schedule: 11 May to 26 Jun 2026, Monday/Tuesday/Wednesday lecture, Thursday lab, Friday quiz day.

## Approach
Put the reusable logic in `backend/app/domains/admin/dev_reseed.py` and keep the CLI in `backend/scripts/reseed_dev_modules.py`. The reseed will validate all safety preconditions before touching rows: explicit confirmation, non-prod environment, local/container database URL, and Alembic version `0021`.

The tool snapshots existing modules and memberships, deletes dependent rows in FK-safe order, recreates modules using the same schedule generator used by module creation, restores memberships without duplicating the owner active membership, publishes the first recreated lab, uploads deterministic PDF and `.ipynb` bytes through the storage provider, and inserts matching `SectionAsset` rows. It reports counts and fixture IDs as machine-readable JSON so the report can quote exact proof.

## Test strategy
Unit/integration tests use the backend test DB and fake storage to prove the destructive logic without touching local dev data. The dev-stack run proves the command path against the actual `0021` database and configured storage. Post-run SQL asserts that recreated modules have 28 schedule-stamped sections and that the published lab fixture has both `processable` and `attachment` assets.

## Risks & mitigations
- Deleting the wrong database -> require explicit confirmation, non-prod environment, local/container DB URL, and Alembic `0021`.
- Storage upload failure after DB changes -> perform storage writes before transaction commit and best-effort delete uploaded fixture objects on failure.
- Existing module IDs change -> this is intentional replacement, not migration; report it clearly.
- Empty dev DB -> fail clearly rather than silently inventing the 29 modules. A separate explicit bootstrap option would be future work if needed.
- Membership duplicates -> skip duplicate active owner membership created by the module creation flow; restore the rest.

## Open questions
- Runtime dev reseed may still be blocked by shared-port ownership or missing Supabase storage. If that happens, leave the tool/test proof complete and record the runtime blocker separately instead of claiming a dev-stack run.
