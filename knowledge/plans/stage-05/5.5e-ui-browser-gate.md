## Scope confirmation
This delivers the Stage 5.5 closing UI and proof: preview-backed module creation, resolver-backed by-week curation, metadata edit, lab upload/download UI, the new 5.5e browser gate, full active E2E green, and evidence-backed docs closeout.

It does not build quizzes, schedule regeneration, section structure editing, student uploads, grading, overdue logic, calendar UI, holiday modeling, notebook execution, or production/staging reseed.

The developer explicitly directed "proceed with 5.5e"; this plan is filed as approved for that scoped work.

## Approach
Expose the two missing read surfaces as thin backend endpoints instead of duplicating backend logic in React: schedule preview calls `generate_sections`, and by-week views call `resolve_sections_by_weeks`. Regenerate the client and wire the existing admin/lecturer/student pages through the wrapper. Keep UI dense and utilitarian, matching the current panels and tables.

For the browser proof, add one focused 5.5e Playwright spec that uses real product UI where required, and targeted API checks where the spec asks for backend absence/rejection. Then run the complete active suite serially against the rebuilt `stage-55` E2E stack after stopping sibling stacks.

Migration seam handling is evidence-based: `origin/main` currently has migrations only through `0013`, so `0020.down_revision` cannot be rebased to `0019` in this workspace. Re-check before final report and record pending unless the Stage 5 band is actually present.

## Order of operations
1. Add backend preview/by-week/student metadata DTO support and focused tests.
2. Regenerate the API client and wire the wrapper helpers.
3. Implement the thin admin, lecturer, and student UI changes.
4. Add the 5.5e Playwright gate.
5. Run backend/frontend static verification.
6. Stop sibling stacks, rebuild `stage-55`, run migrations, seed/run the browser gate and then the full active suite serially.
7. Write the evidence report, update status/roadmap/log/architecture, and commit.

## Test strategy
- Backend tests prove preview uses the generator contract and by-week endpoints use stored resolver rows.
- Frontend `tsc --noEmit` proves regenerated client and UI types.
- The 5.5e Playwright spec proves the browser oracle: exactly 28 reference sections, preview counts, resolver grouping, metadata recompute, lab upload/download, and student deadline display.
- The full Playwright suite proves no Stage 4/4.7 browser regressions remain after reseed.
- `alembic heads` and fresh migration round-trip prove the migration graph is honest; if Stage 5 `0014-0019` is absent, the report records the rebase as pending.

## Risks & mitigations
- Full E2E stack port collisions -> stop `kyiv` and other stacks before `stage-55` browser proof; use the runbook and fixed `COMPOSE_PROJECT_NAME`.
- Frontend container stale source -> rebuild the E2E overlay before browser proof.
- Slow AI/embedding specs -> run serially with `--workers=1` as the runbook requires.
- Student download routing could regress PDFs -> route by `assetKind`; PDFs keep signed URLs and attachments use `/download`.
- Admin access is not membership based -> add a separate admin by-week route instead of forcing admin through `require_module_access`.
