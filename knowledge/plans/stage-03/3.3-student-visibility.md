## Scope confirmation
This session adds shared role-aware section read endpoints and signed read URL minting for student-visible content. Students can read only published, active sections in assigned modules and completed asset metadata; lecturers keep full assigned-module read access and gain the same download URL mint path.

Do not build student writes, asset publish flags, public buckets, API download proxying, signed URL caching/persistence, transcript or summary views, download audit logging, pagination, `/student` routes, or `/lecturer` routes.

Resolved confirmations:
- `SIGNED_READ_URL_TTL_SECONDS=300`.
- No existing section-list endpoint or lecturer section-list DTO exists, so 3.3 introduces the shared list.

## Approach
Use `require_module_access(moduleId)` at the router boundary for all new read and download endpoints. Keep `platform/query` as read-model-only code: student projections encode the published/active/completed filters, while service code selects projections by role and applies download authorization before minting a fresh URL through `StorageProvider.create_signed_read_url`.

The API returns one shared `SectionListItem` list DTO for both roles. Detail remains role-aware: lecturers receive the existing `SectionDetail`, while students receive `StudentSectionDetail` with notes and completed asset metadata only. Download URL responses carry `Cache-Control: no-store`.

## Changes, file by file
- `backend/app/platform/query/content_read.py` — add section list/detail projections for lecturers and students, plus a privileged `get_asset_download_ref` row that includes `storage_key` only for service-layer URL minting.

- `backend/app/domains/content/service.py` — add role-aware list/detail services and download authorization/minting; add consistent `404`, `403 CONTENT_FORBIDDEN`, and `409 SECTION_ARCHIVED` behavior.
- `backend/app/api/routers/content.py` — add shared `/modules/{module_id}/sections...` read routes and the download-url endpoint using `require_module_access`.
- `backend/app/platform/config.py` — add validated `SIGNED_READ_URL_TTL_SECONDS` defaulting to `300`.
- `backend/app/platform/storage/supabase.py` — ensure signed URL creation uses the blocking SDK safely via `asyncio.to_thread` and parses supported response shapes.
- `backend/tests/test_content.py` — extend coverage for student visibility, download authz, live re-validation, no-leak responses, ordering, and student write denials.
- `frontend/src/features/content/` — add typed API helpers and presentational student list/detail/download components; do not store signed URLs beyond the click.

- `knowledge/` — add report, ADRs 012-014, storage architecture updates, status/log/open-question closeout, and linked-document maintenance.

## Risks & mitigations
- Risk: student responses accidentally expose storage or processing fields. Mitigation: use separate DTOs and assert field absence in tests.
- Risk: stale signed URLs are cached by clients or proxies. Mitigation: mint per request, never persist, and set `Cache-Control: no-store`.
- Risk: unpublish is expected to revoke already-issued URLs. Mitigation: ADR-013 records the explicit MVP limit that unpublish blocks future minting only.
- Risk: role forks drift into separate route trees. Mitigation: keep all new endpoints under the shared `/modules/{moduleId}/sections...` prefix.
