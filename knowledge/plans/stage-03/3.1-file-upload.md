## Scope confirmation
Implement lecturer PDF upload, list, and replace for predefined module sections using private Supabase Storage through a backend storage abstraction. Do not build publish/unpublish, lecturer notes editing, student visibility, signed read URLs for students, section management, PDF processing, transcript upload, version history, per-section asset slots, bulk uploads, drag-drop, rich text, idempotency keys, frontend auth/session plumbing, or browser login flow.

The existing Stage 2 placeholder `section_assets` table is migrated in place by Session 3.1 with Alembic `0003`; completed Stage 2 migration history is not rewritten.

## Approach
Add the storage abstraction and validation helpers first, then migrate the asset model to storage-backed metadata. Implement service-layer authorization from `CurrentUserContext` and current module membership rules, then expose multipart endpoints. Use fake storage in tests for deterministic compensation, replacement, and failure scenarios; Supabase Storage remains behind the provider boundary.

## Post-review completion patch
The 2026-05-30 review found that the first implementation should not be accepted yet. The remaining plan is:

1. Refactor upload/replace routes to accept `Request`, authorize role/module/section access first, and parse multipart form data only after authorization passes while preserving the OpenAPI multipart request body.
2. Move section asset listing into `platform/query/content_read.py` as projection rows instead of returning raw ORM rows from the domain list path.
3. Add missing tests for auth-before-form-parse, upload DB-failure compensation, replace DB-failure compensation, unexpected provider `502`, independent checksum verification, and replace row-lock coverage.
4. Remove the frontend `File as unknown as string` workaround by using generated multipart `Blob` request types, or by isolating upload/replace in a deliberate typed multipart wrapper if the generator cannot emit a file type.
5. Re-run Docker backend verification, API client generation, frontend type-check, and `git diff --check` before restoring the spec to `done`.

## Test strategy
Run targeted backend content tests first, then migration round-trip, full backend tests, OpenAPI generation, and frontend type-check. Expected proof: upload/list/replace paths pass, all seven negative authz cases return specified statuses, migration upgrades/downgrades cleanly, the generated TypeScript client supports multipart endpoints, and frontend components compile without adding auth/session plumbing.

## Risks & mitigations
- Storage and DB lack a shared transaction -> compensate with idempotent deletes and structured warnings.
- The placeholder Stage 2 schema conflicts with 3.1 ownership language -> migrate in `0003` and document the correction instead of rewriting history.
- Multipart handling can read unauthorized bodies -> keep route dependencies resolving module access and lecturer role before passing files into service logic.
- Supabase SDK details can leak into domain code -> domain code accepts only `StorageProvider` and tests assert fake-provider behavior.
