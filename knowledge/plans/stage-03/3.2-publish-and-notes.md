## Scope confirmation
Deliver lecturer-only notes editing plus publish/unpublish transitions for assigned lecturers. Do not build student read visibility, storage changes, events, capability differentiation, rich text, section structure management, optimistic locking, `published_at`, or history/audit behavior.

## Preflight / stop conditions
- 3.1 content routes are in `backend/app/api/routers/content.py`, registered without an `/api` prefix, and are module-scoped root routes: `/modules/{module_id}/sections/{section_id}/assets`.
- 3.2 will therefore use module-scoped routes and validate that `module_id` and `section_id` refer to the same section. Mismatches return `SECTION_NOT_FOUND`.
- DTOs use `CamelModel` with Pydantic camelCase aliases.
- Errors use FastAPI's default `{"detail": ...}` envelope, so stable codes are returned as `detail`.
- `module_sections` already has `publish_status`, `lecturer_notes`, and `status` with the required check constraints and nullability. No migration is planned.
- `CurrentUserContext` does not carry memberships; use the established membership query pattern from content/module access code and do not reimplement auth token lookup.
- ADR numbering was checked; `adr-009`, `adr-010`, and `adr-011` are the next free numbers.
- If any of the above changes before implementation, stop and amend this plan before coding.

## Approach
Add a `SectionDetail` projection in the content read layer, then centralize authorization, archived checks, transition rules, notes normalization, and no-op behavior in the content service. Keep the router thin and module-scoped. Frontend work stays presentational and typed through regenerated OpenAPI helpers.

## Changes, file by file
- `backend/app/platform/query/content_read.py` — add a section detail read row and a lookup that joins module membership, validates module/section coherence, and intentionally does not filter archived sections.
- `backend/app/domains/content/validators.py` — add notes normalization/cap helpers and transition validation constants.
- `backend/app/domains/content/service.py` — add publish/unpublish/notes operations, stable error-code helpers, canonical no-op returns before ORM mutation, and reuse the new section lookup.
- `backend/app/domains/content/schemas.py` — add `UpdateSectionNotesRequest` and `SectionDetail` using camelCase aliases.
- `backend/app/api/routers/content.py` — add `PATCH notes`, `POST publish`, and `POST unpublish`.
- `backend/tests/test_content.py` — extend content tests for auth, transitions, notes, archived sections, module/section mismatch, validation timing, and no-op write behavior.
- `frontend/src/features/content/` — add typed section API helpers and presentational `PublishToggle`/`LecturerNotesEditor` components.
- `frontend/src/lib/api/` — regenerate after backend route/schema changes.
- Knowledge files — add ADRs, then write the report after source changes and verification using `git diff`/command output.

## Order of operations
1. File the approved spec and plan.
2. Add backend read projection, validators, service methods, schemas, and routes.
3. Add backend tests and run targeted checks locally as needed.
4. Add frontend helpers/components and regenerate the API client.
5. Run the required Docker verification sequence in order.
6. Write ADRs, the evidence-based report, `STATUS.md`, and `log.md`; update architecture/open questions only if warranted.

## Test strategy
- Service tests prove transition legality, no-op behavior without writes, notes normalization/cap, and archived blocking.
- Router tests prove the auth matrix, module/section coherence, archived `409`, validation timing after authz, and stable error codes.
- Frontend verification is by generated client plus `docker compose exec frontend npx tsc --noEmit`; no frontend test harness currently exists.
- Final acceptance uses the exact verification commands from the spec.

## Risks & mitigations
- Existing asset authorization returns unassigned lecturers as `403`; the new 3.2 operations will implement the stricter requested `404 SECTION_NOT_FOUND` for inaccessible sections without changing 3.1 asset behavior.
- No-op write assertions can be brittle if SQLAlchemy autoflushes after mutation, so implementation returns before assigning fields for unchanged values.
- Generated client files can create whitespace churn; run `git diff --check` after codegen and normalize only generated trailing EOF issues if needed.
