## Scope confirmation
Deliver 11.5 only: a deterministic student-facing `.ics` snapshot export for the student's active workload plan,
using persisted 11.4 `WorkloadPlanItem` rows and current module deadlines. The feature is a file download only.

Do not build OAuth, external calendar sync, auto-update subscriptions, plan regeneration, editable calendar blocks,
plan-item mutations, AI, Stage 10 reads, or 11.6 grade advice.

Housekeeping completed before planning: the leftover local E2E stack on frontend `:3006` and backend `:8006` was
torn down with the Stage 11 E2E compose files.

## Approach
Add a single download-oriented backend path that exports a caller-owned active plan by `plan_id`. The service will
reuse the existing student ownership boundary from 11.4, load the plan and its module deadlines, serialize a
minimal RFC-shaped iCalendar document, and return it as `text/calendar` with an attachment filename.

Timezone approach: emit all event datetimes as UTC-normalized `YYYYMMDDTHHMMSSZ` values from stored aware UTC
instants. Preserve the module timezone, falling back to `INSTITUTION_TIMEZONE`, as calendar metadata and verify
by converting parsed UTC event times back through `zoneinfo.ZoneInfo`. This avoids hand-writing `VTIMEZONE`
daylight-saving rules while still preventing local-time drift.

Deadline approach: export deadlines as timed `VEVENT` marker events at their `due_at` instant. This preserves the
actual due time instead of flattening deadlines to all-day dates.

No migration is planned. Alembic head should remain `0058`.

## Data flow

```text
StudentWorkloadPlanner
  plan.id from GET /student/modules/{module_id}/workload/plan
        |
        v
GET /student/workload/plans/{plan_id}/calendar.ics
        |
        v
service.export_student_workload_calendar(...)
  require role=student
  load WorkloadPlan where id=plan_id
  if missing -> 404
  if student_id != current_user.user_id -> 403
  if !is_active -> 404 or 409, covered by tests
  load WorkloadPlanItem rows ordered by sort_index
  load module active/published due_at sections ordered by due_at/id
        |
        v
calendar_export.build_workload_calendar(...)
  VCALENDAR
  VEVENT study blocks from plan items
  VEVENT deadline markers from due_at sections
        |
        v
text/calendar attachment
```

## Test strategy
- Serializer:
  - assert CRLF-delimited `VCALENDAR` with `VERSION:2.0`, `PRODID:XYZ LMS`, and required `VEVENT` properties;
  - assert plan-item `UID` is derived from `WorkloadPlanItem.id` and remains stable across exports;
  - assert deadline `UID` is derived from `ModuleSection.id` and remains stable across exports;
  - assert `DTSTAMP` can change while UIDs stay fixed;
  - assert `SUMMARY:Study: <label>` for study items and `SUMMARY:Deadline: <title>` for deadline markers;
  - assert descriptions include reason plus estimate and escape commas, semicolons, backslashes, and newlines;
  - assert scheduled plan items use stored `scheduled_start_at` and `scheduled_end_at`;
  - assert unscheduled tight residual items are not emitted as fake timed events;
  - assert a DST-sensitive timezone case converts parsed UTC event times back to the expected local hour.
- API:
  - owner student downloads an active plan;
  - same-module second student requesting the first student's `plan_id` gets 403;
  - lecturer/admin get 403;
  - missing plan returns 404;
  - superseded plan behavior is explicit and covered;
  - response has `text/calendar`, `Content-Disposition: attachment`, and no other student's data.
- Frontend:
  - no export button before a plan exists;
  - export button after plan load calls the download helper;
  - failed download surfaces a recoverable error;
  - plan item area still has no edit/done/drag/accept/reject affordances.
- Browser:
  - seed student one with an active plan and deadlines;
  - click export from the visible workload planner and capture the download;
  - parse the `.ics` file and assert expected study items, deadlines, UIDs, dates, and times;
  - re-export and assert the same UIDs for the same items/deadlines;
  - seed module/institution timezone edge case and assert parsed UTC times convert to the correct local time;
  - student two direct plan export returns 403.
- Closeout:
  - no Alembic migration expected; confirm current head remains `0058`;
  - no rule-11 smoke because no AI;
  - run 11.5 browser gate and then the full active Playwright suite per rule 14.

## Risks & mitigations
- Timezone drift: using local floating times can import one hour off around DST. Mitigation: export UTC `Z`
  datetimes from stored aware instants and test conversion back to the module/institution timezone.
- Invalid iCalendar escaping: summaries/descriptions can break parsing if commas, semicolons, backslashes, or
  newlines are not escaped. Mitigation: serializer unit tests cover text escaping and line endings.
- UID instability: using export time or sort index in UID would break re-export identity. Mitigation: UID derives
  from persisted item/section ids only.
- Privacy leak: exporting by module alone cannot prove direct object access is blocked. Mitigation: export by
  `plan_id` and explicitly return 403 when the plan exists but belongs to another student.
- Scope creep: an `.ics` feature can turn into sync/subscription work. Mitigation: keep this as one attachment
  download and state snapshot semantics in UI/test copy.
- Hand-written serializer risk: the feature avoids a new dependency, but `.ics` has strict formatting. Mitigation:
  keep the serializer tiny, deterministic, and covered by parser-style tests; revisit a library only if tests show
  the local serializer becoming complex.

## GSTACK REVIEW REPORT

### Step 0 Scope Challenge
- Existing code already solves ownership and active-plan retrieval for 11.4; reuse `WorkloadPlan`,
  `WorkloadPlanItem`, and student membership checks instead of adding tables or a second planner.
- Minimum viable change is one serializer, one service method, one download route, one frontend button/helper, and
  tests. No migration, worker, AI, or external-calendar integration is needed.
- Complexity is acceptable despite touching more than eight files because generated client/tests/docs account for
  much of the file count. Core behavior stays in one serializer and one service path.
- No design doc was found in gstack artifacts; the Stage 11 master spec plus owner kickoff brief are the source of
  truth for this narrow export.
- `TODOS.md` was not present in the repo root.

### Architecture Review
- Recommendation: export by `plan_id`, not by module-only active plan. This creates a concrete direct-object
  authz test where another student requests a known plan id and gets 403.
- Recommendation: use UTC-normalized datetimes in the `.ics`, with timezone metadata and tests converting back to
  local time. This is simpler and safer than generating partial `VTIMEZONE` definitions by hand.
- Recommendation: keep deadlines as timed marker events, not all-day events. The user explicitly called out
  dates/times and timezone drift, so preserving `due_at` time is the safer product behavior.

### Code Quality Review
- Keep serialization in `calendar_export.py`, not inline in the router. The route should only set headers and
  return bytes/text.
- Keep UID builders deterministic and obvious:
  - `workload-plan-item-{item.id}@xyz-lms`
  - `module-deadline-{section.id}@xyz-lms`
- Keep escaping/folding helpers private and unit-tested. If they grow beyond basic RFC text escaping plus line
  folding, reconsider using a dedicated library.

### Test Review
- The timezone edge case is mandatory. The test should not merely assert that a date string exists; it should
  parse `DTSTART`, convert from UTC through `ZoneInfo`, and assert the expected local hour.
- The browser gate should inspect the downloaded file content, not just the presence of a download.
- Re-export UID stability must ignore `DTSTAMP`, because `DTSTAMP` correctly changes per export.

### Performance Review
- Export size is bounded by one active plan and module deadlines. No pagination or async job is needed.
- Use ordered, targeted queries for one plan's items and module deadlines; avoid loading unrelated plans or other
  students' rows.
- Response can be generated synchronously; no cache is needed because the export is explicitly a snapshot.
