## Scope confirmation
Deliver 11.4 only: deterministic student availability and workload planning, persisted as `StudentAvailability`,
`WorkloadPlan`, and `WorkloadPlanItem`, with a read-only, list-first student UI. The algorithm is the owner-locked
six phases: `GATHER`, `BUILD TASKS`, `PRIORITIZE`, `LAY OUT`, `RESOLVE OVERFLOW`, `EMIT`.

Do not build AI, `.ics`, grade advice, risk recomputation, Stage 10 reads, assistant, gamification, drag/edit,
accept/reject, mark-done, auto-rescheduling, or any plan-item mutation UI.

## Prerequisite resolution
The previous 11.4 blocker is resolved by owner decision:
- nearest deadline first;
- bounded soft-cap overflow;
- whole remaining course horizon;
- physically impossible work is scheduled as far as it fits and tight-flagged rather than dropped.
- legacy horizon fallback triggers only when `CourseModule.ends_on` is genuinely unknown; it uses the later of the
  latest known remaining `due_at` and the config-backed fallback horizon, records fallback provenance, and must
  never override or truncate a resolvable course end.

This plan records the algorithm in [[decisions/adr-058-stage-11-workload-planner-algorithm]] and the 11.4 spec.

## Data model approach
Add three models:

- `StudentAvailability`
  - one active row per `(student_id, module_id)`;
  - `study_days` JSONB or compact weekday columns/list with validation for Monday through Sunday;
  - `preferred_window`;
  - `max_study_minutes_per_day`;
  - `availability_version`, starting at 1 and incrementing on changes;
  - timestamps.
- `WorkloadPlan`
  - student/module ownership;
  - `algorithm_version`, `input_hash`, `availability_version`, `source_cutoff_at`;
  - `is_active`, `superseded_at`;
  - unique active partial index per `(student_id, module_id)` where `is_active`;
  - timestamps.
- `WorkloadPlanItem`
  - parent plan id;
  - stable `task_key`;
  - optional `source_section_id`;
  - scheduled date/window plus `scheduled_start_at` and `scheduled_end_at`, nullable only for tight unscheduled
    residual items;
  - display label;
  - stored estimate minutes;
  - reason `deadline | gap`;
  - source reason code and source metadata JSONB;
  - `tight` and optional `tight_message`;
  - deterministic ordering index.

The frontend renders the stored estimate and reason from plan items. It does not calculate display estimates.

## Algorithm implementation approach
Implement planner logic in `backend/app/domains/analytics/workload.py`, keeping cross-domain input reads in
`platform/query/analytics_read.py`.

1. `GATHER`
   - Pin `source_cutoff_at = now()` at request start.
   - Fetch course end from `CourseModule.ends_on`. If it is present, the planning horizon is the whole remaining
     course through that course end, full stop.
   - Legacy fallback triggers only when `CourseModule.ends_on` is genuinely unknown. In that case, use the later
     of the latest known remaining `due_at` and `source_cutoff_at + WORKLOAD_PLAN_LEGACY_FALLBACK_HORIZON_DAYS`,
     and record that fallback in plan provenance. If fallback logic would ever truncate a resolvable `ends_on`,
     fail the implementation/review as a rule-10 bug rather than shipping it.
   - Fetch remaining assessed sections with `due_at >= source_cutoff_at`; include section id, title, type,
     week, and due date.
   - Fetch the latest `StudentRiskSnapshot` for the student/module and use its persisted `risk_reasons` as the
     only gap source. Do not call `classify_risk`.
   - Fetch Stage 9 forecast/progress context through an existing or new read model for input hashing/provenance
     only. Do not generate grade advice, alter ordering from forecast state, or add new forecast math.
   - Fetch availability and its version.
2. `BUILD TASKS`
   - Deadline tasks use config-backed default estimates by section type.
   - Gap tasks use config-backed default estimates by reason code/severity.
   - Convert weak-topic risk reasons with topic/target labels into one gap task each. For `workload-v1`, that is
     the `topic_deadline_gap` risk reason or a later risk reason that explicitly carries a stable topic target.
     Non-topic risk reasons remain in input hash/provenance but do not become invented topic tasks.
   - Dedup by target/section/topic: if a gap maps to or matches an upcoming deadline task, merge the gap metadata
     into the deadline task and keep reason `deadline`.
3. `PRIORITIZE`
   - Sort deadline tasks by due date ascending, then stable task key.
   - Sort gap-only tasks after deadline tasks by severity rank descending, then stable task key.
4. `LAY OUT`
   - Generate candidate study slots from `source_cutoff_at` through course end using selected weekdays and
     config-backed preferred-window start/end times. `no_preference` expands to configured windows in stable
     morning, afternoon, evening order.
   - For deadline-bound tasks, include only slot segments whose scheduled end is `<= due_at`; same-day windows
     after `due_at` are ineligible.
   - Allocate task minutes across slots before each deadline where applicable.
   - Keep per-day assigned minutes tracked from persisted estimates, not UI math.
5. `RESOLVE OVERFLOW`
   - For deadline tasks that do not fit under cap, retry relevant days with an overflow capacity of
     `max_study_minutes_per_day + floor(max_study_minutes_per_day * overflow_percent / 100)`.
   - The overflow retry allocates only the unscheduled residual minutes after the normal-cap pass.
   - Assert per-day scheduled minutes never exceed that bound.
   - If a deadline task still cannot fit, allocate as much as possible before the deadline and mark emitted items
     tight with an honest message. If zero minutes fit, emit a tight unscheduled residual item with null
     start/end, the residual estimate, and an explicit no-available-slot message.
   - Gap-only tasks never exceed the daily cap; they fill spare capacity and continue to later eligible days.
6. `EMIT`
   - Compute a canonical JSON input hash from algorithm version, availability version/values, deadlines, risk
     snapshot id/reasons/input hash, forecast context, and config values. Persist `source_cutoff_at` separately;
     do not include the timestamp itself in the hash so identical gathered inputs hash identically.
   - Supersede the old active plan inside the same transaction and insert the new active plan plus items.

## Plan-eng-review findings
- Persisted estimate and reason on every item are load-bearing; `.ics` in 11.5 must be able to export without
  recalculating the planner.
- Superseding must be transactional: mark old active plan inactive/superseded and insert the new active plan in
  one transaction.
- The planner must keep one deterministic stable ordering path for all task types and slot allocations; no Python
  set iteration or raw DB ordering without explicit order clauses.
- Overflow needs two separate capacities: normal cap for initial layout and overflow cap for deadline recovery.
  Gap-only tasks must never opt into the overflow capacity.
- Use the latest 11.1 `StudentRiskSnapshot` as the gap source. Calling the risk classifier in 11.4 would violate
  the dependency contract.
- `sourceCutoffAt` should be pinned once at generate request start and reused through gather, hash, plan, and
  items.

## Codex adversarial review findings
- Failure mode: a physically impossible deadline could create an unbounded over-cap day. Countermeasure: compute
  overflow capacity from config and assert/item-test no day exceeds it.
- Failure mode: a physically impossible deadline with zero eligible capacity could disappear. Countermeasure:
  emit a tight unscheduled residual item with null start/end and a required message.
- Failure mode: same-day windows can land after `due_at`. Countermeasure: configure concrete window times and
  filter deadline-bound slot segments by `scheduled_end_at <= due_at`.
- Failure mode: input hash changes just because regenerate time changes. Countermeasure: persist
  `sourceCutoffAt` separately and hash gathered source data, versions, and config, not the timestamp itself.
- Failure mode: a same-module student query leaks another assigned student's plan. Countermeasure: every read and
  write filters by both `current_user.user_id` and `module_id`; add same-module two-student tests.
- Failure mode: gap-only tasks could starve behind all future deadlines for the whole course. This is expected by
  owner priority, but the plan should still show gap tasks when spare future days exist; tests should seed spare
  capacity.
- Failure mode: a non-topic risk reason becomes an opaque fake topic. Countermeasure: define weak-topic gaps as
  risk reasons with stable topic targets, initially `topic_deadline_gap`; non-topic reasons stay provenance only.
- Failure mode: dedup by display title can merge unrelated topics. Countermeasure: dedup by source section id or
  stable target key first, label only as fallback.
- Failure mode: read-only UI might still expose form controls for plan items. Countermeasure: availability form is
  separate; plan item rows have no buttons except non-mutating navigation if needed.

## Plan-design-review findings
- The planner should be a working student surface, not a calendar toy. Lead with the list of planned study blocks;
  keep any date grouping secondary.
- Use gentle, practical language: "Plan may not fully fit before Friday" rather than alarming labels.
- Availability is the only editable region. Plan items should read as locked recommendations with provenance.
- Tight items need visible text plus status styling, not color-only treatment.
- The UI must remain compact on My Progress and student module pages; avoid a large marketing-style hero or
  nested cards.
- Absence of edit/done/drag affordances is an acceptance criterion, so component tests should assert those labels
  and draggable attributes are absent.

## Risks and mitigations
- Risk: Stage 10 migration collision. Mitigation: use only `0058`, keep `0059` emergency-only, and stop before
  `0060`.
- Risk: risk gaps are stale compared with live risk. Mitigation: 11.4 explicitly reuses the latest snapshot per
  scope; it does not recompute risk. Regenerate after a new AgentRun updates the gap source.
- Risk: schedule math becomes timezone-flaky. Mitigation: use module/institution timezone consistently and seed
  E2E deadlines with explicit datetimes.
- Risk: frontend accidentally calculates estimates. Mitigation: DTO exposes stored `estimateMinutes` and UI tests
  render that value directly.
- Risk: browser gate becomes brittle around current date. Mitigation: seed deadlines relative to the test run and
  trigger generation immediately; no wall-clock waiting.
