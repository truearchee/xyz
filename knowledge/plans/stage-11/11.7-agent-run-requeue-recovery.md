## Scope confirmation
Deliver the stuck-run recovery exactly as requested: existing `queued` and `failed` `AgentRun`s may be re-enqueued with the stable job id only when RQ does not already hold a live job. Do not rebase, merge, add migrations, add AI work, change Stage 10, weaken rule-15, or expand Stage 11 behavior beyond recovery and knowledge cleanup.

## Approach
Centralize AgentRun enqueue reconciliation in `workers/queues.py`. The helper will inspect the stable RQ job id before enqueueing; live RQ statuses suppress duplicate enqueue, while absent or finished/failed/stopped jobs allow recovery enqueue. Scheduler ticks and admin manual triggers will call the helper for newly created runs and for existing rows in `queued` or `failed`, preserving DB idempotency while fixing the post-commit enqueue gap.

## Changes, file by file
- `backend/app/workers/queues.py` - add AgentRun live-job detection and a requeue-safe enqueue helper that remains on the ingestion/default worker queue.
- `backend/app/platform/scheduler/service.py` - re-enqueue due existing `queued` or `failed` scheduled runs when RQ does not already contain a live job.
- `backend/app/api/routers/analytics.py` - manual trigger re-enqueues the returned run when it is recoverable, not only when newly created.
- `backend/tests/test_scheduler.py` - add tests for enqueue failure after commit followed by later re-enqueue, and duplicate suppression when a stable RQ job already exists.
- `backend/tests/test_analytics_api.py` - update manual trigger expectations and cover manual retry re-enqueue for the same run.
- `knowledge/STATUS.md`, `knowledge/roadmap.md`, `knowledge/log.md`, reports - close the knowledge loop and add missing Stage 11.6 document links.

## Order of operations
1. Add recovery helper and adapt scheduler/manual call sites.
2. Add focused tests for enqueue failure/retry and already-live duplicate suppression.
3. Clean unrelated working-tree files.
4. Run focused backend tests and fix failures.
5. Run 11.1 browser gate and full active Playwright suite.
6. Update status, roadmap, log, and the 11.7 report from actual diff/output.

## Test strategy
- Unit/API tests prove same-row recovery and duplicate suppression without requiring real Redis.
- Existing scheduler tests continue proving advisory-lock skip, idempotent run creation, failed-run recording, and AI-free queue selection.
- Browser gate proves the Stage 11.1 manual trigger, polling, live-on-read, and idempotency path still works end to end.
- Full active Playwright suite satisfies rule 14 after the corrective scheduler change.

## Risks & mitigations
- RQ status vocabulary drift could duplicate jobs - keep the live-status set conservative and tests pin duplicate suppression.
- Failed job status might block recovery - treat finished/failed/stopped/canceled or missing jobs as recoverable.
- Manual retry could enqueue completed runs - gate enqueue attempts on `AgentRun.status in {"queued", "failed"}` only.
