## Approach
Add a read-only assessment insight query in `platform/query/analytics_read.py` and expose it through the
existing analytics router/service. The query joins `QuizDefinition -> QuizAttempt -> QuizQuestion ->
StudentAnswer -> AnswerOption`, filters to the requested module and completed attempts, and aggregates in
Python or SQL without returning student identity.

The item key should be stable across attempt snapshots. Prefer `source_pool_question_id` when present; otherwise
derive a deterministic in-memory key from source provenance, question text, and option/correctness shape. The
derived key is only for read grouping; it is not persisted and is not used for topic mapping.

Small-cohort suppression is applied at the aggregate boundary. If a question or topic has fewer than 3 answer
rows, the response returns the exact small-cohort message and omits percentages/ranked insight for that group.
Most-missed questions are ranked only from eligible question aggregates with at least 3 answers.

## Risks & mitigations
- Risk: grouping by attempt-snapshot `quiz_question.id` would make every item look like a one-answer cohort.
  Mitigation: use `source_pool_question_id` when present and a deterministic read-time fingerprint otherwise.
- Risk: topic mapping becomes a hidden inference problem. Mitigation: use only section/summary provenance and
  return unavailable for null/unresolvable provenance.
- Risk: small cohorts leak identity through tiny counts. Mitigation: suppress percentages and ranked insight for
  answer counts below 3, and do not return student identifiers at all.
- Risk: aggregate query grows expensive. Mitigation: scope by module, completed attempts, and shipped indexes
  first; keep compute-on-read in one bounded query path and add only a narrow index if measured tests prove it
  necessary.
- Risk: inherited Stage 11 analytics code accidentally couples to recommendations or risk. Mitigation: keep 11.3
  service/query methods separate and AI-free; tests should assert no AIRequestLog/provider path is touched.

## Plan-eng-review findings
- The main architecture issue is item identity. The plan must not group by `quiz_question.id` because that is an
  attempt snapshot. Use pool identity first, then a deterministic read-time fingerprint for unpooled snapshots.
- Topic mastery is acceptable only because direct provenance exists. The fallback must be an unavailable state,
  not an inferred topic.
- Small-cohort suppression belongs in the backend response, not only in UI, so API consumers cannot accidentally
  render misleading percentages.
- No migration is the correct default. Add `0058` only if verification proves an index is necessary.

## CSO findings
- Authz boundary: the new endpoint must use the same lecturer active-membership check as existing analytics
  endpoints and return 403 for cross-course access.
- Privacy boundary: response DTOs must not include `student_id`, names, emails, attempt IDs tied to students, or
  per-student answer rows. Aggregates only.
- Injection/secret boundary: 11.3 is deterministic SQL/read-model work only. It must not touch AI providers,
  prompts, model routing, `AIRequestLog`, or secrets.
- Small-cohort boundary: backend suppression is required to avoid exposing meaningful tiny-group performance
  data.
