## Implementation phases

### Phase 1 — Deterministic recommendation lifecycle
1. Add the `Recommendation` ORM model and migration.
2. Add enums/checks for:
   - `status`: `active`, `closed`;
   - `lecturer_state`: `new`, `acted`, `dismissed`;
   - `student_state`: `new`, `shown`, `dismissed`;
   - `ai_status`: `not_requested`, `queued`, `succeeded`, `failed`, `template_fallback`.
3. Add a canonical scoped `target_key` such as `module:<module_id>` or
   `topic:<module_id>:<normalized_topic_title>`.
4. Add a partial unique index enforcing one active recommendation per
   `(student_id, reason_code, target_key)` when `status='active'`.
5. Persist:
   - `agent_run_id`, `student_risk_snapshot_id`, `student_id`, `module_id`;
   - `reason_code`, `target_key`, `target_label`;
   - `deterministic_payload`, `algorithm_version`, `input_hash`, `source_cutoff_at`;
   - separate lecturer/student state timestamps;
   - AI text/provenance fields and optional `ai_request_log_id`.
6. Add a recommendation sync that runs after 11.1 snapshot persistence and can also be called by tests.
   It reads latest `StudentRiskSnapshot.risk_reasons` only; it never calls the risk engine directly.
7. For each risk reason, derive:
   - module-level target for forecast, missed quiz, low quiz, and inactivity reasons;
   - topic-level target from `topicTitle` for `topic_deadline_gap`.
8. Upsert/update an existing active recommendation when the reason persists and the deterministic input changes.
9. Revalidate current deterministic reasons on read, using the same 11.1 live-risk inputs, and hide any
   recommendation whose reason no longer holds even if the persisted row has not yet been closed.
10. Auto-close active recommendations whose `(reason_code, target_key)` no longer appears during deterministic
   sync after a run.
11. Never recreate a recommendation for an audience that previously dismissed the same
   `(student_id, module_id, reason_code, target_key)`.
12. Set `AgentRun.recommendation_count` from deterministic recommendation sync, before any AI work.

### Phase 2 — Template path and APIs
1. Add backend DTOs for lecturer recommendation detail and student nudge summaries.
2. Add deterministic template rendering from `deterministic_payload` for both audiences.
3. Add lecturer-owned-student endpoints:
   - read module student recommendation detail;
   - mark a recommendation acted;
   - dismiss lecturer state.
4. Add student-self endpoints:
   - read active module nudge;
   - read at most one dashboard-banner nudge;
   - dismiss student state with `Not now`.
5. Keep all template/API paths AI-free and independently green before adding AI phrasing.

### Phase 3 — Lazy cached AI phrasing
1. Add `recommendation_copy/v1.yaml` and a new gateway feature, tentatively `recommendation_copy`.
   Route choice: short supportive phrasing uses V2/Cerebras per the Stage 11 spec unless implementation hits a
   provider blocker, which must be recorded before changing route.
2. Add a structured `RecommendationCopy` output schema containing lecturer draft and student nudge text.
3. Add a queue helper/job that:
   - uses BACKGROUND priority;
   - has a stable job id per recommendation/prompt/input hash;
   - makes one gateway call per recommendation;
   - stores `aiProvenance { modelId, promptVersion, inputHash, generatedAt }` and AIRequestLog link;
   - skips if the cached AI result is still current.
4. Trigger lazy AI generation from recommendation reads when current AI text is missing/stale.
5. Return template text immediately while AI is queued or invalid. The page must never block on the AI call.
6. Regenerate only when `inputHash` or prompt version changes.

### Phase 4 — Validators
1. Extend the LLM output validation path for `RecommendationCopy` shape.
2. Add `RecommendationCopyValidationContext` before contextual validation runs:
   - `reasonCodes`;
   - `metricKeys`;
   - `targetLabel`;
   - canonical `allowedNumbers`;
   - canonical `allowedFactPhrases`;
   - audience fields for lecturer and student outputs.
3. Build `allowedNumbers` only from displayed deterministic metrics and template text. Do not derive allowed
   numbers from UUIDs, timestamps, hashes, prompt versions, or hidden metadata.
4. Add `RecommendationNumericConsistencyOutputValidator`:
   - rejects numbers, percentages, fractions, ordinals, and number words absent from `deterministic_payload`;
   - rejects peer comparisons;
   - rejects diagnoses;
   - rejects new risk reasons or unsupported factual claims;
   - checks factual claims against allowed reason codes, metric keys, target labels, and allowed fact phrases;
   - applies to lecturer and student text.
5. Add `StudentCopySafetyOutputValidator`:
   - applies only to student-facing copy;
   - rejects banned ideas such as `at risk`, `critical`, `failing`, `behind the class`,
     `other students`, `mental health`, `diagnosis`, `you are not trying`, and close variants;
   - also rejects tier/shame/pressure language such as `needs support`, `watch`, `risk tier`, `warning`,
     `red flag`, `urgent`, `you are behind`, `not putting in effort`, `lazy`, `unmotivated`, `careless`,
     and `disengaged`;
   - lecturer copy is exempt from this student-only tone guard.
6. Diagnosis/mental-health and unsupported-fact checks apply to both lecturer and student outputs.
7. Validator failure rejects the whole AI result, causes one bounded regeneration attempt, then deterministic
   template fallback. Never persist or return partial AI text where one audience passed and the other failed.
8. Add adversarial unit tests for:
   - invented numbers;
   - peer comparison;
   - diagnosis/mental-health implication;
   - new risk reason;
   - banned student phrase;
   - valid allowed numbers copied exactly from deterministic payload;
   - valid allowed fact phrases such as topic name, deadline phrasing, and lecturer metric values;
   - lecturer copy using direct risk language without tripping the student-only guard.
9. Adversarial cases must include:
   - digits, decimals, percentages, ordinals, fractions, ranges, and number words (`two`, `third`, `half`,
     `2-3`, `1/2`, `90 percent`);
   - peer terms (`class average`, `cohort`, `peers`, `classmates`, `most students`, `everyone else`,
     `below average`, `bottom group`, `compared with`, `rest of the class`);
   - unsupported facts (`attendance`, `missing assignments`, `falling grades`, `poor participation`,
     `late submissions`, invented deadlines);
   - diagnosis/mental-health terms (`anxiety`, `depression`, `ADHD`, `burnout`, `mental health issue`,
     `diagnosed`, `struggling emotionally`, `seems depressed`);
   - combined-output behavior where valid lecturer text plus invalid student text rejects the entire result.
10. Add positive-control tests proving a correct AI draft passes validators and the API/UI renders AI text rather
   than the template.

### Phase 5 — Frontend surfaces
1. Lecturer module roster rows link into a student-detail recommendation view using existing design-system
   primitives.
2. The lecturer recommendation modal contains deterministic reasons, cited metrics, AI/template draft,
   student nudge preview, and exactly three actions:
   - Copy draft;
   - Mark acted;
   - Dismiss.
3. There is no Send button or send endpoint.
4. My Progress `Where you stand` becomes the primary student nudge placement.
5. Student dashboard gets a secondary banner limited to one active nudge.
6. Student nudge supports inline `Not now`; it is never modal, push, email, or toast.
7. Lecturer detail includes a preview of the student-facing nudge.
8. Pending-AI state displays template copy with a quiet generation status for lecturers only; student copy
   should not expose AI machinery.

## Plan-eng-review findings
- Lifecycle correctness is the critical path. The database partial unique index owns one-active enforcement, while
  service tests cover state transitions and audience separation.
- Auto-close must be tied to the latest snapshot-derived reason set during deterministic sync.
- Visibility is the live-on-read exception by design: reads re-run the same 11.1 current-risk aggregation only to
  decide whether a persisted recommendation is currently visible. This prevents stale nudges after a student
  catches up while keeping creation/provenance tied to snapshots.
- Dismissed-never-reshown needs historical lookup, not just active-row filtering. Otherwise a cleared and later
  recurring reason could resurrect a dismissed nudge.
- AI should not be in the `AgentRun` critical path. `AgentRun.recommendation_count` is deterministic; AI copy is
  asynchronous enrichment and must not block the scheduler/manual trigger.
- Performance risk is N+1 detail loading. Add module/student indexes and batch read latest recommendations for
  roster/detail surfaces where practical.
- The current platform `OutputValidator.validate(...)` signature has no recommendation context. Implementation
  must add an explicit contextual validation layer before AI text is persisted or returned.

## Plan-design-review findings
- Lecturer surface should prioritize deterministic reasons and metrics above draft copy so the lecturer can judge
  whether to act manually.
- The student surface should read as a calm progress note, not a warning. No tier labels, peer language, risk
  labels, or modal interruptions.
- The old design plan says "draft-message modal" but does not currently show a literal Send action. The 11.2 spec
  wins over any old send-oriented interpretation. If implementation finds an existing Send affordance, record a
  rule-10 findings note and remove it.
- Required UI states: loading, no active recommendation, template fallback, AI pending, AI succeeded, lecturer
  acted, lecturer dismissed, student dismissed.

## Security and authz plan
- Lecturer detail endpoints must prove both module ownership and student membership in that module.
- Student endpoints must derive the student id from the authenticated user; no user-supplied student id for student
  reads or dismissals.
- Copy and dismiss actions must not disclose whether another student's recommendation exists.
- AI payloads must contain only deterministic facts needed for copy generation, not raw private quiz attempts or
  unrelated student data.
