# Plan — 4.5.1b Brief-from-detailed + gating + backfill + fencing

Grounded in the 4.5.1a code. Integration points confirmed: `insert_summary_jobs`
([summary_service.py](backend/app/domains/transcripts/summary_service.py)) creates both jobs at
embed-complete ([parse_service.py:254](backend/app/domains/transcripts/parse_service.py#L254)); the detailed
handler's success tail already calls `_try_activate_after_summary` (the natural place to fork the brief);
`get_latest_transcript_summaries` ([summary_read.py:19](backend/app/platform/query/summary_read.py#L19)) reads
completed summaries; `retry.py`/`reaper.py` re-enqueue by job_type.

## Build order

### 1 — Brief-from-detailed prompt + schema input (ADR-052)
- New prompt `prompts/brief_from_detailed/v1.yaml` (BRIEF route: model = `LLM_BRIEF_MODEL_ID`, backend
  `cerebras`, small `max_tokens`). `{{transcript}}` slot carries the **detailed summary's content_json**
  serialized (the prompt states it is a detailed summary to compress, not a transcript). Record its hash in
  CHECKSUMS.json.
- `SummaryFeature` += `brief_from_detailed`; extend the `ck_ai_request_logs_feature` check (migration 0016).

### 2 — Rewire the brief handler to read the completed detailed
- `_claim_summary_job` for the BRIEF spec: instead of loading + truncating segments, load the latest
  completed eligible DETAILED `GeneratedLectureSummary`; if none exists yet, the brief job is NOT ready —
  re-defer (return a "not-ready" sentinel that leaves the job queued without consuming an attempt) so the
  detailed-completion fork (step 3) is the trigger. Carry the detailed `content_json` + source detailed id as
  the brief's input; `input_hash` derived from the detailed row's identity.
- `_generate_summary_async` BRIEF branch: `gateway.complete(prompt_key=brief_from_detailed/v1,
  output_schema=BriefSummary, context_refs.transcript_text=<serialized detailed content>, feature=
  brief_from_detailed)`.
- `_persist_summary_success` brief: `generation_strategy='derived_from_detailed'`, `truncated=false`,
  `generation_metadata={ sourceDetailedSummaryId, briefPromptVersion }`.
- **OB1 fallback:** when `ENABLE_DETAILED_SUMMARY` is false, the brief keeps the existing transcript-based
  single-call (Option-A truncated) path (`single_call`). One branch on the setting (confirm OB1 first).

### 3 — DAG: detailed-completes → enqueue brief
- `insert_summary_jobs` at embed-complete: create the DETAILED job (and the brief job row, left queued but
  NOT enqueued onto the worker, OR not created until detailed completes — decide in OB-resolution). Simplest:
  create both rows as today, but only ENQUEUE detailed; the detailed handler enqueues the brief on success.
- detailed handler success tail (next to `_try_activate_after_summary`): `enqueue_generate_brief_summary` for
  the same transcript's brief job. Idempotent (the brief job's claim no-ops if already completed/running).
- This makes the order embed → detailed (map-reduce) → brief → activation-ready. Activation already fires
  from whichever leaf is last (4.6 F-4.6b-2); confirm the brief leaf triggers activation.

### 4 — Full-coverage gating predicate (§0.1)
- Pure `is_full_coverage_detailed(summary) -> bool` in the summary domain (e.g. `summary_eligibility` or a new
  `summary_gating`): `generation_strategy == 'map_reduce' and not truncated`. Unit-tested truth table.
- Surface the boolean on the read projection / summary DTO so an admin/consumer can see "full coverage"
  (no Stage-5 consumer built — this is the gate Stage 5 will call).

### 5 — Backfill re-enqueue command
- `app/domains/transcripts/backfill.py` (+ a worker/CLI entrypoint): select ACTIVE-transcript detailed
  summaries where `generation_strategy != 'map_reduce' OR truncated = true`; for each, reset+enqueue its
  detailed IngestionJob (reuse `_ensure_summary_job` / `enqueue_summary_job`). Bounded by a per-run cap;
  `dry_run` reports the selection without enqueuing; idempotent (skip transcripts whose detailed is already
  map_reduce+¬truncated). Unit-tested on the deterministic adapter (stale rows → re-enqueued → regenerated
  map_reduce; dry-run writes nothing; cap honored).

### 6 — Stale-write fencing audit (§6.1)
- Add the not-superseded + checksum/partition-hash guard before each `MapUnitSummary` write (map_reduce.py)
  and the brief write, mirroring the existing final-persist fence; reuse 4.6 helpers. Test: a forced
  mid-flight supersede writes no partial, no reduce row, no brief.

### 7 — AIRequestLog chain helper
- A read helper returning the correlated chain (map units + reduce + brief) for one detailed
  `ingestion_job_id`, ordered; test asserts completeness + correlation.

## Tests (BACKEND VERIFIED)
Brief derives from detailed (input asserted = detailed content, not transcript); brief deferred until
detailed exists then forked on detailed-completion; gating truth table; backfill selection/idempotency/cap/
dry-run; fencing (mid-flight supersede writes nothing); AIRequestLog chain; activation readiness under the new
ordering; OB1 fallback path when detailed disabled; migration 0016 round-trip. Full suite stays green.

## Risks
- **R-b1 brief-deferral correctness** — the brief must not consume an RQ attempt while waiting for detailed
  (else it exhausts retries before detailed finishes). The not-ready sentinel + fork-on-completion avoids a
  poll/retry storm. Verify the brief job is enqueued exactly once, after detailed.
- **R-b2 retry/reaper under the new DAG (OB3)** — a brief-only retry must read the existing completed detailed,
  not re-summarize the transcript; a reaper re-enqueue of a stuck detailed must re-fork the brief.
- **R-b3 detailed-disabled mode (OB1)** — keep the transcript-based brief fallback so the cost-control mode
  still produces a brief, unless the developer chooses "no summaries when detailed off."

## Out of scope (4.5.1c)
Running the backfill on real data; the real-provider smoke; the browser gate; the 4.7 re-gate; the reaper
per-job-type threshold fix (F-4.5.1a-3, a 4.5.1c prerequisite).
