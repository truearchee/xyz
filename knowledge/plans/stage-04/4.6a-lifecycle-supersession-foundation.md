## Scope confirmation
Delivers ONLY the 4.6a foundation: lifecycleState migration + lineage, one-pending index,
section-locked creation/activation, fencing-friendly per-row provenance stamps, the
`summary_eligibility` domain service + read-only `ActiveTranscriptSummaryResolver`, and the env-gated
fault-injection harness. Does NOT deliver: retry + fenced deletes (4.6b), reaper/reconciliation/
MaintenanceRun (4.6c), lecturer UI + preview endpoint + browser gate (4.6d), student surface (4.7).

Key reconciliations with current code (verified before building):
- `GeneratedLectureSummary` is success-only (no `status` column) → "generated" maps to row existence.
- detailed `summary_type` is `'detailed_study'`; summary prompts are `v1`.
- Fencing token is the existing `IngestionJob.attempts`; the "current job" pointer is the one-active
  partial-unique index (embed/summary only — parse/chunk deferred, see Risks).
- `TranscriptMeta` did not expose `is_active`, so removing it is not a wire change; adding
  `lifecycleState` is a deliberate, reviewed contract addition.

## Approach
Clean pre-MVP cut from boolean `is_active` to `lifecycle_state` enum-via-CHECK. Replacement becomes a
`pending` row created under a `module_sections` `FOR UPDATE` lock (serializes double-click / two
co-lecturers); the prior pending is discarded first. Activation (`try_activate_pending_transcript`)
gates on `overall_state == 'summarized'` (existing projection) + exactly-one-eligible brief+detailed
(domain eligibility) and swaps old→superseded / pending→active atomically. The eligibility predicate
lives in the domain and is wrapped read-only in `platform/query`. Provenance is stamped on the happy
path so 4.6b fencing/audit has the data. The fault harness is a new env-gated module, separate from
the LLM-transport `LLM_FAULT_INJECTION`.

`test_pipeline_stamps_per_row_provenance` in `test_transcript_worker.py`. Rewrite the three 409 tests

Maps to the spec Verification: fresh-DB migration round-trip, `pytest`, `tsc --noEmit`.

## Risks & mitigations
- **Unit-of-work flush order** can violate the one-active / one-pending partial-unique index → flush
  the demotion before the promotion (activation + discard both).
- **Discard lineage FK** points at the not-yet-inserted new row → demote with NULL pointer, insert,
  then back-fill.
- **parse/chunk one-active index** would break the tested two-chunk-jobs-coexist replacement path →
  DEFERRED to 4.6b (embed/summary indexes already exist).
- **Lock ordering** (section→transcript) kept consistent between upload and activation to avoid deadlock.
