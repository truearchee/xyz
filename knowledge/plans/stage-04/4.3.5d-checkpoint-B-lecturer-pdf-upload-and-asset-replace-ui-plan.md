## Scope confirmation
Implement only the lecturer-side PDF upload and asset-level replace UI for existing generated module sections. Do not add publish/unpublish controls, student UI, signed URL opening, backend changes, generated-client changes, section create/delete/reorder, transcript/AI work, or Stage 3 FULLY VERIFIED status.

## Implementation approach
1. Run the required pre-checks from the checkpoint spec.
2. Confirm `frontend/src/lib/api/upload.ts` exports `uploadSectionAsset` and `replaceSectionAsset`, uses multipart field `file`, attaches bearer auth, does not set multipart `Content-Type`, and returns `SectionAssetResponse`.
3. Confirm the asset read path. If section detail/list does not include assets, use `GET /modules/{moduleId}/sections/{sectionId}/assets` through a wrapper method.
4. Add a thin `api.content.listAssets` wrapper only if needed for re-fetching asset rows.
5. Add lecturer content components:
   - `SectionUploadControl.tsx`
   - `SectionAssetList.tsx`
   - `SectionAssetRow.tsx`
6. Update `LecturerModuleDetail.tsx` to load assets for each section from backend data, render upload controls, render asset rows, and re-fetch after upload/replace.
7. Keep `publishStatus` rendering on the section and `processingStatus` rendering on each asset row with asset-id-specific test IDs.
8. Render upload/replace failures as `role="alert"`.
9. Run the required verification commands and browser smoke.
10. Update report/findings/status/architecture/log and commit.

## Verification plan
- Rebuild/restart the frontend container before Docker-backed frontend verification if source was copied into the image.
- Run `docker compose exec frontend npx tsc --noEmit`.
- Run `docker compose exec frontend npx next build`.
- Run the direct-fetch scan and expect no matches outside generated core and `upload.ts`.
- Run the JWT/session-role scan and expect no matches.
- Run `git diff --name-only -- backend` and expect no backend changes.
- Run `bash scripts/generate-api-client.sh` and confirm no generated model/service/core/index drift. If `wrapper.ts` changed, treat that as intentional hand-written wrapper work, not generated client drift.
- Run browser smoke with local Supabase overlay and prove PDF upload, asset row render, processing status, asset-level replace, non-PDF backend rejection alert, and no publish/student scope drift.

## Risk controls
- Do not duplicate multipart upload logic in components; import only from `frontend/src/lib/api/upload.ts`.
- Do not rely on local selected file state as success; always re-fetch backend asset data after mutations.
- Do not introduce client-only PDF validation as the proof path; browser smoke must observe backend rejection for a non-PDF upload.
- Do not change backend read projections. Use the existing asset-list endpoint if assets are not in section detail/list.

## Open questions
None. If the current code lacks an asset read path, stop and record a finding instead of expanding backend scope.
