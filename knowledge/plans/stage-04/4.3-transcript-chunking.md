## Scope confirmation
Implement the approved Session 4.3 spec as a worker-internal chunking stage after transcript parsing. A successful parse creates a recoverable queued `chunk` job in the parse-completion transaction, enqueues that job by id after commit, and the chunk worker persists ordered normalized `transcript_chunks` while leaving `transcript_segments` unchanged.

Do not build HTTP endpoints, OpenAPI regeneration, frontend UI, embeddings, summaries, retry/backoff, recovery sweepers, user-facing chunk DTOs, semantic/AI filtering, or segment splitting.

## Approach
Add the schema and model surface first, then pure chunking code, then worker handoff and handler code. Use `ingestion_jobs.idempotency_key` as the correctness surface, persist worker result counts in `result_metadata`, and serialize chunk replacement by locking both the job row and owning transcript row. Use the completed parse job's persisted `processor_version` in the chunk idempotency key, never the current parser code constant.

## Implementation rules
- `TranscriptChunk` rows include `updated_at`, nullable embedding placeholders, ms timestamps mirroring ADR-018, and no redundant standalone `ix(transcript_id)`.
- Normalization is minimal and structural only: collapse whitespace, remove deterministic VTT leftovers, skip empty normalized segments, and drop adjacent duplicates.
- If one segment exceeds `TARGET_CHUNK_WORDS` but is below `HARD_CHUNK_WORDS`, emit it as a normal singleton with `is_oversized=false`; if it is `>= HARD_CHUNK_WORDS`, emit an oversized singleton with `is_oversized=true`.
- Chunk replacement is atomic. Transaction A deletes old chunks, inserts fresh chunks, updates transcript/job success state, and commits. If transaction A errors, rollback preserves any prior completed chunk set. Transaction B then marks job/transcript failed and stores sanitized error.
- Before relying on idempotency, assert the DB has uniqueness for `ingestion_jobs.idempotency_key`; if absent in the target schema, add it before chunk concurrency logic.

## Risks & mitigations
- Enqueue can fail after DB commit — keep the queued chunk job durable for 4.6 recovery and log sanitized metadata only.
- Transaction failure during replacement could wipe old chunks — use a single replacement transaction and record failure in a separate transaction after rollback.
- Version drift can create false chunk keys — read `processor_version` from the completed parse job that produced the segments.
- `vector(384)` has no Python package — use a SQLAlchemy-compatible custom/raw column type rather than adding a dependency unless unavoidable.
