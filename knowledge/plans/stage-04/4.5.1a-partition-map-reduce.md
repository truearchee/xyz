# Plan — 4.5.1a Partition + Map + Reduce

Grounded in a read of the real integration surface (sacred rule). Key code facts that shape every step:
- **Gateway is the only provider path** ([gateway.py:112](backend/app/platform/llm/gateway.py#L112)) — every map/reduce call is a
  `gateway.complete(prompt_key, output_schema, context_refs, priority="background", feature, attempt_number)`. It owns
  logging/limiting/validation. No new transport.
- **`registry.render` has exactly two slots** `{{transcript}}` + `{{section_type}}` ([registry.py:132](backend/app/platform/llm/registry.py#L132), placeholder
  required at load). Map feeds the unit text into `{{transcript}}`; reduce feeds the **serialized partials** into `{{transcript}}`.
  No registry change.
- **Validator dispatches on schema identity** (`output_schema is DetailedSummary`, [validation.py:132](backend/app/platform/llm/validation.py#L132)). → **OA1 RESOLVED:** a
  distinct `DetailedSummaryPartial` type is REQUIRED (not for different fields — for lenient dispatch). Reduce reuses the strict
  `DetailedSummary` path unchanged.
- **Deterministic provider keys canned output off `rendered.prompt_key.name` and can read `rendered.content`** ([provider.py:322](backend/app/platform/llm/provider.py#L322)).
  → map/reduce canned output is **input-aware** (marker-echo) so the late-topic-sentinel proves content flow-through provider-free,
  not just structurally.
- **Superseded-fence already exists** in `_claim_summary_job` + `_persist_summary_success` ([summary_service.py:358](backend/app/domains/transcripts/summary_service.py#L358),L439). 4.5.1a
  reuses it for the final write; the per-write checksum/partitionHash GUARD is the C3 resume rule (in scope); the comprehensive
  fencing audit is 4.5.1b.

## Build order (each step compiles + has tests before the next)

`LLM_SUMMARY_MAP_UNIT_CHAR_BUDGET` (default **9000**), `LLM_SUMMARY_MAP_UNIT_TOKEN_BUDGET` (**2200**, est via the D2 chars/3.5),
`LLM_SUMMARY_REDUCE_INPUT_CHAR_BUDGET` (**9000**) + `_TOKEN_BUDGET` (**2200**), `LLM_SUMMARY_MAX_MAP_UNITS` (**20**),
`LLM_SUMMARY_MAP_CONCURRENCY` (**1**, hard-cap 2). All `_int(..., minimum=1)`. Budget = HEADROOM under the 115–145s ceiling, not 12K.

### 2 — Schemas ([summary.py](backend/app/platform/llm/models/summary.py))

(semantically: a unit may legitimately lack examples/definitions). Add `PARTIAL_SCHEMA_VERSION = "detailed-partial-v1"`. No change to
`DetailedSummary`/`BriefSummary`.

### 3 — Prompt type + feature Literal ([prompt.py:8](backend/app/platform/llm/models/prompt.py#L8))
`SummaryFeature = Literal["summary_brief", "summary_detailed", "detailed_summary_map", "detailed_summary_reduce"]`
(`brief_from_detailed` is added in 4.5.1b). This Literal flows into the gateway + logging unchanged.

### 4 — Validator ([validation.py](backend/app/platform/llm/validation.py))
Extend the three type unions (`validate` param, gateway `output_schema`, `CompletionResult.parsed`) to include `DetailedSummaryPartial`.
Add runtime dispatch `if output_schema is DetailedSummaryPartial: return self._validate_detailed_partial(raw_text)` →
last-valid-object select (reuse `_candidate_objects`/`_select_last_valid`) → `model_validate` + **refusal-marker reject only**; empty
sections ALLOWED, NO lab-notes check. Reduce path = the existing strict `_validate_detailed` (lab-notes enforced there).

### 5 — Prompts (new YAML, mirror [detailed_summary/v2.yaml](backend/prompts/detailed_summary/v2.yaml))
- `prompts/detailed_summary_map/v1.yaml`: model `MBZUAI-IFM/K2-Think-v2`, backend `nvidia`, `max_tokens` ~3000 (a partial is small),
  `{{section_type}}` + `{{transcript}}` (= ONE consecutive PORTION). Instructs: summarize ONLY this portion; sections may be empty if
  absent from this portion; same camelCase contract; no cross-portion speculation.
- `prompts/detailed_summary_reduce/v1.yaml`: backend `nvidia`, `max_tokens` ~8000, `{{transcript}}` = **partial summaries of consecutive
  portions of ONE lecture** (state this explicitly — it is NOT raw transcript). Instructs: MERGE + DEDUPE + resolve cross-portion
  references into ONE coherent `DetailedSummary`; preserve every portion's distinct content; same output contract.
  (`{{section_type}}` retained for lab-notes coherence.)

### 6 — Migration `0015_map_reduce_summaries.py` (existence-checked, mirror [0014](backend/alembic/versions/0014_summary_truncation_flag.py); revision "0015" down "0014")

- New table `map_unit_summaries`: `id` uuid7 PK; `transcript_id` FK→transcripts CASCADE; `unit_index` Int; `start_segment_id`/

  **UniqueConstraint(transcript_id, unit_index, partition_config_hash, source_transcript_checksum)** = partition-bound identity (C3);

- `ai_request_logs`: DROP + recreate `ck_ai_request_logs_feature` to add `detailed_summary_map`, `detailed_summary_reduce`. Mirror the

- New ORM model `MapUnitSummary` + export from `app/platform/db/models/__init__.py`; add `generation_strategy`/`generation_metadata`

### 7 — Engine `backend/app/domains/transcripts/map_reduce.py` (new; pure where possible)
- `partition_segments(segments, *, char_budget, token_budget, chunker_version, transcript_checksum) -> Partition` — group CONSECUTIVE
  normalized segments into the FEWEST units each under BOTH caps (reserve prompt overhead); never split a segment; deterministic;
  `partition_config_hash = sha256(budget + chunker_version + transcript_checksum)`. Raise `SummaryGenerationError` if units >
  `MAX_MAP_UNITS` (cost guard). Pure → directly unit-tested.
- `reduce_with_guard(partials, *, char_budget, token_budget, call) -> (DetailedSummary, coverage_manifest)` — **§3.3 C1 GUARD**:
  serialize partials; WITHIN budget → one `call` (a reduce gateway.complete); OVER → **tiered**: greedy-fill ordered groups each under
  budget → reduce each group (each a gateway.complete, feature=`detailed_summary_reduce`, output=`DetailedSummary`) → recurse on the
  group-summaries until one fits. Each group carries a coverage sub-manifest; the final `coverage_manifest` unions them so it lists
  EVERY original `unit_index`. Deterministic grouping (**OA2 RESOLVED**: greedy by serialized size, stable order).
- `run_map_reduce_detailed(...)` orchestrator (called by the detailed handler): partition → for unit in units (SEQUENTIAL, honor
  `MAP_CONCURRENCY`): **C3 resume** — reuse an existing `succeeded` `MapUnitSummary` iff `partition_config_hash` AND
  `source_transcript_checksum` match, else `gateway.complete(map)` (feature=`detailed_summary_map`, output=`DetailedSummaryPartial`,
  `context_refs.transcript_text=unit.text`) → persist `MapUnitSummary` (with the map call's `ai_request_log_id`). Superseded-fence
  before each unit write (reuse existing check). Then `reduce_with_guard(partials)`. Return a `MapReduceOutcome` { reduce result
  (parsed `DetailedSummary` + reduce `ai_request_log_id`), `map_unit_count`, `source_map_unit_summary_ids`, `coverage_manifest`,
  `map_prompt_version`, `reduce_prompt_version` }.

### 8 — Wire the detailed handler ([summary_service.py](backend/app/domains/transcripts/summary_service.py))
- `_claim_summary_job`: for the **detailed** spec, do NOT apply `_truncate_for_summary` — carry the FULL `normalized_full` for
  partitioning (brief keeps Option-A truncation in 4.5.1a; brief→detailed rewiring is 4.5.1b). For detailed, set
  `input_hash = sha256(normalizationVersion + fullText + partition_config_hash)` so a budget change ⇒ distinct provenance row and no
  collision with the prior truncated single-call row.
- `_generate_summary_async`: branch the detailed spec to `run_map_reduce_detailed(...)` instead of the single `gateway.complete`.
  Brief path unchanged.
- `_persist_summary_success`: accept the `MapReduceOutcome`; write `generation_strategy="map_reduce"`, `truncated=false`,
  `generation_metadata={mapPromptVersion, reducePromptVersion, mapUnitCount, sourceMapUnitSummaryIds, coverageManifest}`. Provenance
  (model/prompt/backend) copied from the REDUCE call's `AIRequestLog` (the artifact-producing call), as today. `truncated_fallback`
  branch (Option A) retained ONLY if map-reduce cannot proceed — logged, strategy=`truncated_fallback`.
- **queues.py job_timeout** ([queues.py](backend/app/workers/queues.py)): the detailed job now runs N sequential calls. Raise its `job_timeout` to
  `MAX_MAP_UNITS * LLM_DETAILED_TIMEOUT_SECONDS + reduce_overhead + buffer` (a CEILING; a real ~60-min lecture is ~7–9 units). This is
  the same RQ-SIGKILL class of bug fixed before — must scale or the work-horse dies mid-map.

### 9 — Deterministic adapter ([provider.py:322](backend/app/platform/llm/provider.py#L322) `_render_output`)
Add input-aware branches:
- `detailed_summary_map`: return a `DetailedSummaryPartial`-shaped JSON whose `keyConcepts` includes a stable **unit marker** derived
  from `rendered.content` (e.g. `"unit-marker: " + <first ~40 normalized chars of the {{transcript}} body>`) + static filler. Distinct
  per unit, traceable to input. `forced_invalid` drops a key (mirror the existing pattern).
- `detailed_summary_reduce`: scan `rendered.content` (= serialized partials) for ALL `"unit-marker: …"` strings and emit them in
  `keyConcepts` + a static `overview`/sections → the reduce output literally contains every unit's content (the late-topic sentinel's
  final-unit marker among them). `forced_invalid` drops a required section.
This makes coverage **content-provable without a real provider** (the spec's §7 goal).

## Tests (BACKEND VERIFIED, deterministic adapter — `backend/tests/...`)
- **partition**: deterministic for (content, budget, chunkerVersion); FEWEST units; both caps respected; never splits a segment;
  `partition_config_hash` stable & budget-sensitive; MAX_MAP_UNITS cost-guard raises.
- **map**: writes valid `MapUnitSummary` partials; empty sections allowed (lenient validator); `forced_invalid` → InvalidOutput.
- **reduce guard**: a dense fixture trips the tiered path; a small one stays single; final `coverage_manifest` covers all unit indices
  in BOTH paths.
- **coverage / late-topic sentinel**: multi-unit (≥2) fixture + a fixture whose FINAL unit text = `LateLectureSentinelTopic`; assert
  (a) `coverage_manifest` includes the final `unit_index` (structural), AND (b) the reduce output content contains the sentinel marker
  (content flow-through, via the input-aware adapter).
- **resume (C3)**: a `succeeded` unit with matching hash+checksum is reused (no second map call); a `partition_config_hash` OR
  `source_transcript_checksum` mismatch forces a re-map.
- **persist**: detailed map-reduce success ⇒ `generation_strategy=map_reduce`, `truncated=false`, `generation_metadata` populated;
  provenance from the reduce log; no collision with a prior truncated single-call row (distinct input_hash + prompt_version).
- **migration**: fresh-DB upgrade/downgrade round-trip; the extended feature CheckConstraint accepts the two new features and the ORM
  string matches.
- Frontend untouched (no UI in 4.5.1a). NO real provider. The §3A deterministic gate guard is unaffected.

## Risks
- **R-mr1 job_timeout** — if the detailed `job_timeout` isn't scaled to N map calls, the work-horse SIGKILLs mid-map (the exact prior
  bug). Step 8 scales it; a test asserts the computed ceiling ≥ MAX_MAP_UNITS×detailed timeout.
- **R-mr2 reduce is the new 408 surface** — the reduce input (serialized partials) could itself exceed the ceiling; the §3.3 guard +
  tiered fallback is the mitigation, exercised by the dense fixture. Real-provider confirmation of reduce latency/tokens is 4.5.1c §9(c).
- **R-mr3 provenance collision** — without folding `partition_config_hash` into the detailed `input_hash`, a re-budget could collide on
  `uq_gen_summaries_provenance`. Step 8 folds it in; a test covers re-budget ⇒ distinct row.
- **R-mr4 input-aware adapter drift** — the marker-echo must stay simple/deterministic; keep it to a single short marker so it never
  trips `max_tokens` or schema bounds.
