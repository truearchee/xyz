After Checkpoint A review, the developer explicitly approved Checkpoint B only. This adds the Supabase browser client, session provider, token-per-request API wrapper, and throwaway `/login` page.

Do not build `/tracer`, Playwright tests, dashboards, app shell, navigation, role routing, admin UI, lecturer UI, student UI, transcript summaries, AI, or new storage behavior.

Local Supabase is not required before Checkpoint A unless backend tests unexpectedly require it. It is required before Checkpoint C.

## Approach
Add a narrow backend router for `/me` that composes the existing `get_current_user` active-user gate with a read of active course memberships. The frontend must later consume backend-confirmed application context from this endpoint instead of JWT/session claims.

Admins return no active module memberships; if bad data gives an admin active memberships, the endpoint logs a warning and excludes them. Lecturer/student users receive only active memberships on active modules.

Checkpoint B configures Supabase browser auth without adding a custom auth endpoint. `OpenAPI.TOKEN` is assigned an async resolver that calls `supabase.auth.getSession()` for every generated-client request, so the access token is not captured at module initialization.

## Test strategy
- `tests/test_me.py` proves the five required backend cases using existing mocked JWT/JWKS fixtures.
- OpenAPI route proof confirms the router is mounted before client generation.
- Generated-client diff inspection prevents unrelated DTO/service churn.
- Full backend pytest confirms no regression.
- Frontend type-check proves Checkpoint B compiles.
- Static scan proves page/component code did not add direct `fetch()` calls.
- Runtime sign-in proof is deferred until deterministic Supabase fixtures exist.
- Checkpoint C proves fixture creation only. Browser login, CORS, multipart upload, signed URL, and full client-edge proof remain deferred to Checkpoints D/E.
