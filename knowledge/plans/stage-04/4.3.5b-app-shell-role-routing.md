# Session 4.3.5b — Implementation Plan — App Shell + Role Routing

- Spec: [[specs/stage-04/4.3.5b-app-shell-role-routing]]

## Scope confirmation
Build the recovery shell only: route groups, `/me`-driven session state, role routing, route guards, 401/403 wrapper behavior, E2E hooks, `/tracer` env gating, and browser proof across the listed checkpoints.

Do not build dashboards, product UI, backend endpoints, Next.js middleware, Redux/Zustand/TanStack Query, or design-system work.

For Checkpoint A only, implement the route group structure, `AppShell`, standalone `AccessDenied`, root provider-only layout, `(app)/layout.tsx` wired through `ProtectedAppLayout`, and `/login` outside the AppShell.

## Approach
Promote the 4.3.5a tracer-era frontend into a minimal application frame while preserving the approved auth model. `SessionProvider` becomes the source for bootstrap state, `ProtectedAppLayout` owns protected-app shell rendering, and product route pages remain placeholders until later checkpoints/sessions.

Generated API files and backend code remain untouched.

## Changes, file by file
- `frontend/src/app/layout.tsx` — keep global providers only.
- `frontend/src/app/(auth)/login/page.tsx` — render login outside AppShell.
- `frontend/src/app/(app)/layout.tsx` — mount `ProtectedAppLayout`.
- `frontend/src/app/(app)/admin/page.tsx`, `frontend/src/app/(app)/lecturer/page.tsx`, `frontend/src/app/(app)/student/page.tsx`, `frontend/src/app/(app)/unauthorized/page.tsx` — minimal route placeholders.
- `frontend/src/components/shell/AppShell.tsx` — header with user identity, role nav placeholder, logout, and content outlet.
- `frontend/src/components/auth/AccessDenied.tsx` — standalone forbidden bootstrap state with logout.
- `frontend/src/lib/routing/ProtectedAppLayout.tsx` — Checkpoint A loading/auth shell decisions only.
- `frontend/src/lib/session/SessionProvider.tsx` — expose the explicit session state needed by Checkpoint A.

## Order of operations
1. File approved spec and plan documents, recording `5f92698` as 4.3.5a implementation and `7f06f471` as docs-only cleanup baseline.
2. Restructure frontend route files into `(auth)` and `(app)` route groups.
3. Add shell, access-denied, protected-layout, and session-state code for Checkpoint A only.
4. Run `docker compose exec frontend npx tsc --noEmit`.
5. Stop and report changed files, exact verification output, and findings/blockers.

## Test strategy
Checkpoint A is accepted by frontend type-check:

```bash
docker compose exec frontend npx tsc --noEmit
```

Later checkpoint static scans, backend regression, and Playwright assertions are not run until their checkpoints.

## Risks & mitigations
- Route guard behavior can drift into Checkpoint B: keep Checkpoint A limited to loading, unauthenticated, forbidden, and authenticated shell rendering.
- Login may accidentally render inside the shell: place it under `(auth)` and keep AppShell only in `(app)/layout.tsx`.
- `/me` bootstrap may create noisy redirects: only `ProtectedAppLayout` redirects unauthenticated app-route users in Checkpoint A.
