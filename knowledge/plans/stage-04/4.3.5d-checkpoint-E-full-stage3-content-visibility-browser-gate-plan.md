## Scope confirmation
Run only the final Stage 3 browser gate. Do not add product features. If a gate bug is discovered, fix only the narrow bug required for the gate and document it. Do not mark Stage 3 FULLY VERIFIED unless every required browser, API, auth, status-separation, signed URL, student 403, and verification assertion passes.

## Execution approach
1. Run required pre-checks against the E2E compose overlay.
2. Ensure the E2E/local Supabase stack is running with `docker compose -f docker-compose.yml -f docker-compose.e2e.yml up -d`.
3. Ensure deterministic E2E auth/app users exist with the existing fixture seed. The fixture may establish old baseline rows, but the gate module itself must be created through the admin UI/product path.
4. Execute a Playwright browser gate with separate admin, lecturer, and student contexts:
   - admin context creates a fresh module through the admin UI and assigns the student
   - lecturer context logs in, opens the fresh module, proves generated sections, notes, upload, replace, invalid upload rejection, publish, and status separation
   - student context logs in, opens the same module, proves published-only UI and server response, signed URL open, no mutation controls, student upload 403, and session still active
5. Capture screenshot and structured PASS/BLOCKED evidence.
6. Run post-gate verification commands.
7. If and only if the gate and checks pass, update Checkpoint E report, findings, STATUS, architecture/log, and mark Stage 3 FULLY VERIFIED.
8. Commit the knowledge/status updates and any approved narrow gate-bug fixes.

## Browser gate implementation notes
- Use separate browser contexts for lecturer and student.
- Do not reuse a browser session and switch accounts.
- Use the real frontend UI for lecturer upload, replace, notes, publish, and student signed URL open.
- Use `page.waitForResponse()` for the student `GET /modules/{moduleId}/sections` response and assert draft/unpublished sections are absent from JSON.
- Use an authenticated student request to `POST /modules/{moduleId}/sections/{sectionId}/assets` with multipart data to prove 403. Re-check `/me` afterward to prove the session remains active.
- Do not construct storage URLs manually; use only the UI signed URL flow and verify the returned URL fetches with HTTP 200.
