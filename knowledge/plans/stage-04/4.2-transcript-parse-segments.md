## Scope confirmation
Implement the first transcript worker stage: enqueue an RQ parse job after successful transcript upload, read the raw VTT/TXT object from storage, parse immutable ordered transcript segments, and track idempotent execution in `ingestion_jobs`.

Do not build normalization, chunking, embeddings, summaries, AI/K2Think calls, retry/backoff, replacement/supersession, new endpoints, frontend features, a second job type handler, `0004` edits, or `processor_version` in the idempotency key.

## Approach
Add the database spine first, then storage read support, pure parser modules, queue wiring, upload enqueue behavior, and the parse handler. Treat RQ as at-least-once delivery and make the database job claim the correctness surface. The handler uses short DB transactions for claim and persist/failure, parses outside a transaction, and applies the claimed-attempt guard on both success and failure paths.

## Risks & mitigations
- RQ enqueue can fail after upload commit — log sanitized metadata and leave `uploaded` for the future sweeper.
- Mid-parse crash can leave `parsing`/`running` — accepted 4.6 recovery gap; status index is added now.
