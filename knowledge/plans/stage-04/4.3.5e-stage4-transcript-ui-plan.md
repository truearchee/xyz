## Scope confirmation
This session makes Stage 4.1-4.3 transcript upload, parse, and chunk persistence browser-verifiable through a thin real lecturer UI.

This plan does not include embeddings, summaries, AI infrastructure, retry/recovery, transcript replacement, student transcript/summaries, new transcript statuses, raw transcript/segment/chunk exposure, frontend transcript UI during Part 2, Playwright tests during Part 2, or `/tracer` deletion during Part 2.

## Session sequence
- Part 1 — Checkpoint 0 investigation: completed.
- Part 2 — prerequisite repair: completed; targeted/full backend tests and generated-client freshness passed.
- Part 3 — Checkpoint A/B frontend transcript UI: completed; frontend type-check/build and guardrail scans passed.
- Part 4 — Checkpoint C/D tracer removal + teardown: completed; `/tracer` removed, E2E bridge retained, run manifest/teardown verified.
- Part 5 — Checkpoint E Playwright browser gate + status closure: future work.

## Approach
Part 2 first removes the prerequisite blockers: persist the session spec/plan and repair the worker terminal status so a later browser UI can poll to `completed`. The backend repair is deliberately narrow: change only the successful chunk job completion status, keep job metadata, and update targeted tests.

Later frontend checkpoints should add wrapper/upload-helper transcript surfaces, mount a section transcript control on lecture/lab sections, poll existing active transcript status, and keep transcript status visually and structurally separate from section publish status and asset processing status.

- `bash scripts/generate-api-client.sh && git diff --exit-code frontend/src/lib/api` proves no OpenAPI client drift.

## Risks & mitigations
- Risk: `completed` could be confused with embeddings/summaries completion. Mitigation: ADR states `completed` means Stage 4.1-4.3 completion only until Stage 4.4 extends the pipeline.
- Risk: chunk reruns could regress terminal status. Mitigation: existing completed/running job guards remain intact, and tests assert idempotent rerun behavior.
- Risk: frontend code might treat chunk counts as product API fields. Mitigation: findings preserve test-level DB read guidance for counts.
