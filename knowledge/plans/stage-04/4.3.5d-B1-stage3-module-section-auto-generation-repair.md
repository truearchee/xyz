## Scope confirmation
Deliver the narrow backend repair that makes admin module creation generate predefined `ModuleSection` rows in the same transaction. Use the temporary MVP default policy: Lecture 1, Lecture 2, Lab 1, Assignment 1.

Do not build lecturer/student UI, upload UI, publish UI, section CRUD/reorder endpoints, schedule/template systems, transcript/AI work, or the Stage 3 Playwright gate. Do not mark Stage 3 FULLY VERIFIED.

## Approach
Add a small admin-domain section generation helper that owns the MVP default definitions and creates `ModuleSection` models without committing. Call it from `create_module` after the `CourseModule` has an id and before the service's final flush/refresh. The route already commits after `create_module`, so module, owner membership, and generated sections remain one transaction.

Tests will prove the product path creates sections and that existing content read/publish visibility rules work with those generated sections.

## Changes, file by file
- `backend/app/domains/admin/section_generation.py` - define the MVP default policy and `generate_initial_sections`.
- `backend/app/domains/admin/service.py` - call the generator during module creation before the final flush.
- `backend/tests/test_admin.py` - assert admin module creation generates the default section set and retains non-admin protections.
- `backend/tests/test_content.py` - add coverage that generated sections are lecturer-readable, student-hidden while draft, and student-visible only for the single published section.
- `knowledge/architecture/db-spine.md` - record module creation now creates default section rows through backend service code.
- `knowledge/steps/stage-04/4.3.5d-content-ui-final-report.md`, `knowledge/STATUS.md`, `knowledge/log.md`, and `knowledge/archive/stage-04/4.3.5d/4.3.5d-B1-section-generation-repair.md` - close the loop from evidence.

## Order of operations
1. File spec and plan.
2. Add admin-domain section generation helper.
3. Wire helper into admin module creation.
4. Add targeted admin/content tests.
5. Run targeted tests, full backend tests, frontend type-check, generated-client freshness, and diff guards.
6. Update findings, status, architecture, log, and report.
7. Commit only after verification passes.

## Test strategy
- Targeted admin tests prove `POST /admin/modules` creates exactly four default sections with stable order, draft visibility, active status, and null notes.
- Targeted content tests prove generated sections flow through existing lecturer and student read/publish visibility paths.
- Full backend suite catches regressions.
- Frontend type-check and generated-client diff prove no frontend/API contract drift.
- `git diff --name-only -- frontend` must be empty.

## Risks & mitigations
- Existing tests may assume module creation has no sections -> update only assertions that conflict with the new product requirement.
- Accidental API contract drift -> keep DTOs unchanged and verify generated client has no diff.
- Atomicity regression -> keep section creation inside the existing module creation transaction and do not commit in the generator.

## Open questions
- Future policy remains open: schedule-driven vs template-driven generation. This session explicitly chooses temporary MVP default only.
