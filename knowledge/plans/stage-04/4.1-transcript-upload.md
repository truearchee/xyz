## Scope confirmation
Deliver lecturer-only VTT/TXT transcript upload and active-transcript status read for lecture/lab sections, storing raw bytes unchanged through `StorageProvider`, persisting one active `transcripts` row at `status='uploaded'`, and returning only the public `TranscriptMeta` fields.

Do not build workers, queueing, processing, parser/chunker/embedder/summarizer tables, replacement/supersession logic, checksum deduplication, student transcript/summary views, signed transcript URLs, language detection, AI/K2Think calls, or any status transition beyond `uploaded`.

## Approach
File and keep the spec as the source of scope, then run P1-P6 before source edits. Implement a separate transcript domain that consumes a platform/query authorized section context and a storage provider; it must not import content-domain service logic or directly call Supabase. Use the existing module-scoped route convention, existing private bucket setting, and the repository's `text + CHECK` database convention.

## Guardrails
- P6 governs the FK target: inspect `section_assets.uploaded_by_user_id` in the model/live DB and make `transcripts.uploaded_by_user_id` point to the same target.
- Use the transcript key pattern documented in ADR-016/storage; only adapt to the Stage 3 module/section prefix if the existing convention requires it. Do not add an optional nonce when `transcriptId` already ensures uniqueness.
- `TranscriptMeta` exposes no `storageKey`, `checksum`, `isActive`, or `supersededAt`.
- Preserve the lost-race test: storage put succeeds, DB insert hits the partial unique index, cleanup deletes the loser object, response is `409`, and exactly one active row remains.

## Test strategy
Run the spec verification sequence exactly: `docker compose build`, `docker compose up -d`, `docker compose exec backend alembic upgrade head`, `docker compose exec backend pytest tests/test_transcripts.py -v`, `docker compose exec backend pytest -v`, `bash scripts/generate-api-client.sh`, frontend rebuild/restart, `docker compose exec frontend npx tsc --noEmit`, and `git diff --check`.

Targeted tests must explicitly prove public field omission, no body parse for unauthorized requests, no worker/status drift, raw bytes stored unchanged, `checksum` present in DB but absent from response, and cleanup for both general DB failure and partial-index lost race.

## Risks & mitigations
- Multipart parsing could happen before authz → keep route signatures on `Request`, authorize before `request.form()`, and test with a monkeypatched parser/storage mock.
- Duplicate uploads can race → keep both the pre-check and partial unique index; handle `IntegrityError` by deleting the loser object and returning `409`.
- Storage key policy could drift from Stage 3 → record the final key pattern in ADR-016 and `storage.md`; keep the key deterministic from transcript ID plus sanitized filename.
- Generated OpenAPI types may change multipart shapes → regenerate via `scripts/generate-api-client.sh` and keep frontend helpers typed against generated services.
