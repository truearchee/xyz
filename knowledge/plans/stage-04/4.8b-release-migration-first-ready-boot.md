## Scope confirmation
**Delivers (in-repo):** `env.py` prefers `DIRECT_DATABASE_URL`; an idempotent first-admin + seed-identity
bootstrap CLI (Supabase service-role ensure + `AppUser` upsert, admin via the sanctioned direct path,
USERS ONLY, staging-gated); `scripts/release.sh` (migrate + bootstrap) wired as the backend fly
`release_command`; `scripts/deploy-staging.sh` (env gate → immutable images → deploy → verify); a
committed `deploy/staging-runbook.md` with rollback; tests. End state in-repo: the release path is
runnable + idempotent locally; the **hosted first ready boot** is the developer's run.

**Does NOT build (from the spec):** the hosted deploy execution / first-ready-boot verification
(**human handoff**); SSE probe + staging-origin CORS + §8 proof (**4.8c**); the hook-free smoke +
memberships + fixture (**4.8d**); `allow_credentials` removal (**4.9**); any new migration / pgcrypto.

## Approach
Keep the laptop path untouched: `env.py` and the engine fall back to `DATABASE_URL` when no direct
URL is set, so local `alembic upgrade head` and the e2e stack are unchanged. The bootstrap is a small
idempotent CLI that reuses the exact primitives the e2e seed already uses (`get_supabase_admin_client`
+ `auth.admin.create_user(email_confirm=True)` + an `app_users` upsert), so it inherits a proven
create-or-update shape — the only new thing is doing it in Python in the release phase and creating
the `admin` row (which the API forbids). The release is a thin shell wrapper so Fly's single
`release_command` runs both steps and aborts on either failure. The deploy script is the repeatable
front door; the runbook is the human-facing ordered procedure.

## Changes, file by file

  load the identity set (O1), open an async session over the **direct** engine
  (`create_direct_engine()`), and for each identity `await ensure_identity(...)`:
  (1) ensure CONFIRMED Supabase user (create; on "already exists" fetch the id — mirror
  `ensureAuthUser`); (2) upsert `AppUser` (insert if absent by email, else update
  role/full_name/auth_provider_id, keep `is_active=True`). Commit once. Log counts (created/updated)

- `backend/app/platform/config.py` — bootstrap settings: the identity inputs (O1) + a

- `scripts/release.sh` (NEW, `chmod +x`) — `#!/usr/bin/env sh`, `set -eu`; `cd` to the app root;

- `deploy/staging-runbook.md` (NEW) — ordered procedure, start-ordering, extension-privilege

- `backend/tests/test_bootstrap_identities.py` (NEW) — mock `get_supabase_admin_client`; against the

- `backend/tests/test_alembic_url.py` (NEW) — `env.py::get_database_url()` prefers
  `DIRECT_DATABASE_URL`, falls back to `DATABASE_URL`.

## Order of operations
1. `env.py` direct-URL preference + unit test.
2. `bootstrap_identities.py` + config inputs/gate + idempotency/gate tests (against local Supabase + DB).
3. `scripts/release.sh`; run locally against the e2e stack (migrate no-op + bootstrap) → exit 0.
4. `backend/fly.toml` release_command uncomment.
5. `scripts/deploy-staging.sh` (+ `--dry-run` env-gate path).
6. `deploy/staging-runbook.md`.
7. Local verification sweep (pytest; release.sh; check-staging-env; full active suite 11/11).
8. Hand off to the developer for the first hosted deploy; record `/health/ready` + workers-process.

## Test strategy
- **Bootstrap idempotency** — the load-bearing test: two runs, assert no duplicate Supabase calls for
  an existing user (mock asserts create-or-fetch), AppUser rows correct + stable, admin row has
  `role='admin'`, exit 0 both times.
- **Admin-creation path** — assert the bootstrap creates an `admin` AppUser (the API path would 400);
  proves the sanctioned bypass works and is bootstrap-only.
- **Staging gate** — seed identities refused when `ENVIRONMENT` prod-like / flag off; admin handling
  per O1.
- **env.py** — direct-preference unit.
- **Secrets hygiene** — capture logs in the idempotency test; assert no password substring.
- **No runtime regression** — full active Playwright suite 11/11 (deploy-time changes only).

## Risks & mitigations
- **Supabase egress / "user already exists" races in the release phase** → the ensure step is
  create-or-fetch (idempotent); a transient Supabase error aborts the release (correct — deploy
  retried). Mitigate by keeping the bootstrap last in release.sh so a schema migration isn't redone.
- **Extension privilege on Supabase** → the migration `IF NOT EXISTS` is a no-op if the SQL-console
  fallback was used; runbook documents both. Verified on first deploy (F006).
- **release_command single-command limit** → wrapped in `release.sh`; `set -e` ensures either failure
  aborts.
- **Bootstrap connection** → use the **direct** engine (release VM is single, short-lived; avoids
  pooler prepared-statement quirks for the few upserts) — see O3.
- **deploy-staging.sh assumes flyctl auth + two app names** → operator-provided; the script reads app
  hostnames from env/args, no secrets inline.
