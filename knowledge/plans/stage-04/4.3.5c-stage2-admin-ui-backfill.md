## Scope confirmation
Deliver real Stage 2 admin UI and browser proof. Option A is selected and approved: add the narrow read-only admin module-membership projection required for real assign/remove UI.

Do not build migrations, assignment/removal write-path changes, auth redesign, backend login, content/PDF/section/upload UI, transcript UI, module detail UI, SSO, self-registration, invitations, bulk import, audit log, or `/tracer` deletion.

## Approach
Keep backend change limited to a read-only projection endpoint under the existing admin router. Regenerate the OpenAPI client after the backend contract exists, then build feature-level client components that call the generated client through `frontend/src/lib/api/wrapper.ts`. Use simple, dense, operational UI that fits the existing app shell. Re-fetch after every mutation and render backend failures visibly.

Subagents are quality gates only. The primary agent owns code changes; subagents inspect and report at the required checkpoints.

## Changes, file by file
- `backend/app/domains/admin/schemas.py` - add `ModuleMemberResponse`.
- `backend/app/domains/admin/service.py` - add read-only `list_module_members` query; do not modify write-path behavior.
- `backend/app/api/routers/admin.py` - add `GET /admin/modules/{module_id}/members`.
- `backend/tests/test_admin.py` - add projection tests for success, joined fields, inactive member visibility, archived/admin exclusion, non-admin 403, unknown module 404, and deterministic ordering.
- `frontend/src/lib/api/` - regenerate generated client after backend endpoint exists.
- `frontend/src/lib/api/wrapper.ts` - expose admin user/module/member actions through wrapper methods.
- `frontend/src/features/admin/users/*` - create user list, lecturer/student creation forms, deactivate action, and reset-password form.
- `frontend/src/features/admin/modules/*` - create module list, create-module form, assign-member forms, real module member list, and removal action.
- `frontend/src/features/modules/AssignedModulesList.tsx` - shared read-only assigned-module list for lecturer and student areas.
- `frontend/src/app/(app)/admin/page.tsx` - render admin user and module panels without becoming a one-giant-page implementation.
- `frontend/src/app/(app)/lecturer/page.tsx` and `frontend/src/app/(app)/student/page.tsx` - render assigned-module list only.
- `tests/e2e/4.3.5c-stage2-admin.spec.ts` - implement Stage 2 browser gate using run-scoped identities and separate browser contexts.
- `knowledge/steps/4.3.5c-stage2-admin-ui.md`, `knowledge/steps/findings-4.3.5c.md`, `knowledge/architecture/frontend.md`, `knowledge/STATUS.md`, `knowledge/log.md`, `knowledge/steps/stage-02/2.3-admin-flows.md` - close loop from evidence.

## Test strategy
- Backend projection: `docker compose exec backend pytest backend/tests/test_admin.py -v`, then `docker compose exec backend pytest`.
- Frontend: `docker compose exec frontend npx tsc --noEmit`, direct fetch scan, and JWT-role scan.
- Generated client: `bash scripts/generate-api-client.sh`, then `git diff --exit-code frontend/src/lib/api`.
- Browser gate: rebuild E2E stack, seed admin fixture, and run `npx playwright test tests/e2e/4.3.5c-stage2-admin.spec.ts`.
- Stage 2 FULLY VERIFIED is written only after all tests pass and all findings are resolved.

## Risks & mitigations
- Projection scope drift -> keep backend diff limited to schema, read query, router endpoint, tests, and generated client.
- Owner auto-membership could pollute "only assigned" proof -> create separate owner lecturers for Module A and Module B, and separate main lecturer assigned only to Module A.
- Stale UI state could fake success -> re-fetch users, modules, and members after each mutation.
- Inactive-user behavior could differ between 401 and 403 -> assert observed behavior and document it in findings.
- E2E cleanup could delete too much -> cleanup only run-scoped E2E-prefixed identities using an EmailStr-accepted E2E domain.
