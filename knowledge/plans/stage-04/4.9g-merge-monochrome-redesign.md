# Session 4.9g - Implementation Plan - Merge Monochrome Redesign

- Spec: [[specs/stage-04/4.9g-merge-monochrome-redesign]]

## Scope confirmation
This merge lands the Stage 4.9f monochrome design system onto current `main` while preserving all current backend, schema, generated API client, and Stage 5-9 functionality.

It does not push, open a PR, accept stale backend behavior, delete Stage 5-9 frontend features, introduce backend behavior, or intentionally change migrations.

## Approach
Run the merge on `merge-4.9f-redesign`, resolve explicit conflicts according to category, then run a broader non-conflict safeguard pass because `stage/4.9f` deletes many newer Stage 5-9 files without conflict. For frontend files shared by both branches, keep `main` data flow and API usage, but replace inline styling with the 4.9f semantic token and UI primitive style. For backend and generated API client files, restore `main` wholesale.

## Changes, file by file
- `backend/**`, `backend/alembic/versions/**`, `backend/prompts/**`, `backend/tests/**` - restore from `main` unless only a docs/report file is involved outside backend.
- `frontend/src/app/globals.css`, `frontend/src/components/ui/**`, `frontend/postcss.config.mjs`, `frontend/scripts/check-*.mjs`, `frontend/vitest.*` - accept 4.9f design foundation.
- `frontend/package.json` - union current runtime dependencies with 4.9f design/tooling dependencies.
- `frontend/package-lock.json` - regenerate after the manifest union.
- `frontend/src/app/layout.tsx` - use 4.9f global styling and portal roots while preserving the current session provider and KaTeX import.
- `frontend/src/app/(app)/student/page.tsx` - keep current dashboard links and Stage 5-9 features; restyle with semantic tokens.
- `frontend/src/features/admin/**` - keep current admin behavior and API contracts; restyle shared surfaces and controls.
- `frontend/src/features/content/**` - keep current lecturer/student content, metadata, summary, quiz, assistant, and progress wiring; restyle shared surfaces.
- `frontend/src/features/quiz/**`, `frontend/src/features/glossary/**`, `frontend/src/features/assistant/**`, `frontend/src/features/progress/**`, `frontend/src/lib/api/**` - restore from `main` if the merge deletes or stales them, then restyle only where touched deliberately.
- `knowledge/**` - union history where safe; current `main` remains source of truth for Stage 5-9 completion and Alembic `0039`.
- `.conductor/**`, `.github/**`, `.husky/**`, `deploy/**`, root deploy scripts, `docker-compose*.yml`, root package files - reject stale 4.9f non-design infra unless required by verification.

## Order of operations
1. Run `git merge --no-commit --no-ff stage/4.9f`.
2. Resolve backend/config conflicts by restoring `main`.
3. Accept 4.9f design foundation and restore current Stage 5-9 frontend domains/routes/API client.
4. Manually restyle conflicted shared frontend files.
5. Regenerate `frontend/package-lock.json`.
6. Run guard diffs and preservation checks.
7. Run verification commands.
8. Write the report from `git diff` and command output.
9. Update `STATUS.md` and append one line to `log.md`.

## Test strategy
- Backend full suite proves the stale branch did not regress Stage 5-9 backend.
- Alembic heads proves the migration chain still ends at `0039`.
- Frontend `tsc --noEmit` proves the design dependency union and restyled components compile.
- `git diff --check` proves whitespace/conflict hygiene.
- Route/file preservation checks prove `/student/glossary`, `/student/progress`, assistant panel, and quiz modes were not deleted.
- Backend guard diff against `main` proves backend/migrations/prompts/generated-client behavior was not silently changed.

## Risks & mitigations
- Risk: non-conflicting stale deletes remove Stage 5-9 files. Mitigation: restore `main` for backend, generated client, and Stage 5-9 frontend domains before verification.
- Risk: package lock drift. Mitigation: regenerate lockfile from the merged `frontend/package.json`.
- Risk: design ADR numbering collisions. Mitigation: keep main ADRs and record duplicate-number risk instead of renumbering during this merge.
- Risk: frontend restyle accidentally changes behavior. Mitigation: preserve imports, API calls, callbacks, props, route files, and existing tests; use type-check and route/file checks.
