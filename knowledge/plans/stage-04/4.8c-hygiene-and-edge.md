## Scope confirmation
**Delivers:** the gated `/internal/sse-probe` (admin-only, `text/event-stream`, 3–5 chunks/<10s/no
compression/anti-buffering — C2 acceptance) validated over D1; the staging `CORS_ORIGINS` value +
preflight check; the §8 two-surface proof bound to the **Dockerfile.prod** artifact (frontend hook
build-time absent via a call-site guard; backend flags-absent + fail-loud + no-surface, reframed); and
`check-staging-env` proven a **hard** deploy gate.

**Does NOT build (from the spec):** the hook-free **staging smoke** + memberships + fixture (**4.8d**);
`allow_credentials` removal (**4.9**); a real product SSE endpoint (**8.3**); any product feature.

## Approach
The SSE probe is a tiny self-contained router included only behind `ENABLE_INTERNAL_SSE_PROBE`, so the
route literally does not exist when the flag is off (the cleanest "absent in prod" — a 404, not a
disabled handler). It carries the exact SSE headers a real Stage-8.3 stream would, so it tests the true
transport. The §8 frontend absence is a one-line **call-site guard** at `SessionProvider.tsx:90` that
lets the prod minifier eliminate the hook — the binding proof is `window.__xyzE2E === undefined` in the
Dockerfile.prod bundle; the bundle grep is the secondary surface. The §8 backend "proof" is reframed to
the surfaces that actually exist (no fault route → nothing to 404). `check-staging-env`-as-hard-gate is
already true by construction (`set -e` + non-zero exit in `deploy-staging.sh`); 4.8c adds the rehearsal
that *proves* it rather than asserts it.

## Test strategy
- **SSE probe** — the load-bearing new surface: `404` when `ENABLE_INTERNAL_SSE_PROBE` unset; with it
  set, `401`/`403` for unauth/non-admin, and for an admin: `Content-Type: text/event-stream`, ≥3
  `data:` events, `X-Accel-Buffering: no`, no `Content-Encoding`. Consume the `StreamingResponse`
  body to count events without real wall-clock waits (small/zero delay in the test path).
- **§8 frontend** — build the **Dockerfile.prod** bundle with `NEXT_PUBLIC_E2E_TEST_HOOKS` absent;
  assert no reachable `__xyzE2E` token-override (grep). Re-run the active suite (dev build) to prove
  hooks still register → the regression guard for the call-site change.
- **§8 backend** — `test_fault_boot_check` (already green) + `check-staging-env` forbids the flags.
- **Hard gate** — `deploy-staging.sh --dry-run` returns non-zero on each bad env.

## Risks & mitigations
- **Frontend call-site guard breaks the e2e suite** (highest regression risk) → the guard evaluates at
  runtime under `next dev` (env true → registers); proven by re-running the full active suite 11/11.
- **Minifier leaves inert `__xyzE2E` strings** in the prod bundle → the binding proof is the runtime
  `window.__xyzE2E === undefined`; if strings remain, resolve accept-with-rationale (build-false ⇒
  unreachable) or strengthen by excluding the module from the prod build (O1).
- **GZip/compression on the event-stream** → ensure no `GZipMiddleware` wraps the probe (none today);
  the probe sets no `Content-Encoding`; `X-Accel-Buffering: no` defeats proxy buffering.
- **Probe present in staging** is intentional (validation + the 4.8d smoke use it); it is gated +
  admin-only. A future PROD build leaves `ENABLE_INTERNAL_SSE_PROBE` off (O2).

## Open questions
- **O1 (decision — yours).** If the prod minifier leaves inert `__xyzE2E` strings (hook function
  defined but unreachable), accept-with-rationale (runtime `undefined` is the binding proof) — or
  strengthen by excluding the e2e modules from the prod build entirely? Recommend accept-with-rationale
  unless the grep must be byte-clean.
- **O2.** `ENABLE_INTERNAL_SSE_PROBE` is **allowed ON in staging** (gated + admin-only) for the C1
  validation and the 4.8d smoke; OFF in a future prod. So `check-staging-env` does NOT forbid it (unlike
  the E2E hook / fault flags). Confirm (vs. requiring it off in staging too).
- **O3.** Probe auth: **admin-only** (satisfies §7.C2 "admin-authenticated AND/OR rate-limited") vs.
  also wiring the Redis limiter. Recommend admin-only; the probe is cheap + admin-gated + 3–5 chunks.
