# Session 2.2 — Implementation Plan — Supabase Auth Integration & CurrentUserContext

## Scope confirmation
Deliver auth infrastructure only using Supabase asymmetric signing keys and JWKS: `PyJWT[crypto]` verification through `jwt.PyJWKClient`, frozen `CurrentUserContext`, the `get_current_user` FastAPI dependency, one authenticated health smoke endpoint, JWKS/audience/issuer settings, environment placeholders, and tests proving JWT validation plus DB-backed identity resolution.

Do not build legacy HS256/shared-secret verification, user provisioning, user update/deactivation endpoints, admin flows, lecturer or student flows, Supabase Admin API calls, password/reset logic, refresh token handling, route-level role guards, frontend changes, new database tables, or migrations. Do not add `supabase-py`.

The developer confirmed the Supabase project uses asymmetric signing keys (ECC P-256 / JWKS) and that the legacy JWT secret has been migrated. This plan therefore implements JWKS verification only.

## Approach
Keep authentication logic isolated under `backend/app/platform/auth/`. JWT verification will use `jwt.PyJWKClient(settings.SUPABASE_JWKS_URL)` to fetch/cache keys by token `kid`, then decode with asymmetric algorithms `RS256` and `ES256`, configured audience, configured issuer, and required `sub`, `exp`, and `role` claims. All verification failures return generic 401 responses.

The trusted request identity is resolved from the application database after token verification. `app_users` supplies email, full name, app role, timezone, and active state; active course memberships joined to modules supply module access, ownership, and publish flags. The health route remains thin and only consumes `Depends(get_current_user)`.

Tests will generate a local asymmetric keypair and avoid live network calls by mocking `jwt.PyJWKClient` or its fetch path. Test JWTs will be signed with the generated private key, while DB-touching tests continue to use `TEST_DATABASE_URL`.

## Test strategy
JWT unit tests prove JWKS-backed signature validation, audience validation, issuer validation, expiry handling, invalid-signature rejection, and generic 401 behavior. FastAPI tests prove header parsing, unauthenticated `/health` stability, authenticated smoke behavior, inactive and unknown user handling, and the `WWW-Authenticate: Bearer` header. DB-backed context tests prove the app database wins over JWT role claims, archived memberships are excluded, and `is_owner`/`can_publish` are computed from active membership plus module ownership.

Final verification maps to the spec:
- `docker compose exec backend pytest tests/test_auth.py -v` must show 13 passed.
- `docker compose exec backend pytest` must pass the full backend suite.
- `docker compose exec frontend npx tsc --noEmit` must exit 0.

## Risks & mitigations
- `PyJWKClient` performs HTTP fetching when a key is not cached → tests will mock the client or fetch path and use generated keys so no live network dependency exists.
- Supabase may sign with ES256 because the project uses ECC P-256 → allow `ES256` and `RS256` in decode, while trusting PyJWT key matching and token validation.
- Existing test DB fixture behavior may not yet support API tests with seeded rows → reuse or extend `conftest.py` narrowly, keeping all DB-touching auth tests on `TEST_DATABASE_URL`.
- Settings may already exist in another form → extend the existing pattern rather than introducing competing config objects.
- Membership joins can drift into route logic → keep all JWT parsing and DB user lookup in `platform/auth/`, with the health endpoint only consuming `CurrentUserContext`.
