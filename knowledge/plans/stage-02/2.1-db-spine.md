# Session 2.1 — Implementation Plan — DB Spine

## Scope confirmation
Deliver the schema-only database spine: five SQLAlchemy models, one Alembic migration, pgcrypto in local Postgres init, isolated DB-spine tests, and required knowledge artifacts. Do not add endpoints, auth/JWT logic, Supabase clients, service/repository layers, product flows, storage integration, seed data, or runtime imports from application routers/services into individual model modules. `.env.example` is included as an approved exception because the tests require `TEST_DATABASE_URL` documentation.

## Approach
Use SQLAlchemy 2.x declarative models and a hand-written Alembic migration so database constraints and indexes are explicit. UUIDs are generated in Python via `uuid6.uuid7`, with UUID columns left without database-side defaults. Tests run Alembic against `TEST_DATABASE_URL` only, creating the isolated test database if needed and never falling back to `DATABASE_URL`.

## Changes, file by file
- `backend/pyproject.toml` — add `uuid6` for UUIDv7 generation.
- `backend/app/platform/db/models/` — add `Base` and the five model classes, then import them from `__init__.py` for metadata registration.
- `backend/alembic/env.py` — import `Base` from `app.platform.db.models` and expose `Base.metadata` to Alembic.
- `backend/alembic/versions/0002_db_spine.py` — create/drop the five tables, check constraints, unique constraints, FKs, and indexes.
- `backend/tests/test_db_spine.py` — add exactly three tests covering migration round-trip, schema/index/constraint shape, and DB constraint enforcement plus UUIDv7 model default behavior.
- `docker/postgres/init/001-create-vector.sql` — add `pgcrypto` next to `vector`, without moving extension creation into migrations.
- `.env.example` — add `TEST_DATABASE_URL` pointing to a separate test database.
- `knowledge/architecture/db-spine.md`, `knowledge/STATUS.md`, `knowledge/log.md`, and final report — close the development-memory loop from real diff and command output.

## Order of operations
1. Create approved spec and plan artifacts with links.
2. Add dependency/config changes and SQLAlchemy model files.
3. Add Alembic metadata import and migration.
4. Add isolated migration/schema tests.
5. Build/restart Docker services as needed, then run migration and test verification commands.
6. Write architecture and report from `git diff` and actual command output.
7. Update spec/plan/report links, `STATUS.md`, and `log.md`.

## Test strategy
Map verification to the spec's done criteria: Alembic upgrade/downgrade/upgrade proves migration reversibility; `test_db_spine.py` proves isolated DB use, table presence, representative schema shape, check enforcement, no DB UUID defaults, and UUIDv7 Python defaults; full backend pytest proves Stage 1 health still passes; frontend type-check proves the frontend remains unaffected.

## Risks & mitigations
- Docker init scripts only run on first database volume creation → record the managed/hosted Postgres extension bootstrap caveat in the report.
- `TEST_DATABASE_URL` may not be present in the running backend container → run verification with an explicit test URL environment override if the local `.env` has not been updated.
- Existing dirty Stage 1 report is unrelated → leave it untouched.
