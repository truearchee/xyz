## Scope confirmation
Build DB-backed participant module access and minimal typed module views. Do not add migrations, module content, publish actions, module pagination, Redis caching, per-module roles, admin branches under `/modules`, or frontend auth/session plumbing.

The plan intentionally follows live schema behavior: `course_modules.is_active`, partial active-membership uniqueness, historical archived membership rows, and role-derived `canPublish`.

## Approach
Move module authorization out of request identity and into a reusable per-module dependency. Keep `get_current_user` focused on token validation plus app-user identity. Add read-model queries under `platform/query`, then compose those projections in a thin participant router.

Frontend work is limited to reusable presentational components typed against generated DTOs. Backend authorization remains the source of truth.

## Changes, file by file
- `backend/app/platform/auth/context.py` - remove `ModuleMembership`; add frozen `ModuleAccessContext`.
- `backend/app/platform/auth/dependencies.py` - stop loading memberships in `get_current_user`.
- `backend/app/platform/auth/guards.py` - keep `require_role`; add `require_module_access`.
- `backend/app/platform/query/modules.py` - add projection dataclasses and read-only active-access/detail/list queries.
- `backend/app/api/routers/modules.py` - add participant `/modules` list and detail endpoints plus DTOs.
- `backend/app/main.py` - mount the modules router.
- `backend/tests/test_auth.py` - update identity-context assertions.
- `backend/tests/test_modules.py` - add focused module-access and endpoint coverage.
- `frontend/src/lib/api/` - regenerate OpenAPI client after backend routes exist.
- `frontend/src/features/modules/` - add typed presentational list/detail components using generated DTO types.
- `knowledge/decisions/adr-003-module-authz-db-lookup.md` - record DB-backed access, `404` convention, `is_active`, partial uniqueness, historical reassignment rows, and role-derived `canPublish`.
- `knowledge/architecture/auth-current-user-context.md` - document identity-only context and module-access guard.
- `knowledge/steps/stage-02/2.4-module-base-views.md`, `knowledge/STATUS.md`, `knowledge/log.md`, prior reports - close the evidence loop after verification.

## Test strategy
`backend/tests/test_modules.py` will prove active access, non-member/archived/inactive hiding, unauthenticated behavior, malformed UUID validation, admin participant-surface behavior, revocation without token refresh, reassignment with historical rows resolving the active `membership_id`, and co-lecturer parity.

The full backend suite protects prior admin/auth behavior. Client regeneration plus frontend type-check proves API and presentational component types.

## Risks & mitigations
- Risk: changing `CurrentUserContext` could break prior auth assumptions. Mitigation: update auth tests and keep app-user identity fields unchanged.
- Risk: `404` hiding could be bypassed by a router-level query. Mitigation: route detail through `require_module_access` and run the membership grep gate.
- Risk: frontend scope drifts into auth. Mitigation: presentational components only; no route fetching, token storage, or fake providers.
