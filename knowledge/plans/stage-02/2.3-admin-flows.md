# Session 2.3 — Implementation Plan — Admin Flows

## Linked documents
- Spec: [[specs/stage-02/2.3-admin-flows]]
- Plan: [[plans/stage-02/2.3-admin-flows]]
- Report: [[steps/stage-02/2.3-admin-flows]]

## Scope confirmation

Deliver the approved admin API slice: admin-only user provisioning, user
deactivation, password reset through Supabase Auth Admin, course module creation
with lecturer owner membership, membership assignment/removal, paginated admin
lists, generated OpenAPI client updates, and knowledge closure.

Do not build lecturer/student endpoints, user reactivation, section generation,
content upload, file storage, frontend UI, migrations, new tables, invitations,
forced password change, self-registration, or Supabase-side banning.

## Approach

Add the route-level role guard first, then implement a single Supabase Admin
client module and a service-owned business layer. Keep route handlers thin and
let Pydantic schemas own wire casing. Use SQLAlchemy async sessions for all DB
work, explicit `flush()` calls for response objects, and route-level `commit()`
after successful service calls. For create-user, perform the local email
pre-check before Supabase, then roll back the Supabase user if local insertion
fails after creation.

The pasted spec's stale knowledge paths are normalized to current repository
paths: `AGENTS.md`, `knowledge/architecture/auth-current-user-context.md`,
`knowledge/architecture/db-spine.md`, and `knowledge/steps/stage-02/...`.

## Changes, file by file

- `backend/app/platform/auth/guards.py` — add `require_role(*roles)` dependency
  wrapper around `get_current_user`.
- `backend/app/platform/supabase_client.py` — add one async Supabase Admin
  client singleton using `SUPABASE_SECRET_KEY`.
- `backend/app/domains/admin/` — add camelCase schemas and service functions
  for all admin business rules.
- `backend/app/api/routers/admin.py` and `backend/app/main.py` — expose and
  mount the nine `/admin` endpoints.
- `backend/pyproject.toml` and `.env.example` — add `supabase`,
  `pydantic[email]`, and `SUPABASE_SECRET_KEY`.
- `backend/tests/test_admin.py` — add the 25 required admin-flow tests with a
  mocked Supabase Admin client.
- `frontend/src/lib/api/` — regenerate the committed OpenAPI client.
- `knowledge/` — write the report, update auth/backend architecture, status,
  log, and prior-session change histories from real evidence.

## Order of operations

1. File the approved spec and this approved plan before source edits.
2. Add dependencies, settings, role guard, Supabase client singleton, admin
   schemas, service, router, and router mount.
3. Add admin tests and run the focused test command.
4. Fix issues until `tests/test_admin.py` passes.
5. Run the full backend test suite.
6. Regenerate the frontend OpenAPI client and type-check the frontend.
7. Write the evidence-based report and close knowledge links/status/log/history.

## Test strategy

- Use existing ES256/JWKS fixtures and `TEST_DATABASE_URL`; never mint HS256
  tokens.
- Mock only the Supabase Admin client methods: `create_user`,
  `update_user_by_id`, and `delete_user`.
- Cover the spec's 25 cases exactly in `backend/tests/test_admin.py`.
- Verify no response includes passwords, secret values, or `auth_provider_id`.
- Run:
  - `docker compose exec backend pytest tests/test_admin.py -v`
  - `docker compose exec backend pytest`
  - `scripts/generate-api-client.sh`
  - `docker compose exec frontend npx tsc --noEmit`

## Risks & mitigations

- Supabase async Admin return shapes may vary. Mitigation: isolate extraction of
  the created Supabase user id in the service and keep tests shape-tolerant.
- Container dependencies may be stale after editing `pyproject.toml`.
  Mitigation: rebuild/recreate backend containers before Docker verification if
  imports fail.
- OpenAPI generation needs the backend running with the new dependency set.
  Mitigation: regenerate only after backend tests and container refresh succeed.
